# Agile Testing: A Practical Guide for Testers and Agile Teams
**Authors:** Lisa Crispin and Janet Gregory
**Domain:** Testing

## Core Concepts

- **The Agile Testing Quadrants (the book's signature model, credited originally to Brian Marick, developed at length here).** A 2x2 grid crossing two axes: *business-facing vs. technology-facing* (who the test speaks to/for) and *supporting the team vs. critiquing the product* (whether the test guides development or evaluates the finished thing). Q1 (technology-facing, supporting the team) = unit and component tests. Q2 (business-facing, supporting the team) = functional tests, examples, story tests — what this repo calls BDD/Gherkin scenarios and example mapping. Q3 (business-facing, critiquing the product) = exploratory testing, usability testing, UAT, showcases. Q4 (technology-facing, critiquing the product) = performance, load, security, "-ility" testing. The point of the model is not the boxes themselves but that **all four are needed and none substitutes for another** — a team heavy on Q1/Q4 automation but with no Q2/Q3 presence still ships things that are technically correct and wrong for the user.
- **Whole-team testing responsibility.** The book's central thesis: quality is not a phase or a role, it is a property the *whole team* is accountable for, continuously, not a bug-hunting activity handed to a separate department after coding is "done." Testers are full team members embedded from the start of a story's life (in refinement/planning), not gatekeepers stationed at the end of the pipeline. "Testing is everyone's job" is the recurring frame — developers write tests, POs define examples, testers coach and probe, and the team collectively owns the Definition of Done.
- **Shift-left as a lived practice, not a slogan.** Crispin and Gregory operationalize shift-left specifically as: testers participate in story kickoff and example-mapping-style conversations *before* code is written, acceptance tests are agreed as executable examples ahead of implementation (their "Power of Three" — business/dev/tester collaboration on examples), and automated checks run continuously in CI rather than in a post-development "test phase." The book predates the term "shift-left" itself (which entered wider currency later) but its practices are exactly what that term now names.
- **"Testers are not gatekeepers, they are guides, coaches, and investigators."** A tester's job in an agile team is reframed away from "find bugs at the end and block release" toward: help the team build the right thing (asking clarifying questions early, using examples to expose ambiguity), build quality in continuously, and probe the finished increment with exploratory testing to catch what confirmatory/scripted tests structurally cannot.
- **Checking vs. Testing (a distinction the book popularized, building on James Bach/Michael Bolton's terminology).** "Checking" is confirmatory: does the system do what we already specified (automatable, deterministic, pass/fail). "Testing" is investigative: exploring the system with human judgment to discover what we didn't think to specify (not fully automatable — requires a skilled human). Both are necessary; conflating them leads teams to believe 100% automated "checking" coverage means testing is done, when exploratory investigation is a structurally different activity that automation cannot replace.
- **Exploratory testing as a first-class, skilled discipline** — not "unstructured clicking around." The book treats session-based exploratory testing (time-boxed charters, structured note-taking, debrief) as a rigorous technique that finds classes of defects (usability friction, unanticipated interactions, edge cases nobody wrote an example for) that scripted/automated tests by design cannot, because scripted tests only check what someone already thought to specify.
- **The Definition of Done as a testing artifact.** The book pushes teams to make "done" mean tested-and-accepted, not "code complete" — DoD should explicitly enumerate which test quadrants/levels must pass before a story counts as finished, closing the gap between "developer thinks it's done" and "it actually works."
- **Test automation as a whole-team investment, prioritized like the Test Pyramid** (the book discusses the pyramid concept, generally crediting Mike Cohn's formulation, as the shape automation investment should take) — heavy unit/component automation, lighter UI/E2E automation, because UI-level automated checks are the most brittle and expensive to maintain. This is consistent with (not the origin of) what later became widely known simply as "the Test Pyramid."
- **Collaboration rituals that produce Q2 tests before code**: the book describes practices very close to what this repo calls Example Mapping (Crispin/Gregory's own term of art is closer to "specification workshops" / "the three amigos" collaboration — three amigos being business, dev, and test perspectives meeting before a story starts) to turn a story into concrete, agreed examples that become both the acceptance criteria and, once automated, the regression suite.

## Actionable Techniques

- **Quadrant self-audit per release**: for a given increment, check whether all four quadrants have real coverage (not just Q1+Q4 automation with no Q2/Q3 human/business-facing presence) — a checklist a skill or agent could run explicitly.
- **Three-amigos session before a story starts**: business, developer, and tester perspectives converge on concrete examples before code is written — the examples become both spec and (once automated) regression tests.
- **Checking vs. testing labeling**: explicitly tag a test suite's contents as "checks" (automated, confirmatory) vs. "tests" (exploratory, human-judgment) so a 100%-green CI run is never mistaken for "we've tested everything."
- **Session-based exploratory testing charters**: time-boxed (e.g., 60–90 min), a stated mission/area of focus, structured notes captured during the session, and a debrief afterward — treated as a repeatable, schedulable practice, not an ad hoc afterthought.
- **DoD quadrant mapping**: write the Definition of Done so it names which test quadrant(s)/levels a story must pass, rather than a vague "tested" bullet.
- **Whole-team quality retro question**: periodically ask "who owns testing on this team?" — if the honest answer is "the tester" rather than "everyone," that's a signal the whole-team model has eroded back into a gatekeeper model.
- **Test the risk, not just the requirement**: use exploratory charters specifically to probe areas the team suspects are risky/ambiguous/under-specified — a deliberate complement to confirmatory checks that only verify what was already made explicit.

Candidates worth checking against `skills/glossary-management/` for inclusion in the Ubiquitous Language, if not already present:

- **Checking vs. Testing** — the confirmatory/automatable vs. investigative/human-judgment distinction; genuinely useful shared vocabulary currently entirely absent from `test-pyramid`, which uses "testing" to mean only the former.
- **Exploratory Testing** — appears nowhere in the four skills read; would be new vocabulary if Q3 coverage is added.
- **Whole-Team Testing / Whole-Team Quality** — the book's organizing principle; worth a glossary entry distinguishing it from this repo's necessarily different single-agent-per-role reality (see Caveats).

## Caveats

- **"Whole-team testing" assumes live human collaborators this repo doesn't have.** The book's central practice — testers, developers, and POs collaborating continuously, in the same room/stand-up, on the same stories — presupposes a multi-person team with distinct human perspectives in real time. This repo has one agent per role (`test-strategist` for testing, `requirements-analyst` for requirements/UAT, `backend-engineer`/`frontend-engineer` for code) operating largely in sequence via handoffs, not a synchronous cross-functional team. The book's collaboration rituals (three amigos, specification workshops) need to be adapted here as **structured cross-agent review steps** (e.g., "test-strategist reviews the requirements-analyst's acceptance criteria before converting them to feature files" — which the existing handoff description already implies but doesn't name as a check) rather than literally reintroduced as live meetings.
- **Exploratory testing assumes a skilled human explorer.** The book is explicit that exploratory testing is a skilled discipline requiring human judgment and cannot be meaningfully automated. In this repo, the closest available executor is a UAT participant (a design partner or Shafi as internal proxy, per `uat-plan`) or possibly a `test-strategist`-run scripted approximation — but an LLM agent "exploring" a UI has real limits compared to the book's model of a trained human tester probing for the unexpected. Any adoption of exploratory-testing practice here should be honest that its highest-value form (a skilled human charter session) is bottlenecked on the same scarce human bandwidth (Shafi, or a design partner) that already constrains `uat-plan`.
- **The book predates this repo's specific tooling and terminology** (Testcontainers, mutation testing tooling, OpenTelemetry trace correlation, Consumer-Driven Contracts as a named practice) — its Q1/Q4 technical content is generally superseded in specificity by `test-pyramid`'s and `test-strategist`'s modern, Go-native tooling choices. Cite this book for the *quadrants model*, the *whole-team philosophy*, and the *checking-vs-testing distinction* — not for modern test-automation tooling specifics, which this repo's existing skills already handle in more current and more concrete form.
- **The Agile Testing Quadrants are widely attributed to Brian Marick**, with this book being the work that most thoroughly popularized and extended them into a practical methodology — worth citing accurately as "the quadrants model this book develops at length" rather than implying Crispin/Gregory originated the 2x2 itself.
