# Unit Testing: Principles, Practices, and Patterns
**Author:** Vladimir Khorikov
**Domain:** Testing

## Core Concepts

- **The four pillars of a good unit test, held as competing trade-offs.** Khorikov's central analytical tool: every test scores on (1) *protection against regressions* (does it catch a real bug?), (2) *resistance to refactoring* (does it stay green through a pure refactor, or throw false positives?), (3) *fast feedback*, and (4) *maintainability* (cheap to read and change). The book's key claim is these are **multiplicative, not additive** — a test that's fast and maintainable but has zero resistance to refactoring (because it asserts on private state) is still a bad test, because the missing pillar zeroes out the value of the others. Most test-design mistakes in the book are diagnosed as over-optimizing one pillar (usually speed, via heavy mocking) at the expense of resistance to refactoring.
- **Resistance to refactoring is scored by false positives, and is won by testing observable behavior instead of implementation details.** A test that fails when internal structure changes but external behavior doesn't is a false positive — it trains the team to distrust red tests. The fix is disciplined: assert only on the public contract (return values, resulting state, or an observable side effect crossing the application's boundary), never on private fields, call counts on incidental collaborators, or code paths taken.
- **Classical (Detroit/Chicago) school vs. London (mockist) school of unit testing.** The London school isolates *every* class under test from *all* its collaborators via mocks, verifying interactions. Khorikov argues strongly for the classical school instead: isolate the *test case* from other test cases (no shared mutable state), not the *class* from its collaborators — a "unit" of test is a unit of observable behavior, which may legitimately span several classes exercised together with only true external dependencies replaced. This directly determines how much a codebase ends up mocking.
- **Mocks vs. stubs are not interchangeable, and mixing their roles is the single most common test-design mistake in the book.** A **mock** is used to verify that the system under test *called* a dependency in a particular way — it tests an interaction. A **stub** is used only to feed a canned input into the system under test so a scenario can be set up — its calls are never asserted on. "Asserting on a stub" (verifying a call that was only there to supply data) is named explicitly as an anti-pattern: it couples the test to an implementation detail with zero behavioral payoff.
- **Only mock unmanaged dependencies; treat managed dependencies as internal implementation detail.** A *managed dependency* is one your application fully controls and which is reachable only through your own code (your own database, your own file store) — the outside world never observes it directly, so it should be verified through its *end state* (an integration test against the real thing, or a state-based fake), never mocked. An *unmanaged dependency* is one whose side effect is observable to something outside your app's control (an SMTP server, a message queue, a third-party payment API) — here the interaction itself is the behavior that matters, so a mock verifying "this call happened, with these arguments" is the correct and only tool.
- **Code falls into four quadrants by domain significance × number of collaborators**, and the quadrant should decide test strategy: (1) *domain model / algorithms* — high complexity, few collaborators — the best return on unit-test investment, and where nearly all unit-testing effort belongs; (2) *trivial code* — low complexity, few collaborators — not worth testing at all; (3) *controllers* — low complexity, many collaborators — best covered by a handful of integration tests, not exhaustive unit tests; (4) *overcomplicated code* — high complexity *and* many collaborators — the danger zone, untestable as-is, and the correct fix is to decompose it until the complex logic moves into quadrant 1 and what's left in the controller is quadrant 3.
- **The Humble Object pattern is the standard technique for rescuing quadrant-4 code.** Extract the complex decision logic out of the thing that's structurally hard to unit test (a framework entry point, a UI layer, a cron handler, an HTTP handler) into a plain object with no framework dependency — leaving behind a "humble," deliberately untested-by-unit-test wrapper that just plumbs calls through (covered instead by a thin integration/e2e test). This is the general technique behind, e.g., MVP/MVC's split of view from presenter logic, but it generalizes to any hard-to-test glue layer.
- **Coverage metrics are a lagging, gameable indicator — never a target.** Line/branch coverage tells you code executed, not that a meaningful assertion ran; a test with no assertions inflates coverage to 100% while catching nothing. Coverage can suggest under-tested code (a useful negative signal) but can never certify a suite is good, because it cannot measure resistance to refactoring or maintainability at all, and can be gamed on the regression-protection axis too.
- **Overspecification** — an interaction-based test that verifies more of the interaction than the behavior actually requires (exact call order when order isn't a real invariant, or asserting on calls to a collaborator that isn't the point of the test) — is named as a direct cause of low resistance to refactoring: the test breaks on refactors that don't change any observable behavior.
- **The test pyramid's shape is a consequence of pillar-optimization, not a target ratio to hit.** Khorikov endorses "more unit tests, fewer integration tests, fewest end-to-end" as the natural *outcome* of pushing complex logic into a well-isolated domain model (quadrant 1) and keeping controllers thin (quadrant 3/Humble Object) — not as a number to engineer toward independent of how the code is actually shaped.

## Actionable Techniques

- Score any test-design decision against all **four pillars** explicitly before accepting it — a proposed test or mock that clearly sacrifices one pillar for another (usually speed for resistance to refactoring) should be flagged, not accepted by default.
- Before writing a test, classify the code into one of the **four quadrants** (domain/algorithm, trivial, controller, overcomplicated) — this decides whether it gets exhaustive unit tests, a couple of integration tests, no tests, or a refactor (Humble Object) before it gets tests at all.
- **Never assert on a stub.** If a double's calls are only there to arrange input, don't verify them; if you find yourself wanting to verify them, it's actually functioning as a mock and the test should say so.
- **Mock only unmanaged, externally-observable dependencies** (message publish, email send, third-party API call). For managed dependencies (your own repository/DB), use a state-based fake or a real instance via an integration test — never a call-count mock.
- Apply **Humble Object** whenever complex decision logic is trapped inside something structurally hard to unit test (HTTP handler, message consumer, CLI entry point, UI code) — extract the logic to a plain, framework-free type; leave the wrapper thin and covered by integration tests instead.
- Treat a **survived mutant or a false-positive test failure** as a direct pillar-score signal: a survived mutant is a regression-protection gap; a test that fails on a pure refactor is a resistance-to-refactoring gap. Don't treat these as generic "flaky test" noise.
- Name tests after the **observable behavior in plain language** ("returns error when tenant is empty"), not after the method signature or internal call sequence — a name tied to implementation wording is itself an early symptom of an implementation-coupled test.
- Never test **private methods or private state** directly. If a private method feels important enough to need its own test, that's a signal it's a hidden abstraction that should be extracted into its own public, independently-testable type.
- Watch for **multiple Act sections in one test** — arranging, acting, and asserting more than once in a single test function usually means the test is secretly an integration test, or should be split into two unit tests.

## Caveats

- **The book is C#/.NET-centric**, with examples built around xUnit, Moq, and constructor-based dependency injection — its code samples don't transfer directly to Go. This repo's `mock-generation` skill already correctly substitutes Go-idiomatic tooling (`moq`/`mockgen`, generated from consumer-defined interfaces) for Moq, and that adaptation should be preserved — cite Khorikov for the *principles* (managed/unmanaged, mock-vs-stub, classical school), not for tooling specifics.
- **Khorikov's examples are one-test-per-case (xUnit `[Theory]`/`[InlineData]` at best), not Go's table-driven-with-subtests idiom.** The book's AAA (Arrange-Act-Assert) discipline maps naturally onto each row of a Go table-driven test, but this repo's `go-unit-test` skill already has its own well-developed table-driven convention — apply AAA as a lens on each subtest case, don't try to restructure the skill around the book's per-test-function shape.
- **The book only briefly touches mutation testing** and does not cover fuzzing at all (fuzzing is a Go-native technique with no .NET-community equivalent at the time of writing) — `go-mutation-test` and `go-unit-test`'s fuzz-testing section are already more current and more Go-specific than the book on these points; cite Khorikov only for the coverage-is-gameable argument that motivates mutation testing, not for mutation-testing mechanics.
- **The book's fixture guidance (xUnit constructor/`IDisposable` setup-teardown-per-class) doesn't map onto Go's `t.Cleanup`/per-test-function model** — `test-fixture-design`'s existing `t.Cleanup` discipline is a reasonable Go-native substitute and needs no correction from the book; the book is worth citing here only for its general warning against over-shared fixture state coupling test cases together, which is compatible with, not a change to, what's already written.
