# Software Testing: A Craftsman's Approach
**Author:** Paul C. Jorgensen
**Domain:** Testing

## Core Concepts

- **Error, Fault, Failure, Incident — a precise causal chain, not synonyms.** Jorgensen insists on this vocabulary: a human **error** produces a **fault** (a defect in an artifact — code, design, spec), a fault *may* produce a **failure** (observable wrong behavior at execution), and a failure produces an **incident** (the symptom a user or operator actually sees). Testing's job is to find faults by provoking failures; a fault that never executes under any test never surfaces as an incident. This is a stricter causal chain than most agile-testing books bother to define, and it grounds the book's insistence that test design should be systematic rather than "try things until it feels covered."
- **Verification vs. validation, applied at every level of abstraction.** Verification asks "are we building the artifact right" (does the code match the design, does the design match the spec); validation asks "are we building the right artifact" (does the spec match the customer's actual need). Jorgensen maps distinct testing activities onto distinct levels of translation (requirements → design → implementation), which is the conceptual ancestor of this repo's phase-gated Strategy → Design → Implement structure, though the book frames it as a V-model rather than agile phases.
- **Functional test-case design as a formal, derivable procedure — not intuition.** The book's signature technique is **Boundary Value Analysis (BVA)** done with mathematical precision rather than ad hoc "check the edges": for each input variable, hold all others at nominal and vary the target variable through its min, min+ε, nominal, max−ε, and max values. The book formally distinguishes four tiers: **normal BVA** (valid boundaries only, one variable perturbed at a time — `4n+1` test cases for `n` variables), **robust BVA** (adds just-outside-boundary invalid values — `6n+1`), **worst-case BVA** (Cartesian product of all variables' boundary values simultaneously, not one-at-a-time — catching interaction bugs single-variable BVA misses), and **robust worst-case** (the Cartesian product including invalid values). Choosing among these tiers is itself a documented, risk-based decision in the book, not a single fixed recipe.
- **Equivalence Class Testing, with the same four-tier rigor.** Partition each input's domain into classes where the program is expected to behave uniformly (valid classes and invalid classes). **Weak normal** picks one representative per class and covers classes with as few test cases as possible; **strong normal** takes the full Cartesian product of one representative from every valid class (proving class interactions, not just each class in isolation); **weak/strong robust** add invalid classes analogously. The book is explicit that *weak* equivalence testing is what most engineers actually do when they say "equivalence partitioning" informally, and that it is provably weaker than the strong forms at catching multi-variable interaction faults.
- **Decision-table-based testing for combinational business logic.** Where a rule depends on multiple independent Boolean conditions (e.g., "if the asset is classified AND the tenant is active AND the requester has clearance, allow"), Jorgensen has a formal procedure: enumerate every condition as a stub, build all `2^n` rules (collapsing truly "don't-care" conditions to keep the table tractable), map each rule to its action, and verify the table is both **complete** (every possible condition combination has a rule) and **consistent** (no two rules with identical conditions produce different actions). This is the direct formal ancestor of Gherkin's `Given/When/Then` scenario tables, but predates and is more rigorous about combinational completeness than most BDD treatments.
- **Structural (white-box) testing organized around the control flow graph and a strict coverage hierarchy.** The book builds testing bottom-up from statement coverage → decision (branch) coverage → condition coverage → decision/condition coverage → **basis path testing**, and proves formal **subsumption** relationships between them (e.g., all-edges/branch coverage subsumes all-nodes/statement coverage, but neither subsumes nor is subsumed by condition coverage alone). **Basis path testing** uses McCabe's **cyclomatic complexity**, `V(G) = E − N + 2` (edges minus nodes plus 2, for a connected graph with one entry/exit), to compute the exact minimum number of linearly independent paths ("the basis set") that, once each is tested, guarantee every edge has been exercised — a mathematically precise floor for "how many path-based tests are enough," not a heuristic.
- **Data flow testing — a criterion the branch-coverage-first world usually skips entirely.** For every variable, find every point it is *defined* (assigned) and every point it is later *used* (in a computation or a predicate) before being redefined — a **def-use (DU) pair**. The book defines a nested family of criteria on top of these pairs: **all-defs** (every definition reaches at least one use), **all-uses** (every def reaches every one of its uses), and **all-DU-paths** (every simple path from a definition to a use, not just one). This catches a specific bug class that pure control-flow coverage is structurally blind to: a variable computed correctly but read from the wrong place, or a stale value read after a definition was skipped on some path.
- **A unifying graph theory across functional, structural, integration, and specification-based testing.** The book's structural insight is that "coverage" is always really **graph coverage** — only what the nodes and edges *represent* changes: statements/branches for structural testing, program states for finite-state-machine/specification-based testing, and modules/call sites for integration testing. This is why the book can define **node coverage**, **edge coverage**, and **path coverage** once, abstractly, and reuse them across every testing level — including **N-switch coverage** for state-machine testing (covering every sequence of N consecutive state transitions, not just every single transition).
- **Integration testing as an explicit build-order strategy, independent of unit correctness.** Top-down (stub the lower modules, drive from the top), bottom-up (driver harnesses call up from tested leaves), sandwich/collaborative (meet in the middle), and big-bang (assemble everything at once, the strategy Jorgensen explicitly warns against for anything but trivial systems) are named strategies for *the order modules are assembled and tested together*, each with a documented stub/driver cost trade-off — a different axis entirely from "mocked vs. real dependency," which is what "integration testing" has come to mean in most modern Go/microservice usage.

## Actionable Techniques

- For any function/handler with numeric, range, or ordered-enum inputs: derive test cases mechanically via **BVA** (min, min+, nominal, max−, max per variable) rather than intuiting "a few edge cases" — pick normal BVA for low-risk code, worst-case BVA (full boundary Cartesian product) for logic where variable *interactions* at the edges are plausible and costly if wrong.
- For any input with distinct valid/invalid categories (not just numeric ranges — e.g., an enum, a status field, a tenant-tier flag): build explicit **equivalence classes** first, then decide weak (one test per class) vs. strong (Cartesian product across classes) based on whether the classes are suspected to interact.
- For any rule combining ≥3 independent Boolean conditions: build a **decision table** — enumerate the full `2^n` rule space, collapse genuine don't-cares, and check the table for completeness and consistency *before* writing test cases — this catches missing/contradictory business rules that ad hoc example-based tests (including Gherkin scenarios written from intuition) routinely miss.
- Compute **cyclomatic complexity** (`V(G) = E − N + 2`) for any function with nested conditionals/loops as a concrete, numeric floor on "how many test cases are structurally necessary" — and treat a high number as a refactor signal (extract logic) rather than a test-writing burden to grind through.
- For any function where a variable is computed in one branch and consumed in another (a common shape in stateful domain logic), trace its **def-use pairs** explicitly and write at least one test per pair — this is the systematic version of "test the state edges" that catches "right coverage number, wrong data flow" bugs.
- When assembling multiple modules/services into a working slice, pick an explicit **integration build order** (top-down/bottom-up/sandwich) up front rather than defaulting to big-bang "wire it all up and see what breaks" — matters most when a bounded context has several collaborating aggregates/handlers being built in parallel.
- For any Aggregate with a nontrivial state machine (a lifecycle with multiple named states and guarded transitions), model it explicitly as a finite state machine and apply **N-switch coverage** — cover not just every individual transition but every short sequence of N consecutive transitions — to catch bugs that only manifest across a transition sequence (e.g., classify → reclassify → delete).

## Key Terminology

Candidates worth checking against `skills/glossary-management/` for inclusion in the Ubiquitous Language:

- **Boundary Value Analysis** (Normal / Robust / Worst-Case) — absent from `go-unit-test`, which currently names the practice informally as "boundary...analysis" with no tiered vocabulary.
- **Equivalence Class Partitioning** (Valid Class / Invalid Class, Weak/Strong) — absent everywhere; would sharpen `go-unit-test`'s edge-case guidance.
- **Decision Table** (Condition Stub/Entry, Action Stub/Entry, Completeness, Consistency) — not in the existing Testing terms list in `CLAUDE.md` (which has `Feature File`, `Scenario`, `Given/When/Then` but no decision-table vocabulary); this is the formal ancestor of Gherkin scenario tables and a real gap in the current glossary.
- **Cyclomatic Complexity** / **Basis Path Testing** — absent from `go-mutation-test` and `test-pyramid`, both of which discuss code-quality/coverage without this concrete, computable metric.
- **Def-Use Pair** / **Data Flow Testing** (All-Defs, All-Uses, All-DU-Paths) — absent everywhere; names a coverage dimension (data flow, not control flow) this repo's testing skills don't currently address at all.
- **Coverage Criteria Subsumption** — absent; would give `test-pyramid`'s "coverage is a signal, not a target" section a precise vocabulary for *which* stronger criterion to reach for, and why it's provably stronger.

## Caveats

- **This is an academic textbook with heavy mathematical rigor** — full BVA/equivalence-class/decision-table/basis-path/data-flow derivation for every function would be disproportionate for a frugal, solo-operator PM project per `CLAUDE.md`'s Budget and Frugality section. Adopt the *formal derivation method* selectively — for complex domain logic (Jorgensen's own emphasis is that rigor pays off precisely where logic is dense and consequential, which lines up with this repo's Aggregate invariants and classification rules) — and keep the current lighter, intuitive boundary-checklist approach for straightforward CRUD/glue code.
- **The book predates microservice and event-driven architectures.** Its integration-testing chapter (top-down/bottom-up/sandwich/big-bang) assumes a monolithic call tree of modules compiled and linked together with stubs/drivers standing in for not-yet-built pieces — it does not contemplate independently-deployed services communicating over a broker with eventual consistency. Its build-order vocabulary is worth borrowing for how a single service's collaborating handlers/aggregates are assembled during development; it does not map onto `go-integration-test`'s actual subject (Testcontainers verifying real Postgres/Redpanda behavior), which is a fundamentally more modern concern the book doesn't address at all.
- **Mutation testing, fuzzing, and consumer-driven contract testing are absent or only nascent in this book** — these are later lineages this repo's `go-mutation-test`, `go-unit-test`'s fuzz section, and (unreviewed here) `go-contract-test` already handle with more currency and Go-specific depth than Jorgensen could. Don't cite this book for those topics' mechanics — only for the general "coverage is gameable, structural rigor matters" argument that motivates all of them.
- **The book's object-oriented testing chapter targets Java/C++-style class hierarchies with significant inheritance** — this repo's Go stack favors composition over inheritance, so class-hierarchy-specific techniques (testing overridden methods, polymorphic dispatch coverage) don't transfer directly. The state-machine/N-switch coverage idea, however, transfers cleanly and usefully onto Go domain Aggregates with real lifecycle state machines (e.g., a `DataAsset`'s classify → reclassify → archive transitions) — cite the state-transition-coverage concept, not the class-hierarchy mechanics around it.
