## Core Concepts

- **Verification vs. Validation (the foundational distinction a dictionary with this exact title would open with).** Verification asks "are we building the product right?" — confirmation, through objective evidence, that specified requirements have been fulfilled (conformance to a specification, a contract, a design). Validation asks "are we building the right product?" — confirmation that the product, when used as intended, meets the actual needs of its users/stakeholders (fitness for intended use). This is ISO 9000's canonical phrasing and it is the single most load-bearing distinction the discipline has — verification can be satisfied while validation fails (a system that perfectly conforms to a wrong specification), and this repo's own phase model (`Quality` produces verification-shaped artifacts like test suites and coverage gates; `Customer Validation` is a distinctly named phase for the validation half) already tracks the shape of this distinction structurally, even though the terms themselves are never defined as a pair anywhere in the glossary.
- **The Error → Fault/Defect → Failure chain (IEEE 610.12 / ISO 24765).** Four distinct, non-interchangeable concepts, commonly conflated in casual usage:
  - **Error (or Mistake)** — a human action that produces an incorrect result (a developer misunderstands a requirement, a programmer mistypes a comparison operator).
  - **Fault (or Defect)** — a flaw in a work product (code, design, requirement) that results from an Error. A Fault is static — it sits in the artifact whether or not it is ever executed.
  - **Failure** — the manifestation of a Fault during execution: the system deviates from its specified or expected behavior, observable at runtime.
  - Precise relationship: **Error causes Fault; Fault, when executed under the right conditions, causes Failure.** A Fault can exist for years without ever causing a Failure (dead code, an untriggered edge case). "Bug" is colloquial and imprecise — it is used loosely for any of the three and should never appear in a precise artifact without being mapped to one of the above.
- **Test vs. Test Case vs. Test Procedure vs. Test Suite (ISTQB glossary distinctions).**
  - **Test** — the general activity of executing a component/system to evaluate a property.
  - **Test Case** — a specific, documented set of preconditions, inputs, and expected results designed to exercise a particular objective (a requirement, a code path, a scenario). A Test Case is a specification; it need not yet be automated.
  - **Test Procedure (or Test Script)** — the sequence of actions for the execution of a Test Case, often the executable/scripted realization of one or more Test Cases.
  - **Test Suite** — a collection of Test Cases/Procedures grouped for a shared execution context (e.g., a regression suite).
  - These four are a hierarchy, not synonyms: a Test Case is *what* is being verified and under what conditions; a Test Procedure/Script is *how* it is executed; a Test Suite is the *grouping*.
- **Coverage types — Statement, Branch, and Path (structural/white-box coverage criteria).**
  - **Statement Coverage** — the percentage of executable statements exercised by a test suite at least once. The weakest criterion; can be 100% while entire branches (e.g., the `else` of an `if`) are never taken.
  - **Branch (Decision) Coverage** — the percentage of decision outcomes (each `if`/`switch` branch, both true and false) exercised at least once. Strictly subsumes Statement Coverage.
  - **Path Coverage** — the percentage of distinct execution paths (every combination of branch outcomes through a routine) exercised. The strongest and most expensive criterion; combinatorially explodes with nested conditionals and is rarely pursued exhaustively in practice.
  - These are structural/white-box measures, distinct from Test Pyramid layer proportions (unit/integration/e2e), which are about test scope and cost, not code-structure exercise.
- **Regression Testing (precise definition).** Testing performed after a modification to a system or component to confirm that the modification has not introduced new defects and that previously working functionality still functions as intended (ISTQB). The precision that is easy to lose: regression testing is defined by *when* and *why* it runs (after a change, to protect existing behavior), not by *which* tests compose it — any Unit, Integration, Contract, or E2E test can be "a regression test" the moment it is run for that purpose. Regression Testing is not a fifth pyramid layer; it is a purpose a test from any existing layer can serve.
- **Test Oracle** — the mechanism (specification, prior known-good output, a simpler reference implementation, a human judgment) used to determine whether a test's actual result is correct. Every test has an oracle, even if implicit; a test with no defined oracle beyond "it doesn't crash" is a weak test.
- **Smoke Testing vs. Sanity Testing (frequently conflated, precisely distinct in standard usage).**
  - **Smoke Testing** — a shallow, broad set of tests run to confirm a build is stable enough to accept further, deeper testing (does the system even start, do the critical paths respond at all). Breadth over depth.
  - **Sanity Testing** — a narrow, focused check that a specific piece of functionality or a specific fix behaves as expected, after a narrow change, without doing a full regression pass. Depth over breadth, scoped to what just changed.
- **Severity vs. Priority (a defect-triage distinction, not synonyms).** Severity is the objective impact of a Failure on the system (data loss, crash, cosmetic glitch) — a technical judgment. Priority is the order in which a defect should be fixed relative to other work — a business/scheduling judgment. A low-severity defect (a typo on a rarely-seen screen) can have high priority (it's on the CEO's demo path); a high-severity defect (data corruption in a deprecated, soon-to-be-removed module) can have low priority.
- **Static vs. Dynamic Testing (verification technique classification).** Static testing examines a work product without executing it (code review, static analysis, requirements walkthroughs, linting). Dynamic testing requires executing the code with specific inputs and observing actual behavior. Both are Verification techniques (and static review, applied to a requirement itself, can also be a Validation technique — confirming the requirement itself reflects real user need).

## Actionable Techniques

Because this is a terminology dictionary rather than a practice book, its "techniques" are precision disciplines — places where this repo's own testing vocabulary currently blurs a distinction the standard terminology keeps sharp, and where tightening the language would improve traceability and reduce ambiguity in artifacts a PM (Shafi) must review without IDE tooling:

- **Stop using "Verification" and "Validation" as unmarked synonyms of "check"/"confirm."** CLAUDE.md itself says "Nothing advances without passing validation" (line 31) when it means phase-gate *verification* against required artifacts and structure — not validation in the ISO 9000 sense (fitness for actual user need). The word choice is currently accidental, not precise, even though the plugin's own phase model (`Quality` vs. `Customer Validation`) already implicitly tracks the real distinction.
- **Separate "Test Case" (the specification) from "Test" (the activity) from "Test Procedure/Script" (the executable realization) in any skill or agent output that describes what a test-authoring agent produces** — currently these three ideas are used interchangeably across `test-pyramid` and `methodology-review` (e.g., "test file," "test," "test coverage" all used loosely for what are, precisely, different things).
- **Name Statement/Branch/Path coverage explicitly wherever "coverage" is discussed as a gate.** `test-pyramid/SKILL.md`'s Coverage Philosophy section says "Branch over line" (line 108) — correct instinct, imprecise vocabulary; "line coverage" is colloquial for Statement Coverage, and Path Coverage (the strongest, most expensive criterion) is never mentioned at all, which is fine as a choice but should be a stated choice, not an absent category.
- **Introduce the Error → Fault → Failure chain wherever "bug," "defect," or "issue" is used informally** — this precision would resolve a real, current terminology collision (see Applies To below): `methodology-review/SKILL.md` defines "Defect" with a governance-specific meaning ("a methodology requirement that applies to an artifact but is absent," line 26) that is a legitimate, deliberate, self-contained usage for that skill — but it directly collides with the standard testing-discipline meaning of Defect (a flaw in a work product, synonym of Fault) used nowhere else, with no cross-reference connecting or distinguishing the two. A reader moving from `methodology-review`'s "Defect" to `test-pyramid`'s "Every Bug Gets a Regression Test" section has no signal that "Defect" (governance) and "Bug"/Fault (testing) are different concepts sharing the same conceptual neighborhood.
- **Add Smoke Testing as a canonical, defined term** — this repo has literally built and named a smoke-test suite (`tests/run-smoke-tests.sh`, described at length in CLAUDE.md's "Running Tests" section, and this very branch is `feature/19-smoke-test-suite`) without the term "Smoke Testing" ever appearing as a defined entry in the canonical Testing and Quality glossary. This is the single most concrete, actionable gap this research surfaced.

## Key Terminology

Proposed additions/corrections to `skills/glossary-management/references/ubiquitous-language.md`'s "Testing and Quality" category, each as a precise one-line definition in the file's existing style:

- **Verification** — Confirmation, through objective evidence, that specified requirements have been fulfilled; "are we building the product right?" Distinct from Validation.
- **Validation** — Confirmation, through objective evidence, that the requirements for a specific intended use have been fulfilled; "are we building the right product?" Distinct from Verification; a system can pass Verification while failing Validation.
- **Error (Mistake)** — A human action that produces an incorrect result. The root cause that introduces a Fault into a work product.
- **Fault (Defect)** — A flaw in a work product (code, design, requirement) resulting from an Error. Static — exists whether or not it is ever executed. Note: this repo's `methodology-review` skill uses "Defect" in an unrelated, governance-specific sense (a missing required practice) — the two meanings must not be conflated.
- **Failure** — The observable deviation of a system from its specified or expected behavior, occurring when a Fault is executed under triggering conditions.
- **Test Case** — A documented specification of preconditions, inputs, and expected results designed to exercise a specific test objective. Distinct from the Test Procedure that executes it.
- **Test Procedure (Test Script)** — The documented or automated sequence of actions that executes one or more Test Cases.
- **Test Oracle** — The mechanism used to determine whether a test's actual result is correct (a specification, a known-good prior result, a reference implementation, or human judgment).
- **Statement Coverage** — The percentage of executable statements exercised at least once by a test suite. The weakest structural coverage criterion.
- **Branch (Decision) Coverage** — The percentage of decision outcomes (both branches of every conditional) exercised at least once by a test suite. Subsumes Statement Coverage.
- **Path Coverage** — The percentage of distinct execution paths through a routine exercised by a test suite. The strongest and most expensive structural coverage criterion.
- **Smoke Testing** — A shallow, broad test pass confirming a build is stable enough to accept deeper testing. Breadth over depth. (This repo's own `tests/run-smoke-tests.sh` is a live instance of this practice and should be the worked example in this glossary entry.)
- **Sanity Testing** — A narrow, focused test pass confirming a specific change behaves as expected, without a full regression pass. Depth over breadth, scoped to what changed.
- **Severity** — The objective technical impact of a Failure on the system. Distinct from Priority.
- **Priority** — The scheduling order in which a defect should be addressed relative to other work. Distinct from Severity; the two can diverge in either direction.
- **Static Testing** — Verification performed on a work product without executing it (review, static analysis, walkthrough).
- **Dynamic Testing** — Verification or Validation performed by executing a work product with specific inputs and observing actual behavior.

## Caveats

- **This is an SAE (automotive/systems engineering) reference applied to a software SDLC context, and the fit is imperfect.** SAE's V&V vocabulary (and Quigley's actual documented body of work) is grounded in automotive/embedded systems engineering — configuration management, hardware-software integration V&V, requirements traceability across physical and electronic systems, safety-standard verification (functional safety, ISO 26262-adjacent territory). Software-only web/API/event-driven systems (this repo's actual domain — Go services, event-driven microservices, React frontends) share the terminology's *foundations* (the Error/Fault/Failure chain, Verification/Validation, coverage criteria all originate in and remain identical across both domains) but not its specific concerns (hardware-in-the-loop testing, physical safety certification, embedded systems timing verification are not this repo's problem). Use this source for the precise, standardized vocabulary; do not import automotive-specific V&V process models (e.g., V-model gate structures tied to physical build stages) without adapting them to a continuous-deployment software context that `test-pyramid` and `ci-pipeline`/`cd-pipeline` already handle in a more current, software-native form.
- **A dictionary defines terms; it does not adjudicate between competing usages that already exist deliberately in this repo.** The most important finding here — the Defect collision between `methodology-review`'s governance sense and the testing-discipline sense — is not "wrong" in either location; it is an unmarked collision. The fix is a disambiguation note, not necessarily a rename, since `methodology-review`'s usage is now embedded in its own Pass/Defect/Warning rubric and renaming it would ripple through every skill and hook that cites "defect" in that governance sense.
- **Precision-for-its-own-sake is not the goal.** Not every colloquialism needs replacing — "bug" in casual prose inside a skill's narrative text is harmless; the actionable finding here is specifically about *canonical glossary entries and phase-gate/methodology vocabulary*, where imprecision has real governance consequences (a hook checking for "Defects" that could mean either sense, a gate named "validation" that actually performs verification).
