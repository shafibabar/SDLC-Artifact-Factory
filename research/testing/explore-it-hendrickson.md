# Explore It! Reduce Risk and Increase Confidence with Exploratory Testing
**Author:** Elisabeth Hendrickson
**Domain:** Testing

## Core Concepts

- **Exploratory testing is simultaneous learning, test design, and test execution — a skilled discipline, not "clicking around."** Hendrickson's foundational reframe: in scripted testing these three activities are separated in time (someone designs a test, then later someone/something executes it). In exploratory testing they happen in the same moment, in the explorer's head — what you learn from the last action informs the next test you design, which you then immediately execute. This is the book's answer to "how is this different from just poking at the app," and it is the single idea everything else in the book operationalizes.
- **The test charter — a structured mission statement for a session, not a free-for-all.** Hendrickson's own template, stated explicitly in the book: **"Explore [target/area], with [resources], to discover [information]."** A charter names *what* to explore (a feature, a workflow, a component), *what to bring* (test data, tools, environment access, personas), and *what you're trying to learn* (risks, unknowns, a specific quality concern) — without prescribing the exact steps, which is what preserves the tester's freedom to follow what they find. A charter is deliberately narrow enough to focus a session and open enough to allow discovery.
- **Session-based structure: charter → time-boxed execution → debrief.** Sessions are typically time-boxed (Hendrickson discusses roughly an hour as a practical default, adjustable), produce running notes captured *during* the session (not reconstructed afterward from memory), and end with a debrief that surfaces what was learned, what bugs/risks were found, and what new charters the session itself suggests. This is a lighter-weight, more practical cousin of the formal Session-Based Test Management (SBTM) that James and Jonathan Bach developed — Hendrickson's version emphasizes the charter/session/debrief loop as a repeatable habit rather than SBTM's fuller metrics-and-reporting apparatus.
- **Touring as a way of varying your angle of attack.** Hendrickson devotes a chapter to "touring" the product — the idea that exploring the same feature from different vantage points (a new user's perspective, a power user's perspective, a "what's the most valuable data here" perspective) surfaces different classes of problems than a single straight-through pass. This is a lighter, more general treatment of the same metaphor James Whittaker's *Exploratory Software Testing* develops at much greater length with an enumerated catalog of named tours (Money Tour, Landmark Tour, Garbage Collector's Tour, etc.) — cite Hendrickson for the *charter-driven, angle-varying habit*; cite Whittaker if a future research pass wants the fuller named-tour catalog.
- **Heuristics for generating test ideas when you don't know where to start.** The book's most directly reusable content is a set of named heuristics, each a compact trigger for a class of test idea:
  - **CRUD** — for any persistent thing, test Create, Read, Update, and Delete, and critically the *interactions between them* (can you delete something that's referenced elsewhere? does an update survive a re-read?).
  - **Boundaries** — the edges of any range: minimum, maximum, zero, one below/above a limit, empty, off-by-one.
  - **Position / "Goldilocks"** — first, last, only-one, none, many; too few, too many, just right.
  - **Interruptions** — start a workflow, then stop mid-way (navigate away, lose connectivity, let a session expire, switch users) and resume or abandon it; interruption is one of the highest-yield heuristics for finding state-management bugs scripted tests rarely cover because a script normally runs a workflow start-to-finish.
  - **Configuration variance** — the same feature under different browsers, devices, locales, permission levels, feature-flag states, or environment configurations.
  - **The SFDIPOT model** (Structure, Function, Data, Interfaces, Platform, Operations, Time — credited to James Bach, adopted and applied by Hendrickson as a coverage-category checklist) — a way to ask "have I even considered testing this dimension of the system at all," used to spot blind spots in what's been explored so far, not to generate individual test ideas the way CRUD/boundaries do.
  - **The saboteur mindset** — deliberately trying to break things a well-behaved user never would (wrong input types, out-of-order steps, malformed data) as a distinct stance from "does the happy path work."
- **Exploratory testing is a distinct activity from confirming a known requirement — it exists to find what nobody thought to specify.** Scripted/automated tests (including BDD scenarios) can only check the things someone already wrote down. Hendrickson's exploratory testing exists specifically to surface the risks, edge cases, and interactions that *no one wrote a scenario for* — not because scripted testing is deficient, but because it is structurally incapable of finding what it didn't anticipate.
- **Testing without a story or a spec.** A dedicated part of the book addresses testing a feature (or an entire legacy system) that has no requirements document at all — using heuristics and touring to construct a mental model of expected behavior on the fly, which is directly relevant to any product with under-specified or missing requirements.

## Actionable Techniques

- **A charter template a skill can adopt directly:** `Explore [target/area], with [resources], to discover [information about risk/quality].` Concrete example in this repo's domain: "Explore the sensitivity-classification workflow, with a mix of scanned low-quality PDFs and edge-case file types, to discover whether misclassification or crashes occur outside the happy-path documents already covered by scripted UAT scenarios."
- **A session record template:** Charter · Time-box (e.g., 45–90 min) · Tester · Environment · Notes (running log of actions and observations, captured live) · Bugs/Issues found · New charters/questions surfaced · Debrief summary.
- **A heuristics checklist a tester (or a UAT executor) can literally run down** when generating test ideas for a feature: CRUD, Boundaries, Position/Goldilocks, Interruptions, Configuration variance, Saboteur — each phrased as a one-line prompt ("What happens if I interrupt this workflow halfway and come back later?").
- **The SFDIPOT coverage-category audit**: before closing an exploratory pass on a feature, check whether Structure, Function, Data, Interfaces, Platform, Operations, and Time have each been touched by at least one charter — a way to catch "we only ever explored the happy-path UI, never the underlying data/platform/timing dimensions."
- **Charter-generates-charter discipline**: every debrief should produce zero or more follow-on charters for what the session surfaced but didn't have time to pursue — treating exploration as an ongoing, branching activity rather than a single closed session.

## Key Terminology

Candidates worth checking against `skills/glossary-management/` if the gap above is acted on:

- **Test Charter** — Hendrickson's specific `Explore/with/to discover` structure for scoping an exploratory session; not present anywhere in the four skills read.
- **Exploratory Session** (charter + time-box + notes + debrief) — the unit of exploratory work; currently has no artifact shape anywhere in this repo's Customer Validation vocabulary, which otherwise only has `uat-scenario` (scripted, pass/fail).
- **Touring** — the practice of re-exploring the same feature from varied vantage points; would be new vocabulary, and should be introduced carefully given the term is more fully developed (with named tours) in Whittaker's book — worth a glossary note distinguishing "touring as a general habit" (Hendrickson) from "the enumerated named-tour catalog" (Whittaker) if both are ever cited.
- **SFDIPOT** — Structure/Function/Data/Interfaces/Platform/Operations/Time coverage-category heuristic (originally Bach, applied by Hendrickson); useful as a completeness checklist term if exploratory coverage tracking is ever formalized.

## Caveats

- **Session-based exploratory testing, as the book describes it, assumes a live human tester exercising real-time judgment.** Hendrickson is explicit that the value of exploration comes from a skilled person's in-the-moment curiosity and pattern recognition — the same-moment loop of "learn something, immediately design the next probe based on it" is precisely the part hardest to hand to an agent running a fixed prompt. This repo's existing UAT model already has the same bottleneck (a design partner or Shafi-as-proxy per `uat-plan`), so adding exploratory sessions does not introduce a *new* scarce-human-bandwidth problem — it makes the existing one more visible, because a charter session cannot be scripted away the way a `uat-scenario` execution nominally could be handed to a proxy following a script.
- **A single agent (or an agent-facilitated session) "running" an exploratory charter against a live system is a different activity than the book describes, and this repo should be honest about that rather than pretend equivalence.** A plausible adaptation: an agent (or the `requirements-analyst` acting as facilitator) drafts the charter and heuristics checklist, and the actual exploring is still done by a human executor (design partner or Shafi-proxy) using the charter as a mission brief — which keeps the book's core claim (exploration needs human judgment) intact while giving this repo a concrete artifact (the charter) an agent *can* author. An agent attempting to "explore" a UI autonomously (clicking through screenshots, say) would be simulating scripted testing under an exploratory label, not doing what the book describes — this distinction should be stated explicitly if this gap is ever closed, not glossed over.
- **The heuristics (CRUD, boundaries, interruptions, configuration, SFDIPOT) are the most directly and safely reusable part of the book for this repo**, because they are checklist-shaped and heuristic-application is exactly the kind of "apply criteria" reasoning this repo's CLAUDE.md assigns to skills/agents already — unlike the live-session mechanics, adopting the heuristics list requires no adaptation for the single-agent-per-role model.
- **Session-Based Test Management (SBTM) proper (Bach/Bach) is a heavier formalism than what Hendrickson prescribes** — full SBTM includes session metrics, debrief reports scored against a rubric, and session-count-based coverage estimation. Hendrickson's book is lighter-weight and more approachable; if this repo adopts a session artifact, it should cite Hendrickson's simpler charter/session/debrief loop rather than importing full SBTM's reporting overhead, consistent with this repo's frugality principle.
