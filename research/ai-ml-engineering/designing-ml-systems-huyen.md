# Designing Machine Learning Systems
**Authors:** Chip Huyen
**Domain:** AI/ML Engineering (classical ML system design — training data, features, models, deployment, monitoring)

## Core Concepts

- **This is the *trained-model* lifecycle, not the *foundation-model* lifecycle — a different angle from book 1 even though it's the same author (Ch. 1-3, throughout).** `research/ai-ml-engineering/ai-engineering-huyen.md` (book 1, *AI Engineering*) covers adapting a pre-trained foundation model via prompting/RAG/fine-tuning. This book predates that shift and covers the classical alternative: collecting and labeling your own training data, engineering features, training and selecting a model from scratch (or fine-tuning as the *primary* path, not an escalation), and running the full MLOps lifecycle around it. Both are legitimate answers to the same problem this repo has left unnamed — `data-classification`'s "Automated detection" (SKILL.md lines 68-81) and `data-pipeline-implementation`'s `extractUnit(ctx, unit)` (line 164) never say which. Book 1 argues for prompting as the frugal default; this book supplies the discipline needed *if* a trained classifier is chosen instead — and per CLAUDE.md's frugality principle ("open-source over paid tooling," "prefer simpler solutions... when outcomes are equivalent"), a small trained model called locally at effectively zero marginal cost is a real frugality contender against a per-document LLM API call at estate scale, once training-data and monitoring discipline exist to support it.
- **Training data has its own provenance problem, distinct from this repo's operational data lineage (Ch. 4, "Training Data").** `data-lineage-design/SKILL.md` (read in full) implements OpenLineage — Dataset/Job/Run — to answer "where did this compliance finding come from," and explicitly keeps input file *content* transient (line 58), recording only that an entity was derived from a file, never the file's content. That is exactly right for operational lineage. But Huyen's book is about a different provenance question entirely: which labeled examples, collected when, from which sampling strategy, went into *training the classifier itself* — a training-set version that must be tracked, reproduced, and audited independently of any single document's processing run. This repo currently has no artifact for that question at all; `data-lineage-design`'s OpenLineage model answers "what happened to customer document X," never "what data trained the model that classified customer document X."
- **Sampling and class imbalance shape what a model can learn, and must be a stated design decision (Ch. 4, "Sampling").** Huyen catalogs sampling strategies (random, stratified, weighted, importance, reservoir) and treats class imbalance (rare classes swamped by common ones) as a first-class problem with named techniques (resampling, class weights, synthetic minority oversampling) — not something a training pipeline back-and into. This maps with unsettling precision onto `data-classification`'s own detection table (SKILL.md lines 72-77): `SSN`, `NATIONAL_ID`, `HEALTH_TERM` are exactly the rare, highest-consequence classes in a real document corpus (most documents are ordinary business content; genuinely Restricted entities are the minority), while `EMAIL`/`PHONE` are comparatively common. A classifier trained naively on a corpus of crawled documents would see this same imbalance and, uncorrected, learn to under-detect the rare classes that matter most — the identical asymmetric-error-cost concern book 1 raised for evaluation thresholds, but here as a *training-data-collection* problem that must be solved before a threshold can even be meaningfully tuned.
- **Feature engineering and the train/serve skew failure mode (Ch. 5, "Feature Engineering").** For any classical (non-prompted) classifier, features must be computed identically at training time and at inference time — the book's most-cited production failure mode is a feature computed one way in an offline training pipeline (e.g., Python/pandas) and a subtly different way in the online serving path (e.g., Go), producing silently wrong predictions that no test catches because both paths "work." `data-pipeline-implementation`'s own Python Exception section (SKILL.md lines 259-277) already anticipates a Go/Python split for exactly this kind of ML workload ("OCR/ML-based extraction... a document-layout or OCR model") — but it frames the exception purely as a *language/library* choice, never naming train/serve feature-computation consistency as a distinct risk that crossing a language boundary specifically introduces.
- **Model development is inseparable from experiment tracking and versioning (Ch. 6, "Model Development and Offline Evaluation").** Every training run — data version, feature set, hyperparameters, resulting metrics — needs to be recorded and reproducible, and every deployed model needs an identity (a version) distinct from the code that serves it. Book 1 already flagged "model/prompt version" as a missing audit-trail field for *prompted* classifications (its Key Terminology, and Actionable Technique #7); this book extends the same gap to *trained* models with more machinery behind it: a model registry (which artifact is version 14, what data and code produced it, is it currently serving), not just a version string in a log line. Nothing in `data-classification`, `data-pipeline-implementation`, or `data-quality-rules` names a registry, a training-run record, or a rollback path for a classifier.
- **Deployment patterns are a model-specific decision, not a generic service decision (Ch. 7, "Model Deployment and Prediction Service").** The book distinguishes batch prediction (score everything on a schedule, serve from a precomputed table — high throughput, tolerates latency) from online/streaming prediction (score on demand — low latency, higher infra cost) as an explicit trade-off, and adds **shadow deployment** (a candidate model runs alongside the current one on live traffic, its predictions logged but never acted on, compared offline before it ever serves a real decision) as the safest way to validate a new model with zero user-facing risk. `data-pipeline-design`'s choreographed, async pipeline topology (per `data-pipeline-implementation`, read in full) is already a batch-shaped architecture — extraction happens stage-by-stage over an estate scan, not synchronously per user request — but nothing in the repo's skills names this as a deliberate "batch prediction" choice, and `canary-deployment/SKILL.md` (checked: its Gate Metrics section, lines 69-111, gates on RED-style service health — error rate, latency — not on a model's *prediction* quality) has no concept of shadow deployment or a champion/challenger comparison for a model specifically, only staged traffic-percentage rollout for a service.
- **Monitoring and continual learning is the book's most distinctive contribution, and it is almost entirely absent from this repo (Ch. 8-9, the book's capstone chapters).** Huyen names three failure modes ordinary service monitoring cannot see: **data drift** (the statistical distribution of production inputs shifts away from what the model was trained/evaluated on — e.g., the estate starts ingesting a new document format or language mix), **concept drift** (the relationship between input and correct label changes even if inputs look the same — e.g., a new regulation redefines what counts as Restricted, or a business unit starts producing a new kind of contract), and **model staleness** (a model's accuracy silently decays over time as the world moves on, with no code change and no error thrown). None of these are detectable by `opentelemetry-instrumentation`'s RED/USE metrics (SKILL.md, checked in full: Rate/Errors/Duration and Utilization/Saturation/Errors are entirely about *service health*, never about *prediction quality*) or by `prometheus-metrics-design`'s metric taxonomy (checked in full: "Interface metrics," "Resource metrics," "Platform metrics" — three classes, none of them "model quality"). A drifting classifier that starts calling Restricted documents Public produces *zero* RED/USE signal — latency is fine, error rate is fine, the service is "healthy" by every metric this repo currently defines, while quietly committing the exact compliance failure the whole product exists to prevent.
- **The closest thing this repo has to drift detection is a biased, ex-post proxy — not a monitoring design (data-quality-rules, metrics-instrumentation-plan).** `data-quality-rules/SKILL.md` (checked in full) names a "model-accuracy proxy" metric (line 127): "Steward override rate (quarantine resolved as confirm/correct/reject)... a high correction rate suggests the extraction model itself needs work." `metrics-instrumentation-plan/SKILL.md` (checked in full) formalizes this as "Classification accuracy proxy" (line 76): `1 − (steward-corrected classifications ÷ total steward reviews)`, target ≥ 0.85. Huyen's book would flag the fatal flaw in this design immediately: it is computed *only over documents a human already reviewed* — i.e., only over low-confidence, quarantined cases (`data-quality-rules`'s three-way gate, lines 81-97). It cannot see a confident-but-wrong classification that never triggered review at all — precisely the "Restricted misclassified as Public with high confidence" scenario that is this product's worst compliance failure mode, and the one ordinary confidence-threshold gating is structurally blind to. This is a **selection bias** in the existing "model-accuracy proxy," not a comprehensive evaluation, and the book's monitoring chapters are what would surface and fix it — via drift detection on the *full* production input/output distribution, not just the reviewed subset.
- **Natural labels and feedback loops make continual learning tractable here — but only if designed for, not left as an accident (Ch. 4 "Natural Labels," Ch. 9 "Continual Learning").** A natural label is ground truth the system acquires for free from user behavior, sometimes with a delay — this product already has one: `DataAssetReclassified` events where a steward's manual decision *disagrees* with the automated provisional level are, functionally, delayed natural labels for the classifier's own past prediction. Huyen's continual-learning framing treats this exact signal as the input to a retraining/re-prompting loop with a defined cadence and trigger condition — not merely an after-the-fact accuracy metric to watch trend downward. `data-classification/SKILL.md`'s audit discipline (manual overrides stored separately from computed level, escalation-past-override triggers re-review, lines 92-98) already captures the *events* needed for this loop; nothing in the repo currently closes the loop by feeding them back into a retraining or re-evaluation trigger.

## Actionable Techniques

- Add a **training-data provenance record**, distinct from `data-lineage-design`'s OpenLineage operational lineage: for any trained (not purely prompted) classifier, record the training-set version, its collection window, its sampling strategy, and its class balance — reproducible and auditable independently of any single document's processing lineage.
- Before training or fine-tuning any classifier on the detected-entity distribution in `data-classification`'s table, explicitly check and correct for class imbalance (the rare-but-highest-consequence classes — `SSN`, `HEALTH_TERM`, `NATIONAL_ID` — are structurally underrepresented in ordinary business content) using a named technique (stratified sampling, class weighting, or targeted oversampling of Restricted-category examples) — never train on the raw natural document distribution and hope thresholds fix it later.
- For any pipeline stage that computes features in one language/runtime for training and another for serving (the exact Go/Python split `data-pipeline-implementation`'s Python Exception already anticipates for OCR/ML stages), add an explicit train/serve parity check — a test asserting the same feature-computation code path (or a shared, versioned feature-computation library) is used in both, not two independently maintained implementations that can silently diverge.
- Require a **model registry entry** for every trained classifier version: training-data version, code version, offline evaluation metrics, and current serving status (active / shadow / retired) — extending book 1's "add prompt/model version to the audit trail" recommendation with the actual registry machinery a trained model (not just a prompt string) needs.
- Before promoting any new model version to serve real classification decisions, run it in **shadow deployment** — scoring live production documents in parallel with the currently-serving model, logging predictions without acting on them, and comparing the two offline — as the default validation gate, cheaper and safer than `canary-deployment`'s staged-traffic-percentage rollout for a decision this compliance-sensitive.
- Add a **model quality metric class** to the observability stack, parallel to but distinct from RED/USE (`opentelemetry-instrumentation`) and product metrics (`metrics-instrumentation-plan`): data drift (statistical distance between the current window's input feature/document distribution and the training/reference distribution), prediction drift (shift in the classifier's output class distribution over time, e.g., the Confidential/Restricted/Public/Internal mix drifting from its historical baseline), and model staleness (time since last retrain/re-evaluation against a fresh sample) — each as its own named Prometheus metric and alert, not folded into the existing `pipeline_documents_processed_total`/`pipeline_dlq_depth` families.
- Fix the **selection bias in the existing accuracy proxy**: `data-quality-rules`' and `metrics-instrumentation-plan`'s steward-override-rate metric measures only reviewed (quarantined) cases. Add a periodic, randomly-sampled audit of *passed* (non-quarantined, confidently-classified) documents, re-reviewed against a small held-out labeled sample, specifically to catch the confident-but-wrong failure mode the quarantine gate structurally cannot see.
- Design a **continual-learning trigger**, not just an accuracy-trend chart: define the condition (e.g., data drift metric exceeds a threshold for N consecutive days, or the fixed-audit-sample accuracy drops below a floor) that automatically opens a retraining/re-prompting review task, closing the loop `DataAssetReclassified`'s delayed natural labels already make possible.
- Name the **deployment pattern** (`data-pipeline-design`'s async, stage-by-stage estate scan is a batch-prediction pattern in Huyen's terms) explicitly in the pipeline architecture document, and evaluate whether any future feature (e.g., a "classify this one uploaded file right now" interactive path) would need a genuinely different, online/synchronous prediction service — a distinct deployment decision from the existing batch pipeline, not an assumed drop-in extension of it.
- Where training data would need to be pooled across tenants to get enough examples of a rare class (e.g., enough labeled `HEALTH_TERM` examples), require an explicit ADR — this repo's physical multi-tenancy stance (per-tenant data isolation, one Prometheus per tenant, per-tenant storage) does not obviously permit cross-tenant training-data pooling without an explicit, reviewed decision, and Huyen's book takes such pooling for granted in a way this repo's compliance posture cannot.

## Candidate New Skills/Agents

Three candidates. All three assume book 1's proposed `ai-systems-engineer`/`ml-engineer` agent gets built — none of them propose a new, competing agent; they extend that agent's portfolio with the classical-ML-lifecycle work book 1 never covered. None of these re-propose book 1's `llm-classification-design`, `ai-system-evaluation`, `prompt-engineering-and-versioning`, or `rag-pipeline-design` — they are additive, and one of them (`model-drift-monitoring`) is explicitly scoped *against* `ai-system-evaluation` below, per the task's instruction to make that boundary call honestly rather than silently duplicating it.

1. **`ml-training-pipeline-design` (skill, design-phase, owned by data-architect or the ai-systems-engineer agent).** Training-data provenance (distinct from `data-lineage-design`'s operational OpenLineage lineage — this is "what data trained the model," not "what happened to this customer's document"), sampling strategy and class-imbalance handling (directly addressing the rare-but-highest-consequence entity types `data-classification` already names), feature engineering and the train/serve-skew failure mode for any stage crossing the Go/Python boundary `data-pipeline-implementation`'s Python Exception already anticipates, and train/validation/test split discipline with explicit data-leakage checks (e.g., never letting documents from the same source asset span both train and test splits). This is real, new ground book 1 never touches — book 1's world has no "training data" at all, only prompts and retrieved context.

2. **`model-lifecycle-and-deployment-patterns` (skill, owned by the ai-systems-engineer agent, handing off to platform-engineer for infra).** A model registry concept (version, training-data version, code version, offline eval metrics, serving status), batch vs. online vs. streaming deployment-pattern selection (naming `data-pipeline-design`'s existing async architecture as the batch-prediction choice it already is, and flagging when a future interactive feature would need a distinct online path), and **shadow deployment** as the default pre-promotion validation gate for a new model version — extending `canary-deployment`'s service-level staged-rollout pattern with a prediction-comparison gate `canary-deployment`'s Gate Metrics (error rate, latency) structurally cannot express. This is the deployment half of what book 1 only sketched as "add model version to the audit trail" — a full lifecycle around that version, not just a field.

3. **`model-drift-monitoring` (skill, owned by the ai-systems-engineer agent for the model-quality metric design, with platform-engineer implementing the Prometheus recording rules/alerts exactly as the existing RED/USE-vs-product-metrics split already works).** **Explicit scope call against `ai-system-evaluation` (book 1's proposed skill), as instructed:** `ai-system-evaluation` is periodic, *labeled* evaluation — golden dataset, precision/recall, run on a schedule, requires ground truth. `model-drift-monitoring` is continuous, *unsupervised* production monitoring — statistical distance between the current input/output distribution and a reference distribution, requiring no new labels at all, and therefore able to raise a signal far faster and cheaper than waiting for the next scheduled labeled evaluation. They are complementary, not overlapping: drift monitoring is the cheap, continuous tripwire; `ai-system-evaluation` is the expensive, periodic confirmation of *why* the tripwire fired. This is the single most consequential candidate in this file for the reason stated in the task: a data-sensitivity classifier that silently drifts from correctly classifying Restricted documents to missing them produces zero signal in `opentelemetry-instrumentation`'s RED/USE metrics, zero signal in `prometheus-metrics-design`'s three-class taxonomy (Interface/Resource/Platform), and a *biased* signal in the existing "model-accuracy proxy" (which only sees reviewed, not passed, documents) — this repo currently has no mechanism that would catch this specific compliance failure at all.

## Key Terminology

Worth considering for `skills/glossary-management/`'s Ubiquitous Language if any of the candidates above are built. None of these duplicate book 1's terms (`Foundation Model`, `Prompt Engineering`, `RAG`, `Golden Dataset`, `Hallucination`, `Model/Prompt Version`, `Guardrails`):

- **Data Drift** — a shift in the statistical distribution of production model inputs away from the distribution the model was trained/evaluated on, detectable without new labels. The specific failure mode `opentelemetry-instrumentation`'s RED/USE and `prometheus-metrics-design`'s three metric classes have no vocabulary for at all.
- **Concept Drift** — the relationship between input and correct label changing over time even when the input distribution looks stable (e.g., a new regulation redefines what "Restricted" means). Distinct from data drift: data drift is "the inputs look different," concept drift is "the same-looking inputs now mean something different."
- **Model Staleness** — the gradual decay of a deployed model's accuracy over time purely from the world changing around it, with no code change and no thrown error — the reason a model needs a retraining cadence, not just a one-time evaluation at ship time.
- **Train/Serve Skew** — the production failure mode where a feature is computed differently in the offline training pipeline than in the online serving path, producing silently wrong predictions no test catches because both paths individually "work." Directly relevant to any Go-training/Python-serving (or vice versa) split this repo's Python Exception already permits.
- **Natural Label** — ground truth acquired for free (sometimes with a delay) from user/system behavior, rather than purpose-collected. This repo already has one: a Data Steward's `DataAssetReclassified` override is a delayed natural label for the classifier's own prior prediction — currently used only as a trailing accuracy ratio, not as a designed continual-learning input.
- **Class Imbalance** — a training-data distribution where some classes are far rarer than others, requiring a named correction technique (stratified sampling, class weighting, oversampling) rather than naive training on the raw distribution. Maps directly onto `data-classification`'s rare-but-highest-consequence entity types (`SSN`, `HEALTH_TERM`, `NATIONAL_ID`).
- **Shadow Deployment** — running a candidate model on live production traffic in parallel with the currently-serving model, logging its predictions without acting on them, to validate it with zero user-facing risk before promotion. Distinct from `canary-deployment`'s percentage-based live traffic-shifting, which is a service-health rollout pattern, not a prediction-comparison one.
- **Champion/Challenger** — the paired naming for the currently-serving model (champion) and a candidate replacement being validated (challenger, typically via shadow deployment) before a promotion decision. Needed the moment `model-lifecycle-and-deployment-patterns` names a registry with more than one model version.
- **Training-Data Provenance** — the record of which labeled examples, collected when, via which sampling strategy, trained a specific model version — distinct from `data-lineage-design`'s OpenLineage *operational* lineage, which tracks what happened to a specific customer document, never what trained the model that classified it.

## Caveats

- **This book assumes you are training or fine-tuning a model at all — that is a decision this repo has not yet made, and book 1 would default it toward "no."** Everything in this file (training data, features, class imbalance, a registry, shadow deployment) is conditional on the technique-selection decision (book 1's proposed `llm-classification-design`) landing on "trained classifier" for some entity type rather than "prompted foundation model." Given CLAUDE.md's frugality stance and book 1's own caveat that fine-tuning is "the most expensive, least-default recommendation," treat this file's content as the discipline required *if and when* that decision is made for a specific, high-volume, stable-shape classification task — not a mandate to go train models generally. The one place this book's case for a trained model is genuinely strong on frugality grounds: a small, self-hosted, locally-served classifier can have a lower marginal cost per document at estate scale than a per-call LLM API, once the training/monitoring discipline this book supplies actually exists to support it.
- **The existing "model-accuracy proxy" metric is not wrong so much as incomplete — do not read this file as "delete it."** `data-quality-rules`' and `metrics-instrumentation-plan`'s steward-override-rate metric is a real, useful signal for the subset of documents it covers (quarantined, reviewed cases); the gap this book exposes is what it *cannot* see (confidently-passed, never-reviewed documents), not that the metric itself is a bad idea. The fix is additive — a sampled audit of the passed population — not a replacement.
- **Physical multi-tenancy constrains training-data collection in a way this book's examples never consider.** Huyen's worked examples generally assume a single pooled dataset an ML team can freely sample from. This repo's per-tenant isolation stance (separate Prometheus, separate storage, no cross-tenant data sharing implied anywhere else in the repo) means pooling documents across tenants to get enough rare-class training examples is not a default — it is, at minimum, an ADR-worthy decision with its own privacy/consent review, and quite possibly out of scope entirely for a SOC 2-target product. Any `ml-training-pipeline-design` work must confront this constraint explicitly rather than importing the book's assumption of a shared corpus.
- **Do not conflate this file's monitoring proposal with the existing RED/USE and product-metrics frameworks — the point is that a fourth class is missing, not that the existing three are wrong.** `opentelemetry-instrumentation`'s RED/USE and `metrics-instrumentation-plan`'s product metrics are both sound, well-specified frameworks for what they cover; this book's contribution is naming the thing neither covers (model *prediction quality* over time), not correcting either of them.
- **This book, like book 1, is written for general ML-application builders and does not know this repo's DDD/event-driven vocabulary.** Its drift-detection and continual-learning patterns need translating into this repo's existing idiom exactly as book 1 already did for evaluation and RAG — e.g., a continual-learning retraining trigger is naturally expressed as a new kind of Domain Event (`ModelDriftDetected` or similar) feeding the same outbox/alerting machinery every other operational signal in this repo already uses, not a bolt-on cron job with its own notification path.
- **As with book 1: do not confuse this with the `research/ai-agent-engineering/` cluster.** That cluster is about how this factory's own Claude Code agents/skills are engineered; this book (like book 1) is about the *product's* own use of ML to classify and extract from customer documents. Different layer, never cross-cited as answering the same question.
