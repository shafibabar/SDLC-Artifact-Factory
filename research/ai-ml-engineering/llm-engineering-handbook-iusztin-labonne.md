# The LLM Engineering Handbook
**Authors:** Paul Iusztin, Maxime Labonne
**Domain:** AI/ML Engineering (LLMOps — the production engineering discipline around LLM systems specifically: data pipelines, fine-tuning, CI/CD, and serving)

## Core Concepts

- **This is the *LLMOps* lifecycle, not the foundation-model-application lifecycle (book 1) or the classical-ML lifecycle (book 2) — a third, distinct angle.** `research/ai-ml-engineering/ai-engineering-huyen.md` covers adapting a pre-trained model via prompting/RAG/fine-tuning as an *application* concern; `research/ai-ml-engineering/designing-ml-systems-huyen.md` covers the *classical* trained-model lifecycle (your own labeled data, features, monitoring). This book sits at the intersection and is opinionated about the *engineering scaffolding*: a three-pipeline architecture (data collection → feature/training pipelines → inference pipeline), a specific fine-tuning technique family (LoRA/QLoRA, parameter-efficient fine-tuning of open-weight models), and — its single most distinctive contribution relative to both prior books — **LLM-specific CI/CD**: prompt versioning, dataset versioning, model registries, and automated evaluation gates as deploy-blocking checks, the direct LLM analog of this repo's `tdd-gate` hook applied to prompts and model artifacts instead of source code.
- **The "FTI" (Feature / Training / Inference) pipeline architecture (throughout, the book's organizing structural idea).** The authors decompose any production ML/LLM system into three independently deployable, independently scalable pipelines communicating only through a shared, versioned artifact store: a **feature pipeline** (collects and transforms raw data into a feature store — for an LLM system, this is the *data collection and instruction-dataset-generation* pipeline), a **training pipeline** (reads versioned features/datasets, fine-tunes, evaluates, and pushes a versioned model to a registry), and an **inference pipeline** (reads a registry-approved model and a feature/vector store, serves predictions). This is structurally the same shape as `data-pipeline-design`'s choreographed, topic-decoupled stages (read in full: "each stage is an independent, separately deployable consumer... communicate only through Redpanda topics") — but that skill's stages move *documents through classification*, never *data through model training*. Nothing in `data-pipeline-design` or `data-pipeline-implementation` (also read in full) names a training pipeline or a feature store as a pipeline *stage type* — the FTI split is a genuinely new pipeline topology this repo has never modeled.
- **Data collection, generation, and deduplication for instruction fine-tuning (the book's data-engineering chapters).** Before any LoRA/QLoRA fine-tuning run, the book insists on a dedicated data pipeline: collect raw examples (or generate synthetic instruction/response pairs via a stronger "teacher" model — distillation), deduplicate aggressively (exact-hash and near-duplicate/embedding-based dedup, since duplicate training examples both waste compute and bias the model toward over-represented patterns), and version the resulting instruction dataset as an artifact with its own identity, separate from the code that trains on it. `designing-ml-systems-huyen.md`'s already-proposed `ml-training-pipeline-design` candidate covers training-data provenance, sampling, and class imbalance in general classical-ML terms; this book's *specific* techniques (synthetic generation via a teacher model, near-duplicate detection at the instruction-example level, treating the dataset itself — not just a database row — as a versioned pipeline artifact with a hash/tag) are the LLM-specific fill of that same slot, not a competing concept.
- **LoRA and QLoRA make fine-tuning frugal, but the resulting artifact still needs a registry and a version (the book's fine-tuning chapters).** Low-Rank Adaptation freezes the base model's weights and trains a small number of low-rank adapter matrices instead — cutting GPU memory and storage requirements by orders of magnitude versus full fine-tuning; QLoRA adds 4-bit quantization of the frozen base weights to fine-tune models on a single consumer GPU. This is directly relevant to this repo's stated frugality principle ("open-source over paid tooling," "prefer simpler solutions... when outcomes are equivalent") — it is the technique that makes "fine-tune an open-weight model" a genuinely cheap option rather than the "most expensive, least-default recommendation" book 1's caveat warned against, *if* the training/registry/eval discipline this book and `designing-ml-systems-huyen.md` both call for actually exists first.
- **LLM-specific CI/CD — the book's headline distinctive content, and the thing this repo's CI/CD skills provably have zero vocabulary for.** `skills/ci-pipeline/SKILL.md` and `skills/cd-pipeline/SKILL.md` were read in full end to end. Both are excellent, real, working Go/React pipelines — but every gate, artifact, and promotion unit in both files is **code**: `go vet`, race tests, coverage, `govulncheck`, a container image pushed and promoted **by digest**. Neither file contains the words "prompt," "model," "dataset," "fine-tune," or "evaluation" even once. This book's core CI/CD claim is that an LLM system has a *second, parallel class of artifact* that needs the exact same discipline `ci-pipeline` already gives container images: a **prompt template** (versioned, tested for regressions against a golden/eval set before merge — the direct LLM analog of `ci-pipeline`'s race-test gate), a **fine-tuned model checkpoint** (registered, evaluated, and *promoted by an immutable version/hash* — the direct LLM analog of `ci-pipeline`'s "promotion by digest, never by mutable tag" rule), and a **training/instruction dataset** (versioned as its own artifact, the direct LLM analog of a `go.sum`-pinned dependency). None of these three artifact classes, or a gate that blocks a prompt/model/dataset change the way `tdd-gate` blocks an implementation file with no preceding test, exist anywhere in this repo's `ci-pipeline`, `cd-pipeline`, or `CLAUDE.md`'s Phase Gate Rules section.
- **Automated evaluation as a deploy-blocking gate, not a training-time nicety (the book's evaluation-in-CI chapter).** The book's LLMOps loop closes with: every prompt or model change triggers an automated evaluation run (against a held-out set, scored by exact-match, a rubric, or an LLM-as-judge) *inside the CI pipeline itself*, and a regression below a stated threshold fails the build exactly the way a coverage regression fails `make ci` today. This is structurally identical to `ci-pipeline`'s own "Coverage below the enforced threshold (≥80%) fails the build" gate (row 4 of its Gate Sequence table) — the book's contribution is naming the LLM-world's equivalent threshold-gated check as a first-class CI gate, not an offline, only-sometimes-run notebook experiment.
- **Inference serving is a distinct engineering problem from inference *cost/latency as a design input* (book 1's Ch. 9 territory).** `ai-engineering-huyen.md`'s Core Concepts already flagged "inference cost and latency are architecture-shaping constraints" as a decision-level concern (batching, caching, model tiering). This book goes one layer deeper into the *mechanics*: quantization formats for serving (GGUF, AWQ, GPTQ) that trade a small accuracy loss for large memory/throughput wins, continuous/dynamic batching (grouping concurrent requests to maximize GPU utilization without blocking any one request on the slowest in its batch), and KV-cache management (the dominant GPU-memory cost of serving a transformer, and the parameter that most constrains how many concurrent requests one GPU can serve). None of this mechanical detail is in book 1, and none of it is in this repo's `helm-chart`, `container-diagram`, or `platform-engineer.md` (Kubernetes/Helm/observability owner) — self-hosting an open-weight model at estate scale would need this vocabulary and this repo currently has zero of it.
- **The Python exception this repo already carved out for OCR/ML extraction is the same seam an LLM training/fine-tuning pipeline would need — but at a different scale of consequence.** `data-pipeline-implementation/SKILL.md`'s "Python Exception" section (read in full) already anticipates "a document-layout or OCR model only available via a mature Python library" as the canonical justified non-Go stage, with a narrow, ADR-recorded, still-transactionally-disciplined exception. A training/fine-tuning pipeline (PyTorch, Hugging Face `transformers`/`peft`/`bitsandbytes` for LoRA/QLoRA, a training-loop orchestration framework) is not a single stage's library dependency the way an OCR call is — it is an *entire toolchain* (GPU scheduling, distributed training, experiment tracking) with no meaningful Go equivalent at all, several orders of magnitude larger a departure from the Go-first default than a single `extractUnit` call reaching into a Python OCR library. `CLAUDE.md`'s Tech Stack Defaults table (read in full) has ten rows — Backend language, API framework, Frontend, Primary database, Message broker, Graph database, CI/CD, IaC, Service mesh, Container orchestration, Observability — and **not one of them names Python, ML/LLM training, or a training framework anywhere**, even as an acknowledged exception. The only place in the entire repository that Python is named at all, checked directly, is that one skill's "Python Exception" section — CLAUDE.md itself, the file every session reads first and calls "the non-negotiable rules," is silent on the question this book's entire second half assumes is already settled.
- **`backend-engineer.md`'s toolchain (read in full) is Go end to end, with no seam for this book's workload at all.** Every skill in its `skills:` list is `go-*` or observability-instrumentation; its Behavioral Directives are Go-idiom-specific (goroutine lifecycle, `errgroup`, escape analysis, `unsafe` forbidden without a benchmark, `go test -race`); its Quality Checklist item is `make ci` green. Nothing in its Owns/Does Not Own tables, or in `data-engineer.md`'s (referenced by both prior research files' ownership-gap finding), claims a training/fine-tuning pipeline written in Python with a completely different toolchain (PyTorch, CUDA, experiment tracking, a GPU-aware CI runner) — confirming, a third time across three books now, that this is a genuinely foreign toolchain this repo has never accounted for, not merely an underspecified corner of an existing agent's scope.

## Actionable Techniques

- Adopt the **FTI (Feature / Training / Inference) three-pipeline split** as the named topology for any future LLM training/fine-tuning work, and cross-reference it explicitly against `data-pipeline-design`'s existing choreography topology in whichever skill formalizes it — the two are the same *shape* (independently deployable stages, communicating only through a versioned artifact store) applied to a different payload (features/datasets/models instead of documents/entities).
- Treat an **instruction dataset as a versioned pipeline artifact** with its own identity (a hash or semantic version), never an unversioned file or a training-notebook side effect — the LLM-specific instance of `designing-ml-systems-huyen.md`'s already-proposed training-data-provenance record, but with this book's specific mechanics: exact and near-duplicate (embedding-based) deduplication before the dataset is frozen, and an explicit record of any synthetic/distilled examples' teacher-model source.
- Default to **LoRA/QLoRA** over full fine-tuning whenever a fine-tuning decision is made at all — this is the frugality-aligned default the same way book 1 defaulted to prompting over fine-tuning in the first place: prompting first, then LoRA/QLoRA fine-tuning of an open-weight model as the next-cheapest escalation, full fine-tuning only with an ADR justifying why parameter-efficient methods proved insufficient.
- Add three new deploy-blocking gates modeled directly on `ci-pipeline`'s existing Gate Sequence table, for any product surface with prompts or fine-tuned models: a **prompt regression gate** (re-run the golden/eval set against a changed prompt template; block merge on a scored regression, the same shape as the existing coverage gate), a **dataset version gate** (block a training run against an unversioned or unhashed dataset, the same shape as the existing freshness/lockfile-drift gate), and a **model promotion gate** (a fine-tuned checkpoint is registered and evaluated before it can be referenced by an inference pipeline — promoted by an immutable version/hash, never a mutable "latest" model alias, mirroring `ci-pipeline`'s own "promotion by tag" anti-pattern almost verbatim).
- Require every model-serving decision to name its **quantization format** (GGUF/AWQ/GPTQ or none) and its **batching strategy** (static, dynamic/continuous) explicitly, with the accuracy/throughput trade-off stated — never leave "we serve the model" as an unstated implementation detail the way `extractUnit` currently leaves its extraction technique unstated (book 1's Actionable Techniques already flagged this exact anti-pattern for classification; this is its serving-layer counterpart).
- Add an explicit **Tech Stack Defaults row (or an adjoining exception note) in `CLAUDE.md`** naming Python as the sanctioned language for ML/LLM training, fine-tuning, and evaluation-tooling workloads — promoting `data-pipeline-implementation`'s one skill-level "Python Exception" section into the same always-on, session-start-read standards document that governs every other tech-stack default, so the exception is discoverable without having read one specific data-phase skill.
- Before self-hosting any open-weight model for inference, require a stated choice between a metered third-party API and self-hosted serving, evaluated against this repo's frugality principle exactly as book 1's fine-tuning caveat already models: self-hosting wins on marginal per-call cost at volume but requires the GPU-scheduling/quantization/batching discipline this book supplies to actually realize that saving; an unmanaged self-hosted deployment can easily cost more than the API it was meant to replace.

## Key Terminology

Worth considering for `skills/glossary-management/`'s Ubiquitous Language if any of the candidates above are built. None of these duplicate book 1's terms (`Foundation Model`, `Prompt Engineering`, `RAG`, `Golden Dataset`, `Hallucination`, `Model/Prompt Version`, `Guardrails`) or book 2's (`Data Drift`, `Concept Drift`, `Model Staleness`, `Train/Serve Skew`, `Natural Label`, `Class Imbalance`, `Shadow Deployment`, `Champion/Challenger`, `Training-Data Provenance`):

- **LLMOps** — the discipline this whole book names: applying MLOps/DevOps engineering rigor (versioning, CI/CD, registries, monitoring) specifically to LLM-based systems, as distinct from generic "AI engineering" (book 1's broader adapt-a-foundation-model scope) or generic "MLOps" (book 2's classical-model scope). The umbrella term this file's contribution sits under.
- **FTI Pipeline Architecture (Feature / Training / Inference)** — the book's three-pipeline decomposition: a feature pipeline (data collection/generation into a versioned store), a training pipeline (reads versioned data, produces a versioned model), and an inference pipeline (reads a registry-approved model to serve). Structurally parallel to, but a distinct payload from, `data-pipeline-design`'s existing choreographed document-processing topology.
- **LoRA (Low-Rank Adaptation)** — a parameter-efficient fine-tuning technique that freezes a base model's weights and trains only small, added low-rank adapter matrices, drastically cutting the compute/memory/storage cost of fine-tuning versus updating every weight.
- **QLoRA** — LoRA fine-tuning combined with 4-bit quantization of the frozen base model, enabling fine-tuning of large models on a single consumer-grade GPU — the book's specific answer to "how does fine-tuning stay frugal."
- **Instruction Dataset** — a collected or synthetically generated (distilled from a stronger "teacher" model) set of instruction/response pairs used to fine-tune a model for instruction-following, versioned and deduplicated as its own artifact — the LLM-specific instance of `ml-training-pipeline-design`'s general training-data-provenance concept.
- **Model Registry** — a system of record for trained/fine-tuned model artifacts: which version, from which data and code, with what evaluation scores, and its current serving status — named in book 2's Core Concepts but not carried into that file's Key Terminology list; included here because this book's CI/CD chapters make the registry a deploy-time *gate*, not just a bookkeeping system.
- **Prompt/Dataset/Model Versioning as CI/CD Artifacts** — this book's specific framing that a prompt template, a training dataset, and a model checkpoint are each independently versioned pipeline artifacts requiring their own promotion gate — the direct LLM analog of `ci-pipeline`'s existing "promotion by immutable digest, never by mutable tag" discipline for container images.
- **Quantization (GGUF / AWQ / GPTQ)** — post-training or serving-time weight compression formats that trade a small, measured accuracy loss for large memory-footprint and throughput gains — the mechanism that makes self-hosted serving of a large model financially and operationally tractable.
- **Continuous (Dynamic) Batching** — a serving-time technique that groups concurrent inference requests to maximize GPU utilization without forcing every request in a batch to wait for the slowest one to finish — the LLM-serving-specific throughput lever underneath book 1's more general "batching" mention in its inference-cost chapter.
- **KV Cache** — the per-request memory the transformer's attention mechanism must retain across generated tokens; the dominant GPU-memory cost of serving an LLM and the primary constraint on how many concurrent requests one GPU can serve — a mechanical detail neither prior book names at this level.

## Caveats

- **This book assumes you are fine-tuning an open-weight model at all — same conditional caveat both prior books already state, now for a third technique.** Book 1 defaults toward prompting over fine-tuning; book 2 conditions its entire training-lifecycle content on "if a trained classifier is chosen instead of a prompted one." This book's LoRA/QLoRA/registry/CI-CD content is conditional on that same upstream decision (book 1's proposed `llm-classification-design`) landing on "fine-tune" for some task. Treat this file's content as the LLMOps discipline required *if and when* that happens — LoRA/QLoRA make fine-tuning cheaper, they do not make it free, and per CLAUDE.md's frugality principle, prompting remains the default to beat.
- **The CI/CD gap this file identifies is real and specific — it is not a claim that `ci-pipeline`/`cd-pipeline` are poorly designed.** Both files, read in full, are excellent, coherent, real working pipelines for what they were built to cover (Go/React code and container images). The gap is that neither was ever asked to cover a second artifact class (prompts, datasets, models) because this repo has not yet built anything that produces one — this is a forward-looking gap, not a defect in existing work.
- **The CLAUDE.md Tech Stack Defaults gap is the most consequential finding in this file and should be raised as a direct proposal, not silently patched.** Per this repo's own working discipline (`sdlc-context.json` open questions, chunk-by-chunk approval), a change to CLAUDE.md's always-on standards is exactly the kind of decision that needs Shafi's explicit "go," not something to fold quietly into a skill's prose. This file surfaces the gap; it does not resolve it.
- **Do not conflate this file's `llm-cicd-pipeline-design` candidate with `research/software-architecture/fundamentals-of-software-architecture-richards-ford.md`'s "fitness function" concept, even though they rhyme.** That book's fitness functions are automated checks that an *architecture characteristic* still holds (e.g., a lint rule enforcing layering). This book's prompt/dataset/model gates check that a specific *AI artifact* meets an evaluation bar before promotion — a narrower, artifact-specific instance of the same "automated, continuous verification" spirit, worth citing as a precedent if `llm-cicd-pipeline-design` is ever built, but not the same mechanism.
- **As with both prior books: do not confuse this with the `research/ai-agent-engineering/` cluster.** That cluster is about how this factory's own Claude Code agents/skills are engineered; this book, like the two before it in this cluster, is about the *product's* own use of LLMs — here specifically the engineering scaffolding (data pipelines, fine-tuning, CI/CD, serving) around that use, not the product's classification/extraction logic itself (book 1's territory) or its classical-ML monitoring lifecycle (book 2's territory). Different layer, never cross-cited as answering the same question.
- **The book's worked examples assume a team with dedicated ML infrastructure (GPU clusters, experiment-tracking SaaS) this repo's solo-operator, frugality-first context does not have.** Its CI/CD and serving chapters need translating down to this repo's actual scale: a GitHub-Actions-hosted eval gate rather than a dedicated ML platform, and a metered API as the honest default over self-hosted GPU serving until volume justifies otherwise — exactly the translation both prior books' caveats already modeled for their own content.
