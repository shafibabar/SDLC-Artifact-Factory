# Build a Large Language Model (From Scratch)
**Authors:** Sebastian Raschka, PhD
**Domain:** AI/ML Engineering (mechanistic fundamentals — tokenization, attention, transformer architecture, pretraining, and fine-tuning, implemented from first principles in PyTorch)

## Core Concepts

- **This is the mechanistic layer underneath all three prior books in this cluster — not application engineering, not the classical-ML lifecycle, not LLMOps scaffolding, but the actual architecture and math inside the model those three books treat as a callable black box.** `ai-engineering-huyen.md` covers adapting a pre-trained foundation model (prompting/RAG/fine-tuning as a decision); `designing-ml-systems-huyen.md` covers the classical trained-model lifecycle (data, features, monitoring); `llm-engineering-handbook-iusztin-labonne.md` covers the LLMOps engineering scaffolding (FTI pipelines, LoRA/QLoRA, CI/CD for prompts/datasets/models). This book builds a GPT-style model's tokenizer, embeddings, self-attention, transformer blocks, and pretraining loop from scratch, then fine-tunes it — one layer beneath everything the other three books assume already exists and works. Training a foundation model from scratch is an enormous compute/data undertaking, explicitly out of scope for this repo's frugal, solo-operator context (see Caveats) — the honest yield here is narrower and more specific than the prior three books', and this file says so plainly rather than force-fitting a skill-sized proposal to match their volume.
- **Tokenization (Byte Pair Encoding) turns text into subword units — and this determines what an entity extractor can even "see" as one thing.** BPE starts from individual bytes/characters and iteratively merges the most frequent adjacent pairs into subword tokens, so common words become single tokens while rare or structured strings (part of a SSN like `078-05-1120`, or an email's local-part) are split across several subword pieces. This has a direct, concrete consequence for `data-classification`'s and `data-pipeline-implementation`'s entity extraction: if the underlying technique is a transformer-based tagger, a detected entity is not scored as one atomic unit in one shot — it is a span across multiple tokens, each carrying its own output distribution. Reconstructing the single confidence number `data-classification`'s worked example reports per entity ("1 × SSN ... confidence 0.41", SKILL.md lines 130-135) from several token-level scores requires an aggregation policy (minimum across the span, geometric mean, per-token product) that is a real design decision — currently unstated anywhere in either skill.
- **Self-attention is the mechanism, not just the buzzword, behind why a model can make the semantic judgments `ai-engineering-huyen.md` already distinguished from cheap regex detection.** Self-attention computes, for every token, a weighted combination of every other token in the context window, with weights learned to reflect relevance — this is what lets a transformer integrate meaning across an entire paragraph or document (e.g., a mention of "diagnosis" three sentences away) when deciding something like "is this paragraph about health information," rather than matching only local patterns the way a regex or bag-of-words check would. `ai-engineering-huyen.md`'s Ch. 5 distinction (rule-based detection for deterministic patterns like EMAIL/SSN format; model-based detection for semantic judgments) asserted this split without explaining *why* a model succeeds where regex fails. This book supplies that mechanism — genuinely new information relative to the first three books in this cluster, not a restatement of any of them.
- **Confidence scores are softmax outputs over logits — a relative ranking within one forward pass, not a calibrated probability of correctness — and this book's architecture chapters make that distinction unavoidable.** The book is explicit, end to end: the model's final layer produces raw, unbounded logits (over the vocabulary for next-token prediction, or over class labels once a classification head replaces that layer), and softmax normalizes them to sum to 1 — a mathematical convenience for producing a distribution, not evidence that the resulting number reflects how often that prediction is actually correct. A model can be confidently wrong: `softmax(logits) = [0.97, 0.02, 0.01]` says the top class dominates the model's own internal scoring for that one input, not that the model is right 97% of the time it says this. `data-quality-rules`' entire confidence-threshold-and-routing design (SKILL.md lines 62-97, thresholds of 0.85 and 0.90) treats these scores as if they already carry that probabilistic meaning — comparing a `0.62` SSN score against a `0.90` threshold implicitly assumes "0.62 means roughly 62% likely to be a real SSN," which softmax alone never guarantees. The book's decoding-strategies material (temperature scaling to control the randomness of generated text) is the same mathematical lever used, differently applied, in the standard fix for this exact problem: **calibration** (temperature scaling, or Platt/isotonic scaling, fit against a held-out *labeled* set) — a step that must happen after training or prompting and before a raw score is ever compared to a business threshold.
- **Fine-tuning a small, task-specific classifier by attaching a classification head to a small pretrained transformer — the book's own worked chapter (SMS spam/ham classification via a small pretrained GPT-2) — is a real, frugal, and directly transferable alternative to an ongoing paid classification API.** The book demonstrates loading a small (124M-parameter) open-weight pretrained model, replacing its output projection with a small classification head, and fine-tuning only the last transformer block plus the new head on a modest labeled dataset (a few thousand examples) — cheap enough for a single consumer GPU, or patiently on CPU, with no distributed training cluster and no paid third-party API dependency once trained. This is structurally the same shape as this repo's actual, live need: classifying a document's sensitivity, or an extracted string's entity type, in-house at effectively zero marginal per-document cost, instead of paying per-call for a hosted classification API — squarely inside CLAUDE.md's Budget and Frugality section ("open-source over paid tooling," "no external paid APIs added without explicit approval").
- **But this technique is training, and training carries exactly the governance obligations `designing-ml-systems-huyen.md` already assigned to any trained classifier — this book adds mechanics to an already-open question, it does not open a new one.** A fine-tuned classification head is precisely the "trained classifier" case book 2's proposed `ml-training-pipeline-design` skill scoped: it needs a training-data provenance record, explicit correction for the same rare-but-highest-consequence entity classes `data-classification` already names (`SSN`, `HEALTH_TERM`, `NATIONAL_ID`), and a model registry entry per book 2's and book 3's already-proposed candidates. This book's contribution is *how* the fine-tuning mechanically happens (freeze most of the network, train a small head over a handful of epochs on a small labeled set) — the newest, most concrete instance of a slot both prior books already opened, not a reason to open a competing third slot.
- **Instruction fine-tuning (the book's closing chapters) is a different technique aimed at a different problem this repo does not currently have.** The book's instruction-tuning material adapts a pretrained model to follow natural-language instructions and produce open-ended text (the mechanism behind "chat" models) — the generative counterpart to book 3's LoRA/QLoRA content, not the classification-head technique above. This repo's stated, actual need (classify a document, extract an entity) is a narrow, closed-set classification problem, not open-ended generation — so this is the least transferable part of the book to this repo's near-term roadmap. It becomes relevant only if a future feature needs a document-estate chat/Q&A assistant (book 1's speculative `rag-pipeline-design` territory), not to the classification/extraction work already planned.
- **Tokenizer context-window limits interact badly with this repo's existing checkpoint-granularity design, if a transformer-based extractor is ever chosen.** `data-pipeline-implementation`'s checkpointing is built around document-native units — `pdfPages`, `docxParagraphs`, `xlsxSheets` (SKILL.md lines 293-298) — each processed and checkpointed independently, by design, for resumability. A transformer has a fixed context window measured in tokens, not pages or paragraphs; an entity straddling a page break or a paragraph boundary (a SSN split across a page, a sentence continuing across a checkpointed unit) could be systematically invisible to a per-unit transformer call, since each unit is tokenized and processed with no access to an adjacent unit's tokens. This is a concrete, mechanically-grounded gap in the existing checkpoint design that none of the three prior books in this cluster surfaced, because none of them reason at the token/context-window level.

## Actionable Techniques

- Wherever a "confidence score" threshold appears (`data-quality-rules`' 0.85/0.90 table, `data-classification`'s worked example), state explicitly whether the score is calibrated (temperature or Platt/isotonic scaling fit against a held-out labeled set) or is a raw, uncalibrated softmax/logit-derived ranking — and if uncalibrated, document that the threshold is empirically tuned against steward-corrected outcomes rather than treated as a probability of correctness.
- Add an explicit entity-span confidence aggregation policy wherever an entity-level confidence number appears (`data-classification`'s and `data-quality-rules`' worked examples): name whether a multi-token entity's reported confidence is the minimum, geometric mean, or per-token product across its constituent tokens — as unstated today as `extractUnit`'s underlying technique itself (`ai-engineering-huyen.md`'s Actionable Techniques already flagged the latter; this is the former's necessary companion the moment the technique is a token-level model).
- If `extractUnit` (`data-pipeline-implementation`, line 164) is ever implemented as a transformer, evaluate the checkpointed unit granularity (page/paragraph/sheet) against that model's actual context window, and add a concrete test for an entity spanning a unit boundary — either overlap adjacent checkpointed units by a token margin, or confirm the chosen model's context window comfortably exceeds the largest realistic unit, before trusting that no entity is silently lost at a boundary.
- Before fine-tuning any classification head on a small open-weight model, treat it exactly as `designing-ml-systems-huyen.md`'s proposed `ml-training-pipeline-design` skill already requires: a training-data provenance record, an explicit class-imbalance correction for the same rare, high-consequence entity types `data-classification` already names, and a model registry entry — the fine-tuning mechanics being cheap (per this book) is not a reason to skip that governance.
- Do not conflate this book's "embedding layer" (the token-to-vector mapping trained jointly as the first stage of the transformer architecture) with a RAG system's "embedding model" (a separate model producing document/query vectors for nearest-neighbor retrieval, book 1's speculative `rag-pipeline-design` territory) — if that skill is ever built, state this distinction explicitly in its glossary entry so the two senses of "embedding" are never used interchangeably.
- When (and only when) fine-tuning a classification head is chosen over prompting a hosted foundation model for a specific classification task, record the decision as the ADR-gated escalation `ai-engineering-huyen.md`'s Actionable Techniques already require ("before choosing fine-tuning ... require an ADR showing prompting was tried and found insufficient") — this book does not relax that gate; it only supplies the mechanics for the cheapest version of the escalation.
- If any future work touches this book's territory (fine-tuning code, an evaluation harness, PyTorch training scripts), flag the same `CLAUDE.md` Tech Stack Defaults gap `llm-engineering-handbook-iusztin-labonne.md` already identified — no row or exception paragraph anywhere in the ten-row Tech Stack Defaults table sanctions Python or any ML-training toolchain — rather than resolving it silently. This book reinforces, a fourth time across this cluster, that the gap is real and general, not specific to LLMOps CI/CD.

## Key Terminology

Worth considering for `skills/glossary-management/`'s Ubiquitous Language only if the fold-in above (or a related skill) is actually built. None of these duplicate the three prior files' terms (`Foundation Model`, `Prompt Engineering`, `RAG`, `Golden Dataset`, `Hallucination`, `Model/Prompt Version`, `Guardrails`; `Data Drift`, `Concept Drift`, `Model Staleness`, `Train/Serve Skew`, `Natural Label`, `Class Imbalance`, `Shadow Deployment`, `Champion/Challenger`, `Training-Data Provenance`; `LLMOps`, `FTI Pipeline Architecture`, `LoRA`, `QLoRA`, `Instruction Dataset`, `Model Registry`, `Prompt/Dataset/Model Versioning as CI/CD Artifacts`, `Quantization`, `Continuous (Dynamic) Batching`, `KV Cache`):

- **Byte Pair Encoding (BPE)** — the subword tokenization algorithm that iteratively merges the most frequent adjacent character/byte pairs into a fixed vocabulary, splitting rare or structured strings (part of a SSN, an email's local-part) across multiple tokens rather than treating them as one unit. The mechanical reason an "entity" detected by a token-level model is a multi-token span, not an atomic scored unit.
- **Self-Attention / Multi-Head Attention** — the mechanism by which a transformer computes, for every token, a learned weighted combination of every other token in its context window, letting the model integrate meaning across an entire input rather than matching only local patterns. The concrete "why" beneath `ai-engineering-huyen.md`'s assertion that semantic judgments need a model where regex does not.
- **Logit** — the raw, unbounded output of a model's final linear layer before any normalization — the actual number the model computes per class/token, as distinct from the score any downstream consumer sees after softmax.
- **Softmax** — the function that normalizes a vector of logits into a probability-*shaped* distribution (non-negative, summing to 1) — a mathematical convenience for producing a distribution, not a guarantee that the resulting numbers reflect real-world correctness frequency. The precise term for what `data-classification`'s and `data-quality-rules`' "confidence score" actually is, absent a stated calibration step.
- **Calibration (Temperature Scaling)** — a post-hoc adjustment (dividing logits by a learned temperature before softmax, or an equivalent Platt/isotonic fit) applied against a held-out labeled set so that a model's output scores approximate true correctness probability — the missing step between "the model produced a softmax score" and "this score can be safely compared to a business threshold like 0.85."
- **Context Window** — the fixed maximum number of tokens a transformer can attend over in a single forward pass. The limiting factor `data-pipeline-implementation`'s page/paragraph/sheet checkpoint granularity must be checked against if the extraction technique is ever a transformer, to avoid silently losing entities that straddle a checkpointed-unit boundary.
- **Classification Head** — a small, newly-initialized output layer (replacing a pretrained model's original next-token-prediction head) trained to project the model's internal representation onto a fixed set of class labels — the specific, cheap fine-tuning technique this book's worked spam-classifier chapter demonstrates, and the LLM-specific third worked example `ml-training-pipeline-design` should carry if it is ever built.

## Caveats

- **Training a foundation model from scratch is out of scope for this repo, plainly and without qualification.** This book's central exercise — implementing tokenization, embeddings, attention, a full transformer, and a pretraining loop, then pretraining a GPT-style model on a text corpus — is an educational, from-first-principles exercise requiring compute, data, and time this repo's frugal, solo-operator context (`CLAUDE.md`'s Budget and Frugality section) cannot and should not fund. Nothing in this file should be read as a step toward pretraining anything; the book's *pretraining* chapters are read here for mechanism (how tokenization/attention/softmax work), never as a recommendation to build one.
- **The one genuinely actionable idea — fine-tuning a small classification head — is still training, and still needs the same ADR-gated escalation and governance discipline the first two books in this cluster already established.** Cheap fine-tuning mechanics (per this book) do not exempt the decision from `ai-engineering-huyen.md`'s "prompting first, fine-tuning only with an ADR" discipline, or from `designing-ml-systems-huyen.md`'s training-data-provenance/class-imbalance/model-registry requirements for any trained classifier. Treat this book's contribution as *how cheap the escalation can be*, not as a reason to skip evaluating whether prompting a hosted model already meets the bar.
- **The confidence-score/softmax distinction is a real, load-bearing finding, but it is a documentation and design fix to two existing skills, not a new skill.** `data-quality-rules`' and `data-classification`'s confidence thresholds are not wrong as a design shape (per-entity-type risk-scaled thresholds, quarantine routing) — the gap is that nothing states whether the numbers being thresholded have been calibrated to mean what "confidence" implies. Fix this by adding a calibration-status field to each threshold table, not by building new tooling.
- **Do not conflate this book's "embedding" with a RAG system's "embedding."** The token embedding layer this book implements (a lookup table mapping token IDs to dense vectors, trained jointly with the rest of the model) answers a completely different question from a RAG embedding model (a separate model producing semantically comparable vectors for retrieval, book 1's speculative `rag-pipeline-design` territory). This repo has no vocabulary distinguishing the two senses yet; if `rag-pipeline-design` is ever built, it must not inherit this book's embedding-layer mechanics as if they were retrieval-embedding guidance.
- **Even "small" fine-tuning still needs a toolchain this repo has not sanctioned anywhere.** `llm-engineering-handbook-iusztin-labonne.md` already found that `CLAUDE.md`'s ten-row Tech Stack Defaults table names no Python/ML-training exception at all. This book's cheaper fine-tuning technique (a classification head on a 124M-parameter model) still requires PyTorch and, practically, at least a modest GPU to be tractable — smaller than a full LLM fine-tune, but still a real toolchain and hardware decision this repo's standing defaults do not currently acknowledge. This file reinforces that finding; it does not resolve it, and any resolution belongs in a direct proposal to Shafi per that book's own caveat, not a silent addition to a skill.
- **As with all three prior books in this cluster: do not confuse this with the `research/ai-agent-engineering/` cluster.** That cluster is about how this factory's own Claude Code agents/skills are engineered; this book, like the three before it in `ai-ml-engineering/`, is about the *product's* own use of ML/LLM mechanics to classify and extract from customer documents. Different layer, never cross-cited as answering the same question.
- **This book, unlike the other three, genuinely is the odd one out in this cluster — an educational/mechanistic text, not an applied engineering handbook.** Its practical yield for this repo is real but narrow: two concrete corrections to existing skills (calibration, span-aggregation), one context-window caution for pipeline design, and one confirmed non-candidate (fold fine-tuning-a-classifier into the already-proposed `ml-training-pipeline-design` rather than forking it). Treat it as source material that sharpens two existing skills, not as the basis for new domain expertise this repo needs to build.
