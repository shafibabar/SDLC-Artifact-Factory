# AI Engineering
**Authors:** Chip Huyen
**Domain:** AI/ML Engineering (building applications on top of foundation models)

## Core Concepts

- **AI engineering is building *with* foundation models, not training them (Ch. 1).** Huyen's central distinction: classical ML engineering trains models from labeled data for a narrow task; AI engineering composes, prompts, retrieves for, evaluates, and guards a pre-trained foundation model (an LLM or multimodal model) to perform a task it was never explicitly trained on. This repo has zero skills of either kind — but its actual product need (`data-classification`'s sensitivity assignment, `data-pipeline-implementation`'s `extractEntities`) is squarely an AI-engineering problem: classifying a never-before-seen document's sensitivity and pulling structured entities out of unstructured PDF/DOCX/XLSX content are exactly the "adapt a foundation model to a task via prompting/fine-tuning" problems this book addresses, not problems solvable by a hand-written rule engine alone.
- **Prompt engineering vs. fine-tuning vs. RAG — a decision, not a default (Ch. 5, 6, 8).** The book frames these as three points on a cost/effort/data-requirement spectrum for adapting a foundation model, and insists the choice be justified per task: prompting (cheapest, no training data needed, most flexible, weakest guarantees), RAG (adds retrieval to ground the model in your own data without retraining), and fine-tuning (most expensive, needs labeled data, best for narrow high-volume tasks with stable requirements). Nothing in `skills/data-classification/SKILL.md` or `skills/data-pipeline-implementation/SKILL.md` names any of these three — "automated detection" and `extractUnit(ctx, unit)` are stated as facts, not decisions with a stated technique or a rationale for that technique.
- **Structured output and function calling (Ch. 6, "Structured Outputs").** Modern foundation-model APIs can be constrained to emit a validated schema (JSON matching a defined type) rather than free text, which is what makes an LLM usable as a component in a deterministic pipeline instead of a text generator whose output must be re-parsed and hoped-for. This is the mechanism that would make an LLM-based entity extractor slot cleanly into `data-pipeline-implementation`'s `Result{Entities: entities}` return shape — but the skill's worked example never mentions a schema-constrained call, prompt, or model at all; `extractUnit` is a black box that "finds" entities by unstated means.
- **Evaluation is the hardest and most neglected part of AI engineering (Ch. 3-4, the book's most emphasized theme).** Unlike traditional software, an AI system's output is often non-deterministic and has no single ground truth — "correct" is graded, not asserted. The book insists every AI system needs: (a) a golden/labeled evaluation dataset representative of production inputs, (b) an evaluation method appropriate to the task (exact-match, human rating, LLM-as-judge, or task-specific metrics), (c) evaluation run continuously against production traffic (not just at model-selection time), and (d) evaluation criteria that reflect the actual cost of different error types — a false positive and a false negative are rarely equally bad. `skills/data-quality-rules/SKILL.md` already has real evaluation-adjacent machinery (per-entity-type confidence thresholds, a "model-accuracy proxy" metric from steward override rates, "confirm/correct/reject" resolution tracking) — but it never names a golden dataset, an evaluation harness, or a systematic accuracy-measurement methodology; the confidence score itself is treated as an already-given fact ("Automated extraction... produces a confidence score, not certainty"), never as the output of a specific, chosen, and periodically re-validated model or prompt.
- **Asymmetric error cost is a first-class evaluation design input (Ch. 4, "Evaluation Criteria").** The book insists evaluation criteria be tied to the actual business cost of each error type, not a single blended accuracy number — a false negative in a fraud-detection model and a false positive cost the business very differently, and the eval methodology (thresholds, precision/recall targets, human-review triggers) should be set accordingly. This is exactly the shape of `data-classification`'s core compliance risk: classifying a Restricted document as Public (false negative on sensitivity) is a compliance failure with legal exposure, while classifying a Public document as Confidential (false positive) is only a productivity cost (unnecessary access friction). `data-classification`'s confidence-threshold-plus-review design already behaves as if it knows this asymmetry (escalation flags for review, de-escalation always audited) — but the skill never states the asymmetry as an explicit evaluation design principle that should drive where thresholds are set per entity type, leaving `data-quality-rules`' "confidence threshold ... set per entity-type by the risk of getting it wrong" (line 73) as the only place this reasoning surfaces, informally.
- **RAG (Retrieval-Augmented Generation) grounds a model in your own data at query time (Ch. 8).** RAG retrieves relevant documents/chunks (typically via vector/semantic search over embeddings) and injects them into a model's context so it can answer questions grounded in specific, current data it was never trained on — the standard architecture for "chat with your documents" or "ask a question over your data estate" features. This repo's `data-model-design` and `data-retention-policy` mention Elasticsearch only as a full-text/keyword search index (`data-model-design` line 160: "Full-text search, ranking, alert storage") — there is no vector/embedding store, no chunking strategy, and no retrieval-then-generate pattern anywhere in the repo. If a future feature ever lets a user ask a natural-language question over their data estate (a highly plausible feature for a data-estate-mapping product), RAG is the standard architecture for it and this repo currently has zero vocabulary or design skill for it.
- **AI system architecture: orchestration, guardrails, and caching around the model call (Ch. 10).** A production AI system is never a bare model call — it is a pipeline: input guardrails (validate/sanitize input, detect prompt injection), the model call itself (possibly multiple, chained, or routed by task complexity/cost), output guardrails (validate structure, check for policy violations, detect hallucination signals), and caching (exact-match or semantic caching to cut redundant inference cost). `data-pipeline-implementation`'s worker architecture (idempotent processing, checkpointing, DLQ, backpressure) is a strong, correct pattern for the *plumbing* around any single unit of work — but if that unit of work becomes an LLM call, none of the book's model-specific guardrail concerns (adversarial/injected content in a crawled Google Drive file being fed to a model as "data" that the model might interpret as an instruction; validating the model's structured output before trusting it downstream) appear anywhere in the skill or in `security-architect`'s threat-modeling scope as read.
- **Inference cost and latency are architecture-shaping constraints, not afterthoughts (Ch. 9, "Inference Optimization").** The book treats cost-per-inference-call and latency as first-order design inputs: batching, prompt caching, model tiering (a cheap/fast model for easy cases, an expensive/slow one only when needed), and choosing self-hosted open-weight models vs. paid APIs are all evaluated against a frugality and throughput budget. `data-pipeline-implementation`'s backpressure section already reasons carefully about per-unit CPU/memory cost for extraction concurrency (`w.pool.SetLimit(cfg.ExtractionConcurrency)`, sized "much lower than a typical lightweight consumer") — but if the underlying `extractUnit` call is (or becomes) an LLM API call, the true bottleneck and cost driver is per-call inference cost and provider rate limits, not local CPU/memory — a completely different backpressure/concurrency-sizing problem the skill has no vocabulary for yet.
- **Fine-tuning is a narrow, data-hungry, high-maintenance option — not a "better" default (Ch. 7).** The book is explicit that fine-tuning should be reached for only when prompting genuinely cannot meet accuracy/latency/cost targets for a narrow, stable, high-volume task, because it requires curated labeled data, retraining discipline as the task drifts, and its own versioning and evaluation overhead. Given this repo's CLAUDE.md frugality principle ("open-source over paid tooling," "prefer simpler solutions... when outcomes are equivalent"), this maps cleanly onto a repo-native default: prefer prompting a foundation model (or, where the pattern is truly deterministic — email/SSN/IBAN regex — prefer rule-based detection over any model at all) and only escalate to fine-tuning with an explicit ADR, exactly the same escalation discipline the repo already applies to the Go→Python exception in `data-pipeline-implementation`.

## Actionable Techniques

- For every place "the model classifies" or "the extractor finds" currently appears as an opaque verb (`data-classification`'s "Automated detection," `data-pipeline-implementation`'s `extractUnit`), require a named technique with a stated rationale: regex/rule-based (for genuinely deterministic patterns like EMAIL/SSN format), a fine-tuned classifier, or an LLM prompt with structured/schema-constrained output — never left unstated.
- For every automated classification or extraction decision, require a golden evaluation dataset (a labeled, representative sample of real document types/sensitivity levels) and a stated evaluation methodology (precision/recall per entity type, not one blended accuracy number) before the technique ships — this converts `data-quality-rules`' "model-accuracy proxy" metric from an informal side-effect of steward corrections into a designed, first-class evaluation artifact.
- Make error-cost asymmetry an explicit, named input to every confidence threshold: state the cost of a false negative (Restricted classified as lower) vs. a false positive (over-classification) per entity type, and set the threshold from that stated cost — not from a single "accuracy" target.
- Treat any content ingested from an external, user-controlled source (a Google Drive file, an S3 object) that is fed to a model as untrusted input requiring input guardrails — specifically, prompt-injection resistance (the document's text should never be interpretable as an instruction to the extraction/classification model) — and add this as a named threat-modeling entry, not an assumption that crawled content is safe because it's "just data."
- Require schema-constrained/structured output (a defined JSON schema, validated before the result is trusted) for any model-based classification or extraction call — never free-text output re-parsed with best-effort string matching — so the model's output slots into the pipeline's existing typed `Result{Entities: entities}` contract exactly as a rule-based extractor's would.
- Before choosing fine-tuning for any classification/extraction task, require an ADR showing prompting was tried and found insufficient (accuracy, latency, or cost) — mirroring the repo's existing Go-first/Python-exception discipline, so "train our own model" doesn't become a default reached for out of familiarity rather than necessity.
- If/when a natural-language search or Q&A feature over the data estate is proposed, require a RAG design pass (chunking strategy, embedding/vector store choice, retrieval evaluation) before implementation — distinct from the existing Elasticsearch full-text search design, which answers a different question (keyword match, not semantic relevance).
- Add prompt/model version to every classification decision's audit trail (alongside the existing override reason and who/when) — a compliance-critical AI decision must be traceable to *which* model/prompt version produced it, not just *when* and *by what confidence score*, so a later model upgrade's effect on historical classifications can be audited.
- Add a caching strategy (exact-match on file hash, at minimum) to any model-based pipeline stage before it ships, to control per-document inference cost at estate scale — re-classifying an unchanged file on every rescan is a frugality defect once inference has a per-call cost.

## Candidate New Skills/Agents

Five candidates, reflecting that "classify this document" and "extract these entities" are genuinely AI-engineering problems this repo has never named, plus the ownership gap confirmed above:

1. **`llm-classification-design` (skill, owned by data-architect or the new agent below).** Teaches the technique-selection decision (rule-based vs. fine-tuned classifier vs. LLM prompt) per entity/category, with a decision table analogous to `nfr-specification`'s selection tables — cost, latency, data-requirement, and accuracy-ceiling trade-offs per technique, plus the asymmetric-error-cost framing that should set confidence thresholds. This directly fills `data-classification`'s "Automated detection" black box.
2. **`ai-system-evaluation` (skill).** Golden/labeled dataset design per classification/extraction task, evaluation methodology (precision/recall per entity type, not blended accuracy), human-in-the-loop review workflow design for compliance-critical low-confidence decisions, and drift monitoring (re-run eval on a schedule, alert on accuracy regression). This is the most consequential candidate: it converts `data-quality-rules`' informal "model-accuracy proxy" metric into a designed governance artifact, and gives `compliance-design` a control category it currently lacks entirely.
3. **`prompt-engineering-and-versioning` (skill).** Prompt template structure, schema-constrained/structured output design, prompt version tracking tied to the audit trail of every classification decision (so a model/prompt upgrade's effect on historical decisions is auditable — directly extending `data-classification`'s existing audit discipline for manual overrides to cover automated ones too), and regression testing prompts against the golden dataset before a prompt change ships.
4. **`rag-pipeline-design` (skill, speculative — gate on an actual future feature).** Chunking strategy, embedding/vector store selection (distinct from the existing Elasticsearch full-text index), retrieval evaluation, and grounding/citation design for any future "ask a question over your data estate" feature. Not urgent today (no such feature exists in the roster), but flagged because it is the standard architecture the moment natural-language search-over-the-estate is proposed, and the repo currently has zero vocabulary for it.
5. **New agent: `ai-systems-engineer` (or `ml-engineer`).** The ownership gap is real and structural, not cosmetic: `data-architect` owns the classification *scheme* (taxonomy, propagation, control mapping) and explicitly does not implement; `data-engineer` owns pipeline *plumbing* (idempotency, checkpointing, backpressure, DLQ) and explicitly does not set the scheme — neither agent's "Owns" table includes "the actual model/prompt that performs classification and extraction," and neither should absorb it without diluting their existing, coherent boundaries. A dedicated agent — owning `llm-classification-design`, `ai-system-evaluation`, `prompt-engineering-and-versioning`, and (if built) `rag-pipeline-design` — mirrors this repo's existing pattern of a narrow specialist agent for a technically deep cross-cutting concern (`security-architect` vs. `security-engineer` is the closest precedent: design authority vs. implementation, split the same way this new agent would split from `data-engineer`). Its outputs would hand off to `data-engineer` (the concrete model/prompt call slots into `extractUnit`, same as any other implementation dependency) and to `security-architect` (prompt-injection threat modeling, third-party-API data-residency review) exactly the way `data-architect`'s classification scheme already hands off to `security-architect` today.

## Key Terminology

Worth considering for `skills/glossary-management/`'s Ubiquitous Language if any of the candidates above are built:

- **Foundation Model** — a large, pre-trained model (LLM or multimodal) adapted to a task via prompting, RAG, or fine-tuning rather than trained from scratch for that task. Not used anywhere in the repo; needed the moment any classification/extraction technique is named explicitly instead of left as "the model."
- **Prompt Engineering** — the discipline of designing, testing, and versioning the input given to a foundation model to reliably produce a desired output — distinct from "writing a prompt" as a one-off act. Needed for `prompt-engineering-and-versioning` if built.
- **Retrieval-Augmented Generation (RAG)** — grounding a model's output in retrieved, current data (via embedding-based semantic search) at query time, as opposed to relying solely on the model's training data. Distinct from the "full-text search" already named in `data-model-design`'s Elasticsearch entry — the repo currently conflates or simply lacks vocabulary for the difference between keyword and semantic retrieval.
- **Golden Dataset / Evaluation Set** — a labeled, representative sample of real inputs (with known-correct outputs) used to measure an AI system's accuracy systematically and repeatably, as distinct from ad hoc spot-checks or steward-correction rates observed in production. The single most load-bearing missing term for `ai-system-evaluation`.
- **Hallucination** — a foundation model producing plausible-sounding but factually incorrect or fabricated output, not detectable by a schema check alone (the output can be perfectly well-formed and still wrong). Directly relevant to any LLM-based entity extraction: a well-formed but fabricated `PERSON_NAME` or `SSN` is a distinct failure mode from a low-confidence-but-honest miss, and needs its own detection strategy (e.g., requiring extracted values to be verifiably present as a substring of the source document).
- **Model/Prompt Version** — the specific, identified version of a model or prompt that produced a given output, tracked alongside (not instead of) the existing confidence score and audit fields, so a later upgrade's effect on historical decisions is traceable.
- **Guardrails** — input/output validation layered around a model call (input sanitization against prompt injection; output schema/policy validation before trusting the result) — distinct from `zero-trust-design`'s network/access guardrails, and currently absent from any threat-modeling or pipeline-implementation vocabulary in this repo.

## Caveats

- **The book assumes classical rule-based detection is sometimes still the right answer — do not read it as "replace regex with an LLM everywhere."** Huyen's own cost/latency chapter (Ch. 9) would favor deterministic, cheap, zero-latency regex for genuinely pattern-based signals (EMAIL, SSN format, IBAN) over any model call — `data-classification`'s existing detection table (EMAIL/PHONE/SSN/ACCOUNT_NUMBER) is very plausibly *already* correct as rule-based detection for these specific types. The real gap this book exposes is the *semantic* judgments the current skills leave equally opaque (is this paragraph's content health-related enough to be Special-category? does this whole document's tone and content make it Confidential vs. Restricted beyond its individual detected entities?) — that is where prompting/fine-tuning genuinely earns its cost, not as a blanket replacement for cheap deterministic checks.
- **Fine-tuning is the book's most expensive, least-default recommendation — align it with CLAUDE.md's frugality principle explicitly.** Given "open-source over paid tooling," "no external paid APIs without approval," and "prefer simpler solutions when outcomes are equivalent," any future `llm-classification-design` skill should default to prompting a foundation model (open-weight, self-hosted where budget allows, or a metered API with an explicit approval) and treat fine-tuning as an ADR-gated escalation — mirroring the repo's existing Go→Python exception discipline in `data-pipeline-implementation`, not a parallel, unrelated decision process.
- **Sending customer document content to a third-party foundation-model API is a data-residency and processing-agreement decision, not merely a "new technology" PIA checkbox.** This repo's product is physically multi-tenant with a SOC 2 target; the book (written for general AI-application builders) does not carry this repo's specific compliance stakes — its guardrails/architecture chapter provides the vocabulary (guardrails, self-hosted vs. API trade-offs) but the actual data-residency judgment call belongs to this repo's own `privacy-design`/`compliance-design`/`security-architect`, not to the book.
- **Do not confuse this cluster with the existing `research/ai-agent-engineering/` cluster.** That cluster (`context-engineering-for-ai-agents-zhu.md`, `mastering-claude-code-agent-skills-gale.md`) is about this factory's *own* agents and skills — the meta-level of how Claude Code agents are engineered to build the product. This book, and this new `ai-ml-engineering` cluster, are about the *product's own* use of AI/ML to classify and extract from customer documents — a completely different layer, and the two should never be merged or cross-cited as if they answered the same question.
- **The book is written for general AI-application builders and does not know this repo's DDD/event-driven conventions.** Its evaluation, RAG, and guardrail patterns need translating into this repo's existing vocabulary (Domain Events, the outbox pattern, Bounded Contexts) exactly as every other research file here has done for its source book — e.g., "model/prompt version" belongs in the audit trail the same way `DataAssetReclassified` already carries an override reason, not as a bolt-on logging concern.
