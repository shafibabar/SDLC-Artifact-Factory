# Clean Architecture
**Authors:** Robert C. Martin (Uncle Bob)
**Domain:** Software Architecture

## Core Concepts

- **The Dependency Rule (Ch. 22).** Source code dependencies must point only inward, toward higher-level policy. Nothing in an inner circle can know *anything* — not a name, not a signature, not a data format — about anything in an outer circle. This is the book's central, load-bearing idea and it is stricter than plain Dependency Inversion: DIP is a class-level SOLID technique (depend on an abstraction, not a concrete type) used *at a specific crossing point*; the Dependency Rule is the architecture-wide invariant that every crossing, in every direction, must obey — DIP is one tool the Dependency Rule uses when a lower-level detail must be called from a higher-level policy (an interface defined inward, implemented outward), but the Rule itself also governs things DIP alone doesn't reach, such as data formats never leaking outward-to-inward and control flow being allowed to run counter to source-dependency direction only via that same inversion.
- **The four concentric rings — Entities, Use Cases, Interface Adapters, Frameworks & Drivers (Ch. 20–23).** Entities hold enterprise-wide business rules and are the least likely to change for any reason external to the business itself. Use Cases hold application-specific business rules — they orchestrate the flow of data to and from Entities and direct those Entities to use their enterprise-wide rules to achieve a use case's goal. Interface Adapters convert data between the format most convenient for Use Cases/Entities and the format most convenient for an external agency (a database, the web) — controllers, presenters, and gateways all live here. Frameworks & Drivers is the outermost ring: the actual web framework, the actual database, glue code — "generally all detail." Each ring is a **level of policy**: the further in, the higher-level and more general the policy; the further out, the more mechanical and concrete the detail.
- **Screaming Architecture (Ch. 21).** A codebase's top-level structure should scream what the system *does* for its users, not what framework or delivery mechanism it uses. Uncle Bob's litmus test: could you look at the top-level directory names of a health-care system and tell it's a health-care system — or would you only be able to tell it's "a Rails app" / "an MVC app"? Framework, database, and web-delivery mechanism are details Screaming Architecture says should be deferrable, invisible, and never load-bearing in the top-level layout.
- **Boundary-crossing data and the Humble Object pattern (Ch. 22, Ch. 26).** When control flow crosses a boundary (say, from a Use Case into a database gateway), the data that crosses should be in the simplest form convenient for the inner circle — typically plain structs/DTOs — and never a framework or database type, and never an Entity object passed straight through untranslated. The book pairs this with the **Humble Object** pattern: split code that's hard to test (anything touching a GUI, a database, or another external boundary) from the logic that's easy to test, by extracting a thin "humble" wrapper that contains almost no logic (verified, if at all, by integration/acceptance tests) around the boundary, leaving all real decision-making in a plain object that's fully unit-testable in isolation. A repository implementation and a Presenter/View are the book's own canonical examples of humble objects.
- **Policy vs. detail — the database, the web, and frameworks are all details (Ch. 1, Ch. 30–31, "The Database Is a Detail," "The Web Is a Detail").** Business rules are the reason the software exists; everything else — which database engine, which web framework, which UI — is a deferrable, replaceable detail that should not shape the architecture. The book's blunt formulation: a good architect maximizes the number of decisions *not* made (framework choice, database choice) for as long as possible, because those are the decisions most likely to be revisited, and Clean Architecture is precisely the discipline that keeps them revisitable without touching business rules.
- **`main` as the ultimate, dirtiest detail (Ch. 28, "The Main Component").** The composition root is not a first-class architectural ring — it is a plugin *to* the architecture, sitting outside every ring, responsible only for instantiating and wiring dependencies (dependency injection) before handing control to the outermost ring it constructed. High-level policy has no knowledge that `main` exists at all.
- **Component cohesion and coupling principles (Ch. 12–14, briefly).** REP (Reuse/Release Equivalence), CCP (Common Closure), and CRP (Common Reuse) govern what belongs in the same deployable component; ADP (Acyclic Dependencies), SDP (Stable Dependencies), and SAP (Stable Abstractions) govern how components should depend on each other. These principles are aimed at multi-component/multi-module systems (large monoliths splitting into independently versioned libraries) and have limited direct bite in a one-Go-module-per-microservice repo where the "component" boundary is already the service boundary — noted here for completeness, not pursued as an actionable technique below.

## Actionable Techniques

- Whenever a skill or agent cites "Dependency Inversion Principle" for an architecture-wide, cross-layer inward-dependency rule, distinguish it from the Dependency Rule: DIP is the class/interface-level mechanism (Domain defines the port, Infrastructure implements it); the Dependency Rule is the broader invariant that governs every ring crossing, including ones DIP alone doesn't reach (e.g., data formats, naming, and the "outermost ring is the most detail-laden, least policy-laden" ordering itself).
- When naming this repo's four Go layers against the book's four rings, state the mapping explicitly rather than leaving it as parallel-but-uncited conventions: `domain/` = Entities, `application/` = Use Cases, `handlers/` + `infrastructure/` = Interface Adapters merged with Frameworks & Drivers (the repo does not separate these last two rings — see Caveats).
- Add a named "Fitness Function(s)" line or section wherever a diagram or skill asserts the Dependency Rule, pointing at the concrete, already-existing mechanical check (`go-makefile`'s `arch` target / `scripts/check-imports.sh`) rather than leaving the enforcement as prose in an anti-pattern table cell.
- For any future skill or agent describing a service's top-level file layout, state explicitly that Screaming Architecture is satisfied one level up — at the Bounded-Context/service-name boundary (`classification-service/`, decided at the Container Diagram) — not inside a single service's `internal/` tree, so the intentionally generic `cmd/`+`internal/{domain,application,infrastructure,handlers}` skeleton is not mistaken for a Screaming Architecture violation.
- Where a skill already separates a "thin, logic-free" boundary wrapper (a repository, a chi handler) from a fully unit-tested inner object (an Aggregate, a command handler), name that separation as the Humble Object pattern in the skill's prose so future readers recognize the pattern instead of re-deriving "why is this method so thin and only integration-tested" from first principles each time.
- Add "Entity" to the canonical glossary's Domain and DDD table, scoped precisely to the DDD sense already used throughout `aggregate-design` (an object with identity and mutable state inside an Aggregate), with an explicit disambiguation note against Clean Architecture's "Entities" ring (enterprise-wide business rules, a broader architectural concept) so the two senses of the same word never get silently conflated in a future Clean-Architecture-flavored skill or diagram.

## Key Terminology

Worth considering for `skills/glossary-management/`'s Ubiquitous Language:

- **Dependency Rule** — Clean Architecture's specific, named term for "source code dependencies point only inward, toward higher-level policy" — broader than Dependency Inversion Principle (a DIP crossing is one mechanism the Dependency Rule uses, not a synonym for the Rule itself). Not currently named anywhere in the repo; `go-project-structure` and `component-diagram` both practice it correctly but currently call it "the dependency rule" generically or conflate it with DIP.
- **Entity (Clean Architecture sense)** — the outermost ring: enterprise-wide business rules least likely to change for reasons outside the business itself. Distinct from the DDD sense of "Entity" (an object with identity inside an Aggregate) already used in `aggregate-design`. Neither sense is in the glossary today; if either is added, both must be present and cross-referenced to prevent drift.
- **Interface Adapters / Frameworks & Drivers** — the book's two outermost rings, kept conceptually distinct (adapters translate; frameworks/drivers are raw external detail) even though this repo's `infrastructure/` package deliberately merges both into one Go package per external system (see Caveats) — worth naming precisely so a future reviewer understands the merge is a deliberate simplification, not a missed ring.
- **Screaming Architecture** — the principle that a codebase's top-level structure should reveal its business domain, not its framework or delivery mechanism. Not currently named in the repo; worth adopting as a citable term specifically to explain (per the Actionable Techniques above) why this repo's generic per-service skeleton is compliant rather than a violation, once someone asks the question.
- **Humble Object** — a pattern splitting hard-to-test boundary code (a database gateway, a presenter) from fully unit-testable logic, verified instead by integration/acceptance tests. Already practiced correctly by `go-repository-pattern` but never named.

## Caveats

- **This repo already treats the domain layer's purity correctly; the book mainly supplies precise names for what's already built, not new mechanics.** Unlike the Fundamentals-of-Software-Architecture research (which surfaced a genuinely missing capability — fitness functions as a formal artifact) or the Clean-Code research (which found real code-shape defects to fix), this book's yield here is overwhelmingly *naming and citation* work: the Dependency Rule, the four rings, DTOs at boundaries, and the Humble Object pattern are all already implemented correctly in `go-project-structure`, `component-diagram`, `go-service-layer`, and `go-repository-pattern` — they are simply cited generically ("Clean/Hexagonal architecture," "dependency rule") rather than precisely. Treat this document's Applies To table as a set of small, targeted tightenings to existing skills, not a rewrite.
- **This repo deliberately merges Clean Architecture's two outermost rings, and that is a reasonable simplification, not a defect.** The book keeps Interface Adapters (translation/gateway code) and Frameworks & Drivers (the raw web framework, the raw DB driver) as two separate rings; this repo's `infrastructure/` package contains both a repository's translation logic (Reconstitute mapping, error translation) and its raw pgx calls in the same file. Uncle Bob himself notes this separation is often collapsed in practice when the marginal benefit of a further split is small — do not propose splitting `infrastructure/` into two rings without a concrete driver (e.g., swapping database engines while keeping the same adapter logic) that would actually benefit from it.
- **No new skill or agent is warranted by this book.** Every distinctive idea here (Dependency Rule, four rings, Screaming Architecture, Humble Object, boundary DTOs) maps onto a gap that is either already correctly implemented in an existing skill (needing only a naming/citation tightening) or belongs in the glossary as a term/disambiguation, not a new artifact type. Specifically considered and rejected: a standalone "screaming-architecture" skill (the concern is already satisfied at the Bounded-Context/service-naming level owned by `container-diagram`, not a gap inside `go-project-structure`'s per-service skeleton); a standalone "humble-object-pattern" skill (the pattern is already fully practiced by `go-repository-pattern` and `go-chi-handler`; it needs a name, not new content); a standalone "entity-terminology" skill (a two-term glossary edit, not an artifact). The MVP-dependency-pull threshold used elsewhere in this campaign (pull in a new skill only when a real, currently-unserved dependent need exists) is not met by anything in this book.
- **Component cohesion/coupling principles (REP, CCP, CRP, ADP, SDP, SAP) are not actionable at this repo's current scale.** These principles govern how a large system splits into independently versioned components/libraries — this repo is one Go module per microservice, with no internal component-splitting problem to solve yet. Cite only if a future product genuinely needs to split a single service's `internal/` tree into separately versioned Go modules; forcing this content into today's single-module services would be ceremony with no problem to solve, the same caution earlier research gave for style catalogs and organizational content with no current audience.
- **Java/procedural-era examples and some chapters don't translate literally.** The book's illustrations (Ch. 25's "clean embedded architecture," parts of the case-study chapters) are written against languages and domains (embedded C, Java EE-era frameworks) this repo's Go/chi/pgx/Redpanda stack doesn't share; treat the book as a source of ring vocabulary and boundary discipline, not literal Go implementation guidance — that already lives correctly in `go-project-structure` and its sibling `go-*` skills.
