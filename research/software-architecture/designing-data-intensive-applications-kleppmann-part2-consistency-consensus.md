**Author:** Martin Kleppmann
**Domain:** Software Architecture — Data Systems
**Part 2 of 2** — Transactions, Isolation, Consistency Models, Consensus, Batch/Stream Correctness. (Part 1: `designing-data-intensive-applications-kleppmann.md` — reliability/scalability/maintainability, storage engines, replication, partitioning, encoding.)

## Core Concepts

- **Transactions and weak isolation anomalies (Ch. 7).** ACID's "I" (Isolation) is precisely not one thing — it's a spectrum, and most databases (including PostgreSQL, by default) do **not** run every transaction fully serializable because of the performance cost. Kleppmann names the concrete anomalies weaker isolation levels allow: **dirty reads** (reading another transaction's uncommitted write), **dirty writes** (overwriting another transaction's uncommitted write), **read skew / non-repeatable read** (re-reading the same row mid-transaction sees a different value because another transaction committed in between), **lost update** (two transactions read-modify-write the same object concurrently; one update silently overwrites the other), **write skew** (two transactions read overlapping data, make disjoint writes based on what they read, and together violate an invariant neither write alone would violate), and **phantom reads** (a write changes the result of another transaction's search query). **Read Committed** (Postgres's default) prevents dirty reads/writes but not read skew, lost update, or write skew. **Snapshot Isolation** (what Postgres calls `REPEATABLE READ`, implemented via MVCC) prevents read skew and (with care) lost update, but **not write skew** — this is the anomaly Kleppmann spends the most space on precisely because it's the one people don't expect. Only **true serializability** (via Serializable Snapshot Isolation — Postgres's `SERIALIZABLE`, added in 9.1 — or two-phase locking, or actual single-threaded serial execution) prevents write skew. Kleppmann's recommended remedies for lost update specifically: atomic compare-and-set / atomic update operations (exactly the version-column CAS pattern), or explicit row locking (`SELECT ... FOR UPDATE`); write skew has no such easy fix short of `SERIALIZABLE` or an explicit, application-enforced lock on the invariant's scope.
- **Distributed consistency models, precisely defined (Ch. 9).** **Eventual consistency** is a weak, almost trivial guarantee: *if writes stop, replicas will eventually converge* — it says nothing about read-your-writes, ordering, or how long "eventually" takes. **Causal consistency** is stronger: it preserves the **happens-before** relationship — if event B was caused by (or created in response to) event A, every observer that sees B must have already seen A; concurrent (causally unrelated) events may be observed in different orders by different observers, and that's fine. **Linearizability** is the strongest and most expensive: the system behaves as if there were only a single copy of the data, and every operation appears to take effect atomically at some point between its start and end, consistent with real-time ordering — this is what's needed for correctness properties like "only one node believes it's the leader," a unique-username constraint enforced across nodes, or a distributed lock. Kleppmann's key practical point: **linearizability is expensive and often unnecessary** — most application correctness needs are actually satisfied by the much cheaper causal consistency (which requires only tracking happens-before via mechanisms like version vectors or explicit causal metadata), reserving linearizability for the few operations that genuinely need it (leader election, uniqueness, locks).
- **Distributed transactions and Two-Phase Commit (Ch. 9).** 2PC makes cross-node atomic commit possible but at a severe cost: it is a **blocking protocol** — if the coordinator crashes after participants vote "yes" but before it broadcasts commit/abort, participants are stuck holding locks indefinitely, unable to safely resolve on their own. This is precisely why the book (and this repo's design) favor the Transactional Outbox + asynchronous event propagation over distributed transactions: atomicity is achieved *locally* (one transaction, one node) and cross-service effects are propagated as an async, idempotently-retriable event stream instead of a synchronous, blocking cross-node commit.
- **Consensus (Ch. 9).** Total order broadcast, leader election, and distributed locking all reduce to the same underlying problem: **consensus** — getting a set of nodes to agree on a single value despite failures. Algorithms like Raft, Paxos, and Zab solve this correctly (unlike naive approaches such as picking "the node with the lowest ID" or a timeout-based leader election, both of which are unsafe under network partitions/GC pauses). The book's key point for practitioners: **consensus is extremely hard to get right — use a proven implementation, never hand-roll it.**
- **Exactly-once semantics, honestly assessed (Ch. 11).** Kleppmann is explicitly skeptical that "exactly-once" is achievable as a literal end-to-end guarantee across independent systems — what's actually achievable is **effectively-once processing**: at-least-once delivery combined with idempotent, deduplicating consumers, such that the *observable business effect* of processing a message twice is the same as processing it once. True exactly-once requires either restricting to a single system's internal transaction (e.g. a stream processor with transactional state and transactional output within one Kafka cluster) or accepting the cost of distributed transactions/2PC. The **end-to-end argument**: deduplication only truly matters at the ultimate side-effect boundary (e.g., don't charge a customer twice) — an idempotency token must be threaded through the *entire* chain from the true origin all the way to that final effect; deduplicating only at one intermediate hop doesn't prevent duplication at the final one if a retry re-enters the chain downstream of that hop.

## Actionable Techniques

- For every multi-row or multi-aggregate invariant enforced by "read current state, then decide, then write" (rather than a single atomic statement), explicitly ask whether it is subject to **write skew** — two concurrent transactions each reading a consistent snapshot, each individually valid, jointly violating the invariant. If yes, either scope it to `SERIALIZABLE`, add an explicit lock/uniqueness constraint covering the invariant, or redesign it as a single atomic statement.
- Name the isolation level a design relies on, not just "we use transactions." State explicitly: this repo's default optimistic-concurrency CAS pattern (a `version` column check-and-increment) is a targeted fix for **lost update** specifically — it does not, by itself, prevent write skew across two different Aggregates.
- For every cross-node or cross-service data guarantee, ask "does this actually need linearizability, or does causal consistency suffice?" — default to the cheaper guarantee (causal, via happens-before metadata already present in the event envelope) and reserve anything genuinely requiring linearizability (a global uniqueness constraint, a distributed lock) as a named, deliberate exception.
- Never hand-roll leader election or distributed locking with ad hoc heuristics (lowest ID, timeout race) — use a proven consensus-backed primitive (e.g., a database's own transactional uniqueness constraint when a single Postgres instance is authoritative, which trivially provides linearizability since there is only one node; or a battle-tested distributed lock service if one is ever needed across nodes).
- When claiming "exactly-once," restate it as "effectively-once via idempotent consumer + at-least-once delivery" and identify the true end-to-end side effect the idempotency key must reach — not just the first hop that happens to check it.
- Thread one idempotency/deduplication key end-to-end through a multi-hop pipeline rather than minting a new one at each stage, wherever the same logical operation must not double-fire its final real-world effect.

## Key Terminology

Worth considering for `skills/glossary-management/`'s Ubiquitous Language — this is the area the book is most precise about, and where this repo's current vocabulary is thinnest:

- **Dirty Read**, **Dirty Write** — basic isolation anomalies, useful baseline vocabulary even though Postgres's default already prevents them.
- **Read Skew (Non-Repeatable Read)** — the anomaly Snapshot Isolation exists to prevent.
- **Lost Update** — the anomaly this repo's `version`-column CAS pattern specifically targets; naming it lets `data-model-design` state its guarantee precisely instead of leaving it implicit.
- **Write Skew** — the anomaly *not* prevented by the version-column pattern; the single highest-value term to add given the repo's current reliance on per-Aggregate optimistic concurrency.
- **Phantom Read** — relevant to any invariant expressed as "no row matching X may exist" checked via a query rather than a constraint.
- **Snapshot Isolation** — the precise name for Postgres's `REPEATABLE READ`; distinguishing it from full serializability clarifies what the default transaction behavior actually guarantees.
- **Serializable Snapshot Isolation (SSI)** — the precise name for Postgres's `SERIALIZABLE` mode; worth naming as the actual mechanism to reach for when write skew must be prevented.
- **Linearizability** — the strongest consistency model; currently absent from the glossary entirely despite the repo needing it implicitly for things like tenant uniqueness constraints (trivially satisfied today only because each constraint lives on one Postgres instance).
- **Causal Consistency** — the model this repo's `correlationId`/`causationId` envelope fields are implicitly built for; naming it lets a future review check whether that promise is actually kept end-to-end.
- **Eventual Consistency** — already a CLAUDE.md ubiquitous term, but currently undefined precisely anywhere in the read skills; the book's precise (weak) definition — "if writes stop, replicas eventually converge, with no promise about ordering or timing" — would sharpen every place this term is invoked to describe the CQRS read side.
- **Happens-Before Relationship** — the formal relation causal consistency preserves; useful shorthand for "this event could only exist because that one happened first."
- **Two-Phase Commit (2PC)** — worth naming explicitly as the pattern this repo's outbox+Saga approach deliberately avoids, so the avoidance reads as a stated decision rather than an absence.
- **Consensus** — the general problem (total order, leader election, distributed locking) that Raft (used internally by Redpanda) solves; useful for explaining *why* Redpanda's partition-leader guarantees can be trusted without this plugin re-deriving them.
- **Effectively-Once Processing** — the honest, achievable alternative to "exactly-once"; already the repo's de facto default (per `data-pipeline-design`) but never named as such.
- **End-to-End Argument** — the principle that a guarantee (like deduplication) is only real if it's enforced at the true endpoint, not at an intermediate layer that merely looks like it enforces the guarantee.

## Caveats

- **This repo does not need to hand-roll consensus, and should say so explicitly rather than by omission.** Raft (inside Redpanda) and PostgreSQL's own single-node transactional guarantees already provide every consensus-dependent guarantee (leader election for partitions, atomic commit for a single Aggregate) this plugin's default stack requires. The book's consensus chapter is background for *why* those guarantees hold, not a design task for `data-architect` or `enterprise-architect` — no skill should ever propose building a custom leader-election or locking mechanism.
- **Linearizability is cheap here specifically because of physical multi-tenancy, not in general.** The book treats linearizability as expensive because it usually implies coordination across many nodes. In this repo, most operations that would need linearizability (a uniqueness constraint, "only one active X") are trivially linearizable for free, because each tenant's data lives on a single PostgreSQL instance — there is only one node to be consistent with. This stops being free the moment a single tenant's data is ever partitioned across multiple database nodes (see Part 1's sharding caveat) — at that point, uniqueness constraints become a genuine distributed-consistency problem the book's Ch. 9 material would need to be revisited for.
- **Write skew is a live risk specifically at Aggregate boundaries, which this repo enforces strictly.** Because `data-model-design` mandates that cross-Aggregate references are by ID only (never a cross-Aggregate transaction), any invariant spanning two Aggregates is, by construction, checked and enforced across two separate transactions — precisely the shape Kleppmann identifies as vulnerable to write skew. This is a direct consequence of this repo's own DDD discipline, not a generic database concern; it deserves specific attention wherever a cross-Aggregate invariant exists (e.g., in Saga design or canonical-model matching), rather than being treated as a rare edge case.
- **The book's exactly-once skepticism is already this repo's stated default — a rare case of full alignment, not a gap needing correction.** No caveat is needed here beyond noting it: `data-pipeline-design`'s "Exactly-once everywhere" anti-pattern entry independently arrives at the same conclusion Kleppmann reaches in Ch. 11.
