# Fundamentals of Software Architecture
**Authors:** Mark Richards and Neal Ford
**Domain:** Software Architecture

## Core Concepts

- **Architecture is the stuff that's hard to change later (Ch. 1).** The authors' working definition of software architecture is the combination of structure (architecture style/topology), architecture characteristics ("-ilities"), architecture decisions, and design principles. What separates an "architecture decision" from a "design decision" is cost of change: architecture decisions constrain what the codebase can easily do later (e.g., synchronous vs. event-driven communication, one database per service vs. shared), while design decisions (e.g., which loop construct to use) are cheap to reverse. This repo's own `CLAUDE.md` component-hierarchy language ("Command → Agent → Skill") and its hard rules ("one database per service," "asynchronous is the default across Bounded Contexts") are exactly this kind of load-bearing, hard-to-reverse decision — the book gives vocabulary for *why* they deserve ADRs and others don't.
- **Architecture characteristics are first-class, not afterthoughts, and they trade off against each other (Ch. 4).** The book's core move is to reframe "-ilities" (availability, scalability, elasticity, performance, security, testability, deployability, etc.) as measurable, competing forces rather than universal virtues to maximize. Every architecture decision is a trade-off among characteristics — you cannot simultaneously maximize all of them (the "iron triangle" expands into an "iron -ility spectrum"). A characteristic only counts as architectural if it's (a) not purely domain-specific, (b) has structural/design impact, and (c) is critical or important to application success. This maps directly onto this repo's `nfr-specification` skill, which already forces measurability but does not yet ask *which characteristics conflict with which* for a given design.
- **Trade-off analysis, not absolutes ("Architecture is the stuff that's hard to change" + "everything is a trade-off," Ch. 1–4).** The book's most repeated line is some version of "there are no right or wrong answers in architecture — only trade-offs." An architect's job is to identify what characteristics matter most for a *specific* system and to make that prioritization and its cost explicit, rather than pursuing best practices as if they were free. This is the direct ancestor of good ADR-writing (options considered, honest cons) — but the book goes further by asking architects to explicitly *rank* competing characteristics (e.g., "we are choosing availability over consistency here, and here is why") rather than just listing pros/cons per option.
- **Architecture styles as a catalog, each with a "fitness" for certain characteristics (Ch. 10–17).** The book catalogs major styles — layered (n-tier), pipeline, microkernel, event-driven (broker and mediator topologies), space-based, service-based, service-oriented, and microservices — and for each one, discusses which architecture characteristics it naturally supports well vs. poorly, and its typical failure modes. Microservices are presented as one style among several, chosen when characteristics like independent deployability, evolvability, and elasticity dominate — not a default "best practice." This directly matters here: this repo's default stack (event-driven microservices, Go/chi, Redpanda, one-DB-per-service) is a valid, deliberate style choice by the book's own criteria (frugality, single-team-ish scale, need for independent service evolution) — but the book would insist the choice be *justified against the specific characteristics that matter for this product*, not simply asserted as the default.
- **Event-driven architecture: broker vs. mediator topology (Ch. 14).** The book distinguishes two event-driven topologies: **broker topology** (event flows are choreographed — each service reacts to events and emits new ones, no central coordinator) and **mediator topology** (a central mediator/orchestrator coordinates a complex, multi-step business process by directing which services act in what order). Broker topology gives high performance, scalability, and decoupling but sacrifices error handling, workflow control, and recoverability for complex multi-step processes; mediator topology sacrifices some decoupling and scalability for workflow visibility and centralized error handling. `integration-design`'s "Default rule: use asynchronous (events) for cross-Bounded-Context communication" and its anti-pattern table already push toward broker-style choreography, but never once uses "mediator" or names orchestration as a legitimate, deliberately-chosen alternative for genuinely complex multi-step sagas — the book would flag any workflow with several ordered steps and compensating actions as a mediator-topology candidate, worth an ADR either way.
- **Architectural fitness functions (Ch. 5, drawing on *Building Evolutionary Architectures*).** A fitness function is any mechanism that provides an objective, automated assessment of whether an architecture characteristic still holds — a governance test, not a one-time review. Examples: an ArchUnit-style test that fails the build if `domain/` imports a database driver; a load test gating a deploy on P95 latency; a dependency-freshness check. The book frames these as executable, continuously-run guardrails for architecture decisions, not just diagrams and prose. This is the single most actionable idea missing from every diagram skill read here — `component-diagram`'s "Dependency direction rule" is stated as prose and an anti-pattern table entry ("enforce with a lint rule or import test in CI") but never formalized as a fitness function the enterprise-architect is responsible for specifying, alongside the diagram, as an artifact.
- **Architecture decisions vs. architecture principles (Ch. 18, "Architecture Decision Records").** The book (which is largely credited with popularizing the modern ADR format alongside Michael Nygard) distinguishes a *decision* (a specific, concrete choice made for this system, in this context, recorded once) from a *principle* (a general rule of thumb that guides many future decisions, e.g., "prefer asynchronous communication," "encrypt data at rest"). Principles are living guidance documents; decisions are point-in-time records. `adr-authoring` already implements the ADR half of this distinction thoroughly and well — but this repo has no distinct artifact type for *principles* (the closest analog is scattered through `CLAUDE.md`'s "Non-Negotiable Methodology" and "Tech Stack Defaults" tables, which function as principles but are never named as such or given their own authoring discipline).
- **The "Laws of Software Architecture" — everything is a trade-off, and "why" is more important than "how" (Ch. 1).** The book opens with two laws: (1) everything in software architecture is a trade-off (if you think you've found something that isn't, you just haven't found the trade-off yet), and (2) *why* is more important than *how* — understanding the reasoning behind a decision matters more than the mechanics of the decision itself, because reasoning is what lets a future architect know when the decision no longer applies. This is precisely the philosophy behind `adr-authoring`'s Context/Rationale split and its anti-pattern "the retrofitted ADR" — the book gives this existing discipline a named first-principle rationale rather than leaving it as an unexplained convention.
- **Architects need "T-shaped" technical breadth over depth, and must understand business drivers (Ch. 2).** An architect must have broad exposure across many technologies (breadth) with deep expertise in a few (the vertical bar of the T), must analyze the business/domain to identify which characteristics actually matter (not just take a menu order), and must balance architecture with hands-on coding to avoid "ivory tower" architecture divorced from implementation reality. This is a direct parallel to `enterprise-architect.md`'s explicit "owns/does not own" boundaries and its NFR-first "Decision Process" step 2 ("Read NFR architecture handoff... first inputs considered, not the last") — the agent file already embodies this without citing it.
- **Diagramming discipline — the "architecture as vitamins, not painkillers" framing and diagram/decision separation (Ch. 19–20).** The book distinguishes diagrams (structure) from decisions (ADRs) from characteristics (the ilities) as three separate, cross-referenced artifact types that must stay synchronized — a diagram implies characteristics and decisions, and neither should silently drift from the other. This repo's C4-diagram skills (system-context, container, component) and `adr-authoring` are already exactly this separation of concerns — the book validates the existing three-artifact-type split rather than suggesting a new one.

## Actionable Techniques

- For every architecture decision (not just ADRs already being written), name the 2-3 architecture characteristics being traded off against each other explicitly — e.g., "chose eventual consistency over strong consistency to gain availability and partition tolerance" — as a required line in the ADR's Rationale section, not left implicit.
- Before drawing a Container Diagram, run a fitness-function inventory pass: for each hard rule the diagram will encode (one DB per service, inward-only dependencies, sync resilience requirements), name the automated check (lint rule, ArchUnit-style import test, load test threshold) that will continuously verify it stays true — and record that check as part of the diagram's output, not just as an aside in an anti-pattern table.
- For every event-driven flow with more than ~2-3 ordered steps or any compensating/rollback logic, explicitly evaluate mediator (orchestration) topology as an alternative to default choreography, and write the topology choice as its own ADR either way — "we chose broker/choreography" is itself a decision worth recording, not just the exception case.
- When selecting an architecture style (even though this repo defaults to event-driven microservices), require a short justification block naming which 2-3 characteristics from the NFR architecture-handoff section specifically drove that choice for *this* Bounded Context — this converts "we use microservices because that's the default" into a traceable, falsifiable claim.
- Separate "decision" artifacts (ADRs — point-in-time, specific) from "principle" artifacts (durable, cross-cutting rules of thumb) explicitly; when an ADR's rationale generalizes beyond the single decision at hand ("we always prefer async for cross-context flows"), promote that generalization into a named principles document rather than leaving it buried in one ADR's prose.
- Add an explicit trade-off matrix (rows = characteristics like availability/consistency/latency/cost, columns = options considered) to the ADR template's "Options Considered" section for decisions with 3+ competing characteristics — text-only pros/cons lose the multi-dimensional trade-off the book emphasizes.
- For every "-ility" in the NFR specification's 8 categories, check whether the chosen architecture style (or a specific service's container/component structure) has a *known* weakness against it per the book's style catalog (e.g., microservices' known weakness in distributed transaction consistency, or layered architecture's known weakness in deployability) — surface that weakness explicitly as an accepted trade-off, not a silent gap.

## Key Terminology

Worth considering for `skills/glossary-management/`'s Ubiquitous Language:

- **Architecture Characteristic** — the book's precise term for an "-ility" that has structural impact and is critical to success (distinct from a merely "nice to have" quality). This repo's `nfr-specification` already lists 8 categories of NFRs but never uses this term to distinguish an architecturally-significant NFR from a purely functional or cosmetic one — naming it would sharpen the existing "Architecture Handoff" section's selection criteria.
- **Fitness Function** — an automated, objective mechanism (test, linter rule, load-test threshold) that continuously verifies an architecture characteristic or decision still holds. Not currently named anywhere in the repo, despite the repo already practicing an informal version of it (the `tdd-gate` hook, lint-rule asides in anti-pattern tables) — formalizing the term would let future skills reference "fitness function" precisely instead of reinventing "enforce with a lint rule" prose each time.
- **Mediator Topology / Broker Topology** — the two named event-driven architecture topologies (orchestrated vs. choreographed). `integration-design` and `event-driven-patterns` (not read here, but referenced by `enterprise-architect.md`'s "Event-Driven Pattern Decisions ← choreography vs orchestration per flow" step) already use "choreography" and "orchestration" informally — adopting the book's paired terms would align this repo's language with the most widely cited architecture reference on the subject.
- **Architecture Principle** (as distinct from Architecture Decision) — see the ADR gap above; if adopted, this term needs a precise glossary definition distinguishing it from `ADR` before any skill starts producing "principles" documents, to avoid the two categories blurring in review.

## Caveats

- **This repo has already committed to a style; the book's "choose the right style" framing must become "justify/validate the chosen style."** The book spends roughly a third of its content (Ch. 10–17) walking through *how to choose* among layered, microkernel, event-driven, space-based, service-based, and microservices architectures based on which characteristics matter most. This repo's `CLAUDE.md` Tech Stack Defaults table has already made that choice (event-driven microservices, Go/chi, Redpanda, one-DB-per-service) as a standing default. Do not re-run the book's style-selection process from scratch per product — instead, use the book's per-style characteristic trade-off tables as a checklist to *validate* the default against each product's specific NFRs, and require an ADR (not a full re-selection) only when a specific Bounded Context's NFRs genuinely conflict with the default (this repo's existing "Architecture Handoff" escalation path in `enterprise-architect.md` already anticipates exactly this narrower use).
- **Space-based architecture, microkernel, and pipeline styles are close to irrelevant here and should not be force-fit.** The book gives substantial space to architecture styles (e.g., space-based architecture for extreme elasticity/high-volume bursty traffic, microkernel for plugin-based product-line systems) that have no plausible fit for this repo's default event-driven microservices stack or its stated product context (data estate mapping/compliance). Cite these styles only as contrast/comparison points in a fitness-function or trade-off discussion, never as candidates to actually adopt without a very specific, unusual NFR driving it.
- **The book is technology-agnostic and pre-dates several specifics this repo has already fixed.** It illustrates concepts with Java/Spring-style examples and does not address Go's specific idioms (its `pgx`/`chi`/Redpanda stack, its layered Go package structure) at all — the mapping from the book's general layered/microservices/event-driven concepts onto this repo's Go conventions is this document's own extrapolation, not literal book content. Treat the book as a source of *vocabulary and trade-off discipline*, not of Go-specific implementation guidance (that already lives correctly in `component-diagram` and the `go-*` skills).
- **"Architect elevator" and organizational/soft-skills chapters (Ch. 2-3, "Modularity," parts of the "Soft Skills" chapters) are largely non-actionable for a solo-operator, single-agent-driven plugin.** Content about team dynamics, negotiating with stakeholders, and career-ladder framing for human architects doesn't translate to this repo's context (one PM, one engineering agent) and should be skipped as source material rather than awkwardly adapted.
