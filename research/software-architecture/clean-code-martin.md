# Clean Code
**Author:** Robert C. Martin (Uncle Bob)
**Domain:** Software Architecture & Implementation

## Core Concepts

- **Meaningful Names (Ch. 2).** Names should be intention-revealing, pronounceable, and searchable. Avoid disinformation (a variable named `accountList` that isn't a `List`), avoid noise words (`data`, `info`, `theAccount`), avoid single-letter names except as loop indices in tiny scopes, and avoid mental-mapping (`r` instead of `region`). Class/type names are nouns; method/function names are verbs. Pick one word per concept and use it consistently (don't mix `fetch`/`retrieve`/`get` for the same kind of operation).
- **Functions Should Be Small, and Do One Thing (Ch. 3).** A function should be small — small enough that it does not need internal blank lines separating "sections." If a function is doing more than one thing, extract those things into their own well-named functions. The **Single Level of Abstraction (SLA)** rule: a function's statements should all sit at the same conceptual level — don't mix a high-level orchestration step with a low-level bit-fiddling step in the same body. Functions should have few arguments (0–2 ideal; 3 is a smell; more needs an argument object). **Flag arguments are a smell** — a boolean parameter that makes the function do one of two different things is a signal to split it into two functions. Functions should have **no side effects** hidden behind an innocent-looking name. **Command-Query Separation (CQS):** a function either does something (command, returns void/error) or answers something (query, returns a value) — never both.
- **Comments Are a Last Resort, Not a Virtue (Ch. 4).** "Don't comment bad code — rewrite it." A comment is often an apology for not expressing intent in code. Good comments are rare: legal notices, warnings of consequences, TODOs, amplification of something non-obvious. Bad comments are common: redundant restatements of the code, misleading comments that drift out of sync, commented-out code (delete it — version control remembers it), and journal/changelog comments (that's what commit history is for).
- **Formatting and the Newspaper Metaphor (Ch. 5).** A source file should read like a newspaper article: the name at the top tells you what it is, the highest-level concepts and algorithms appear first, and detail increases as you read downward. Related concepts should be kept vertically close; the file should not force the reader to jump around to follow one train of thought.
- **The Boy Scout Rule (Foreword/throughout).** "Leave the campground cleaner than you found it." Every time you touch a file, improve something small — a better name, a smaller function, one less duplication — rather than only adding to it.
- **Error Handling Is a Separate Concern from Business Logic (Ch. 7).** Error handling code should not obscure the logic it's guarding. Provide context with every error so its origin is diagnosable. Don't return `null` (forces every caller into defensive null-checking) and don't pass `null` into APIs. Define error/exception types in terms of the *caller's* needs, not the underlying failure mechanism — group errors by how a caller would want to react, not by where they came from.
- **Objects vs. Data Structures, and the Law of Demeter (Ch. 6).** Objects hide their data and expose behavior; data structures expose data and have no meaningful behavior. Don't create hybrids ("half object, half data structure") — they're the worst of both. The Law of Demeter: a method should only talk to its immediate collaborators, not reach through one object to get to another (`a.getB().getC().doSomething()` is a violation — "train wrecks").
- **Classes Should Be Small and Cohesive (Ch. 10).** Single Responsibility Principle at the class level: a class should have one reason to change. Cohesion is maximized when each method uses most of the class's instance variables — low cohesion is a sign the class should be split. Organize classes so that extending behavior means *adding* code, not *editing* existing code (Open/Closed).
- **Emergent Design — Kent Beck's Four Rules of Simple Design (Ch. 12).** In priority order: (1) passes all tests, (2) reveals intention (expressive naming and structure), (3) no duplication, (4) fewest elements (classes/methods). Duplication is "the primary enemy of a well-designed system."
- **Unit Tests and F.I.R.S.T. (Ch. 9).** Tests are production code — kept clean, refactored, and cared for. F.I.R.S.T: Fast, Independent, Repeatable (any environment), Self-Validating (pass/fail, no manual inspection), Timely (written just before the production code — i.e., TDD). One assert (or, more precisely, one *concept*) per test.

## Actionable Techniques

- Rename any variable/function whose purpose isn't obvious from its name alone, with no need to read its body.
- Extract any function body that mixes levels of abstraction until each extracted function does exactly one thing, at one level.
- Cap function argument counts at 2; bundle 3+ related parameters into a named struct/object.
- Replace boolean/flag parameters that branch internal behavior with two separately named functions.
- Delete commented-out code on sight; if a comment restates what the code already says, delete the comment instead.
- Where a language allows returning `nil`/`null`, prefer a documented sentinel, an explicit "not found" error, or an empty collection over letting `nil` silently propagate into a caller that will dereference it.
- Audit method chains for Law-of-Demeter violations (`x.Y().Z().W()`); introduce a well-named method on the immediate collaborator instead of reaching through it.
- Before adding a new class method, check cohesion: does it use most of the class's fields? If not, it likely belongs elsewhere.
- Treat duplication (even structural, not just literal, duplication) as a defect to remove before adding new code on top of it.
- Apply the Boy Scout Rule as a per-PR checklist item: name one small thing improved beyond the requested change.

## Key Terminology

Worth considering for `skills/glossary-management/`'s Ubiquitous Language, if the campaign wants implementation-level (not just domain-level) shared vocabulary:

- **Command-Query Separation (CQS)** — a method-level discipline (do vs. answer, never both), distinct from this repo's existing `CQRS` term (an architectural write/read split). Naming both explicitly would prevent the two from blurring together in review — CQRS is already glossary-adjacent via `Command`/`Read Model`; CQS is not yet distinguished anywhere in the read files.
- **Law of Demeter** — directly relevant to the "consumer-defined interfaces" and layering rules already enforced in `go-project-structure`; naming it would let reviews cite it precisely instead of reinventing "don't reach through objects" prose each time.
- **Single Level of Abstraction (SLA)** — a precise, citable name for the function-extraction technique above; useful shorthand in code review comments.

## Caveats

- **Java-centric examples don't translate literally.** The book's Ch. 7 ("Prefer Exceptions to Error Codes") is written against Java's checked/unchecked exception model. Go has no exceptions in that sense — `go-error-handling`'s existing "errors are values, `panic`/`recover` only at boundaries" standard is the correct idiomatic reinterpretation, not a literal application of the book's chapter. Cite the book's underlying *goal* (don't let error-handling code obscure business logic; provide context) rather than its Java mechanism.
- **"Don't return null" needs Go-specific adaptation, not adoption.** Go's `nil` is a legitimate, idiomatic zero value (nil slices are usable, nil pointers signal absence deliberately, `(T, error)` pairs already force the caller to check `err` before touching a possibly-nil `T`). The book's blanket "avoid null" advice, written for Java where a `null` reference is an unchecked landmine, should become a narrower Go-specific rule about consistency (e.g., "no-results" queries return empty slices, not nil) — not a wholesale ban on nil.
- **Chapter 6 (Objects and Data Structures) and Chapter 10 (Classes) are inherently OOP/class-based** and do not map cleanly onto Go, which has no classes or inheritance. The book's SRP-for-classes and cohesion arguments translate reasonably well to Go structs-with-methods and to package cohesion (already covered by `go-project-structure`'s package-naming rules), but the data/object anti-symmetry argument (built around Java's encapsulation model) is weaker in a language where structs are usually plain data and behavior lives in free functions or small interfaces — treat this chapter as directional inspiration, not a rulebook to port verbatim.
- **Chapter 13 (Concurrency) is written against Java's `synchronized`, thread pools, and `ExecutorService`**, none of which exist in Go. This repo's `go-concurrency-patterns`/`go-service-layer` skills already express the same underlying goal (bounded, well-scoped, independently reasoned-about concurrent units) through goroutines, channels, and `errgroup` — a case where the repo's existing content is already a legitimate, idiomatic translation rather than a gap to fill.
- **Chapter 14 (Successive Refinement)** is a single worked Java case study (the `Args` class) with no reusable content outside that specific example — skip it as source material.
- The book predates Go, TypeScript, and React entirely (2008) — all React/TypeScript-specific applications above (hooks, discriminated unions, component composition) are this document's extrapolation of the book's general principles, not content literally in the book.
