# Building Microservices
**Authors:** Sam Newman
**Domain:** Software Architecture — Microservices (Foundational/Practitioner Text)

## Core Concepts

- **Strangler Fig as a set of concrete execution techniques, not just a sequencing decision (Ch. 2 / "Splitting the Monolith").** Newman's book is the origin of "Strangler Fig" as a named pattern for the microservices context (borrowing the metaphor from Martin Fowler): a new service grows up around an existing monolith capability and gradually intercepts its traffic until the old code path can be deleted, with the monolith running throughout. The book gives three concrete mechanics for actually doing this: **Branch by Abstraction** (introduce an abstraction/interface inside the monolith in front of the capability being extracted, so callers depend on the abstraction while the implementation is swapped from in-process to remote-service behind it, all without a big-bang cutover), **Parallel Run** (run the old and new implementations side by side against the same input, compare outputs, and only cut traffic over once the new implementation's results have been verified to match in production-realistic conditions — a verification technique, not just a rollout technique), and **edge-level strangling via an intercepting layer** (a reverse proxy or edge router in front of the monolith selectively routes specific URLs/capabilities to the new service while everything else still reaches the monolith unchanged). This repo already names "the Strangler Fig pattern" in `skills/subdomain-distillation/references/legacy-transformation-guidance.md`, but only at the sequencing/prioritization layer (*which* subdomain to strangle first, and why) — none of these three execution mechanics appear anywhere in the repo. This is the clearest, most concrete gap this book fills.
- **Conway's Law and the (reverse) Conway maneuver (Ch. 10, "The People Factor").** Conway's Law — "organizations which design systems... are constrained to produce designs which are copies of the communication structures of these organizations" — is Newman's lens for why service boundaries drawn purely on technical/domain merit still fail if team structure doesn't match them: a service two teams both need to change becomes a coordination bottleneck regardless of how clean its Bounded Context is. The **reverse Conway maneuver** flips this deliberately: restructure teams first, to shape the architecture you want. This repo's `container-diagram`'s "Deciding Service Boundaries" table already has a bare "Team ownership" row ("If different teams own different capabilities, those capabilities are in different containers") and `bounded-context-mapping` lists "team ownership" as one of five boundary justifications — but neither cites Conway's Law by name, and (per the Caveats below) the concept has limited direct applicability to this repo's actual operating model.
- **Smart endpoints, dumb pipes.** Newman's inherited principle (via Fowler/Lewis's original microservices article, which Newman's book operationalizes at length) that business logic belongs inside services, and the communication mechanism between them (broker, service mesh, HTTP) should stay dumb — a "smart pipe" that starts making routing decisions based on message content becomes a centralized, unversioned, untested ESB-in-disguise that recouples every service to it. This repo's `integration-design` and `event-driven-patterns` already keep Redpanda topics free of business logic implicitly (all logic lives in producer/consumer handlers), but neither skill states the principle by name as a rule to check new integrations against.
- **Consumer-Driven Contracts, at the architecture-conceptual level (Ch. 4, "Integration").** Newman (crediting Ian Robinson) is one of the two or three canonical sources for Consumer-Driven Contracts as a microservices integration discipline — the same concept this repo's `context-map-patterns` and `go-contract-test` already implement thoroughly. This is confirmed **not a gap**: `context-map-patterns`' Customer/Supplier pattern already mandates CDC with a named contract owner, a CI gate, and an update process, and `go-contract-test` implements it with more operational precision (schema-based, frugal, provider-verification-loop, deployed-version pinning) than Newman's own generic treatment. No action needed here.
- **The "how small is too small" service-sizing question (Ch. 2, "What is a service?").** Newman's heuristic — size a service around what a team can own end-to-end, and split further only when a specific force (independent scaling, security isolation, rate of change, team size) demands it — is real content in the book, but this repo's cluster **already has a more rigorous, more granular treatment of the identical question**: `software-architecture-hard-parts-ford.md` documents the six granularity disintegrators and four granularity integrators from *Software Architecture: The Hard Parts*, which supersede Newman's looser heuristic list. Per this research task's explicit instruction, this is confirmed **not a new gap** — Newman's sizing content is strictly subsumed by content already in the cluster.
- **Testing strategies across service boundaries, including testing in production (Ch. 7, "Testing").** Newman's test pyramid (unit → integration → contract → end-to-end) plus a fifth, distinctive category — testing in production via synthetic transactions, canary releases, and semantic monitoring (checking business-level correctness signals, not just infrastructure health) — is one of the book's most cited chapters. This repo's `test-pyramid` skill already implements an equivalent (and more detailed) four-layer pyramid with an explicit shift-left/shift-right split, and `canary-deployment` already covers progressive traffic shifting. The one piece not explicitly named anywhere is **semantic monitoring** — verifying business outcomes (e.g., "did compliance gap counts change in the expected direction after this deploy") as a distinct signal from technical health checks (`health-check-design`) or SLOs (`slo-definition`) — a small, terminology-level gap rather than a structural one.
- **Versioning and the Tolerant Reader pattern (Ch. 4, "Integration").** Newman recommends consumers parse only the fields they need and ignore unknown fields, deferring breaking changes as long as possible, and treats semantic versioning plus running multiple API versions concurrently during a deprecation window as the default posture for independently-deployable services. This repo's `api-contract-design` already implements the concurrent-version/deprecation-window mechanics (`/v1/` maintained 6 months after `/v2/` launches, `Deprecation` header) and `go-contract-test` already requires consumers to declare only the fields they read — the substance of Tolerant Reader is present, just not under that name.
- **Organizational and cultural aspects of microservices adoption (Ch. 1, 10-11).** Newman spends real space on why organizations adopt microservices (independent deployability, team autonomy, technology heterogeneity) and what that requires culturally (a mature CI/CD culture, decentralized governance, on-call ownership per team). Almost none of this maps onto this repo's actual operating model — see Caveats.

## Actionable Techniques

- When any future product engagement is not greenfield (a real client's existing monolith, or a legacy system this plugin must integrate with rather than replace), name which of the three Strangler Fig mechanics applies to the specific capability being extracted — Branch by Abstraction (in-process swap behind an interface), Parallel Run (side-by-side output comparison before cutover), or edge-level routing (reverse proxy intercepts specific routes) — rather than invoking "Strangler Fig" as an unspecified label.
- Before cutting any traffic from monolith to new service, require a Parallel Run verification step whenever the capability being extracted has any risk of subtly different behavior (financial calculations, compliance determinations, anything audit-relevant) — compare outputs on real (or replayed) production traffic before the cutover, not just before/after smoke tests.
- Add an edge-routing container to the Container Diagram whenever a transformation is in flight — an explicit reverse-proxy/facade container in front of the legacy system that this plugin's own services sit behind, so the transitional topology (not just the target-state topology) is diagrammed and reviewable.
- State "smart endpoints, dumb pipes" as an explicit check in `integration-design`'s Integration Inventory: does any consumer group, topic, or gateway layer make routing/business decisions based on event or request content? If so, that logic has leaked out of the owning service.
- Name "semantic monitoring" explicitly in `alerting-rules-design` or `metrics-instrumentation-plan` wherever a canary or blue-green rollout's success criteria include a business-outcome check (not just error rate / latency) — e.g., "classification throughput and gap-detection rate stay within expected bounds post-deploy," distinct from and in addition to infrastructure health checks.
- If Conway's Law is invoked at all in a future artifact, cross-reference it explicitly against `container-diagram`'s existing "Team ownership" row and `bounded-context-mapping`'s "team ownership" boundary justification, rather than introducing it as unattached vocabulary — it names a force those two skills already act on without naming.

## Key Terminology

Worth considering for `skills/glossary-management/`'s Ubiquitous Language:

- **Branch by Abstraction** — not currently named anywhere in the repo. Directly actionable only if `strangler-fig-execution` (or an equivalent addition to `legacy-transformation-guidance.md`) is adopted; otherwise skip — don't add glossary terms for concepts no skill uses yet.
- **Parallel Run** — same conditional as above; a specific, nameable verification technique (compare old-vs-new outputs before cutover) worth its own term if the strangler-fig gap is closed, since it is easily confused with a generic "A/B test" or `canary-deployment`'s traffic-shifting mechanics, which it is not (Parallel Run compares outputs on the same input; canary shifts live traffic proportion).
- **Smart Endpoints, Dumb Pipes** — a compact, quotable principle already implicitly practiced (no business logic in Redpanda topics or the gateway) but never named; low-cost, high-clarity addition if `integration-design` adopts the actionable technique above.
- **Semantic Monitoring** — distinguishes a business-outcome-level post-deploy check from an infrastructure health check or a technical SLO; worth adding only if `alerting-rules-design` or `metrics-instrumentation-plan` adopts it as a named category of canary success criteria.
- **Conway's Law / Reverse Conway Maneuver** — real, well-known vocabulary, but given this repo's operating model (one PM, Claude Code agents doing all engineering — no human "teams" whose communication structure could shape the architecture in the way Conway's Law describes), do not add this as glossary vocabulary speculatively. Revisit only if this plugin is ever used to architect a system for an organization with actual multiple human engineering teams.

## Caveats

- **The Strangler Fig / monolith-migration gap is real but conditional on a scenario this repo does not currently have.** Every product this plugin has designed so far (per `CLAUDE.md` and the worked examples throughout the cluster) is greenfield microservices from day one. The gap identified above is genuine — no execution mechanics exist anywhere in the repo — but it is a gap in coverage for a *hypothetical future engagement* (a real client with an existing monolith, or a legacy system this plugin must integrate around rather than replace), not a gap affecting the current Data Estate Mapping & Compliance Intelligence product. Size the investment accordingly: worth capturing once, not worth treating as blocking or urgent.
- **Conway's Law and the organizational/cultural chapters (Ch. 1, 10-11) are the least actionable content in this book for this repo**, more so even than the "architect elevator" caveat already recorded for *Fundamentals of Software Architecture*. Those chapters assume multiple human engineering teams whose communication patterns shape (or should be reshaped to shape) the architecture. This repo's actual operating model is one PM (Shafi) directing Claude Code agents that do all engineering — there is no team-communication-structure axis for Conway's Law to act on. Treat this content as background vocabulary only, and only worth resurrecting if this plugin is ever pointed at a real multi-team organization's system.
- **Service-sizing guidance is explicitly and deliberately not extended here.** This book's "how small is too small" heuristics are real and reasonably good, but `software-architecture-hard-parts-ford.md` already documents a more granular, more rigorous treatment of the identical question (six disintegrators, four integrators) from a later, more specialized book. Re-adding Newman's looser version would be a regression in rigor, not an addition — confirmed via direct comparison, per this research task's explicit instruction to check for this overlap before proposing anything.
- **Consumer-Driven Contracts, the test pyramid, and versioning/deprecation mechanics are all confirmed non-gaps, not merely under-checked.** This is worth stating plainly: this book's three most commonly cited practical contributions (alongside Strangler Fig) are already implemented in this repo, in each case with more operational precision than the book's own generic treatment (`go-contract-test`'s deployed-version-pinning nuance, `test-pyramid`'s explicit shift-left/shift-right split, `api-contract-design`'s concrete 6-month deprecation window). Do not re-derive these from the book — the repo's existing versions are already ahead of it.
- **The book pre-dates this repo's specific stack and is intentionally technology-agnostic.** Newman's examples (2nd edition, 2021) discuss REST/gRPC, Kafka, Kubernetes, and service meshes in general terms but never Go, `chi`, Redpanda specifically, or Apache AGE. As with the other two Ford/Richards books already in this cluster, treat this book as a source of vocabulary and pattern names (especially for the strangler-fig execution mechanics), not of Go-specific implementation guidance — that already lives correctly in the `go-*` skills.
