# Python for Backend Development: Constructing APIs and Server Logic for Modern Web Apps
**Authors:** Corin Halstead
**Domain:** Software Architecture & Implementation (Python, applied/practitioner level)

## Core Concepts

- **Python's `asyncio` event loop is structurally closer to Node's single-threaded event loop than to Go's M:N goroutine scheduler — and this is the single most important fact for mapping the `go-*` roster onto Python.** CPython has a Global Interpreter Lock (GIL): only one thread executes Python bytecode at a time, no matter how many OS threads exist. `asyncio` sidesteps this for I/O-bound work the same way Node's libuv event loop does — one thread, cooperative scheduling via `async`/`await`, non-blocking I/O dispatched as coroutines resume when their socket/file descriptor is ready. Where Go achieves non-blocking I/O by parking a goroutine and multiplexing thousands of them across `GOMAXPROCS` real OS threads (`go-concurrency-patterns`), Python — like Node — achieves it with exactly one thread of Python execution, cooperative rather than preemptive. This repo's already-written Node research file (`research/software-architecture/nodejs-2025-daxton.md`) reached this same "single JS thread" conclusion for a different reason (Node's runtime design); Python reaches an equivalent single-thread-for-Python-code outcome for a different, more fundamental reason (the GIL is an implementation detail of CPython itself, not an architectural choice a future CPython version can casually walk back — see Caveats). The practical consequence for this product is the same as the Node file's: I/O-bound concurrency (HTTP handlers waiting on Postgres, waiting on Redpanda, waiting on S3/Google Drive) scales well under `asyncio`; CPU-bound work (JSON transformation at scale, cryptographic hashing, and — most relevant to this product — document parsing/OCR/entity extraction) does not, and needs a different mechanism entirely (see Caveats).
- **ASGI (Asynchronous Server Gateway Interface) as the async-native successor to WSGI, and the book's central architectural fork.** WSGI (the interface Flask and classic Django are built on) is fundamentally synchronous — one worker thread/process blocks for the duration of a request. ASGI (Starlette, and FastAPI which is built on Starlette) supports `async def` route handlers natively, run under an ASGI server (`uvicorn`, optionally behind `gunicorn` as a process manager). The book treats "pick an ASGI-native framework" as the single highest-leverage decision for a Python API service in 2025, precisely because it is the only way to get `go-chi-handler`/Fastify-style non-blocking I/O concurrency out of Python at all — a WSGI app has no analog to this repo's "never block a goroutine/event-loop tick waiting on I/O" discipline; it just blocks a worker.
- **FastAPI's type-hint-driven request/response validation via Pydantic — the book's chief argument for FastAPI over Flask or Django, and a direct structural cousin of this repo's existing chi/Fastify validation pattern.** A FastAPI route declares its request body, path params, and query params as ordinary Python type hints on Pydantic `BaseModel` classes; FastAPI validates the incoming request against that model *before the handler body runs at all*, returning a structured 422 with per-field errors on failure, and derives the OpenAPI schema from the same models automatically. This is not a looser analog of `go-chi-handler`'s hand-written `validate()` method — it is the same "framework validates before the handler sees anything" strength this repo's Node research file already identified in Fastify, arrived at independently in the Python ecosystem via a different mechanism (runtime introspection of type hints, not JSON Schema authored by hand).
- **`contextvars.ContextVar` as Python's analog to `AsyncLocalStorage` — and, like Node, a real gap relative to Go's `context.Context`.** Go's `context.Context` carries cancellation, deadline, tenant id, and trace span together, implicitly, through every function that takes `ctx` as its first parameter. Python's native cancellation primitive (`asyncio.CancelledError` propagation, `asyncio.timeout()`/`asyncio.wait_for()`) carries only cancellation and a deadline — not request-scoped values. `contextvars.ContextVar` (the same primitive `AsyncLocalStorage` is implemented on top of in Node, conceptually) is the explicit, separate mechanism needed to carry tenant id and trace span across `await` boundaries. A naive Python port of `go-service-layer`'s worked examples that just swaps `ctx context.Context` for "whatever `asyncio` gives you for free" would silently lose tenant-scoping and trace propagation — the exact same correctness gap this repo's Node research file already flagged for `AsyncLocalStorage`, arrived at for the same underlying reason (no single-parameter object carries both cancellation and values).
- **Dependency Injection via FastAPI's `Depends()` system, as the book's preferred way to wire ports into handlers.** FastAPI resolves a handler's declared dependencies (a repository, a policy evaluator, an authenticated `Subject` extracted from a JWT) by calling a provider function and injecting the result — the book uses this pervasively for exactly the pattern `go-chi-handler`'s composition root and `go-service-layer`'s constructor-injected ports already establish: dependencies are explicit, typed, and constructed once, not looked up ad hoc inside a handler body.
- **`SQLAlchemy` (2.0's async engine, via `asyncpg`) as the book's default data-access story — presented as the mainstream, ORM-first choice, not the leanest one.** The book teaches SQLAlchemy's async ORM (declarative models, `AsyncSession`, relationship loading) as the default for Python API backends, with Alembic as its paired migration tool. This is a materially heavier, more "magic" default than this repo's existing `pgx`-direct, no-ORM philosophy (`go-repository-pattern`) or Node's `node-postgres` raw-driver approach (the Node research file's `node-repository-pattern` row) — see the framework/tooling recommendation below for why this repo should not follow the book's ORM-first default as-is.
- **Background tasks and worker offload via Celery (or, in more recent stacks, `arq`/`taskiq` for asyncio-native queues) as the book's mechanism for anything that must not block the request-response cycle.** The book frames this as the escape hatch for exactly the CPU-bound work this repo already treats carefully: a request handler enqueues work onto a broker-backed task queue rather than doing it inline, and a separately-scaled worker pool processes it. This maps directly onto this repo's own `data-pipeline-implementation` stage-worker pattern (a separate deployable, not a goroutine/coroutine embedded in the request-serving process) — the book arrives at the same architectural conclusion this repo already enforces for Go, independently, for Python's specific concurrency constraints.
- **`multiprocessing`/`concurrent.futures.ProcessPoolExecutor` as Python's only true CPU parallelism primitive — heavier than Go's goroutines and heavier than Node's `worker_threads`.** Because the GIL prevents multiple OS threads from running Python bytecode concurrently, Python's only way to use more than one CPU core for *pure Python* CPU-bound work is separate OS processes, each with its own interpreter and memory space, communicating via pickling/IPC (or shared memory for select data types). This is a full process, not a lightweight V8 isolate (Node's `worker_threads`) and certainly not a goroutine (Go's cheap, runtime-scheduled stack). The book is explicit that spinning up a process pool per request is far too expensive — like Node's `piscina` pattern, the idiomatic shape is a long-lived, bounded pool of worker processes, sized well below naive "one per core" defaults once per-process memory overhead is accounted for.

## Actionable Techniques

- **Name the GIL/CPU-bound risk explicitly wherever a future `python-*` skill would touch document parsing, OCR, or entity/classification work** — mirroring the exact discipline this repo's Node research file already established, and reinforcing (not superseding) `data-pipeline-implementation`'s existing narrow Python exception. A `python-event-consumer` skill must never silently port `go-concurrency-patterns`' "bounded worker pool" language onto Python as if an `asyncio.Semaphore`-bounded pool of coroutines were sufficient for CPU-bound per-message work — it is sufficient only for I/O-bound work; CPU-bound work needs a `ProcessPoolExecutor`, and that pool's size must account for full per-process memory overhead (not core count alone).
- **Require `mypy` or `pyright` as a mandatory, CI-enforced gate — not an optional nicety** — because Python's type hints, unlike TypeScript's, are erased at runtime with no compiler in the loop by default; a `python-*` roster that only validates FastAPI's request/response boundary via Pydantic while leaving internal domain code's type hints unchecked has a materially weaker static-safety story than either Go (compiler-enforced) or Node (at minimum, `tsc --noEmit` is a mandatory build step in the Node roster's own `node-npm-scripts` skill). This is a genuine, checkable gap a naive Python port would leave open by default.
- **Adopt `contextvars.ContextVar` as the explicit mechanism carrying tenant id, trace span, and deadline across `await` boundaries**, and call this out in any future `python-service-layer` skill exactly as the Node research file calls out `AsyncLocalStorage` — a handler that reads tenant/trace state from a module-level global instead of a `ContextVar` will leak state across concurrently-scheduled coroutines on the same event loop, a correctness bug with no Go equivalent.
- **Reject the book's SQLAlchemy-ORM-first default in favor of `asyncpg` used directly, with parameterized `$1`-style queries** — a closer structural match to this repo's existing `pgx`-direct philosophy than an ORM would be. `asyncpg` (notably) uses the same `$1`, `$2`... positional placeholder syntax as `pgx`, which makes the CAS-on-`version`/outbox-in-one-transaction worked examples in `go-repository-pattern` translate almost mechanically: `async with pool.acquire() as conn, conn.transaction():` replaces `pool.Begin(ctx)`/deferred rollback, but the SQL text itself barely changes. Alembic (the migration tool, decoupled from SQLAlchemy's ORM layer) is still worth adopting for the migration-runner role even while rejecting the ORM — Alembic runs perfectly well against hand-written SQL migrations with no ORM models attached.
- **Require one Python packaging/dependency tool (`uv`) and one linter/formatter (`ruff`) up front**, rather than leaving the historically fragmented Python tooling landscape (`pip`+`requirements.txt`, `pipenv`, `poetry`, `conda`, `pip-tools`) open — this repo's "pick one sanctioned default per concern" pattern (already applied to chi over gin/echo, `pgx` over `database/sql`, Vitest recommended over Jest in the Node file) argues strongly against a Python roster that leaves packaging ambiguous. `uv` is named specifically because it resolves dependencies, manages virtual environments, and runs scripts through a single fast tool, which is the closest 2025 Python equivalent to `npm`'s single-tool convenience and a better fit for this repo's frugality principle than assembling `pip` + `venv` + `pip-tools` by hand.
- **Treat CPU/ML-bound document parsing and classification as an escalation out of the request-handling/event-consuming Python service entirely, exactly as already established for Go and Node** — this is not a new rule invented for Python; it is `data-pipeline-implementation`'s existing "Python Exception" discipline, which already assumed Python-for-ML-work faces exactly this constraint. A Python-default product's API/orchestration service should treat heavy extraction work as its own separately-scaled service (worker-queue-backed, via Celery/`arq`, or the same stage-worker pattern `data-pipeline-implementation` already specifies) rather than running it inline even inside a bounded `asyncio` pool.

## Candidate New Skills/Agents

**Framework choice made first, since it shapes several skill names above: FastAPI is the sanctioned Python web framework default**, not Flask and not Django/Django REST Framework. Reasoning, held against this repo's own stated preference for "transparency over magic frameworks" (the same reasoning already given for chi over gin/echo, and for Fastify over Express/NestJS in the Node research file):

- **Flask** is the closest in spirit to `net/http`/Express alone — minimal, unopinionated, WSGI-first (synchronous by default; async support exists but is not the framework's native model). It has no built-in request/response schema validation (the Python equivalent of hand-rolling every `validate()` method chi already writes explicitly, except without Fastify's or FastAPI's option to push validation into the framework) and, being WSGI-based, does not give a Python service the non-blocking-I/O concurrency story this repo's event-driven stack depends on without bolting on an ASGI adapter as an afterthought.
- **Django (with or without Django REST Framework)** is a full, opinionated, batteries-included framework — its own ORM, migration system, admin panel, templating engine, and settings machinery — structurally in the same "magic framework" category the Node research file already named for NestJS as the reason to avoid it. Adopting Django would import a second, Django-specific migration and ORM system into a repo whose data-access philosophy (`go-repository-pattern`) already deliberately avoids ORM magic in favor of direct, parameterised SQL — a direct conflict, not just added weight.
- **FastAPI** is schema-first by construction (Pydantic models validate `body`/`path`/`query` before the handler runs at all, deriving the OpenAPI schema from the same models — a *stronger* position than chi's hand-written `validate()`, matching the strength this repo's Node research already found in Fastify), is ASGI-native from the ground up (not an afterthought retrofit the way Flask's async support is), has a single centralized exception-handler mechanism (`@app.exception_handler`) that is the direct one-to-one analog of `writeDomainError`/Fastify's `setErrorHandler`, supports dependency injection (`Depends()`) that mirrors this repo's constructor-injected-ports discipline, and is measurably competitive in throughput with Node/Fastify in independent 2024–2025 benchmark rounds (well ahead of Flask/Django) — aligning with this repo's frugality principle.

## Key Terminology

Worth considering for `skills/glossary-management/`'s Ubiquitous Language, but only once a `python-*` skill actually exists to anchor the definition against (same deferred-addition convention the Node.js research file already established):

- **Global Interpreter Lock (GIL)** — CPython's mutex ensuring only one thread executes Python bytecode at a time, regardless of thread count. The central reason Python's threading module does not provide CPU parallelism for pure-Python code, and the reason `asyncio` (cooperative, single-thread) rather than OS threads is Python's idiomatic concurrency model for I/O-bound work. Needs a precise glossary definition distinguishing it from both Go's goroutine scheduler and Node's event loop, since all three solve "how do I handle many concurrent things" through structurally different mechanisms with different failure modes.
- **ASGI (Asynchronous Server Gateway Interface)** — the async-native successor to WSGI; the interface FastAPI/Starlette implement and `uvicorn` serves. Needs a glossary entry distinguishing it from WSGI the moment `python-fastapi-handler` exists, since a product's choice of ASGI vs. WSGI framework has the same category of architectural consequence as chi vs. gin or Fastify vs. Express.
- **Event Loop (asyncio)** — worth flagging as a term that **already exists in Node's proposed glossary entry** (from `research/software-architecture/nodejs-2025-daxton.md`'s Key Terminology) and needs explicit disambiguation if a product ever runs both a Node service and a Python service: Python's `asyncio` event loop and Node's libuv-driven event loop are similar in shape (single-threaded, cooperative, non-blocking I/O) but are not the same mechanism, have different scheduling primitives (coroutines/`Future`s vs. the microtask/macrotask queue), and are not interchangeable words for "the thing that dispatches concurrent work" — exactly the kind of terminology collision `glossary-management`'s "no synonyms, no drift" rule exists to catch before two per-product overrides' artifacts start using the same word to mean subtly different things.
- **`contextvars.ContextVar`** — Python's explicit mechanism for propagating request-scoped values (tenant id, trace span) across `await` boundaries, since `asyncio`'s native cancellation carries no values. Needs a glossary disambiguation against `AsyncLocalStorage` (Node) and `context.Context` (Go) if a multi-language product's artifacts ever reference all three — same concept, three non-interchangeable names.
- **Free-Threaded Python (No-GIL build)** — the experimental CPython build (introduced as an opt-in variant starting around Python 3.13) that removes the GIL, allowing true multi-core thread parallelism for pure-Python code. Not proposed for adoption now (see Caveats) but worth a forward-looking glossary note once it matures, since it would meaningfully change the `python-async-concurrency` skill's CPU-bound-work guidance if and when it becomes the ecosystem default rather than an opt-in experiment.

## Caveats

- **The GIL is Python's most consequential and most honest caveat, and it is a stronger, more fundamental constraint than either Go's or Node's concurrency stories — this must be named plainly, not softened.** Go's goroutines give true multi-core parallelism via the runtime scheduler; Node's single-threaded event loop is a deliberate architectural choice with `worker_threads` as an escape hatch that at least shares a process. Python's GIL is neither: it is an implementation detail of CPython specifically (not the Python language) that prevents multiple OS threads from running Python bytecode concurrently no matter how the service is architected, and the only true-parallelism escape hatch (`multiprocessing`/`ProcessPoolExecutor`) is a full separate OS process — heavier than a `worker_thread`, with no shared memory by default and pickling overhead on every task boundary. **Named mitigation, in priority order, consistent with the conclusion this repo's own Go default and the Node research file already reached independently:** (1) preserve `data-pipeline-implementation`'s existing escalation discipline exactly — CPU/ML-bound document parsing and classification stay a separate, independently-scaled service regardless of which language the request-handling/event-orchestration services use, so a Python override never needs to solve this problem inline in the first place; (2) for I/O-bound work (the overwhelming majority of API request handling and event consumption), `asyncio`/FastAPI/`aiokafka`/`asyncpg` give perfectly good concurrency with no GIL contention, since the GIL is released during I/O waits; (3) where genuinely necessary, bounded CPU work inside a Python service should go through a `ProcessPoolExecutor` sized to actual per-process memory cost (not core count), never a naive `asyncio`-based "worker pool" that silently degrades to fully serial execution under real CPU load; (4) note, without relying on it as a default plan, that some C-extension libraries (NumPy, most cryptography bindings) release the GIL internally during their compute-heavy calls, meaning specific well-chosen libraries can achieve real thread parallelism even under the GIL — this is a library-specific exception, not a general escape hatch pure Python code can lean on; (5) the experimental free-threaded ("no-GIL") CPython build is a real, active development (opt-in since roughly Python 3.13) that could eventually change this whole calculus, but it is not mature or ecosystem-default enough in 2025–2026 to be this repo's frugal, boring-is-correct standard — track it, do not adopt it yet.
- **Python's type-hint story is honestly weaker than either alternate's default, and this must not be glossed over.** TypeScript at least fails a mandatory `tsc` build step on a type error; Go's compiler enforces types unconditionally. Python's type hints are pure runtime-erased annotations unless a separate tool (`mypy`/`pyright`) is run and its failures are treated as build-breaking — FastAPI's Pydantic validation only covers the request/response *boundary*, not internal domain code. A `python-*` roster that treats `mypy`/`pyright` as optional rather than a non-negotiable CI gate would quietly ship a materially weaker static-safety guarantee than either of this repo's other two sanctioned backend languages — this is named as a required mitigation in Actionable Techniques, not left as an aside.
- **Packaging and deployment maturity lag Go's single static binary, similarly to (arguably slightly worse than) Node's `node_modules` situation.** Python has no static-linking equivalent; a deployed Python service ships the interpreter plus every dependency's installed footprint, and the historically fragmented packaging landscape (`pip`+`requirements.txt`, `pipenv`, `poetry`, `conda`, `pip-tools`) is more genuinely unsettled than Node's `npm`+`package-lock.json` convention, which is itself already a relatively converged default. `uv` is recommended specifically because it is the closest 2025–2026 answer to a single, fast, converged tool — but flag honestly that this is a less battle-tested convention across the wider Python ecosystem than `npm` is for Node or `go.mod` is for Go, and revisit if `uv`'s ecosystem position shifts.
- **This file proposes skills; it does not decide agent ownership, and that decision should not be inferred from this document alone** — identical caveat to the Node.js research file, for the identical reason: whether `backend-engineer` becomes stack-conditional across three languages or Python (like Node) warrants its own agent has consequences for `CLAUDE.md`'s "agents never overlap" rule that this research file is not positioned to resolve.
- **Scope discipline: this file mirrors the `go-*`/`node-*` rosters one-to-one on purpose and resists proposing Django-flavored, Flask-flavored, or GraphQL-flavored alternatives "while we're here."** `CLAUDE.md`'s frugality principle and "one sanctioned default per concern" pattern both argue against a wider net; the 24-skill roster above is scoped as a literal mirror of what already exists and works for Go and the already-proposed Node roster, not a fresh, independent Python architecture exercise.
- **The reconciliation with `data-pipeline-implementation`'s existing Python exception (see Candidate New Skills/Agents) is the most important scope clarification in this file and should not be read as either roster superseding the other.** The narrow ML/OCR exception and this file's general-purpose backend roster answer different questions (which library does this one stage need vs. which language does this whole product's backend use) and can coexist in the same repository, the same product, or even — in principle — the same service's dependency tree without contradiction.
