# The Pragmatic Programmer
**Authors:** Andrew Hunt and David Thomas
**Domain:** Software Architecture & Implementation

## Core Concepts

- **Tracer Bullets (Ch. 1, "The Cat Ate My Source Code" section / Topic 4).** Before building any part of a system to full fidelity, build a thin, working, end-to-end slice through every layer it will eventually need — client to server to storage and back — deliberately simplified but *real* (real network calls, real deploy path, real data format), not a throwaway mock. The point is not "a prototype to be discarded" (the book is explicit that tracer bullets are working code, meant to be built on, unlike a disposable prototype) but a vertical slice that proves the architecture connects end-to-end and gives every stakeholder something running to react to early, before the team commits deeply to any one layer's internals. This is a *sequencing* discipline distinct from a design pattern — it answers "what do we build first," not "how do we structure a class."
- **"Programming by Coincidence" (Ch. 4, Topic 33).** Code that happens to work — because of undocumented behavior, an untested assumption about ordering, an API's accidental side effect, or a lucky timing window — is not code that is *known* to work. The book's test: could you explain, from documentation or a deliberate design decision, *why* this works — or did you just try it and it passed? Relying on coincidence means the next refactor, library upgrade, or optimizer change can silently break something that was never actually guaranteed. The countermeasure is to program deliberately: know why code works, document assumptions explicitly, and prove them (a test, an assertion) rather than trust them.
- **Design by Contract (Ch. 4, Topic 32, credited to Bertrand Meyer/Eiffel).** Every function or module can state, explicitly, three things: a **precondition** (what must be true for the caller to legitimately call it), a **postcondition** (what the routine guarantees to be true when it returns, given the precondition held), and a **class invariant** (what remains true of an object's state before and after every one of its methods runs, for the object's entire lifetime). The discipline's power is in making the contract explicit and, ideally, checked — a violated precondition is the *caller's* bug, a violated postcondition is the *callee's* bug, and the distinction tells you immediately where to look. This is a per-function/per-method contract about inputs and outputs, distinct from — and a finer grain than — a DDD Aggregate's invariant, which is a rule about an Aggregate's *internal state* holding across its whole cluster of objects at rest (see Applies To below for how this repo's `aggregate-design` invariant and `access-control-model`'s Domain Primitive already cover adjacent but not identical ground).
- **"Dead Programs Tell No Lies" — crash early (Ch. 4, Topic 34).** A program that detects an impossible state should stop immediately and loudly, not attempt to soldier on with corrupted assumptions. Limping along after an internal invariant is violated turns one clear failure into a cascade of confusing downstream symptoms, discovered far from the actual bug. This is the same philosophy — bugs vs. failures, `panic` for the unrecoverable — that `go-error-handling` already implements natively as an idiomatic Go boundary rule (see Caveats; this is confirmation, not a gap).
- **Orthogonality (Ch. 2, Topic 9).** Two components are orthogonal if a change to one has no effect on the other — the geometric metaphor (moving along one axis doesn't move you along another) for eliminating hidden coupling. The book's diagnostic questions are practical, not architectural jargon: "if I dramatically change the requirements behind one module, how many other modules are affected?" and "if I unplug this piece and rewrote it from scratch, would the rest of the system still work?" This is the same underlying goal as SOLID's Single Responsibility/Interface Segregation and Clean Architecture's Dependency Rule, expressed as a design *test* to run against a specific pair of modules rather than a principle to state in the abstract.
- **DRY — Don't Repeat Yourself (Ch. 3, Topic 11).** "Every piece of knowledge must have a single, unambiguous, authoritative representation within a system." The book's DRY is broader than "don't copy-paste code" — it includes duplicated *knowledge* expressed different ways (a business rule encoded once in validation logic and again in a database constraint; a value hardcoded in two config files that must always agree). Already a canonical glossary term in this repo (see Applies To).
- **The Broken Windows Theory (Ch. 1, Topic 3).** Borrowed directly from the criminology theory of urban decay: one broken window left unrepaired signals that nobody cares, inviting more damage: quality decays *fastest* once the first crack is visible and tolerated. The book's prescription is the same "Boy Scout Rule" ethos this cluster's Clean Code research already documented and proposed as a per-PR checklist item — worth noting here only because Pragmatic Programmer is this idea's actual origin point in the software-craft literature (Clean Code popularized the "Boy Scout Rule" phrasing; this book coined "broken windows" for the underlying decay dynamic a decade earlier).

## Actionable Techniques

- For the first service built inside a newly-decomposed Bounded Context, build (and demo) a thin end-to-end tracer-bullet slice — one Command through the real HTTP handler, real application handler, real Aggregate, real repository, real Postgres row, real emitted event — *before* the full test-first build-out of every Command/Query in that service, so integration risk between the four layers surfaces in days, not at the end of the phase.
- When a function's correctness depends on the caller having already done something (loaded state in a particular order, called setup first, passed a non-empty value), state that as an explicit precondition in the function's doc comment, and where the violation would otherwise silently produce wrong results rather than an error, add a guard clause that fails loudly (`panic` per `go-error-handling`'s existing "invariant violated" row) rather than proceeding on an unchecked assumption.
- When reviewing why a piece of code "happens to work," require a one-line justification that names the documented API guarantee or the enforced invariant behind it; if no such justification exists, treat it as programming-by-coincidence and add either a test that pins the assumption or a fix that removes the reliance on it.
- Run the orthogonality question ("if I rewrote this component from scratch, would every dependent still work unmodified through its existing interface?") as a design-review question for any two components with a suspiciously wide or frequently-changing shared interface — a concrete diagnostic to add to code review, not a new document.

## Candidate New Skills/Agents

No new skill or agent is warranted. Both genuinely distinct findings from this book are narrow, targeted edits to existing artifacts, not new content types:

1. **Tracer Bullets → `commands/sdlc-implement.md` sequencing.** A one-step addition (an optional early walking-skeleton checkpoint before the full test-first fan-out) to an existing command's prompt body — not a new skill. The book's own emphasis that a tracer bullet is *working code*, not a disposable prototype, means this doesn't create a new "prototyping" artifact type either; it's a sequencing note within the phase already owned by `sdlc-implement`.
2. **Design by Contract → `go-error-handling`'s precondition documentation habit.** A doc-comment convention addition to an existing skill, not a new skill — the panic/recover mechanics that would enforce a violated precondition already exist and are correct; what's missing is only the habit of writing the precondition down before the panic path is needed.

Both were sized against the MVP-dependency-pull threshold this campaign has used elsewhere (only pull in new artifacts when a real, currently-unserved dependent need exists) and neither clears it as a standalone artifact.

## Key Terminology

Worth considering for `skills/glossary-management/`'s Ubiquitous Language, if implementation-level (not just domain-level) vocabulary is ever added:

- **Tracer Bullet** — a thin, real, end-to-end working slice through every architectural layer, built first to prove the layers connect, distinct from a throwaway prototype (which is explicitly meant to be discarded). Not currently named anywhere in the repo; `sdlc-implement.md` has no equivalent sequencing step today.
- **Design by Contract** — precondition/postcondition/class-invariant discipline at the function/method level. Distinct from a DDD Aggregate's invariant (state consistency across a whole cluster, checked continuously) and from a Domain Primitive (construction-time self-validation of a single value) — all three are related "make illegal states impossible or immediately loud" ideas operating at different grains, and would need careful mutual disambiguation if more than one is ever formalized in the glossary.
- **Programming by Coincidence** — code whose correctness rests on an undocumented, unproven assumption rather than a known guarantee. Not currently named in the repo; the underlying discipline (don't trust what you haven't verified) is already practiced correctly in `test-fixture-design`'s Hermetic Principle and `go-unit-test`'s fuzzing guidance, just not named as this book's specific term.
- **Orthogonality** — already DRY's and SOLID's underlying goal (independent, non-overlapping responsibility), expressed here as a concrete review question rather than a new principle. Not worth a distinct glossary entry — it would sit redundantly beside Single Responsibility and Bounded Context, which already cover the same ground for this repo's purposes.

## Caveats

- **This is the broadest, most aphoristic book in the cluster, and most of its ground is already covered by the two Martin books already in this cluster.** DRY is already a canonical glossary term (`skills/glossary-management/references/ubiquitous-language.md`, Design Principles section, line 53). The Law of Demeter, decoupling, and "do one thing" are already covered thoroughly by `clean-code-martin.md`'s Core Concepts and Key Terminology (which already flags Law of Demeter as a glossary candidate — not repeated here to avoid a duplicate proposal). The Boy Scout Rule / code-decay-prevention discipline is already proposed as an actionable technique in the same file. Treat this document as supplying only the handful of ideas genuinely absent from that prior coverage — tracer bullets and Design by Contract — rather than a second pass over shared ground.
- **"Dead Programs Tell No Lies" and the config-as-data chapters (Ch. 4's "Assertive Programming," and the book's advocacy for externalized, metadata-driven configuration) are already correctly implemented, not gaps.** `go-error-handling`'s panic-for-unrecoverable-invariants rule is the idiomatic Go version of crash-early; `environment-config`'s Configuration Taxonomy (build-time/deploy-time/runtime-flag), config/secret split, and Promotion Invariance Check are a considerably more rigorous, automatically-enforced realization of "externalize configuration, don't hardcode it" than the book itself prescribes (the book's chapter predates GitOps/values-file conventions entirely and argues only for pulling constants out of code — this repo's actual practice already exceeds that bar). No action follows from re-reading these chapters against these skills.
- **Reversibility is real but already better served by Richards & Ford's framing, already cited in `adr-authoring`.** This book's version of reversibility ("prefer decisions that are easy to undo") is a looser heuristic than Fundamentals of Software Architecture's precise "architecture is the stuff that's hard to change" — the latter is already the cited authority in this skill's Purpose section, and duplicating a second, vaguer reversibility framing on top of it would blur rather than sharpen the existing citation.
- **The book's specific technology chapters (Ch. 5's shell/Unix-pipe advocacy, Ch. 8's refactoring-tool-of-the-2000s references, the metaprogramming and DSL chapters) are dated and don't translate to this repo's Go/React/GitOps stack** — skip them as source material; the durable ideas (tracer bullets, DbC, programming-by-coincidence, orthogonality, DRY, broken windows) are the ones carried forward above.
- **Chapter 4's "Design by Contract" material is written against Eiffel's native, language-level contract syntax (`require`/`ensure`/`invariant` keywords enforced by the compiler/runtime).** Go has no equivalent language feature — the actionable technique above (doc-comment preconditions + a guard-clause panic) is this document's own idiomatic-Go adaptation of the underlying discipline, not a literal port of Eiffel's mechanism.
