# System Design Interview — An Insider's Guide
**Author:** Alex Xu
**Domain:** Software Architecture — System Design

## Core Concepts

- **Back-of-the-envelope estimation (Ch. 1).** Before any design decision, Xu insists on quantifying scale: daily active users → queries per second (average and peak, typically peak = 2–3x average) → storage growth per day/year → bandwidth. The purpose isn't precision — it's forcing a design conversation to be grounded in a number instead of an adjective ("it needs to scale" is not an estimate). This directly parallels — and is less rigorous than — Kleppmann's **load parameter** framing already cited in this repo's `designing-data-intensive-applications-kleppmann.md` research file; Xu's version is the interview-shorthand version of the same idea, aimed at picking an order-of-magnitude architecture (single server vs. horizontally-scaled fleet vs. sharded-and-CDN'd) rather than at precise capacity planning.
- **The "clarify requirements first" interview framework (Ch. 1).** Xu's four-step process — (1) understand the problem and establish scope (functional requirements, out-of-scope explicitly stated), (2) propose high-level design and get buy-in, (3) design deep-dive on 2–3 components the interviewer cares about, (4) wrap up (bottlenecks, trade-offs, alternatives) — is a scoping discipline more than an architecture technique. Its most transferable idea: **explicitly declare what is out of scope**, not just what is in scope, before drawing any boxes.
- **The standard scaling toolkit (Ch. 1, and reused in every subsequent chapter).** A fixed vocabulary of building blocks Xu layers on top of a single server as load grows: **load balancer** (distributes requests across a stateless web tier; enables horizontal scaling and failover), **database replication** (single leader + read replicas — reads scale horizontally, writes stay on the leader), **cache** (in front of the database, typically read-through or cache-aside, to absorb read-heavy load and reduce database round trips), **CDN** (static/media assets served from edge locations near the user, not from origin), **stateless web tier + sticky-session avoidance** (session state moves to a shared store like Redis so any web server can serve any request — a precondition for horizontal scaling and for load balancers to work without session affinity), **message queue** (decouples producer and consumer, absorbs bursts, enables async processing), **data sharding** (once a single database's write throughput or storage is the bottleneck, partition by a shard key — Xu discusses hash-based sharding and the resulting hot-shard and resharding problems in less formal terms than Kleppmann's Ch. 6 but with the same substance), and **multiple data centers** (geographic distribution for latency and disaster recovery, with a routing layer — GeoDNS — sending users to the nearest healthy region).
- **Rate limiting algorithms (Ch. 4 — "Design a Rate Limiter").** Xu enumerates five concrete algorithms with real trade-offs, not just "add a rate limiter": **Token Bucket** (a bucket refills at a fixed rate up to a capacity; each request consumes a token; simple, memory-efficient, allows bursts up to bucket size), **Leaky Bucket** (fixed-rate outflow via a FIFO queue; smooths bursts into a constant rate but a full queue drops/rejects new requests), **Fixed Window Counter** (count requests per fixed time window, e.g. per-minute; simple but allows up to 2x the limit at window boundaries — the classic edge-burst flaw), **Sliding Window Log** (store a timestamp per request, count requests within the trailing window; accurate but memory-heavy at scale), and **Sliding Window Counter** (a weighted average of the current and previous fixed window's counts; approximates the sliding log's accuracy at fixed-window's memory cost — Xu's recommended default for most cases). He also covers where to enforce it (API gateway vs. per-service middleware), what to return on rejection (HTTP 429 plus `X-RateLimit-*` headers), and that a **distributed rate limiter** needs a shared, fast store (Redis, using `INCR`+`EXPIRE` or a Lua script for atomicity) so limits hold across multiple service instances, not just one.
- **Consistent hashing (Ch. 5, and the mechanism underlying Ch. 6's key-value store).** The specific technique for distributing keys across a *changing* set of nodes without a hash-mod-N remap storm: nodes and keys are hashed onto a ring; a key belongs to the first node clockwise from its hash position. Adding or removing one node only remaps the keys between it and its neighbor — not the whole keyspace. **Virtual nodes** (each physical node gets many positions on the ring) solve the uneven-load problem a small number of physical nodes would otherwise create. This is Xu's answer to the same rebalancing problem Kleppmann's Ch. 6 names (hash-mod-N reshuffles almost everything) — same conclusion, less formal treatment.
- **CAP theorem and the shared/leader-replica database default (Ch. 6, "Design a Key-Value Store").** Xu uses CAP (Consistency, Availability, Partition tolerance — pick two when a network partition occurs) as the framing device for choosing between strongly-consistent single-leader systems and AP quorum-based systems (Dynamo-style, using `N`/`W`/`R` quorum parameters — same vocabulary Kleppmann uses in more depth). Xu's treatment is intentionally shallower — it is a decision heuristic for an interview, not a rigorous distributed-systems argument.
- **Unique ID generation at scale (Ch. 7).** Auto-increment doesn't work across shards/multi-master setups. Xu presents the **Twitter Snowflake** approach: a 64-bit ID composed of a sign bit, a timestamp (milliseconds since a custom epoch), a datacenter/machine ID, and a per-millisecond sequence number — sortable by time, globally unique without coordination between nodes.
- **Notification/pub-sub fan-out systems, news feed generation, and the pull vs. push (fan-out-on-write vs. fan-out-on-read) trade-off (Ch. 11, 12).** For a feed/notification system, **fan-out-on-write** (pre-compute and push the feed to every follower's cache at write time) gives fast reads but is expensive for celebrities with millions of followers ("hot user" write amplification); **fan-out-on-read** (compute the feed at read time by merging each followee's posts) is cheap to write but slow to read. Xu's recommended hybrid: fan-out-on-write for ordinary users, fan-out-on-read (merged at request time) for celebrity accounts above a follower threshold.
- **Proximity/geospatial indexing (Ch. 13, "Design Yelp/Proximity Service").** Geohashing and Quadtrees are presented as the two standard approaches to "find things near this location" without scanning every row — relevant any time a design needs spatial queries.
- **API Gateway as the single entry point (recurring across chapters, esp. Ch. 4 and the microservices-adjacent chapters).** The gateway is where cross-cutting concerns — authentication, rate limiting, request routing, TLS termination, logging — live, so that individual services don't each reimplement them.

## Actionable Techniques

- State a back-of-envelope scale estimate (DAU → QPS average/peak → storage/day → bandwidth) before proposing an architecture, and revisit it whenever an NFR changes — a design should name the load number it was sized for.
- Before drawing any diagram, write down what is explicitly out of scope, not only what is in scope — an unstated scope boundary is a future rework.
- When a synchronous API needs abuse/overload protection, name the specific rate-limiting algorithm chosen (Token Bucket, Leaky Bucket, Fixed/Sliding Window Counter, Sliding Window Log) and *why* — "add a rate limiter" is not a design decision, the algorithm's burst-tolerance vs. memory-cost trade-off is.
- Enforce rate limits at the API Gateway/edge for cross-cutting protection, and additionally at the service level for endpoint-specific limits — document both layers, not just one.
- For any distributed rate limiter, name the shared store (Redis) and the atomicity mechanism (`INCR`+`EXPIRE`, or a Lua script) — a rate limiter with a race condition between check-and-increment is not a rate limiter.
- When a component distributes keys/requests across a variable-size pool of nodes, name whether it uses consistent hashing with virtual nodes, and if not, name what rebalancing cost is being accepted instead.
- For any entity requiring a globally unique, sortable-by-time identifier issued without central coordination, consider a Snowflake-style composite ID (timestamp + node ID + sequence) rather than a database auto-increment or a raw UUID (which sacrifices sortability).
- For any feed/fan-out design, state explicitly whether it's fan-out-on-write, fan-out-on-read, or a hybrid with a stated threshold — and name the "hot key" (celebrity/high-fan-out entity) failure mode the choice is protecting against.
- Put authentication, rate limiting, and routing at the API Gateway; keep domain/business logic out of it — this is the same rule this repo's own `container-diagram` anti-patterns table already states ("Business logic in the gateway").

## Key Terminology

Worth considering for `skills/glossary-management/`'s Ubiquitous Language, if genuinely domain-relevant:

- **Rate Limiter** and its algorithm names — **Token Bucket**, **Leaky Bucket**, **Fixed Window Counter**, **Sliding Window Counter**, **Sliding Window Log** — precise names currently absent from `integration-design`'s resilience vocabulary (which already has Circuit Breaker, Retry and Backoff, Bulkhead named precisely).
- **Consistent Hashing** and **Virtual Node** — the mechanism this repo would reach for if the tenant registry, a cache tier, or any pool of stateful nodes ever needs to scale its membership without a full remap.
- **CDN (Content Delivery Network)** — not currently in `CLAUDE.md`'s Tech Stack Defaults table or Ubiquitous Language platform-terms list, despite the React SPA in `container-diagram`'s worked example being exactly the kind of static-asset workload a CDN serves.
- **Read Replica** — the specific term for a database replica serving read traffic while writes go to the leader; distinct from the repo's existing `Eventual Consistency` glossary term (which names the consequence, not this specific mechanism).

## Caveats

- **This book is written for interview prep at hypothetical, hyperscale (FAANG-adjacent) traffic levels** — Xu's worked examples (design Twitter, design YouTube, design a URL shortener at billions of requests/day) assume traffic and team scale far beyond this repo's actual first product. Treat the *vocabulary and decision framework* (name your load, name your algorithm, name your trade-off) as the transferable part; do not import the assumption that every service needs a CDN, a sharded database, and multiple data centers from day one.
- **This repo's physical multi-tenancy is the opposite default from this book's shared-infrastructure-with-sharding assumption**, as noted in the Applies To table above — apply Xu's scaling toolkit *within* a tenant's dedicated infrastructure only, not as a reason to reconsider the physical isolation model itself.
- **Frugal, open-source-first budget constraint governs which pieces of the toolkit are justified.** Xu's book treats CDN, multiple data centers, and dedicated caching clusters as default best practice at scale; this repo's `CLAUDE.md` Budget and Frugality section requires every added dependency (a CDN vendor, a Redis cluster, a second data center) to justify itself against cost, and any paid-service choice needs Shafi's explicit approval — these are not automatic adoptions just because the book recommends them at hyperscale.
- **Xu's CAP-theorem and quorum treatment (Ch. 6) is intentionally shallow** compared to this repo's existing Kleppmann research file (`designing-data-intensive-applications-kleppmann.md`), which covers the same ground (leaderless replication, quorums, hinted handoff) in more rigorous and more directly applicable depth for this repo's actual PostgreSQL-based, single-leader-per-tenant stack. Where the two books overlap, prefer Kleppmann's treatment for this repo's design decisions; Xu's version is useful mainly as the more approachable framing for a PM-reviewable explanation.
- **Deployment/rollout strategy is entirely out of this book's scope.** Nothing in Xu's book informs `canary-deployment` or `blue-green-deployment` — those skills are already deep and correctly sourced from SRE/progressive-delivery practice, not system-design-interview material. Do not force a connection there.
