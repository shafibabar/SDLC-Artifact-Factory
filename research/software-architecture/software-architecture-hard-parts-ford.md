# Software Architecture: The Hard Parts
**Authors:** Neal Ford, Mark Richards, Pramod Sadalage, Zhamak Dehghani
**Domain:** Software Architecture — Distributed Systems Trade-offs

## Core Concepts

- **"It depends" as a discipline, not an evasion.** The book's opening move (shared with *Fundamentals of Software Architecture*, already summarized in this repo's `fundamentals-of-software-architecture-richards-ford.md`) is that distributed-systems architecture has no universal best practices — only trade-offs made explicit and defensible for a specific context. Where *Fundamentals* applies this to style selection, *Hard Parts* applies it specifically to the problems that appear only once you've already decided to decompose a system: how big should each service be, who owns which data, and how do multi-service transactions actually complete. This repo has already made the decomposition decision (event-driven microservices, one-DB-per-service, physical multi-tenancy) — this book's trade-off discipline is for validating and refining that decomposition, not re-litigating whether to decompose at all.
- **Static coupling vs. dynamic coupling.** The book separates two independent axes of service coupling. **Static coupling** is about the contract a service exposes at rest — how strict or loose it is (a strict, typed contract like gRPC/Protobuf is more coupled than a loose one like JSON-over-REST with tolerant-reader parsing; GraphQL sits in between). **Dynamic coupling** is about what happens at runtime between services in a specific *workflow* and has three independent dimensions: **Communication** (synchronous vs. asynchronous), **Consistency** (atomic vs. eventual), and **Coordination** (orchestrated vs. choreographed). This three-dimensional framing is more granular than a single "sync vs. async" or "choreography vs. orchestration" toggle — it treats communication style, consistency guarantee, and coordination style as three decisions that can each be made independently, and every combination has a name (see Saga Taxonomy below). `integration-design`'s sync/async table and `event-driven-patterns`' choreography/orchestration section each capture one of these three dimensions well, but neither states all three as orthogonal axes, and neither states consistency (atomic vs. eventual) as its own explicit decision separate from communication style.
- **The Saga Pattern Taxonomy — eight named combinations.** Crossing the three dynamic-coupling dimensions (communication × consistency × coordination) yields eight saga archetypes, each given a memorable name in the book: **Epic Saga** (sync, atomic, orchestrated — behaves like a distributed monolith; high coupling, low team independence, but simple to reason about), **Phone Tag Saga** (sync, atomic, choreographed — every service must know the next service to call; extremely brittle, generally avoided), **Fairy Tale Saga** (sync, eventual, orchestrated), **Time Travel Saga** (sync, eventual, choreographed), **Fantasy Fiction Saga** (async, atomic, orchestrated), **Horror Story** (async, atomic, choreographed — the worst combination: no central coordinator *and* an attempt at atomicity across async calls; the book explicitly recommends avoiding this one), **Parallel Saga** (async, eventual, orchestrated — the most commonly recommended shape for complex multi-step business processes: a persistent orchestrator directing async, eventually-consistent steps), and **Anthology Saga** (async, eventual, choreographed — the most scalable and decoupled shape, appropriate when steps are genuinely independent reactions with no need for centralized visibility). `event-driven-patterns`' Saga section already describes something very close to a Parallel Saga (async, orchestrated, persisted state machine) as its default recommendation for complex flows, and its Choreography-based Saga example is close to an Anthology Saga — but it never names the consistency dimension separately from coordination, and never surfaces the other six combinations as named, deliberately-rejected alternatives.
- **Granularity disintegrators — reasons to split a service further.** The book gives a checklist of forces that push toward *smaller* granularity: service scope and function (single, cohesive purpose), code volatility (parts that change at very different rates should not ship together), scalability/throughput (a hot path needs independent scaling), fault tolerance (a component whose failure should not take down its neighbors), security (data or operations requiring stricter isolation than their neighbors), and extensibility (a plug-in surface that needs to evolve without touching the core). This is a genuinely different lens than DDD's Bounded Context boundary (which is about *language and model* cohesion) — granularity disintegrators are about *operational and non-functional* cohesion, and a Bounded Context can still be the wrong *size* even when its language boundary is correct.
- **Granularity integrators — reasons to merge services back together.** The counter-checklist: database transactions (two "services" that must commit together in one ACID transaction are really one service wearing two hats), workflow/choreography overhead (two services so chatty that the network calls between them dominate the design — the "distributed monolith" smell), shared code (a large amount of tightly-coupled shared logic that would otherwise be duplicated or extracted into a fragile shared library), and data relationships (foreign-key-like relationships so tight that splitting them creates constant joins-over-the-network). The book frames granularity as a continuous balancing act between these two checklists, not a one-time decision — a real, useful tool for *stress-testing* an already-drawn Context Map's service boundaries rather than drawing them from scratch.
- **Data ownership patterns for cross-service data access.** Once services are split, the hard problem is what happens when Service B needs data that Service A owns. The book catalogs concrete patterns rather than leaving this as "just call the API": **Single Owner** (the default — one service owns a table, everyone else asks it), **Common/Joint Ownership** (multiple services write the same data — the book treats this as something to avoid or heavily govern, similar in spirit to a Shared Kernel), **Data Domain** (several closely related services share one schema under an explicit joint-governance agreement — again structurally similar to a Shared Kernel, but for data rather than code), **Delegate** (the consumer forwards the read/write to the owning service rather than caching or replicating the data locally), and **Service Consolidation** (if the coupling between two services' data is too tight to cleanly separate, merge the services — an explicit admission that the granularity was wrong). This is the book's most directly reusable content for `data-model-design` and `canonical-data-model`, both of which already assert "one database per service" and "cross-aggregate refs by ID only" as rules but do not name the pattern menu for what a consumer does *next* when it needs another Aggregate's data.
- **Contracts and stamp coupling.** The book distinguishes *strict* contracts (a typed schema like Protobuf/gRPC — safer at compile time but more tightly coupling any producer/consumer pair together) from *loose* contracts (untyped or semi-typed, like plain JSON with tolerant-reader parsing — more resilient to independent evolution but riskier at runtime). Within this, **stamp coupling** is the specific failure mode of passing a large, shared data structure across a service boundary when the consumer only needs a few of its fields — the consumer becomes implicitly coupled to every field in the structure, not just the ones it reads, and any change to the unused fields can still break or need to redeploy the consumer. This maps directly onto a real risk in `canonical-data-model`: a canonical `Person` entity carrying every attribute every consumer might ever want is a stamp-coupling risk if consumers are expected to accept the whole shape rather than a purpose-built projection of it.
- **Managing distributed transactions and eventual consistency without 2PC.** The book explicitly rules out two-phase commit / distributed XA transactions across service boundaries as a viable pattern (availability and scalability cost is too high) and instead catalogs how eventual consistency is actually achieved: **background synchronization** (a batch or scheduled job reconciles data between two owners after the fact), **orchestrated request-based** (a coordinator calls each service in turn as part of the request, similar to an orchestrated Saga but for read-side consistency rather than a business workflow), and **event-based** (services publish events; consumers apply them asynchronously to their own projections — this is exactly what this repo's Read Model / graph-projection design already does). The book is explicit that eventual consistency is not a single mechanism but a menu with different staleness/complexity trade-offs — a distinction `data-model-design`'s "graph is a projection, rebuildable from events" already implements as the event-based pattern, without naming the other two options or stating why event-based was chosen over them.

## Actionable Techniques

- For any multi-step business process already modeled as a Saga (in `event-driven-patterns` or the Integration Inventory), classify it explicitly against the three dynamic-coupling dimensions (communication: sync/async; consistency: atomic/eventual; coordination: orchestrated/choreographed) and name which of the eight saga archetypes it is — not just "choreography" or "orchestration" in isolation.
- Treat "Horror Story" (async communication + attempted atomic consistency + choreographed coordination) as a named anti-pattern to check every choreographed flow against: if a choreographed flow's steps implicitly assume all-or-nothing completion, either accept eventual consistency explicitly or move to an orchestrated (Parallel Saga) shape.
- Before adding a new service or splitting an existing one, run both checklists (granularity disintegrators: scope, volatility, throughput, fault tolerance, security, extensibility; granularity integrators: shared transactions, workflow chattiness, shared code, data relationships) against the proposed boundary and record which forces are pulling which way — this becomes the ADR's trade-off section rather than an unstated assumption.
- When a Read Model, Golden Record, or canonical entity needs data owned by another Bounded Context, explicitly name which data-ownership pattern is in play (Single Owner + Delegate, Data Domain, or — as a red flag requiring escalation — Common/Joint Ownership or a candidate for Service Consolidation) rather than leaving the cross-service data path implicit.
- Audit every canonical entity and every OHS/Published Language payload for stamp coupling: does every field exist because a specific, named consumer needs it, or is the shape a superset built "just in case"? Unused fields in a widely-consumed contract are stamp coupling waiting to break someone's deploy.
- When choosing eventual-consistency mechanics for a Read Model or projection, name which of the three patterns is used (background sync, orchestrated request-based, or event-based) and why the other two were rejected — this is a natural addition to an ADR wherever a projection or Golden Record is introduced.
- For every Context Map relationship, add a static-coupling classification (strict-contract vs. loose-contract) alongside the existing nine-pattern classification — two relationships can both be "Customer/Supplier" yet have very different coupling risk depending on whether the contract is gRPC/Protobuf or tolerant-reader JSON.

## Key Terminology

Worth considering for `skills/glossary-management/`'s Ubiquitous Language:

- **Saga** — used extensively already in `event-driven-patterns/SKILL.md` and `enterprise-architect.md` ("Saga Orchestrator," "Saga pattern"), but **`Saga` does not currently appear in `CLAUDE.md`'s Ubiquitous Language glossary list** (which has `Transactional Outbox`, `Dead Letter Queue`, `Idempotency`, `Event Choreography`, `Eventual Consistency`, `Change Data Capture`, `Circuit Breaker`, `Retry and Backoff` but no `Saga` or `Compensating Transaction`). Given how central Sagas already are to the plugin's mandated event-driven default, this is a genuine, low-risk glossary gap worth closing regardless of any further book adoption.
- **Compensating Transaction** — same gap as above: `event-driven-patterns` builds an entire compensation-rules section on this term without it being canonical glossary vocabulary.
- **Static Coupling / Dynamic Coupling** — the book's precise vocabulary for contract-strictness coupling vs. runtime-workflow coupling. Genuinely useful if the "Actionable Techniques" above (contract-strictness classification per integration) are adopted; otherwise skip — don't add glossary terms for concepts no skill actually uses yet.
- **Stamp Coupling** — a specific, well-known coupling failure mode (a consumer implicitly bound to unused fields of a shared data structure) that is directly actionable for `canonical-data-model` if that skill adopts the projection-per-consumer check above.
- **Architectural Quantum** — the book's term for an independently deployable unit with its own bounded data and synchronous dynamic coupling internally; related to but not identical to this repo's "service" or "Bounded Context." Only worth adding if a future skill needs to distinguish "independently deployable" from "one Bounded Context" (they usually, but not always, coincide) — not urgent enough to add speculatively.

## Caveats

- **This repo has already decomposed its system; don't re-run the book's decomposition process from scratch.** The book spends much of its middle section walking through *how* to pull apart a monolith into services (identifying component seams, then applying the granularity checklists). This repo's Bounded Contexts are already derived from Event Storming and DDD (per `CLAUDE.md`'s non-negotiable methodology), and its service boundaries already exist. Use the granularity disintegrator/integrator checklists as a **validation pass** against an already-drawn Context Map or Container Diagram — "does this boundary still make sense operationally, given its data-transaction needs, chattiness, and isolation requirements?" — not as a from-scratch decomposition exercise that could contradict the domain-modeler's DDD boundaries.
- **The saga taxonomy's whimsical names (Epic Saga, Horror Story, Fantasy Fiction Saga, etc.) are memorable in a book but are not obviously PM-reviewable artifact language.** Per `CLAUDE.md`'s Artifact Standards (reviewable by Shafi without IDE tooling, canonical glossary terms only), if this taxonomy is adopted into a skill, the three underlying dimensions (communication/consistency/coordination) should be the primary classification surface in artifacts — the cute names can appear as a parenthetical cross-reference for quick recognition, not as the artifact's load-bearing vocabulary.
- **Multi-tenancy is genuinely outside this book's scope.** Unlike the other five skills, `multi-tenancy-design`'s connection to this book is analogical (granularity-disintegrator language explaining an isolation decision already made) rather than a direct application of book content. Resist manufacturing a bigger connection than exists — this book is not a source for physical-multi-tenancy design guidance.
- **Data-ownership patterns (Data Domain, Common/Joint Ownership) look structurally similar to DDD's Shared Kernel — treat them as reinforcing vocabulary, not a competing model.** This repo's `context-map-patterns` already owns Shared Kernel's team-governance semantics thoroughly; the book's data-specific patterns should be introduced (if at all) as "the data-layer analog of Shared Kernel," explicitly cross-referenced, rather than as a second, parallel governance model that could drift out of sync with the existing one.
- **The book's examples (Sysops Squad) are Java/Spring-and-REST-centric and pre-date this repo's Go/pgx/Redpanda stack.** As with *Fundamentals of Software Architecture*, treat this book as a source of coupling/trade-off vocabulary and pattern names, not of Go-specific implementation guidance — the Go-specific mechanics already live correctly in `data-model-design`, `event-driven-patterns`, and the `go-*` skills.
