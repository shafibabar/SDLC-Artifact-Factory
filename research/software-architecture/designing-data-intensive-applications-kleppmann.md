# Designing Data-Intensive Applications
**Author:** Martin Kleppmann
**Domain:** Software Architecture — Data Systems
**Part 1 of 2** — Foundations, Storage, Replication, Partitioning, Encoding. (Part 2: `designing-data-intensive-applications-kleppmann-part2-consistency-consensus.md` — transactions, isolation, consistency models, consensus.)

## Core Concepts

- **Reliability, Scalability, Maintainability (Ch. 1).** The book's organizing triad for judging any data system design decision. *Reliability* = the system works correctly even when hardware fails, software has bugs, or humans make mistakes — "fault-tolerant" does not mean "fault-free," it means faults are anticipated and handled. *Scalability* is not a single number — it is described by a **load parameter** (requests/sec, read/write ratio, fan-out, active users) and a response-time distribution, not an average: Kleppmann is emphatic that **percentiles (p50/median, p95, p99, p999)** matter far more than mean latency, because tail latency is what users actually experience, and because in a fan-out request the slowest of many backend calls dominates (tail latency amplification). *Maintainability* decomposes into Operability, Simplicity (managing complexity so new engineers can understand the system), and Evolvability (ease of making future changes).
- **Data models (Ch. 2).** Relational vs. document vs. graph is a genuine trade-off, not a hierarchy: document/schema-flexible models fit one-to-many tree-shaped data well but struggle with many-to-many; the relational model fits arbitrary joins; the **property graph model** (vertices with arbitrary key-value properties, directed labelled edges also with properties) fits highly interconnected data and is queried with declarative languages like Cypher. This is exactly Apache AGE's data model (openCypher over PostgreSQL).
- **Storage engines: log-structured vs. page-oriented (Ch. 3).** Two storage-engine families: (1) **log-structured** (append-only writes, e.g. LSM-trees/SSTables as in Cassandra, LevelDB, RocksDB) — fast sequential writes, background compaction merges and discards overwritten/deleted values, reads may need to check several structures (mitigated with Bloom filters); (2) **page-oriented B-trees** (as in PostgreSQL, MySQL/InnoDB) — fixed-size pages, in-place updates, a write-ahead log (WAL) for crash recovery, generally faster point reads but writes touch the WAL plus the page. Kleppmann is explicit that **the right index structure depends on the read/write pattern**, not on one engine being universally superior. He also distinguishes **OLTP** (many small transactional reads/writes, indexed by keys) from **OLAP** (large scans, aggregates over few columns across many rows), and introduces **star/snowflake schemas** and **column-oriented storage** as the OLAP-specific answer to "one row at a time" being wasteful when a query only touches 4 of 100 columns.
- **Replication (Ch. 5).** Three topologies: **single-leader** (all writes to one node, replicated to followers — reads may go to any replica), **multi-leader** (multiple nodes accept writes, used across datacenters or for offline-capable clients — requires conflict resolution), and **leaderless** (Dynamo-style: writes/reads go to multiple replicas and use quorums, e.g. Cassandra/Riak). Sync vs. async replication is a durability/availability trade-off (semi-synchronous — one sync follower — is the common middle ground). **Replication lag** under async replication creates specific, precisely-named anomalies a client can observe: **read-your-writes consistency** (a user must see their own write immediately, even if it hasn't reached the replica they're reading from), **monotonic reads** (a user must never see time go backwards across successive reads, e.g. by hitting different replicas), and **consistent prefix reads** (if write B causally follows write A, no reader should see B without A). Leaderless replication adds its own vocabulary: **quorums** (writes need `w` acks, reads query `r` replicas, `w + r > n` gives overlap), **sloppy quorums / hinted handoff**, and **read repair / anti-entropy** to converge diverging replicas.
- **Partitioning / sharding (Ch. 6).** Splitting data across many nodes for scale beyond one machine. **Key-range partitioning** (sorted, good for range scans, risks hot spots if keys are sequential like timestamps) vs. **hash partitioning** (spreads load evenly, destroys range-query locality). **Skewed workloads and hot keys** are called out explicitly as a partitioning failure mode a scheme must name and handle (e.g. splitting or salting a celebrity/hot key). Secondary indexes complicate partitioning: **document-partitioned (local) indexes** (each partition indexes only its own data — cheap writes, scatter/gather reads) vs. **term-partitioned (global) indexes** (the index itself is partitioned by term — efficient reads, expensive/distributed writes). **Rebalancing** (moving partitions as nodes are added/removed) should avoid hash-mod-N (which reshuffles almost everything) in favor of a fixed number of partitions assigned to nodes, or consistent hashing.
- **Encoding and schema evolution (Ch. 4).** Precise, load-bearing definitions: **backward compatibility** = newer code can read data written by older code (lets you upgrade readers first); **forward compatibility** = older code can read data written by newer code (lets you upgrade writers first). Schema-based binary formats (Thrift, Protobuf, Avro) formalize this: fields are matched by tag number (Thrift/Protobuf) or by full schema resolution against a writer's and reader's schema (Avro) — in all of them, the safe evolution rule is essentially "add optional/defaulted fields, never repurpose or require a previously-optional field." **Dataflow modes** — through databases, through service calls (REST/RPC), and through asynchronous message-passing (queues/brokers) — each have different backward/forward-compatibility pressure, and message-passing is explicitly framed as the most decoupled and most tolerant of independent deploy schedules.

## Actionable Techniques

- State load parameters explicitly for any scalability claim (requests/sec, read:write ratio, fan-out factor) — never say "it needs to scale" without naming the parameter that grows.
- Measure and report **p95/p99 latency**, not average latency, for anything customer-facing; average hides the tail that dominates fan-out requests.
- When choosing an index structure, name the read/write pattern first (point lookups? range scans? write-heavy? append-only?) — that answer picks B-tree vs. LSM vs. BRIN, not a general preference.
- For any replicated read path, name which of the three anomalies (read-your-writes, monotonic reads, consistent prefix reads) the design must prevent, and how (e.g., route a user's own reads to the leader for N seconds after their write; pin a session to one replica).
- For any partitioned/sharded design, name the partitioning key and check it against the hot-key/skew failure mode explicitly — "partition by X" is incomplete without "and here is what happens if one value of X dominates."
- For any secondary index on partitioned data, state whether it is document-partitioned (cheap write, scatter-gather read) or term-partitioned (cheap read, distributed write) — this determines the read/write cost trade-off up front.
- When defining a wire schema, state explicitly which changes are backward-compatible (reader upgrades independently) vs. forward-compatible (writer upgrades independently), and pick a compatibility mode that matches the actual deployment order of producers vs. consumers.

## Key Terminology

Worth considering for `skills/glossary-management/`'s Ubiquitous Language:

- **Log-Structured Storage / LSM-Tree** — the append-and-compact storage family, as distinct from PostgreSQL's B-tree default; useful shorthand when an ADR ever justifies a non-Postgres store on write-throughput grounds.
- **B-Tree Index** — the precise name for PostgreSQL's default index structure, worth naming explicitly since `data-model-design` already contrasts it implicitly with BRIN.
- **Percentile Latency (p50 / p95 / p99)** — not currently named anywhere as a required NFR vocabulary; "the system should be fast" claims should cite a percentile.
- **Fan-out** — the number of backend calls one request triggers; directly relevant to tail-latency amplification in a microservices/event-choreography architecture like this one.
- **Leader-Based Replication**, **Multi-Leader Replication**, **Leaderless Replication** — precise names for replication topologies, useful once per-tenant Postgres HA is designed.
- **Replication Lag** — the general term; the repo's `Eventual Consistency` term is a consequence of this, but the mechanism causing it isn't currently named anywhere.
- **Read-Your-Writes Consistency**, **Monotonic Reads**, **Consistent Prefix Reads** — the three precisely-named anomalies of asynchronous replication; `cqrs-pattern` already implements read-your-writes (returning the new ID/version from a command) without naming it — naming it would let future reviews cite the guarantee directly.
- **Key-Range Partitioning** vs. **Hash Partitioning** — precise names for the two partitioning strategies; `data-pipeline-design` already uses hash-style partitioning (by `tenant_id`/`aggregateId`) without naming it.
- **Hot Spot / Skewed Workload** — the specific partitioning failure mode already implicitly acknowledged ("caps each tenant's throughput at one partition") but never named.
- **Document-Partitioned (Local) Index** vs. **Term-Partitioned (Global) Index** — relevant the moment Elasticsearch (already in the polyglot table) is partitioned.
- **Star Schema** / **Column-Oriented Storage** — worth reserving for the day a Read Model becomes genuinely analytical rather than transactional.

## Caveats

- **Leaderless replication (quorums, hinted handoff, read repair) is not applicable to this stack.** The book spends a full chapter on Dynamo-style leaderless replication (Cassandra, Riak, Voldemort); this repo's default stack is PostgreSQL (single-leader within each tenant's instance) exclusively. Do not import quorum-tuning vocabulary (`w`, `r`, `n`) into this repo's designs — there is no leaderless store here to tune.
- **Consensus underlying Redpanda is provided, not designed.** Redpanda uses Raft internally to replicate each partition across brokers. This repo's `event-driven-patterns` and `data-pipeline-design` skills correctly treat topic replication as a configuration choice (replication factor, ack settings), not something to design from scratch — the book's consensus material (covered in Part 2) is background knowledge for *why* Redpanda's guarantees hold, not a task for this plugin's agents.
- **Physical multi-tenancy already provides the sharding boundary Ch. 6 solves for.** The book's partitioning chapter assumes one logical database sharded across many nodes to scale past one machine's capacity. This repo sidesteps that problem at the tenant level: each tenant gets its own PostgreSQL instance via `multi-tenancy-design`, so intra-database partitioning (Ch. 6's node-level sharding) is only a live concern if a *single* tenant's data outgrows one Postgres instance — worth flagging as a forward-looking scale question, not a current gap.
- **Batch processing (Ch. 10, MapReduce/dataflow engines) has essentially no footprint in this repo.** `data-pipeline-design` is purely stream/event-choreography (Redpanda, Competing Consumers); nothing in the current skill set does offline batch analytics. If a future reporting/analytics chunk is added, Ch. 10's batch-vs-stream trade-offs (and the "unifying batch and stream" material in Ch. 11) become directly relevant; today they are out of scope.
- **Apache AGE's graph partitioning is out of the book's scope, not just this repo's.** Kleppmann's Ch. 6 partitioning discussion is explicitly relational/document-oriented; graph partitioning (how to split a highly-connected graph across nodes without constant cross-partition traversals) is a known-hard problem the book does not solve. `data-model-design`'s decision to keep AGE as a single-instance, PostgreSQL-co-located extension (not a sharded graph store) sidesteps this rather than needing to answer it — worth noting as a deliberate scope limit, not an oversight.
