# Node.js 2025: Mastering Event-Driven Architecture and Microservices for Scalable Backends
**Authors:** Jaren Daxton
**Domain:** Software Architecture & Implementation (Node.js/TypeScript, applied/practitioner level)

## Core Concepts

- **The event loop as the runtime's single organizing idea, and its direct philosophical kinship with this repo's event-driven default.** Node has exactly one JS execution thread; libuv's event loop dispatches callbacks for completed I/O (sockets, file descriptors, timers) onto that thread, one macrotask/microtask at a time, never blocking on I/O itself. This is a genuinely good fit for a repo whose `CLAUDE.md` tech-stack defaults already commit to Redpanda-based event-driven microservices and whose `go-event-consumer`/`go-event-publisher` skills already treat "never block waiting on I/O" as a first design principle (bounded worker pools, manual offset commits, backpressure). Where Go achieves non-blocking I/O by parking a goroutine and letting the runtime scheduler multiplex thousands of goroutines across `GOMAXPROCS` OS threads (`go-concurrency-patterns`' goroutine-lifecycle rule), Node achieves the same *I/O* non-blocking property with zero threads by design — the cost is that there is no second thread to fall back on when work is CPU-bound rather than I/O-bound (see Caveats).
- **`async`/`await` over the Promise microtask queue as the idiomatic concurrency surface — Node's direct analog to `go-concurrency-patterns`.** The book teaches `async`/`await` as the default way to sequence non-blocking work, backed by `Promise.all`/`Promise.allSettled` for fan-out and `AbortController`/`AbortSignal` for cancellation. The mapping onto this repo's existing Go vocabulary is close but not exact: `Promise.all` is the Node analog of `errgroup.Go` + `g.Wait()` for parallel orchestration (`go-service-layer`'s `errgroup` dashboard example), and `AbortSignal` is the Node analog of `context.Context` for cancellation propagation — but `AbortSignal` carries only cancellation, not values (no tenant ID, no trace span riding along "for free" the way a Go `context.Context` carries both cancellation and request-scoped values in one object). A Node port of `go-service-layer`'s pattern needs a second, explicit mechanism (an `AsyncLocalStorage` context, per the book) to carry tenant/trace-span state across `await` boundaries — this is not optional plumbing, it is the direct Node equivalent of something Go gets for free from one parameter.
- **`worker_threads` as Node's only true parallelism primitive, and the book's explicit chapter on CPU-bound offload.** The book devotes a chapter to `worker_threads` (backed by separate V8 isolates, each with its own event loop, communicating via structured-clone message passing or `SharedArrayBuffer`) as the mechanism for anything genuinely CPU-bound — image/document processing, cryptographic hashing at scale, heavy JSON transformation — precisely because the main event loop thread will otherwise stall on synchronous CPU work and every concurrent request/consumer callback queued behind it waits. This is the single most consequential concept for this specific product, and is addressed head-on rather than glossed over in the Caveats section below.
- **The `cluster` module and multi-process horizontal scaling within one host.** Because one Node process uses one CPU core for its event loop, the book teaches `cluster` (or, more commonly in 2025 production stacks, a process manager like PM2, or simply N container replicas behind a Kubernetes Service/Linkerd mesh) to use all cores on a machine. This repo's tech stack already defaults to Kubernetes + Linkerd for horizontal scaling of Go services — the same replica-based scaling story applies unchanged to Node services; `cluster`/PM2 is a single-host alternative the book covers but this repo's container-per-replica model already supersedes.
- **Idempotent, at-least-once event consumption using `kafkajs` (or the Confluent Node client) against Kafka-API-compatible brokers, including Redpanda.** The book's microservices chapters teach consumer groups, manual offset commit after successful processing, and a dedup table keyed on event id — structurally identical to `go-event-consumer`'s idempotent-consumer pattern, translated into `kafkajs`'s `eachMessage`/`eachBatch` callback shape and a Postgres `processed_events` check done inside the same transaction as the write.
- **The Transactional Outbox pattern, implemented as a polling relay process — same shape as `go-event-publisher`, different runtime idiom.** The book presents the outbox relay as a `setInterval`/`node-cron`-driven poller (or a dedicated lightweight worker) reading committed outbox rows and publishing them via `kafkajs`'s producer, marking them published in the same style as `go-event-publisher`'s `FOR UPDATE SKIP LOCKED` polling loop. The correctness properties this repo requires (state + event committed atomically in one Postgres transaction, publish happens only after commit, retries with backoff) are runtime-agnostic — Postgres owns the atomicity guarantee in both stacks; only the polling/publishing code's language changes.
- **Fastify (and, as the book's chief alternative, Express) as the two competing HTTP framework stories, with the book favoring a schema-first framework for anything production-grade.** The book is explicit that hand-validating request bodies ad hoc (Express's un-opinionated default) does not scale past a handful of routes without either heavy Joi/Zod boilerplate or drift between validation code and the OpenAPI contract — and recommends a framework with native JSON Schema request/response validation (Fastify) specifically so the contract and the runtime validator are the same artifact. This maps directly onto `go-chi-handler`'s "structural validation at the boundary" rule and `go-openapi-codegen`'s "the contract is the single source of truth" principle — see Actionable Techniques and the framework-choice recommendation below.
- **Error handling as Node's most-cited footgun, and the book treats it as a dedicated chapter rather than an aside.** Unhandled promise rejections (a rejected promise with no `.catch`/`try-catch` anywhere in its chain) historically only logged a warning and kept the process alive; since Node 15, the default behavior is to crash the process on an unhandled rejection — which the book frames as a *feature*, not a regression, because a silently swallowed rejection is worse than a fast, visible crash. The book's discipline: every `async` function boundary that isn't awaited by its caller needs an explicit `.catch`, every route handler needs framework-level rejection capture (Fastify does this automatically; Express pre-v5 does not), and custom `Error` subclasses (not string-typed error codes) carry domain error identity across `await` boundaries the way Go's sentinel/typed errors and `errors.Is`/`errors.As` do across `error` returns (`go-error-handling`).
- **Streams and backpressure as the idiomatic way to process large payloads (large document uploads, big query result sets) without buffering the whole thing in memory.** The book covers Node's `Readable`/`Writable`/`Transform` stream primitives and their built-in backpressure signal (`.write()` returning `false` when the internal buffer is full) as the direct analog of `go-event-consumer`'s bounded worker pool and `data-pipeline-implementation`'s checkpointed, backpressure-aware unit processing — the same "never buffer unboundedly" discipline, expressed through a different, stream-native mechanism.

## Actionable Techniques

- **Name the CPU-bound risk explicitly wherever a future `node-*` skill would touch document parsing, OCR, or entity/classification work**, rather than silently porting `go-concurrency-patterns`' "spawn a goroutine, bound its lifetime" pattern onto Node code as if `worker_threads` were interchangeable with a goroutine. They are not interchangeable: a goroutine is cooperative-preemptive and cheap (KBs of stack, scheduled by the Go runtime across OS threads); a `worker_thread` is a full V8 isolate (MBs of overhead, its own GC, its own event loop) and is expensive to spin up per-request — the idiomatic pattern is a bounded **worker pool** (the book recommends `piscina`), not one thread per unit of work, which is the direct Node analog of `go-concurrency-patterns`' "bounded concurrency, never unbounded fan-out" rule applied to a much heavier primitive.
- **Monitor event-loop lag as a first-class health signal**, using `perf_hooks.monitorEventLoopDelay()`, exposed as an OpenTelemetry/Prometheus gauge — this is the Node-specific analog of Go's goroutine-count/scheduler-latency observability this repo's `health-check-design`/`opentelemetry-instrumentation` skills already require of Go services; a Node service with no event-loop-lag metric has a blind spot Go services don't have, because a single slow synchronous callback degrades every concurrent request, not just its own.
- **Adopt `AsyncLocalStorage` as the explicit mechanism carrying tenant id, trace span, and deadline across `await` boundaries**, and call out in any future `node-service-layer` skill that this is *not* automatic the way `context.Context` propagation is in Go — a Node handler that reads tenant/trace state from a module-level variable instead of `AsyncLocalStorage` will leak state across concurrent requests on the same event loop tick, a correctness bug with no Go equivalent (Go's `context.Context` is passed explicitly by value, so this specific failure mode cannot occur the same way).
- **Require framework-level automatic promise-rejection capture** (Fastify's default behavior) rather than the un-caught, silently-swallowed rejections Express's pre-v5 routing permits, as a non-negotiable rule in any future `node-fastify-handler`/`node-error-handling` skill pair — this is the direct Node-runtime-idiom equivalent of `go-chi-handler`'s "never `uuid.MustParse` untrusted input" and "one error mapping point" rules: an uncaught rejection in a route handler is Node's version of an unrecovered panic reaching the transport edge.
- **Treat CPU/ML-bound document parsing and classification as an escalation out of the request-handling/event-consuming service entirely, not an in-process `worker_threads` optimization, whenever that work is genuinely heavy (OCR, layout-model inference, multi-page NER passes)** — mirroring this repo's own existing Go-first discipline, where `data-pipeline-implementation`'s Entity Extraction stage already names "OCR/ML-based extraction" as its one sanctioned "Python Exception" even under the **Go** default. If Node becomes the sanctioned per-product override, the same escalation discipline applies unchanged: a compute-heavy extraction worker is its own service (Go or Python, per the existing exception), not something a Node API/orchestration service should do inline even inside a `worker_threads` pool, once the per-document cost crosses the threshold the existing skill already uses to justify escalating out of Go.
- **Recommend one Node testing framework (Vitest) and one Node package manager/lockfile discipline (`npm` + `package-lock.json`, consistent with the `npm` permission already carved out in `settings.json`) up front**, rather than leaving both open — this repo's whole pattern (`net/http`+chi over gin/echo, `pgx` over `database/sql`, k6 over multiple load-test tools) is "pick one sanctioned default per concern," and a Node roster proposing multiple frameworks/test runners/ORMs per concern would violate that pattern on day one.

## Candidate New Skills/Agents

**Framework choice made first, since it shapes several skill names above: Fastify is the sanctioned Node.js web framework default**, not Express and not NestJS. Reasoning, held against this repo's own stated preference for "transparency over magic frameworks" (the same reasoning `go-chi-handler`'s frontmatter already gives for choosing `net/http`+chi over gin/echo):

- **Express** is the closest in spirit to `net/http` alone — minimal, unopinionated — but it fails the direct comparison on two non-negotiable points this repo already enforces in Go: it has no native request/response schema validation (the Go equivalent would be hand-rolling every `validate()` method chi already writes explicitly, except *without* the option of pushing that validation into the framework, which Fastify offers), and versions before Express 5 (final-released 2024, and not yet the ecosystem default as of this book's 2025 publication) silently swallow rejected promises in route handlers unless a wrapper or try/catch is added by hand at every call site — a direct, checkable violation of `go-error-handling`'s "errors are values, never discarded" standard if ported naively.
- **NestJS** is a full opinionated framework — decorator-based dependency injection, modules, guards, interceptors — structurally closer to Spring or Angular than to chi. This is precisely the "magic framework" category `go-chi-handler`'s own frontmatter names as the reason chi was chosen over heavier alternatives; adopting NestJS would import the same category of complexity for a PM-reviewed, "boring is correct" codebase that `CLAUDE.md` explicitly optimizes against elsewhere (frugality, plain reviewability).
- **Fastify** is schema-first by default (JSON Schema or TypeBox validate `body`/`params`/`querystring` before the handler runs at all — a *stronger* position than `go-chi-handler`'s hand-written `validate()` method, not a weaker one), has a single centralized error handler (`setErrorHandler`) that is the direct one-to-one analog of this repo's existing `writeDomainError` single-mapping-point rule, automatically captures rejected promises in route handlers (closing Express's biggest error-handling gap without a bolt-on library), has a plugin/encapsulation model that mirrors chi's sub-router composition, and is measurably faster/lower-overhead than Express or NestJS in every independent 2024–2025 benchmark round — aligning with this repo's frugality principle (lower compute cost per request at the same hardware tier).

## Key Terminology

Worth considering for `skills/glossary-management/`'s Ubiquitous Language, but only once a `node-*` skill actually exists to anchor the definition against (consistent with how prior research files in this cluster have handled thin, no-current-consumer terminology additions):

- **Event Loop** — the single-threaded dispatch mechanism (via libuv) that processes I/O completions, timers, and microtasks one at a time on Node's one JS execution thread. Not currently named anywhere in the repo's glossary (which is Go-oriented); would need a precise definition distinguishing it from Go's goroutine-scheduler concept, since the two solve a similar-sounding problem (non-blocking concurrency) through structurally different mechanisms (single-thread cooperative dispatch vs. multi-thread preemptive scheduling).
- **Worker Thread** — Node's `worker_threads` module: a separate V8 isolate with its own event loop and heap, communicating via structured-clone message passing or `SharedArrayBuffer`. Needs an explicit glossary disambiguation against "goroutine" if both terms ever appear in the same product's artifacts (a per-product override could plausibly have both a Go extraction service and a Node API service in the same system) — they are not interchangeable words for "a unit of concurrent work," and conflating them would be exactly the kind of terminology drift `glossary-management`'s "no synonyms, no drift" rule exists to prevent.
- **Event Loop Lag** (or **Event Loop Delay**) — the Node-specific health/saturation signal (measured via `perf_hooks.monitorEventLoopDelay()`) analogous to, but not identical to, Go's goroutine-scheduling-latency concern; worth naming precisely once `node-service-skeleton`/`node-async-concurrency` exist and an observability skill needs to reference it.
- **Unhandled Promise Rejection** — a rejected `Promise` with no attached `.catch`/`try-catch` anywhere in its chain; since Node 15 this crashes the process by default. Worth naming in the glossary the moment `node-error-handling` exists, since it is the single most-cited Node-specific failure mode with no Go equivalent (Go has no analogous "silently discarded error crashes the process later" runtime behavior — a discarded Go error just returns `nil`, which is its own, differently-shaped defect already covered by `go-error-handling`'s "errors as values, never discarded" rule).

## Caveats

- **Node's single-threaded event loop is a real, product-relevant concern for this specific product — not a theoretical one — and must be named as an explicit architectural caveat, not silently absorbed.** This repo's own prior research (`research/data-governance-and-mdm/unstructured-data-warehouse-krishnan-inmon.md` and `research/ai-ml-engineering/ai-engineering-huyen.md`) already independently established that this product's entity-extraction/classification pipeline stage is CPU/ML-bound work (`data-pipeline-implementation`'s `extractUnit` dispatching PDF/DOCX/XLSX parsing and OCR/NER-style entity detection) — work heavy enough that the **Go** default already carves out an explicit "Python Exception" for it, because even Go's runtime-scheduled goroutines are not the right tool for OCR/ML inference at scale. If a per-product override selects Node.js for a service that ends up performing that same parsing/classification work synchronously on the main thread, the failure mode is categorically worse than in Go: a CPU-bound synchronous call on Node's single JS thread blocks **the entire event loop** — every concurrent HTTP request, every in-flight Kafka consumer callback, every timer — for the full duration of that call, not just its own logical processor the way a CPU-bound goroutine would only stall progress within its own slice of `GOMAXPROCS`. **Named mitigation, in priority order:** (1) preserve this repo's existing escalation discipline exactly — keep CPU/ML-bound document parsing and classification as a separate service (Go or Python, per `data-pipeline-implementation`'s already-existing "Python Exception"), regardless of which language the request-handling/event-orchestration services use, so a Node override never needs to solve this problem in the first place; (2) where some bounded, genuinely necessary CPU work must run inside a Node service (e.g., a fast, deterministic regex/rule pass, not a full ML inference call), offload it to a `worker_threads` pool (via `piscina`) rather than running it on the main thread; (3) monitor event-loop lag (`perf_hooks.monitorEventLoopDelay()`) as a first-class health/saturation metric in any Node service that could plausibly do meaningful synchronous work, so a regression toward accidental main-thread CPU work is caught by an automated signal rather than discovered as a production incident.
- **`AsyncLocalStorage` is a real, non-optional addition a naive Go-to-Node port would miss.** Go's `context.Context` carries cancellation, deadline, tenant id, and trace span together, implicitly, through every function that takes `ctx context.Context` as its first parameter. Node's `AbortSignal` carries only cancellation. A `node-service-layer` skill that ports `go-service-layer`'s worked examples by simply swapping `ctx` for an `AbortSignal` parameter would silently lose tenant-scoping and trace-correlation propagation — a correctness and observability regression, not a stylistic difference. `AsyncLocalStorage` is the explicit mechanism that must be named to close this gap; it is genuinely more boilerplate-heavy than Go's single implicit parameter, which is worth stating plainly rather than presenting the Node port as a clean, equal-effort translation.
- **The JS/TS ecosystem has no single, canonical equivalent to `benchstat` or Go's native fuzzing, and `node-performance-test`/`node-unit-test`'s fuzz-testing sections will need more original authorship than most of the roster, not less.** Where most of the test-tier mirrors above are "thin" (Testcontainers, Playwright, toxiproxy, k6 are already cross-language, mature, drop-in-compatible tools), Go's benchmark-baseline-regression tooling and native fuzzing are comparatively mature relative to the JS ecosystem's equivalents (`tinybench`, `fast-check`) — flag this honestly rather than assuming parity across the whole roster.
- **This file proposes skills; it does not decide agent ownership, and that decision should not be inferred from this document alone.** Whether `backend-engineer` becomes stack-conditional or a sibling agent is created is a real architectural choice with consequences for `CLAUDE.md`'s "agents never overlap" rule and is explicitly left open above, not resolved here.
- **Scope discipline: this file mirrors the `go-*` roster one-to-one on purpose, and resists the temptation to also propose NestJS-flavored or GraphQL-flavored alternatives "while we're here."** `CLAUDE.md`'s frugality principle and "one sanctioned default per concern" pattern both argue against a wider net; the 24-skill roster above is scoped as a literal mirror of what already exists and works for Go, not a fresh, independent Node.js architecture exercise.</content>
