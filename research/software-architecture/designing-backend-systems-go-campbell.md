# Designing Backend Systems with Go: A Practical Guide to Building APIs, Managing Data, and Structuring Scalable Applications in Go
**Authors:** Rodney S. Campbell
**Domain:** Software Architecture & Implementation (Go, applied/practitioner level)

## Core Concepts

- **A pragmatic, general-purpose layered structure for REST APIs.** The book teaches a `handlers` → `services` → `repository` (often called `store`) three-tier split, with a `models`/`domain` package for shared types — the same layering shape this repo's `go-project-structure` already formalizes as a four-layer Clean/Hexagonal architecture (`handlers` / `application` / `domain` / `infrastructure`). The book's version is looser: it does not enforce an inward-only dependency rule as a checked invariant, and its "service" layer often talks directly to `database/sql` rather than through a consumer-defined port.
- **`database/sql` + a lightweight query helper (commonly `sqlx`) as the default data-access approach**, with raw SQL strings and manual `Scan` calls, occasionally introducing a minimal query builder for readability. This is the mainstream, driver-portable approach general Go backend books teach — deliberately more conservative than reaching for `pgx`'s Postgres-native feature set (binary protocol, `pgx.Batch`, `LISTEN/NOTIFY`, connection-level tracing hooks).
- **Middleware as composable `func(http.Handler) http.Handler` chains**, covering request logging, panic recovery, and — centrally for this book, since it targets APIs consumed by browser-based and third-party clients — **CORS handling** (preflight `OPTIONS` short-circuiting, `Access-Control-Allow-Origin`/`-Methods`/`-Headers`, credentialed-request rules) as a first-class, worked example.
- **`net/http.Server` field-by-field configuration** as a dedicated topic: `ReadTimeout`, `ReadHeaderTimeout`, `WriteTimeout`, and `IdleTimeout` are each explained with the specific resource-exhaustion failure mode each one closes (slow header send, slow body send, slow response write, and idle keep-alive connections respectively) — framed as a checklist every production `http.Server` must set explicitly, not an incidental aside.
- **Rate limiting and basic caching as scaling levers**, covered at a practical, single-library depth: a token-bucket limiter (commonly `golang.org/x/time/rate` or an in-memory sliding window) in front of expensive endpoints, and a Redis- or `sync.Map`-backed cache-aside layer in front of read-heavy database queries, with TTL-based invalidation as the primary (only) invalidation strategy discussed.
- **JWT-based authentication and basic RBAC** as the default auth story — token issuance, validation middleware, and role checks in handlers — considerably shallower than this repo's ABAC-based `access-control-model`/`security-implementation`, but directionally the same shape (auth middleware resolves identity into context; downstream code checks it).
- **Graceful shutdown via `signal.Notify`/`context.WithTimeout` and `srv.Shutdown`**, taught as a capstone "production readiness" topic, alongside Dockerizing the built binary and a brief CI pipeline sketch — all at a single-service, non-distributed level (no outbox, no consumer groups, no multi-tenant migration fan-out).

## Actionable Techniques

- Add a **CORS middleware** to `go-middleware`'s named, ordered chain — positioned to short-circuit preflight `OPTIONS` requests *before* `Authenticate` (a browser's preflight carries no `Authorization` header and no cookie by design; if it reaches an auth-gated stage first, every cross-origin browser client fails with 401 on the preflight, not the real request, which is a confusing failure mode to debug). State explicitly which origins/methods/headers are allowed per environment (dev, since the tech-stack default frontend is a separately-served React app during local development; and prod, if the API and the served frontend are ever on different origins/domains).
- Extend `go-service-skeleton`'s `http.Server` construction to set `ReadTimeout`, `WriteTimeout`, and `IdleTimeout` alongside the already-present `ReadHeaderTimeout`, each with the specific failure mode it closes named in a comment — the same "every goroutine has a bounded lifetime" discipline `go-concurrency-patterns` already mandates for goroutines applies unfinished to the connections themselves: a slow client or hung write currently has no server-side deadline forcing it to release its handling goroutine and connection.
- Where `go-service-layer`'s query handlers front a read-heavy view already flagged by `read-model-design` as a Redis/in-memory caching candidate, show the concrete cache-aside implementation pattern (check cache → miss → query the view → populate cache with TTL) and name the one correctness rule that must accompany it: an invalidation or short-TTL strategy that at minimum tolerates the staleness window already implied by `read-model-design`'s "reads that tolerate short staleness" caveat — the architectural intent exists today; no Go skill shows the code.
- When citing this repo's `database/sql`-adjacent choices against a general Go backend text, state explicitly that `pgx`'s native driver (`pgx.Batch`, binary protocol, `LISTEN/NOTIFY`) is a deliberate, justified departure from the more portable but slower `database/sql`+`sqlx` mainstream default this kind of book teaches — not an oversight to reconcile.

## Candidate New Skills/Agents

None warranted. Both genuine findings are narrow, mechanical additions to existing skills, not new artifact types:

1. **CORS middleware → `go-middleware`'s ordered chain.** A new named stage in an existing skill's already-established chain-and-table format, not a new skill — the skill's whole shape (ordering table, one-purpose-per-middleware rule, anti-patterns list) is designed to absorb exactly this kind of addition.
2. **`http.Server` timeout completeness → `go-service-skeleton`'s server literal.** Three more struct fields with the same one-line comment convention the skill already uses for `ReadHeaderTimeout` — a same-shaped extension, not new content requiring its own home.

The cache-aside pattern (Applies To, `go-service-layer`) was considered and sized against the MVP-dependency-pull threshold used elsewhere in this campaign (pull in new content only when a real, currently-unserved dependent need exists) and does not clear it as a standalone skill: `read-model-design` already owns the architectural decision of *when* to cache, and the Go implementation is a few lines inside an existing query handler, not a pattern with enough independent surface area (retry, ordering, idempotency, DLQ-style failure handling, the kind of depth that justifies `go-event-consumer` as its own skill) to warrant a dedicated `go-caching` skill.

## Key Terminology

Worth considering for `skills/glossary-management/`'s Ubiquitous Language only if a browser-facing API surface becomes a concrete, scheduled need (not proposed now, given the thin, single-item yield):

- **CORS (Cross-Origin Resource Sharing)** — the browser-enforced mechanism requiring a server to explicitly allow cross-origin requests via response headers, with a distinct preflight (`OPTIONS`) step for non-simple requests. Not currently named anywhere in the repo; would only earn a glossary entry once a CORS middleware actually exists to define the term against.
- **Cache-Aside Pattern** — check the cache, fall through to the source of truth on a miss, populate the cache with a bounded TTL. Already practiced in substance wherever `read-model-design` names Redis/in-memory caching, but never named as this specific pattern; low priority for the glossary given the pattern has no Go implementation yet to anchor the definition to.

Neither term is proposed as an addition in this pass — both are noted only so a future skill that does implement CORS or caching doesn't have to re-derive the vocabulary from scratch.

## Caveats

- **This is the thinnest-yield book in the cluster so far, and that is the correct, honest result given what's being checked.** This repo's 24-skill Go roster is already a mature, production-grade implementation of DDD, CQRS, the Transactional Outbox, idempotent consumers, and a verified Clean Architecture dependency rule — considerably deeper than a general practical guide to building APIs in Go needs to reach. Nearly everything the book teaches (layered structure, middleware chains, JWT auth, rate limiting, request validation, graceful shutdown, Dockerizing) is already present and, in most cases, already more rigorous in this repo's existing skills. Treat this document's two genuine findings (CORS, `http.Server` timeout completeness) as real, checkable absences worth a small edit each — not as evidence of a broader gap in the Go skill set's depth or coverage.
- **CORS is a genuine absence, not a naming gap.** Unlike the Clean Architecture and Pragmatic Programmer research in this cluster, which mostly found already-correct patterns needing a precise citation, `go-middleware`'s ordered chain genuinely does not mention CORS anywhere — checked directly across the Purpose, Ordering, and Anti-Patterns sections. Given this plugin's tech-stack defaults name a separately-served React frontend, this is a concrete, product-relevant absence rather than a theoretical one.
- **The `http.Server` timeout gap is real but partial — the skill already gets one field right.** `ReadHeaderTimeout` is present and correctly justified; the gap is specifically the three sibling fields (`ReadTimeout`, `WriteTimeout`, `IdleTimeout`) that close adjacent but distinct resource-exhaustion vectors. This is a completion of an existing, correct pattern, not a missing concept.
- **The caching finding is real but low-urgency.** `read-model-design` already made the architectural call (Redis/in-memory, TTL-bounded, for reads that tolerate staleness); no product requirement in `sdlc-context.json` currently names a query handler with a documented latency problem this pattern would solve. Cite it as a documented, ready-to-use pattern for when that need arises — not as an urgent fix.
- **The book's `database/sql`+`sqlx` default and shallower single-service scope (no outbox, no consumer groups, no multi-tenant migration fan-out, RBAC instead of ABAC) reflect its intended audience — a general practical guide, not a distributed-systems reference — and don't translate as gaps.** Where the book's coverage is shallower than this repo's existing skills (auth, shutdown, validation, data access), that is the expected, correct outcome of checking a practitioner-level book against a already-mature, more sophisticated skill set, not a signal to dilute this repo's skills toward the book's simpler defaults.
