# Master Data Management
**Authors:** David Loshin
**Domain:** Data Governance and Master Data Management

## Core Concepts

- **The business case for MDM is the cost of not having one authoritative version (Ch. 1–2).** Loshin frames MDM not as a technology purchase but as a response to a specific, quantifiable pain: the same real-world entity (a customer, a supplier, an employee) is represented differently across systems, and every downstream process — billing, reporting, compliance — pays a reconciliation tax for it. This repo's first product *is* this pain, professionalised into a service: `skills/canonical-data-model/SKILL.md`'s opening line ("A 'person' in an extracted PDF, a 'user' in the identity system, and an 'owner' in a data source each describe overlapping reality with different shapes") is Loshin's business-case chapter's opening scenario, restated as the product's own architecture problem rather than a customer's. The book's business-case discipline (name the cost of *not* reconciling — duplicate outreach, missed compliance findings split across unmerged identities) is not present anywhere in this repo's discovery-phase skills (`vision-statement`, `business-model-canvas`) as a lens applied specifically to master-data quality — it is a gap in the Strategy/Ideate phases, not the Design-phase skill that already does the reconciliation itself well.
- **Identity resolution is a distinct technical discipline with its own failure modes — not just "deterministic vs. probabilistic" (Ch. 5–6).** `canonical-data-model`'s matching section is a two-row table (deterministic exact-key match vs. probabilistic fuzzy match "on weighted similarity... above a threshold") — correct as far as it goes, but Loshin's chapters on identity resolution go substantially deeper: **blocking/indexing** (comparing every record pair is O(n²) and infeasible at scale — partition records into candidate blocks by a cheap key like postal code or normalised last name first, then only compare within blocks), **similarity functions** chosen per attribute type (edit distance for typos, phonetic encoding like Soundex/NYSIIS for name variants, token-based similarity for addresses), and the **Fellegi-Sunter probabilistic record linkage framework** — the actual mathematics behind "probabilistic fuzzy match": each matching attribute contributes an agreement weight derived from its *m-probability* (chance it agrees given a true match) and *u-probability* (chance it agrees by chance given a non-match), summed into a composite score, then split into three explicit bands (auto-match, auto-non-match, and a clerical-review band in between) rather than a single pass/fail threshold. `canonical-data-model`'s "weighted similarity... above a threshold" sentence gestures at this without ever naming the framework, the three-band outcome, or the blocking step that makes it computationally feasible at the estate-mapping product's expected volume (every document across every customer's Google Drive/S3/PDF estate).
- **The "golden record" concept the repo already implements well is one of several possible MDM architectural styles, not the only one (Ch. 7).** Loshin catalogs four MDM implementation styles: **Registry** (a thin index mapping source-system keys to a shared identifier, with no consolidated golden record stored — queries fan out to sources at read time), **Consolidation** (a physically merged golden record assembled and stored, typically for reporting/analytics — read-mostly, often refreshed on a batch or event cadence), **Coexistence** (a stored golden record *and* the source systems both remain writable, with bidirectional sync), and **Transaction Hub / Centralized** (the master data store becomes the single system of entry — all creates and updates go through it directly, sources become thin clients). `canonical-data-model`'s design — a Golden Record "rebuildable from source records + active match decisions," stored in PostgreSQL, "never hand-edited" — is a textbook **Consolidation style** implementation, done well. But the skill never names it as a style choice, never mentions the other three exist, and therefore never asks whether a lighter Registry style (no stored golden record at all, just an identity map) might better fit a *read-heavy, audit-evidence* product where the golden record's main job is answering "does this document reference the same Person as that document" rather than serving a system-of-entry write path. This is a genuine, nameable architectural decision this repo has made implicitly and never recorded as one.
- **Hierarchy management is a named MDM discipline this repo has zero coverage of, despite naming the entity it applies to (Ch. 9).** `canonical-data-model` lists `Organisation` as one of the first product's three master entities (alongside `Person` and `DataSource`) — but no skill anywhere designs how organisations relate to each other. Loshin's hierarchy management chapter covers exactly this: multiple, simultaneous hierarchy *views* over the same entities (a legal-entity hierarchy for regulatory/ownership purposes vs. an operational/reporting hierarchy for who-manages-whom — the same set of organisations can nest differently depending on which question is being asked), containment vs. reporting-line semantics, and how a golden-record "party" model represents both individuals and organisations under a shared supertype (Loshin's "Party" pattern: a `Person` and an `Organisation` are both a kind of `Party` that can hold roles, relationships, and hierarchy memberships). This product's own graph store (`skills/data-model-design/SKILL.md`'s Apache AGE vertex list: `DataAsset`, `Entity`, `DataSource`, `Person`) has no `Organisation` vertex at all today, despite `canonical-data-model` naming `Organisation` as a master entity one skill over — the graph model and the canonical model have silently drifted apart on what master entities actually exist.
- **System of Record vs. System of Entry — a precise distinction this repo's similar-sounding term does not make (Ch. 4).** Loshin distinguishes the **System of Entry** (any system where a master entity's attributes can be created or edited — there may be several) from the **System of Record** (the one place whose value for a given attribute is authoritative when systems disagree — determined by the survivorship rule, not by which system happens to be the golden-record store). This repo already uses the phrase "system of record" extensively, but in a *different*, event-sourcing sense: `skills/data-model-design/SKILL.md`'s "the relational store is the system of record... a non-PostgreSQL store is never a system of record for an Aggregate" is about which *storage tier* holds authoritative state versus which is a rebuildable projection — a storage-architecture meaning, not an MDM cross-source-authority meaning. Both usages are legitimate and both are needed (this product genuinely has both concerns), but they are homonyms across two different skills owned by the same agent (`data-architect`), which risks being read as the same claim when it is not.
- **Merge, unmerge, and match-history are lifecycle states a master record moves through, not a one-time assembly (Ch. 6, 8).** `canonical-data-model`'s `match_decisions` table with a nullable `retracted_at` already implements Loshin's core insight that a merge must be reversible — this is one of the strongest points of alignment found in this pass. What the book adds beyond what's there: an explicit notion of "survivorship trust" per source *per attribute*, not just per record (a source might be highly trusted for `display_name` but low-trust for `primary_email` — the current skill's survivorship table is per-attribute already, which is correct, but doesn't generalise this into a reusable "source trust score" concept an operator could tune without redesigning each attribute's rule by hand) — and a formal state model for a master record's lifecycle (active → merged-into-another → tombstoned/retired → possibly reinstated on unmerge) that would give `data-model-design`'s eventual `Person` Aggregate a documented state machine instead of an implicit one inferable only from the `match_decisions` retraction mechanic.
- **Data quality for MDM has its own dimension beyond the six generic ones already in this repo — match quality (Ch. 6, cross-referencing Ch. 13 of DMBOK, already synthesised in `research/data-governance-and-mdm/dama-dmbok.md`).** `skills/data-quality-rules/SKILL.md`'s Uniqueness dimension ("no two `Entity` rows for the same `(data_asset_id, entity_type, normalized_value)`") is a *within-record* deduplication check. Loshin's MDM-specific quality concern is different: **match precision and recall** — of all the record pairs the identity-resolution process auto-matched, what fraction were actually the same real-world entity (precision), and of all the true matches that existed in the data, what fraction did the process find (recall)? A high false-merge rate (two different people wrongly consolidated into one golden Person) is a specific, MDM-native failure mode that none of the six generic DQ dimensions names, and it has a specific remediation (tightening the auto-match band, in Fellegi-Sunter terms) that a generic "uniqueness" rule cannot express.

## Actionable Techniques

- When identity resolution volumes grow past trivial (every document in every customer's estate, not a handful of test records), add a **blocking step** before any pairwise comparison: partition candidate records into blocks by a cheap, high-recall key (normalised surname + postal code, or first-3-characters-of-email-domain) so full similarity scoring only runs within a block, not across the whole tenant's record set.
- Replace `canonical-data-model`'s single "weighted similarity... above a threshold" sentence with **three explicit match bands** — auto-match (score above an upper threshold, apply automatically), clerical-review (score in a middle band, route to a Data Steward review queue — the same steward role `data-classification` already assumes but a different queue), and auto-non-match (score below a lower threshold, do not merge) — rather than one cutoff that conflates "confident match" with "worth a human glance."
- Before building the `Organisation` master entity `canonical-data-model` already names but never designs, decide and record which of the four MDM architectural styles (Registry, Consolidation, Coexistence, Transaction Hub) fits each master entity — `Person` and `Organisation` may reasonably get different answers (e.g., `Person` stays Consolidation as already built; `Organisation` might only need a lightweight Registry if the product never needs to *edit* organisation data, only resolve identity across mentions of it).
- Add an `Organisation` vertex (and at least one hierarchy edge type, e.g. `PART_OF` for legal containment) to `data-model-design`'s Apache AGE graph model, closing the drift between `canonical-data-model` naming `Organisation` as a master entity and the graph model never representing it.
- When two hierarchy questions could reasonably be asked about the same organisations (which entity legally owns which, versus which team operationally reports to which), model them as **two named hierarchy edge types** in the graph rather than one generic `PART_OF` — collapsing them loses the ability to answer either question precisely once an org restructures operationally without a change in legal ownership (or vice versa).
- Introduce a **source trust score per (source, attribute)** pair as a first-class, tunable value (not hardcoded survivorship-rule prose) so a Data Steward can adjust "how much do we trust the identity provider's `display_name` vs. an extracted document's" without an engineer rewriting the canonical entity's YAML.
- Add a **match precision/recall or false-merge-rate metric** to whatever eventually extends `data-quality-rules` or `metrics-instrumentation-plan` for identity resolution specifically — distinct from the existing six generic DQ dimensions, and distinct from `data-classification`'s extraction-confidence metrics, because a false merge is a *golden record* correctness failure, not an extraction-confidence or classification failure.
- Before the next skill or agent uses the phrase "system of record," check which of the two meanings applies (MDM cross-source authority vs. storage-tier authority for an Aggregate) and consider whether the MDM sense needs its own term (Loshin's literal phrase, or a repo-specific alternative) to avoid silently overloading a term `data-model-design` already uses precisely for a different purpose.

## Candidate New Skills/Agents

- **`entity-resolution-design` (Skill, owned by `data-architect`, likely graduating a large portion of `canonical-data-model`'s current matching section rather than duplicating it).** Teaches identity-resolution technique in the depth Loshin devotes multiple chapters to: blocking/indexing strategies for making pairwise comparison tractable at estate-mapping scale, similarity-function selection per attribute type (edit distance, phonetic encoding, token similarity), the Fellegi-Sunter probabilistic linkage framework with m/u-probability weights, and the three-band auto-match/clerical-review/auto-non-match outcome model. Justification: this is a genuinely hard, distinct technical problem — the same real-world entity (a person, a customer, a legal-hold subject) will be represented differently across Google Drive metadata, S3 object tags, and extracted document text, and `canonical-data-model`'s current two-row table is a placeholder for a discipline, not the discipline itself. Flag the merge-vs-split decision explicitly: this could instead be written as a much-expanded "Identity Resolution" section *within* `canonical-data-model` rather than a new file — the chunk that takes this on should decide based on how large the resulting section gets.
- **`hierarchy-management` (Skill, owned by `data-architect`, consumed by `domain-modeler` for the `Organisation` Aggregate and by whichever skill ends up owning the Apache AGE graph's `Organisation` vertex).** Teaches Loshin's Party-model pattern (Person and Organisation as two kinds of a shared Party supertype that can hold roles and relationships) and multi-hierarchy-view design: legal/ownership hierarchies vs. operational/reporting hierarchies as distinct, simultaneously valid views over the same entity set, plus containment vs. reporting-line edge semantics in a graph model. Justification: `Organisation` is already named as one of exactly three master entities in `canonical-data-model`, yet no skill designs how organisations relate to each other or to tenants — and this product's compliance premise (data estates typically belong to legally structured, multi-subsidiary customers) makes "which legal entity owns this data source, and does that differ from who operationally administers it" a real, likely-recurring question, not a hypothetical one.
- **`mdm-architecture-style-selection` (Skill, owned by `data-architect` — or, if judged too thin to stand alone, folded as a new section into `canonical-data-model` alongside the identity-resolution material above; flag this merge-vs-split decision for whichever chunk takes it on).** Teaches Loshin's four MDM implementation styles (Registry, Consolidation, Coexistence, Transaction Hub) as a decision table — read/write pattern, latency tolerance, and system-of-entry count each drive the choice — so a future master entity's style is a recorded, ADR-worthy decision rather than an unexamined inheritance of whatever `Person` already does. ## Key Terminology

Candidates for `skills/glossary-management/references/ubiquitous-language.md` — cross-reference `research/data-governance-and-mdm/dama-dmbok.md`'s Key Terminology section first: **Master Data** and **Reference Data** are already proposed there and should not be redefined here, only cited:

- **Golden Record** — the reconciled, deduplicated, authoritative version of a master entity, assembled from multiple sources by survivorship rules. `canonical-data-model` uses this term throughout as a load-bearing concept but it is absent from the canonical glossary.
- **Survivorship** — the set of rules that decide, attribute by attribute, which source's value wins when multiple sources disagree about the same master entity (source priority, recency, completeness, confidence). Already a `canonical-data-model` term of art with no glossary entry.
- **Identity Resolution** — the process of determining that two or more source records describe the same real-world entity, via deterministic (exact key) or probabilistic (weighted similarity) matching. Used extensively in `canonical-data-model`; absent from the glossary.
- **Match/Merge** — the operational pairing of identity resolution's *match* decision (are these the same entity?) with the *merge* action (consolidate their attributes via survivorship) — Loshin treats these as two distinct steps that this repo's `match_decisions` table already implicitly separates (a match decision can be retracted independent of re-running survivorship) without ever naming the two-step structure.
- **Blocking (Record Linkage)** — a pre-filtering step that partitions candidate records into small comparison groups by a cheap key, avoiding the O(n²) cost of comparing every record against every other record during identity resolution. Not present anywhere in this repo; a prerequisite concept for `entity-resolution-design` if built.
- **MDM Architectural Style** — one of Registry, Consolidation, Coexistence, or Transaction Hub/Centralized — the deliberate choice of how a golden record is constructed, stored, and kept writable relative to its source systems. Not currently named anywhere despite `canonical-data-model` having already made one such choice (Consolidation) for `Person`.
- **System of Entry** — any system in which a master entity's attributes can be created or edited; contrasted with **System of Record** (Loshin's specific MDM sense: the source whose value is authoritative for a given attribute, per the survivorship rule) — worth a precise glossary entry specifically to disambiguate from `data-model-design`'s existing, different use of "system of record" for storage-tier authority over an Aggregate.
- **Hierarchy Management** — the discipline of modelling and maintaining one or more simultaneous, potentially conflicting hierarchical views (e.g. legal containment vs. operational reporting) over the same set of master entities. Not present anywhere in this repo despite `Organisation` already being named as a master entity that would need it.
- **Party Model** — Loshin's pattern of modelling `Person` and `Organisation` as two kinds of a shared `Party` supertype, so that roles, relationships, and hierarchy memberships can be expressed once and applied to either. Worth naming if `hierarchy-management` is built, to avoid `Person` and `Organisation` evolving incompatible relationship/role mechanics independently.

## Caveats

- **This repo's `canonical-data-model` is already an unusually mature MDM artifact — this book sharpens and extends it, it does not correct a broken foundation.** Unlike some gaps found in other research passes, the Golden Record, survivorship, tenant-scoped matching, and reversible match-decision design already present are genuinely aligned with Loshin's core teaching, not a naive or incorrect version of it. Treat every recommendation here as *depth to add to a good foundation* (blocking, probabilistic scoring detail, style-naming, hierarchy) — not as a rewrite. The DAMA-DMBOK research file already reached the same conclusion independently ("Best-in-repo coverage of Master Data Management... no meaningful gap against DMBOK's Master Data content specifically") — this file's job was to go one level deeper than DMBOK's survey treatment, which it does.
- **Loshin's book pre-dates modern embedding-based/ML entity-resolution techniques (vector similarity, learned matching models) — treat its Fellegi-Sunter framework as the classical, still-valid statistical baseline, not the state of the art.** If `entity-resolution-design` is eventually built, the book gives correct, durable *vocabulary* (blocking, m/u-probabilities, match bands) and a genuinely sound default algorithm for this product's likely scale, but a future revision should note that learned/ML-based matching is a legitimate alternative for the probabilistic-scoring step specifically — this caveat mirrors the software-architecture research file's caution about not over-indexing on a book's era-specific tooling recommendations.
- **Hierarchy management and the four MDM architectural styles are the parts of this book most likely to feel like premature complexity for a solo-operator, single-tenant-at-a-time MVP — sequence them deliberately, not immediately.** `Organisation` is named as a master entity today but the product's MVP scope (per `agents/data-architect.md`'s existing scope) may not yet need multi-hierarchy views or a formal style decision if the MVP only ever has one flat organisation per tenant. Treat `hierarchy-management` and `mdm-architecture-style-selection` as roadmap items to build when a second, genuinely hierarchical customer or a cross-organisation compliance question first appears — not as gaps to close before the next chunk, echoing this repo's existing frugality constraint (`CLAUDE.md`'s Budget and Frugality section).
- **The "clerical review" band in Fellegi-Sunter matching implies an ongoing human-labor cost this repo's frugal, single-operator context must size honestly.** Loshin's three-band match model assumes a review team exists to clear the middle band; for this repo, that reviewer is Shafi or an agent operating under his explicit delegation (the same pattern DAMA-DMBOK's research file already established for the Data Steward role generally). Any future `entity-resolution-design` skill must set thresholds conservatively enough that the clerical-review queue stays small, or explicitly budget for a steward workflow to clear it — an unbounded clerical queue is a silent, unstaffed backlog exactly like the "quarantine as a black hole" anti-pattern `data-quality-rules` already warns against for a different queue.
- **Confidence level:** I am confident in Loshin's headline structure — the business-case framing, the deterministic/probabilistic/Fellegi-Sunter matching taxonomy, the four MDM architectural styles (Registry/Consolidation/Coexistence/Transaction Hub), the hierarchy-management and Party-model concepts, and the System of Record/System of Entry distinction — these are the book's well-established, widely cited contributions to the MDM literature. I am **less confident on exact chapter numbers**, which are my best approximation of the book's structure, not verified against the physical text page-by-page.
