# Managing Data As a Product: A Comprehensive Guide to Designing and Building Data Product-Centered Socio-Technical Architectures
**Authors:** Andrea Gioia
**Domain:** Data Governance and Master Data Management

## Core Concepts

- **A dataset is a Product, with a Product Owner, a Service Level Objective, and a contract with its consumers — not a byproduct of a pipeline stage.** Gioia's central claim, following and formalising Zhamak Dehghani's Data Mesh thesis, is that any dataset another team consumes must be treated with the same rigor a software team gives a public API: versioned, documented, quality-guaranteed, and owned by someone accountable for its consumers' experience — not merely "whatever the last pipeline stage happened to write to a table." This repo's `skills/data-pipeline-design/SKILL.md` (read in full) designs a rigorous internal pipeline — Transactional Outbox, DLQ, idempotency, per-stage contracts documented in a table (`Stage name / Consumes / Emits / Idempotency key / Delivery semantic / Retry-DLQ / State written / Lineage emitted / SLO`) — but every one of those stage contracts describes an *internal* choreography hop between this product's own pipeline stages (`file-processed` → `entity-extracted` → …), not a dataset offered for consumption by an external analytical consumer (a BI tool, a data scientist, a downstream product team, or eventually a customer's own analytics stack). Gioia's book is about exactly the artifact this repo has never built: an **output port** — a dataset published at a stable boundary, with an explicit data contract, meant to be discovered and consumed by parties who were never part of the pipeline's internal choreography.
- **The API/Event Contract vs. Data Product Contract distinction is real and this repo currently has only the former.** `skills/api-contract-design/SKILL.md` (read in full) rigorously contracts synchronous request/response resources: OpenAPI 3.1, versioning, pagination, idempotency headers, a standard error envelope. `skills/domain-event-catalog/SKILL.md` and `skills/event-schema-design/SKILL.md` (event-schema-design referenced, not re-read here — already covered structurally by `canonical-data-model`'s cross-reference) rigorously contract asynchronous, single-fact wire formats: envelope, versioning by additive/breaking change, compatibility mode. Both are point-in-time, single-record contracts — "here is the shape of one request" or "here is the shape of one event." Gioia's data contract is a different animal: a contract for a **dataset as a whole**, over time — its schema (yes, like an API), but also its **update/freshness cadence** (how often is this refreshed, and what does "current" mean for it), its **quality guarantees** (what completeness/accuracy SLOs does the whole dataset carry, not a single record's confidence score), its **retention/versioning of the dataset itself** (can a consumer pin to a prior version of the whole table/export), and its **discoverability metadata** (what is this dataset, who owns it, how do I find it, is it still maintained). Nothing in this repo's API or event skills covers "publish a queryable/analytical dataset with these guarantees" — `api-contract-design`'s Quality Criteria table has no "freshness SLA" or "dataset versioning" row, and neither does `event-schema-design`'s (single-event) compatibility-mode framing.
- **Domain-oriented data ownership is Data Mesh's headline organizational principle, and it borrows DDD vocabulary this repo already owns precisely — the gap is not the ownership *idea*, it's the missing *artifact* for the dataset boundary itself.** Gioia (following Dehghani) argues each Bounded Context/domain team should own and publish its own analytical data products, rather than centralizing all data engineering in one team that becomes a bottleneck. This repo's DDD discipline is already exactly this shape: `bounded-context-mapping` and `context-map-patterns` define context ownership rigorously, and `CLAUDE.md`'s "one-database-per-service" rule already decentralizes operational data ownership. `research/data-governance-and-mdm/multi-domain-mdm-cervo-allen.md` already proposed `data-domain-ownership-model` for master-data domain ownership (Person/Organisation/DataSource) with Business/Technical Steward roles — that candidate is about *who decides matching/survivorship rules for a master entity*, a fundamentally different question from *who owns the published, versioned, SLA-bearing analytical dataset a Bounded Context exposes for others to query*. The two are adjacent but not the same: a `DataAsset` Aggregate inside the Storage Integration context is operational write-model state; a `compliance-findings-export` dataset published *from* that context for a BI tool or a customer report is a data product with its own lifecycle, entirely undesigned today.
- **Federated computational governance — global rules enforced locally and automatically, not by a central review board — is Data Mesh's governance-scaling answer, and it sharpens (does not replace) the governance gap DAMA-DMBOK's research file already found.** `research/data-governance-and-mdm/dama-dmbok.md`'s Core Concepts already identified that this repo has no artifact for *designing* a governance operating model (decision-rights table, escalation ladder) — DAMA's answer is organizational (a Data Governance Council, scaled down to Shafi). Gioia/Data Mesh's distinct contribution is **computational**: global policies (a sensitivity taxonomy, a naming standard, a minimum-freshness rule) are encoded as automatically enforced checks — schema-validation gates, contract-conformance tests in CI — applied uniformly at every domain's data product boundary, rather than a human reviewing every dataset by hand. This repo already has the *mechanism* for this pattern elsewhere — `event-schema-design`'s CI-gated compatibility check (per `agents/data-architect.md`'s Step 3, "specify the CI gate that enforces evolution rules") and `api-contract-design`'s "Pass schema validation in CI before any other pipeline step" are both federated-computational-governance in miniature, already built, just never applied to an analytical dataset boundary because that boundary doesn't exist as an artifact yet.
- **Self-serve data infrastructure as a platform concern is the third Data Mesh pillar, and this repo already has its infrastructure half-built without the data-product-facing half.** Gioia describes a platform that lets any domain team publish a compliant data product without needing bespoke pipeline engineering per team — schema registries, contract-testing scaffolding, a catalog for discovery. This repo's Redpanda/Apache AGE/PostgreSQL stack plus `event-schema-design`'s registry is real self-serve infrastructure for the *operational/event* layer. The DAMA-DMBOK research file's proposed `metadata-catalog-design` candidate is the closest existing proposal to a discovery catalog, but it was scoped to unifying business/technical/operational metadata about the pipeline's own internal state (per that file: "which pipeline stage last touched this dataset... what does the business call it") — not to a public-facing catalog of *published, consumable data products* a downstream consumer browses to decide what to depend on. These are complementary, not the same: an internal metadata catalog answers "what do we have," a data product catalog answers "what can you subscribe to, with what guarantees."
- **A data product's four architectural quanta — output port, input port, control port, discoverability port — is Gioia's most concrete, buildable technique, and none of the four exist as a named concept anywhere in this repo.** The book (formalizing Data Mesh architecture) describes a data product as having: an **output port** (the actual queryable/consumable data — a table, a file export, a stream — versioned and schema'd), an **input port** (how the data product ingests from its sources, usually invisible to consumers), a **control port** (observability/SLO metrics *about the data product itself* — is it fresh, is it passing its own quality gates — as distinct from the operational infrastructure's RED/USE metrics), and a **discoverability port** (metadata a catalog can index: owner, schema, SLA, sample data, usage docs). `skills/data-pipeline-design/SKILL.md`'s stage contracts are entirely input-port-shaped (they describe internal ingestion between stages); `skills/dashboard-specification/SKILL.md` (read in full) gets closest to an output port in spirit — it already uses the phrase "a precise, buildable widget-level **data contract**" — but that contract is scoped to a single BI widget's query shape for one dashboard, not a discoverable, independently-versioned, SLA-bearing dataset another team or product could depend on without knowing which dashboard it was built for.

## Actionable Techniques

- Before extending `data-pipeline-design`'s pipeline any further downstream (e.g., a compliance-findings export, an analytics feed, anything a BI tool or future customer-facing reporting feature will query), explicitly decide whether that destination is an **internal pipeline stage** (another choreography hop, contracted the way `data-pipeline-design` already contracts stages) or a **data product output port** (a published, versioned, SLA-bearing dataset meant for external/cross-team consumption) — these need different contracts, and conflating them means a downstream dashboard silently depends on pipeline internals that were never meant to be a stable interface.
- When `dashboard-specification`'s "widget-level data contract" phrase is next revisited, decide explicitly whether it should graduate into (or reference) a dataset-level data product contract — schema, freshness SLA, quality guarantee, discoverability metadata — one level above a single widget's query shape, so multiple dashboards/widgets can depend on one governed dataset instead of each dashboard re-deriving its own ad hoc query against pipeline internals.
- For any dataset a Bounded Context intends to expose beyond its own pipeline (an export, a feed, a queryable projection for another context or an external tool), require the same four-part completeness `api-contract-design` already requires for APIs: a schema (what shape), a versioning/compatibility policy (how it evolves without breaking consumers — reuse `event-schema-design`'s additive/breaking taxonomy, which already fits this purpose), an SLA (freshness, completeness, and quality guarantees, reusing `slo-definition`'s SLO/SLA distinction), and discoverability metadata (owner, description, sample, last-updated) that a catalog can index.
- When assigning ownership for a data product's output port, use the same Bounded-Context-owns-its-data discipline `bounded-context-mapping` already enforces for services — the context that owns the source data owns the data product built from it — rather than centralizing data-product publication in a single data team, which is exactly the bottleneck Data Mesh's domain-ownership principle exists to avoid.
- Where a global data-product standard needs enforcing (a minimum freshness SLA, a required discoverability field, a naming convention), prefer a CI-gated automated check over a manual review step — this repo already has two working examples of this pattern (`event-schema-design`'s compatibility-mode CI gate, `api-contract-design`'s CI schema validation) that a future data-product-contract skill should extend rather than reinvent.
- Do not conflate a **data product's control port** (is this dataset itself fresh, complete, and passing its own quality gates) with the **pipeline's operational RED/USE metrics** (`opentelemetry-instrumentation`) or the **data quality gate metrics** (`data-quality-rules`'s pass/quarantine/reject rates) — the first asks "can a consumer trust this specific dataset right now," the second two ask "is the infrastructure healthy" and "did this record pass a quality check," which are related but distinct questions already partially covered by existing skills.

## Candidate New Skills/Agents

- **`data-product-contract-design` (Skill, owned by `data-architect`, consumed by `data-engineer` for implementation and cross-referenced by `dashboard-specification`/`reporting-spec` wherever a dataset feeds a BI artifact).** Teaches how to design a data contract for a published, analytical/cross-context dataset — schema (reusing `event-schema-design`'s additive/breaking taxonomy), freshness and completeness SLAs (reusing `slo-definition`'s SLO/SLA distinction), a versioning policy for the dataset as a whole (not a single record), and discoverability metadata (owner, description, sample data, last-updated) a future catalog could index. Justification: this is the most concrete, directly buildable gap this book surfaces — `api-contract-design` contracts request/response resources, `event-schema-design` and `domain-event-catalog` contract single-fact events, `canonical-data-model` contracts cross-context *meaning*, but nothing in the repo contracts a *dataset as a consumable, SLA-bearing whole* the way this product's eventual compliance-findings exports, analytics feeds, or customer-facing reports will need. This is squarely analogous to `api-contract-design` in rigor and shape, just for data-at-rest instead of a request/response resource — the gap Shafi's prompt anticipated is real and unfilled.
- **`data-mesh-domain-ownership` — not proposed as a standalone skill; folded into the existing `data-domain-ownership-model` candidate (already proposed in `research/data-governance-and-mdm/multi-domain-mdm-cervo-allen.md`) as an explicit additional scope note, not a new artifact.** Gioia/Data Mesh's domain-oriented data ownership principle (each Bounded Context owns and publishes its own data products) is real and distinct from Cervo & Allen's master-data domain ownership (who owns matching/survivorship rules for `Person`/`Organisation`), but the *shape* of the artifact needed — a per-domain accountable owner, an escalation path for conflicts — is identical, just applied to a different kind of asset (a published analytical dataset vs. a master entity's golden-record rules). Building two near-identical "who owns this domain's data" skills would itself be the kind of duplication Shafi's prompt asked to avoid; the honest recommendation is one skill (`data-domain-ownership-model`) with two named scopes (master-entity ownership and data-product ownership) rather than a second, thinly-differentiated skill. Flag this scope-merge decision for whichever chunk builds `data-domain-ownership-model`.
- **`data-catalog-discoverability-design` (Skill, owned by `data-architect`; likely a close sibling or eventual merge candidate with DAMA-DMBOK's already-proposed `metadata-catalog-design`, flagged for the chunk that takes either on).** Teaches the discoverability-port concept specifically: what metadata a data product must publish so a consumer (another Bounded Context's team, a BI tool, eventually a customer's own analytics stack) can find it, understand its guarantees, and decide whether to depend on it — owner, schema link, freshness SLA, sample query, deprecation status. Justification: DAMA-DMBOK's `metadata-catalog-design` candidate was scoped to unifying business/technical/operational metadata *about the pipeline's own internal state* for internal operational visibility ("which pipeline stage last touched this dataset"); this book's discoverability port is scoped to *external consumer-facing* catalog entries for published data products specifically — overlapping in mechanism (both are catalogs) but different in audience and purpose. Flag explicitly as a likely merge (one catalog skill serving both internal-operational and external-consumer discovery) rather than building two catalogs that drift apart, the same caution `multi-domain-mdm-cervo-allen.md`'s Caveats gave for premature multi-domain infrastructure.
- **Not proposed as a new skill, with reasoning: a dedicated "federated computational governance" skill.** This book's governance-scaling technique (encode global policy as automated CI checks rather than manual review) is not a new *kind* of artifact this repo lacks — it is a naming and generalization of a pattern already implemented twice (`event-schema-design`'s compatibility-mode CI gate, `api-contract-design`'s CI schema validation). The correct home for this idea is as an explicit principle cited inside `data-product-contract-design` (whichever CI conformance check enforces the new dataset contract should follow the same shape those two skills already use), not a standalone governance-mechanism skill with no artifact of its own to produce.
- **Not proposed as a new skill, with reasoning: a dedicated "self-serve data platform" skill.** Gioia's third Data Mesh pillar describes platform-level infrastructure (schema registries, contract-testing scaffolding, provisioning automation) that lets any domain team publish a compliant data product without bespoke engineering. This repo's frugal, single-operator context (per `CLAUDE.md`'s Budget and Frugality section) has exactly one team building exactly one pipeline today — the self-serve-platform problem (many competing domain teams needing shared tooling to avoid a bottleneck) does not exist yet at this repo's scale, mirroring the same caution `multi-domain-mdm-cervo-allen.md`'s Caveats gave for organizational-change-management content aimed at competing internal teams that don't exist here. If a second product or a second domain team is ever added, revisit this as a real gap; today it would be platform ceremony with no second team to serve.

## Key Terminology

Candidates for `skills/glossary-management/references/ubiquitous-language.md` — cross-reference the three prior files in this cluster first: **Master Data**, **Reference Data**, **Golden Record**, **Survivorship**, **Identity Resolution**, **MDM Architectural Style**, **System of Entry**, **Hierarchy Management**, **Party Model**, **Multi-Domain MDM**, **Data Domain**, **Data Domain Owner**, **Business Data Steward**, **Technical Data Steward / Data Custodian**, **Data Governance Council**, **Cross-Domain Relationship**, **MDM Program Maturity**, **Decision Explainability**, **Data Trust**, and **Minimum Viable MDM** are already proposed there and should not be redefined here, only cited:

- **Data Product** — a dataset treated as a first-class, discoverable, independently versioned unit with an accountable owner, an explicit contract (schema, freshness, quality guarantees), and a stable interface for its consumers — as distinct from an internal pipeline artifact never meant for consumption outside its own choreography. Not present anywhere in this repo despite `dashboard-specification` already using the phrase "data contract" for an adjacent, narrower concept.
- **Data Contract** (dataset sense) — the versioned, machine-checkable specification of a data product's schema, freshness/completeness SLA, and compatibility policy — the dataset-level sibling of `api-contract-design`'s API contract and `event-schema-design`'s event schema, worth a glossary entry that explicitly disambiguates it from both.
- **Output Port** — the interface through which a data product exposes its data to consumers (a queryable table, a file export, a stream) — the boundary a data contract governs, distinct from a data product's input port (how it ingests from sources) and control port (its own health/freshness observability).
- **Input Port** — the interface through which a data product ingests data from its sources, typically not exposed to or depended on by consumers — contrasted with the output port, which is the stable, contracted interface consumers actually depend on.
- **Control Port** — the observability surface of a data product itself (is this specific dataset fresh, complete, and passing its own quality gates) — distinct from the operational infrastructure's RED/USE metrics (`opentelemetry-instrumentation`) and from a pipeline stage's quality-gate pass/quarantine/reject counts (`data-quality-rules`), both of which answer a different question about a different layer.
- **Discoverability Port** — the metadata a data product publishes so a prospective consumer can find it, understand its guarantees, and decide whether to depend on it: owner, schema reference, SLA, sample data, deprecation status — the interface a data catalog indexes.
- **Data Mesh** — the architectural and organizational philosophy (Dehghani, formalised further by Gioia) of decentralising analytical data ownership to the Bounded Context/domain that produces it, publishing each domain's data as an independently owned data product, governed by federated computational rules rather than a central data team — worth naming explicitly given this repo's DDD/Bounded-Context discipline already satisfies the organizational precondition for it.
- **Federated Computational Governance** — the practice of enforcing global data standards (schema conformance, naming, freshness minimums) via automated checks applied uniformly at each domain's data product boundary, rather than centralized manual review — already present in miniature in this repo (`event-schema-design`'s CI compatibility gate, `api-contract-design`'s CI schema validation) but never named as a generalizable governance pattern.

## Caveats

- **Gioia's book (and Data Mesh generally) assumes a mid-to-large organization with multiple domain teams, each capable of publishing and maintaining its own data products — this repo has exactly one operator and one pipeline today.** The core *technique* — a data contract for any dataset a consumer depends on, with schema/SLA/discoverability — is valuable at any scale and should be built as soon as this product has its first cross-context or external analytical consumer (a compliance-findings export, a customer-facing report, a BI dashboard depending on more than pipeline internals). The *organizational* apparatus (many domain teams competing for a shared self-serve platform, federated governance councils per domain) is not needed yet and should not be built preemptively — this mirrors the sequencing caution `multi-domain-mdm-cervo-allen.md`'s Caveats already gave for cross-domain relationship modeling and MDM architectural style selection: build the artifact when a second real consumer appears, not before.
- **Do not let `data-product-contract-design` become a rebrand of `api-contract-design` or `event-schema-design` under a trendier name.** The distinguishing claim of this book is that a *dataset over time* — with freshness, completeness, and dataset-level versioning concerns — needs a contract shape those two skills don't provide, not that APIs and events need replacing. A future skill that just restates OpenAPI/JSON-Schema conventions with "data product" substituted for "API" would add naming, not substance; the freshness-SLA and discoverability-metadata concerns are the actual new content.
- **"Data as a Product" language is easy to conflate with this repo's existing product/business vocabulary (`business-model-canvas`'s "the product" meaning the compliance platform itself, `vision-statement`'s product-market framing) — a future skill must be explicit that "data product" here means a specific dataset, not the company's product-as-a-whole**, to avoid the same kind of terminology collision `multi-domain-mdm-cervo-allen.md`'s Caveats already flagged between MDM's "Data Domain" and DDD's "Subdomain."
- **Confidence level:** I am confident in this book's headline structure — the Data Mesh four principles (domain ownership, data-as-a-product, self-serve platform, federated computational governance), the output/input/control/discoverability port framing, and the data-contract-as-first-class-artifact thesis — these are the book's well-established, widely cited contributions, largely formalizing Dehghani's original Data Mesh writing. I am **less confident on exact chapter structure and terminology**, which are my best approximation of the book's organization, not verified against the physical text page-by-page.
