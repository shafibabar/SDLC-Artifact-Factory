# User Story Mapping
**Authors:** Jeff Patton with Peter Economy
**Domain:** Requirements & User Stories

## Core Concepts

- **The map's two axes.** Narrative flow runs left-to-right — the order in which a user actually encounters activities and tasks, i.e. "the backbone" of the story. Priority/necessity runs top-to-bottom — not simply "depth of detail" but *what's needed for the story to be minimally whole*. The very top row across the entire backbone is the **walking skeleton** (Patton borrows the term from Alistair Cockburn): the thinnest possible slice that actually works end-to-end, even if ugly, rather than a polished slice of just one part.
- **"The map is a conversation, not an artifact" — Patton's central thesis.** The value of story mapping is the shared understanding a cross-functional team builds *while* mapping together, not the resulting diagram. A map produced by one person and handed to others as a document has captured a picture, not the understanding the book says matters — this is stronger than "keep it lightweight"; Patton treats a map with no shared conversation behind it as having missed the point entirely.
- **The flat-backlog critique.** A prioritized flat list destroys the narrative relationships between stories. Teams "grooming" a flat backlog tend to cut scope by finishing one feature completely (a vertical slice) rather than cutting a thin horizontal slice across the whole journey — exactly the failure mode this repo's `story-mapping` skill already names as "the vertical-slice MVP" anti-pattern.
- **MVP as a *learning* vehicle vs. MVP as a *release* slice — two different things Patton is careful to keep separate.** Drawing on Lean Startup thinking, the book distinguishes "the smallest thing we can build to test a belief" (a cheap, possibly throwaway prototype or experiment) from "the smallest thing we can actually ship to real users" (the walking skeleton). Treating a validated-learning experiment as shippable, or demanding production rigor from a learning experiment, are both named failure modes.
- **Frame the opportunity before mapping.** Before generating tasks and stories, the book has the team state the business opportunity, the target user, and the hoped-for outcome. This sits upstream of the map itself — functionally similar territory to what `impact-mapping` and JTBD already do in this repo, produced *before* the map, not derived from it.
- **Whole-team mapping as a live practice.** Patton is explicit that story mapping is done together, physically or synchronously, by product, design, engineering, and business stakeholders in the same room (or call) — not authored solo and reviewed asynchronously later. This is a practice constraint as much as a technique.
- **Slices as releases, and re-cutting as learning happens.** Horizontal cuts below the walking skeleton mark releases; each slice must itself be viable end-to-end. After a slice ships, the book expects a "what did we learn" checkpoint that may reshape later slices — the map is a living tool revisited across releases, not a one-time discovery output.
- **Three working layers: big picture → backbone → details.** "Big picture" is why you're building this and for whom; "backbone" is the narrative spine of activities and tasks; "details" are the individual stories under each task. This maps closely onto this repo's existing Level 1 (Activities) / Level 2 (Tasks/Epics) / Level 3 (Stories) structure.

## Actionable Techniques

- **Walking-skeleton check.** Read the top row across the *entire* backbone — does it form one complete, functioning (if minimal) path through the whole journey? Any activity whose top row is missing or non-functional means the skeleton is broken, not just incomplete.
- **Separate learning-MVP from release-MVP explicitly.** Before scoping an "MVP," ask which kind is meant: a cheap validated-learning experiment (fake door, concierge test, throwaway prototype) or the real, shippable walking skeleton. Never let one substitute for the other.
- **Run mapping as a live, whole-team session.** Sticky notes or a shared virtual board, with product, design, engineering, and business stakeholders present together — not a document authored alone and circulated for comment.
- **Frame-the-opportunity pass first.** One page stating the business objective, the target persona, and the outcome hypothesis, produced before backbone/task/story generation begins.
- **Post-slice learning checkpoint.** After each release slice ships, hold an explicit checkpoint on what was learned, and be willing to re-cut later slices based on it.
- **Narrate-the-spine test.** Read the activities/tasks aloud left to right as a story ("first the user does X, then Y…"). If it doesn't read as a coherent narrative, the spine is wrong. (Already present in this repo's `story-mapping` skill.)

## Key Terminology

- **Walking Skeleton** (Patton credits Alistair Cockburn) — the thinnest end-to-end functional slice across the full narrative spine. More precise than "MVP slice" as currently used in `story-mapping`; worth distinguishing from Minimum Viable Product in the glossary since the two terms get conflated in practice.
- **Backbone** — the top-level narrative spine of activities and tasks. Check against this repo's existing "Activities" / "narrative spine" wording before adding, so the glossary doesn't end up with two terms for the same thing.
- **Story Slice / Release Slice** — a horizontal cut of the map representing one release. Worth distinguishing explicitly from a MoSCoW priority cut: a slice must be end-to-end viable, not merely "the Must Haves."

## Caveats

- **Two different "MVP" concepts must not be conflated.** The book's learning-MVP (cheap, possibly throwaway, used to validate a belief) is not the release-MVP (the walking skeleton, meant to actually ship). The current `story-mapping` skill only describes the latter, which is fine — but any future skill that borrows "MVP" language for a validated-learning experiment must not blur the two.
- **The book's core practice — live, whole-team mapping — cannot be executed literally under this repo's single-agent-produces-then-Shafi-approves model.** Treat this as an intentional, named adaptation (the same way Cohn's persona/role distinction required adaptation in the sibling research file), not a silently dropped practice. A future refactor should decide explicitly how — or whether — to approximate the conversational value inside a solo-agent workflow, rather than assume the artifact alone captures what the book says only the live conversation captures.
- **Don't attribute MoSCoW, INVEST, or Gherkin/BDD to this book.** Those come from DSDM, Bill Wake/Cohn, and Chris Matts/Dan North respectively. This book's own vocabulary is backbone / walking-skeleton / slices, not those terms.
- **The "skateboard → bicycle → car" MVP sketch**, often cited alongside story-mapping and MVP discussions, is Henrik Kniberg's illustration, not content from this book — don't attribute it to Patton/Economy if a future skill wants to reuse it.
