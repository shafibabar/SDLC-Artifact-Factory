# Getting and Writing IT Requirements in a Lean and Agile World
**Authors:** Thomas and Angela Hathaway
**Domain:** Requirements & User Stories

## Core Concepts

- **The four-level requirement hierarchy — Business, Stakeholder, Solution (Functional + Non-functional), Transition.** This is the Hathaways' organizing taxonomy (consistent across their BA-EXPERTS teaching corpus, of which this book is part): a **Business Requirement** states the business goal/objective ("why we're doing this at all"); a **Stakeholder Requirement** states a specific stakeholder group's need for the solution to satisfy that goal (the bridge layer, often skipped by teams that jump straight from goals to solution detail); a **Solution Requirement** states what the system must do (Functional) or how well it must do it (Non-functional); a **Transition Requirement** states what's needed to move from the current state to the future state — data migration, training, cutover, parallel-run, legacy decommission. Transition requirements are the category most often omitted entirely, because they're neither "what the product does" nor "how well it does it" — they're a one-time cost of getting there.
- **A stated want is not a requirement.** Stakeholders naturally describe solutions, not needs ("I want a dropdown menu," "I want a button that emails the report") — the book's core discipline is treating every raw elicited statement as a *want* to be interrogated, not a requirement to be transcribed. The technique for unpacking it is root-cause questioning ("why do you need that?", repeated) until you reach the underlying capability the stakeholder actually needs, at which point the solution (dropdown, button, email) becomes negotiable again rather than locked in.
- **Premature solutioning is the single most common requirements defect.** Not a rare mistake — the book treats it as the default failure mode of unsupervised elicitation, because people think in solutions, not in capabilities. The BA's job is explicitly to peel the solution back off every requirement statement before it's recorded.
- **"Lean" requirements means just enough, not zero.** The book's title distinction from traditional BA practice (BABOK-style, full BRD/FRS/SRS produced once, upfront, before any build starts) is *progressive elaboration* (also called rolling-wave): elicit and document requirements in only as much detail as is needed for the work about to be planned (the next sprint/iteration), and defer detail for later work until it's closer. This is explicitly not "skip documentation" — it's "match the documentation's completeness to how soon it will be built," which is a different failure mode than either a heavyweight upfront BRD or an undocumented backlog.
- **Requirement format should match audience and lifecycle stage, not a single dogma.** The book does not argue that user stories replace all other requirement formats — it treats stories, use cases, and formal "shall" statements as different points on a spectrum, each appropriate to a different combination of team maturity, regulatory context, and how much of the detail needs to survive as a durable record versus live only in the sprint's working memory.
- **Testability is the requirement's exit gate, not a nice-to-have.** The book's repeated question for any candidate requirement is "how will we know this has been satisfied?" — if that question has no answer, the statement isn't a requirement yet, regardless of how confidently it was stated.
- **The necessity test.** For every candidate requirement, ask "what happens if we don't build this?" If the honest answer is "nothing consequential," the item is not a requirement — it's a preference, and it should be named as such rather than smuggled into the requirements set at equal footing with things that actually gate release.
- **Elicitation technique selection is context-dependent, not one-size.** Interviews (depth, one stakeholder, sensitive topics), facilitated workshops/JAD sessions (consensus across a group, surfacing conflicting stakeholder requirements early), observation/job-shadowing (tacit process knowledge stakeholders can't articulate because they don't consciously notice it), document/system analysis (baseline from what already exists), prototyping (validate understanding when stakeholders can't visualize from prose alone), and surveys (breadth over a large, dispersed population) — each technique surfaces a different failure mode if skipped, and the book pairs "use when" guidance with each rather than picking a single default.

## Actionable Techniques

- **Root-cause / "why" chain on every raw stakeholder statement.** Before recording anything as a requirement, ask "why do you need that?" at least once, ideally twice, and stop only when the answer names a business outcome or capability rather than a UI element, a specific technology, or a vendor name.
- **Solution-smell check.** Flag any elicited requirement that names a widget, button, screen, table, technology, or vendor as still-raw. Push it back through the why-chain until it states a capability instead.
- **Four-level classification pass.** Before drafting, sort every raw input into Business / Stakeholder / Solution-Functional / Solution-Non-functional / Transition. An input that doesn't fit any of the five is either out of scope or not yet understood well enough to record.
- **Necessity test on every candidate requirement.** "What happens if we don't build this?" A requirement with no consequential answer is a preference — label it as such, don't let it compete for Must/Should priority on equal footing.
- **Testability gate.** "How will we know this has been satisfied?" is asked of every requirement before it's considered done being written, not deferred to test design later.
- **Progressive elaboration / rolling wave.** Fully elaborate only the requirements needed for the next planned unit of work; keep everything beyond that horizon at business/stakeholder-level detail and revisit it closer to when it will actually be built.
- **Elicitation technique matched to failure mode being guarded against**: interview for depth/sensitivity, workshop for cross-stakeholder consensus, observation for tacit knowledge, document analysis for an existing-system baseline, prototyping when stakeholders can't visualize from text, survey for breadth across a dispersed population.
- **Transition requirements checklist**, run once per release regardless of how "done" the functional/NFR set looks: data migration, user training, cutover/rollback plan, parallel-run period, legacy system decommission.

## Key Terminology

Candidates worth checking against `skills/glossary-management/` for inclusion in the Ubiquitous Language, if not already present:

- **Business Requirement / Stakeholder Requirement / Solution Requirement / Transition Requirement** — the book's four-level hierarchy; if the glossary doesn't already distinguish these, `requirements-analysis`'s flat Requirement Types table has no vocabulary to express the bridge layer between a business goal and a functional requirement.
- **Progressive Elaboration (Rolling Wave)** — worth naming explicitly if this repo ever wants to justify eliciting NFRs or edge-case detail at lower fidelity for later release slices than for the current one, rather than demanding full detail upfront for everything.
- **Transition Requirement** — currently has no home in this repo's vocabulary or artifact set at all; worth adding even before deciding which agent owns it, so the gap is at least named.

## Caveats

- **This repo's requirements-analyst has no live stakeholders to elicit from.** It reasons over already-written Strategy-phase artifacts (vision, OKRs, stakeholder map, GTM strategy, competitive analysis), not a real person in a room. Several of the book's elicitation techniques — facilitated workshops, observation/job-shadowing, prototyping walkthroughs — assume synchronous access to a stakeholder and don't transfer as-is. The techniques that do transfer cleanly are the ones that operate on already-recorded input: document analysis (this repo's dominant mode), and the root-cause "why" chain (which the agent can run against written stakeholder concerns, even without a live interview).
- **"Lean" in this book still means a durable requirements record exists** — it argues against a heavyweight BRD/FRS/SRS produced once and frozen, not against having a requirements document at all. This repo's FRD + backlog dual-track is consistent with that reading; don't take "lean and agile" as license to drop the FRD in favor of stories-only.
- **Precise chapter-level specifics (page numbers, named case studies, exact terminology the Hathaways use for a given technique) are not claimed here with high confidence** — the concepts above reflect the consistent, well-established themes across the Hathaways' BA-EXPERTS teaching body of work (of which this book is part), not a verified line-by-line citation of this specific title. Treat this file as a reliable conceptual grounding, not a page-referenced excerpt.
