## Core Concepts

All items below are general UX/narrative-design practice, not verified book-specific claims:

- **Story as a communication and empathy tool, distinct from story as an analysis grid.** A persona's bullet-point attributes or a journey map's table rows are *analysis* — they decompose a user's experience into inspectable parts. A *story* — written as prose, with a protagonist, a situation, stakes, and an outcome — is a *communication* artifact: it is how that analysis gets carried into a stakeholder's head so they feel the friction, not just read a severity label. Well-established practice (Quesenbery & Brooks, *Storytelling for User Experience*) treats these as complementary, not substitutes: build the grid to analyze, write the story to persuade and align.
- **Narrative arc (setup → tension → resolution) as a diagnostic.** A UX story that has no tension — no moment where something could go wrong or where the user is stuck — is usually a symptom that the underlying research is too shallow or too flattering. This is the same diagnostic idea as this repo's own `user-journey-mapping` skill's "no valley = flattery, not evidence" anti-pattern, generalized to any narrative artifact, not just the emotional-arc chart.
- **"Day in the life" vignettes for personas.** A short narrative — a paragraph or two, present tense, first- or third-person — that puts the persona in a concrete moment ("It's 8:40am. Maya's third coffee is going cold because a Slack message just told her the customer's security questionnaire is due Friday...") makes the persona's attributes cohere as a person, not a spec sheet. This is standard practice traceable to Cooper's goal-directed personas and Kim Goodwin's scenario-based design.
- **Storyboards and scenario sketches as pre-flow communication.** Before a detailed, branching user flow is drawn (decision nodes, error states, ASCII notation), a 2-4 panel storyboard or one-paragraph scenario narrates the moment in human terms. This isn't a replacement for the technical flow — it's a framing device that keeps the technical flow anchored to why the human is here, useful when presenting to a non-engineering stakeholder.
- **Stories surface tacit knowledge that structured questions miss.** Asking "what are your goals?" produces recited goals. Asking someone to narrate a specific recent occasion surfaces what they actually did, including workarounds, frustrations, and exceptions they wouldn't think to list — the same underlying insight that Domain Storytelling (see Caveats) and Jobs-to-be-Done interviewing both rely on, applied here to UX communication rather than domain/event modelling.
- **Stories are told for a specific audience and purpose, and should be built for it.** A story for engineering emphasizes system states and edge cases; a story for an executive stakeholder emphasizes the business stakes and the emotional cost of the status quo. The same underlying journey can and should be narrated differently depending on who is in the room — a point relevant to how this repo currently has exactly one audience (Shafi) for every artifact.

## Actionable Techniques

- **Write a one-paragraph narrative vignette before/alongside every primary persona.** Present tense, one concrete morning or moment, grounded in the persona's actual trigger and frustration attributes — not a new invention, a dramatization of attributes already captured.
- **Precede a technical user flow with a 2-3 sentence scenario framing.** State who, why now, and what's at stake, before the ASCII decision-node diagram begins.
- **Apply the narrative-arc check to any story-shaped artifact.** Does it have a setup, a moment of friction/tension, and a resolution? A story with only setup and resolution (no tension) should be treated the same way this repo's `user-journey-mapping` skill treats an all-satisfied emotional arc: a warning sign, not a clean bill of health.
- **Name the audience for each narrative before writing it.** One line at the top of any narrative artifact: "written for [Shafi / engineering / a future customer-facing reader]" — because the same journey narrated for different audiences legitimately emphasizes different details.
- **Use a concrete, specific occasion, never a generalized "usually."** ("Last Tuesday, Maya got a Slack ping mid-lunch..." not "Typically, compliance officers get notified...") — generalizations smooth over the exact frictions a story is meant to reveal. (This is the same discipline this repo's `domain-storytelling` skill already enforces for domain-expert narration — see Caveats for why the two are not the same technique.)
- **Read the story back to whoever it represents, when a real source exists.** If a persona or journey vignette is grounded in actual user research (an interview, a support ticket), the narrative should be checked against that source rather than left as an unvalidated dramatization — and should be labeled clearly if it's an assumption rather than validated.

## Caveats

- **Confidence caveat, repeated:** the Core Concepts and Actionable Techniques sections above are general, well-established UX storytelling practice, not verified claims from this specific book. If a future session can access the actual text of *Agile UX Storytelling* by Rebecca Baker, this file should be revised against it rather than treated as authoritative.
- **This is not the same technique as this repo's existing `domain-storytelling` skill, and the two must not be conflated.** `domain-storytelling` is Stefan Hofer and Henning Schwentner's specific, named DDD technique: a domain expert narrates a work process aloud while a modeller draws it live using a fixed pictographic notation (Actor / Work Object / Activity / Annotation / Boundary), run as a structured session with a strict protocol (choose scenario → narrate → read back → clarify → run variations), and its purpose is Bounded Context and Ubiquitous Language discovery for DDD. The UX-storytelling practice referenced in this file has a different purpose (building stakeholder empathy and communicating already-gathered persona/journey data), a different form (written prose narrative or storyboard, not live pictographic drawing), and a different audience relationship (written for a reader/reviewer, not co-created live with a domain expert). Any future skill built from this research file should be named and scoped to avoid the word "storytelling" alone implying Hofer/Schwentner's technique — e.g. keep it under `user-persona`/`user-journey-mapping` as narrative *output formats*, not a standalone `ux-storytelling` skill that could be mistaken for a second domain-storytelling variant.
- **Do not add narrative prose as a required field.** Everything proposed above is additive and optional — a companion rendering of data the skills already require, not a new required artifact. Adding a mandatory narrative section would risk artifact bloat and gives Shafi (a non-programmer PM reviewing without IDE tooling) two versions of the same fact to reconcile; keep it strictly as an aid to the existing approval-gate conversation, not a new deliverable with its own quality gate.
- **Any narrative built from unvalidated assumptions must be labeled as such**, mirroring `user-journey-mapping`'s existing "Invented verbatims" anti-pattern — a persuasive story built on invented details is more dangerous than a dry, honestly-incomplete table, because prose reads as more authoritative than it is.
