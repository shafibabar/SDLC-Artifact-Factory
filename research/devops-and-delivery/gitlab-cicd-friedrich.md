## Core Concepts

- **The pipeline is a DAG of stages and jobs, defined in one `.gitlab-ci.yml`.** Jobs belong to `stages:` (e.g. `build`, `test`, `scan`, `deploy`); by default all jobs in one stage run in parallel and the next stage waits for the whole previous stage to finish. This stage-sequential-by-default model is GitLab's baseline mental model — every other mechanic below is a refinement of it.
- **`needs:` breaks the stage-sequential default into a true DAG.** A job can declare `needs: [job-a, job-b]` to start the moment its specific dependencies finish, without waiting for the rest of its nominal stage — this is the mechanism that turns "wait for the whole `test` stage" into "start `deploy` as soon as `unit-tests` and `lint` are green, even if `integration-tests` is still running." This is the single most load-bearing performance concept in modern GitLab pipeline design.
- **Artifacts pass build output between jobs; cache passes reusable input within a job's own repeated runs.** `artifacts:paths` on one job (e.g. a compiled binary or a Trivy report) makes that output automatically available to every job in a later stage (or every `needs:` dependent). `cache:` is a different mechanism for a different purpose — speeding up repeated dependency installs (`node_modules`, Go module cache) across pipeline runs, not for handing output from job to job. Conflating the two is a common and costly mistake this book would flag.
- **Merge Request pipelines vs. branch pipelines vs. merged-results pipelines.** A branch pipeline runs against the branch tip as pushed. A Merge Request pipeline runs specifically in the context of an open MR and can access MR metadata (target branch, MR labels). A merged-results pipeline goes one step further: it runs CI against a *temporary merge commit* of the MR's changes onto the target branch — catching integration problems the source branch alone wouldn't surface — before that merge actually happens.
- **Merge trains** queue multiple approved MRs and validate each one's merged-results pipeline against the *previous* MR in the queue already merged in — preventing the classic "two individually-green MRs break main when merged together" failure, at the cost of pipeline queue depth on a busy repo.
- **Environments and deployment tiers give the pipeline a live model of what's actually deployed where.** A job with `environment: {name: staging, url: https://staging.example.com}` registers a deployment GitLab tracks — the Environments UI shows, per environment, what's currently deployed, its pipeline, and a one-click "Stop environment" / rollback action. `deployment_tier` (`production`, `staging`, `testing`, `development`, `other`) classifies environments for reporting.
- **Manual jobs (`when: manual`) and protected environments are how GitLab implements human approval gates.** A job set to manual halts the pipeline until a human clicks "play"; combined with a **protected environment** (only specific users/groups may deploy to it), this is GitLab's native mechanism for "a human approves before production" — not a bespoke PR-review convention layered on top of the pipeline.
- **Auto DevOps is GitLab's opinionated, zero-configuration default pipeline.** Given almost no `.gitlab-ci.yml` at all, Auto DevOps auto-detects the language, and runs Build → Test → Code Quality → SAST → Dependency Scanning → Container Scanning → (License Scanning) → Review Apps → DAST → Deploy → Performance testing, deploying to Kubernetes via an auto-generated Helm chart. It is the book's likely showcase for "batteries-included" GitLab CI/CD and the platform's most GitLab-proprietary concept — see Caveats.
- **Parent/child pipelines let one pipeline trigger others, including pipelines generated dynamically at runtime.** A parent pipeline's job can `trigger:` a static child `.gitlab-ci.yml` (useful for splitting a huge monorepo pipeline into per-component files) or, further, generate a **dynamic child pipeline** — a YAML file written by a script during the parent pipeline's own execution — and trigger that generated file as a child pipeline. This is a genuinely GitLab-specific capability with no direct GitHub Actions analog (see translation table).
- **`include:` composes reusable pipeline configuration; `extends:` composes reusable job definitions.** `include:` pulls in whole YAML files (local, from another project, or a public template) the way a program imports a module. `extends:` lets one job inherit and override another job's keys — GitLab's mechanism for DRY job templates within one file, distinct from `include:`'s file-level composition.
- **The built-in Container Registry is a first-class, tightly-integrated GitLab feature**, not a bolt-on — every project gets one automatically, with pipeline variables (`CI_REGISTRY`, `CI_REGISTRY_IMAGE`) pre-wired for push/pull with no separate setup.

## GitLab → GitHub Actions Translation

| GitLab CI/CD concept | GitHub Actions equivalent | Notes |
|---|---|---|
| `.gitlab-ci.yml` (one file, `stages:` + `jobs:`) | `.github/workflows/*.yml` (one file per workflow) | GitHub has no single-file-per-project convention; this repo already uses multiple files (`reusable-go-ci.yml`, per-service callers) — a *closer* fit to GitLab's `include:`-split style than to GitLab's single-file default. |
| `needs:` (DAG within a pipeline) | `needs:` (same keyword, same purpose) | Direct, near-exact translation — GitHub Actions natively supports job-level DAGs via `needs:`, not just stage-sequential ordering. This repo's `reusable-go-ci.yml` already uses `needs: gates` for the `image` job — the DAG concept is present but not yet exploited for parallelism (see Applies To). |
| `artifacts:paths` (pass output between jobs) | `actions/upload-artifact` + `actions/download-artifact` | Same purpose, more explicit: GitHub requires an upload step and a matching download step naming the artifact, rather than GitLab's implicit "available to the next stage." |
| `cache:` (speed up repeated installs) | `actions/cache`, or the built-in `cache:` input on `actions/setup-go`/`actions/setup-node` | This repo already does this correctly (`cache: true` on `setup-go`, `cache: npm` on `setup-node`) — no gap here. |
| Merge Request pipeline | `pull_request` trigger | Direct equivalent — `ci-pipeline`'s PR trigger already plays this role. |
| Merged-results pipeline (CI against a temporary merge of MR into target) | `pull_request` event's default checkout (GitHub checks out the merge commit, `refs/pull/<n>/merge`, by default) | GitHub Actions does this *automatically* for `pull_request` (not `pull_request_target`) — functionally the same integration-catching behavior GitLab makes an explicit, separately-named pipeline type. |
| Merge trains | GitHub **merge queue** (native feature, separate from Actions but integrates with required checks) | Closest real equivalent; not identical semantics (GitHub's merge queue re-validates against a temporary combined ref for each queued PR) — genuinely comparable, not a stretch. |
| `environment:` + Environments UI (what's deployed where) | GitHub **Environments** (`environment:` key on a job, deployment history, environment URL) | Direct equivalent for the *tracking* half. GitHub's Environments feature is less visually rich than GitLab's Environments dashboard but covers the same core need — a live model of what's deployed per environment. |
| Protected environments + `when: manual` (human approval gate) | GitHub Environments' **required reviewers** protection rule | Direct equivalent — a GitHub Environment with required reviewers pauses the job (shows as "Waiting") until an authorized reviewer approves, functionally identical to GitLab's manual-job-in-a-protected-environment pattern. |
| Auto DevOps | **No direct equivalent** | See Caveats — this is the platform's most proprietary capability. |
| Parent/child pipelines (static `trigger:`) | Reusable workflows (`workflow_call`) | Close functional equivalent for the *static* case — this repo's Pipeline of Pipelines (`reusable-go-ci.yml`/`reusable-react-ci.yml` called by per-service files) is architecturally the same idea: shared, versioned building blocks, one caller per unit. |
| Dynamic child pipelines (YAML generated at runtime, then triggered) | **No direct equivalent** | GitHub Actions workflows are resolved from the committed file at trigger time; there is no first-class "generate a workflow file mid-run and trigger it as a new run" primitive. Partial substitutes: a `matrix:` strategy (data-driven parallelism, but the job shape is still static) or a job that writes a script and runs it inline (loses the "separate pipeline run" visibility GitLab gives a child pipeline). |
| `include:` (compose pipeline config from files) | `workflow_call` (reusable workflows) + composite actions | Reusable workflows cover the "call another job sequence" case; composite actions cover the "reuse a sequence of steps within a job" case — together they span most of what `include:` does, split across two GitHub mechanisms instead of GitLab's one. |
| `extends:` (job-level inheritance/override) | **No direct equivalent** in workflow YAML; composite actions or reusable workflows as the practical substitute | GitHub Actions has no keyword-level job inheritance. The Pipeline of Pipelines pattern (`ci-pipeline`) already sidesteps the need by keeping the *shared* logic in one reusable workflow and letting callers vary only their `with:` inputs — a different but equally DRY solution to the same problem `extends:` solves. |
| Built-in Container Registry (`CI_REGISTRY`, zero setup) | GitHub Container Registry (`ghcr.io`) | Also first-class and tightly integrated (this repo already uses it in `ci-pipeline`'s `docker/login-action` step against `ghcr.io` with `GITHUB_TOKEN`) — a genuinely close, already-adopted equivalent, not a gap. |
| Pipeline schedules | `schedule:` trigger (cron syntax) | Direct equivalent — `ci-pipeline`'s `nightly-suites.yml` already uses this. |
| `rules:`/`only`/`except` (conditional job execution) | `if:` conditionals on jobs/steps, plus `on:` trigger filters (`paths:`, `branches:`) | Functionally equivalent; GitHub splits the same decision across workflow-level triggers (`on.pull_request.paths`) and job/step-level `if:` expressions rather than one unified `rules:` block. |
| GitLab Runner executors (`docker`, `kubernetes`, `shell`) | GitHub-hosted runners, self-hosted runners, `container:` key on a job | Equivalent range of options; GitHub's default (GitHub-hosted `ubuntu-latest`) covers this repo's needs today with no self-hosted runner infrastructure to operate. |
| Review Apps (ephemeral per-MR environment, semi-automated) | No built-in feature — hand-built via a workflow that provisions a namespace/environment per PR (e.g. `environment: pr-${{ github.event.number }}`) and tears it down on PR close | GitHub gives you the primitives (dynamic environment names, a `closed` PR event to trigger teardown); GitLab packages the whole lifecycle (create-on-MR-open, auto-stop-on-merge/close, review app URL in the MR widget) as one feature. |

## Key Terminology

Worth considering for `skills/glossary-management/`'s Ubiquitous Language — evaluated for genuine tool-agnosticism, not GitLab-specific jargon:

- **Deployment Environment** (as a tracked, live-state concept — "what is currently deployed where," distinct from an *environment* as mere infrastructure) — both GitLab and GitHub model this as a first-class object; this repo's `environment-config` skill uses "environment" only as a configuration-and-topology noun, never as a *tracked deployment-state* concept the way GitLab's Environments UI or GitHub's Environments feature does. Worth a distinction, not necessarily a new term.
- **Pipeline DAG** — the general concept of jobs depending on specific prior jobs rather than waiting on a whole preceding stage; genuinely tool-agnostic (GitLab's `needs:`, GitHub's `needs:`, and most CI systems' equivalent) and not currently named anywhere in this repo's skills despite `ci-pipeline` already using the mechanism.
- **Protected Environment** — an environment a deploy job can only target with specific, pre-authorized approval; tool-agnostic in concept (GitLab protected environments, GitHub Environment required reviewers) even though the exact configuration mechanics differ. Distinct from the already-glossaried `Principle of Least Privilege` — this is its deployment-pipeline-specific instance.

Everything else genuinely load-bearing in this book (Auto DevOps, merge trains, dynamic child pipelines, `extends:`) is GitLab-specific tooling vocabulary, not tool-agnostic practice — deliberately excluded from this list per the task's instruction to flag, not glossary, GitLab-only jargon.

## Caveats

- **Auto DevOps has no direct GitHub Actions equivalent, and adopting its philosophy wholesale would fight this repo's own architecture.** Auto DevOps's value proposition is *convention-driven, near-zero-configuration* CI/CD — GitLab infers language, builds a Docker image, generates a Helm chart, and deploys, all from almost no YAML. This repo's `ci-pipeline`/`cd-pipeline` are the opposite: explicit, hand-authored reusable workflows plus a hand-authored GitOps reconciliation model, chosen deliberately for control and auditability over the "batteries included, trust the platform's defaults" tradeoff Auto DevOps makes. Do not read this book's Auto DevOps chapters as a template to replicate here — the two are opposite design philosophies, not translatable mechanics.
- **Dynamic child pipelines (a YAML file generated at pipeline runtime and triggered as a new pipeline) are a genuinely GitLab-specific capability.** There is no GitHub Actions primitive that resolves a workflow definition generated *during* a run and re-triggers it as a fresh, separately-tracked run. If this repo ever needs highly dynamic, data-driven pipeline generation (e.g. a variable number of per-tenant deploy jobs decided at runtime), the practical GitHub substitute is a `matrix:` strategy computed from a prior job's output (`fromJson()` on a generated list) — closer in spirit but a materially different mechanism, not a drop-in replacement.
- **The built-in Container Registry's tight coupling (`CI_REGISTRY_IMAGE`, zero-setup auth) is a GitLab-hosting-specific convenience, not a portable concept.** This repo already has a functionally equivalent setup with GHCR (`ghcr.io`, `GITHUB_TOKEN` auth) — the *convenience* doesn't transfer as a concept because GitHub's registry is equally first-class; this is a non-issue for this repo, not a gap, but worth noting so a reader doesn't look for a "missing" registry integration that in fact already exists under a different name.
- **Merge trains solve a real problem (compounding merge-order failures) this repo has not yet encountered at its current single-operator scale.** GitHub's merge queue is the closer equivalent, but adopting it before multiple concurrent contributors exist would be premature tooling relative to the Budget and Frugality principle in `CLAUDE.md` — flagged here as "translatable, not yet needed," not as a missing gap to fill immediately.
- **This book's security-scanning chapters (SAST, DAST, Dependency Scanning, Container Scanning, License Scanning as GitLab-branded, license-tiered features) largely restate practice this repo already implements with open-source tooling** (Trivy for container scanning, `govulncheck`/`npm audit` for dependency scanning, per `ci-pipeline`) — cite this book, if at all, for the *practice* of wiring scanning into pipeline stages, never for its specific GitLab-product feature names, several of which (DAST, some SAST rulesets) are gated behind paid GitLab tiers and would conflict with this repo's frugality constraint if adopted as named.
- **I could not confirm this book's exact chapter numbering or edition-specific content with confidence**, unlike the Humble/Farley and Kim et al. sibling files which cite specific chapters. Every claim above is anchored to stable, long-documented GitLab platform mechanics (the `.gitlab-ci.yml` model, `needs:`, environments, Auto DevOps, parent/child pipelines) that a book with this title would necessarily cover, rather than to a specific chapter or page — treat this file as a reliable guide to *GitLab CI/CD mechanics broadly*, not as a verified summary of this book's specific prose or structure.
