# Site Reliability Engineering: How Google Runs Production Systems
**Authors:** Betsy Beyer, Chris Jones, Jennifer Petoff, Niall Richard Murphy (eds.)
**Domain:** DevOps & Delivery — SRE and Observability Discipline

## Core Concepts

- **Service Level Indicators, Objectives, and error budgets as the reliability contract (Ch. 4, "Service Level Objectives," Chris Jones/John Wilkes/Niall Murphy/Cody Smith).** An SLI is a carefully specified quantitative measure of some aspect of the service (a ratio of good events to valid events); an SLO is a target range for an SLI; an SLA is the SLO's external, financially-backed cousin, deliberately looser so the internal target trips first. The book's central move is turning "how reliable should this be?" from an unbounded aspiration ("as reliable as possible") into a *budget*: `1 − SLO` is the amount of unreliability the system is allowed to spend, and that budget is a resource to be spent deliberately — on launches, on risky migrations, on experiments — not hoarded. This repo's `skills/slo-definition/SKILL.md` already implements this chapter closely and accurately: its SLI-type table (availability/latency/freshness/correctness), its error-budget arithmetic worked example (`0.5% × 40,320 minutes ≈ 201.6 minutes`), and its explicit "when the budget is exhausted, feature deploys pause" spend policy are a faithful, well-grounded rendering of Ch. 4's argument — not a gap, a **confirmation** (see Applies To).
- **Burn rate as the enforcement mechanism, and the multiwindow, multi-burn-rate alerting method — with an important attribution nuance.** Chapter 4 introduces burn rate conceptually: `burn rate = observed error ratio / budget fraction`, where a burn rate of 1 exhausts the budget exactly at the end of the SLO window and a burn rate of *N* exhausts it in `window / N`. This is the concept `slo-definition` already states correctly ("Burn rate 14.4 — the budget is gone in ~2 days"). But the **specific, canonical multiwindow multi-burn-rate table** this skill and `alerting-rules-design` both use verbatim — the paired long/short windows, the 14.4×/6×/1× multipliers, the 2%/5%/10% budget-consumption thresholds for a 28-day SLO — was formalized two years later in the **SRE Workbook (2018)**, Google's sequel to this book, specifically Chapter 5, "Alerting on SLOs" (Steven Thurgood and David Ferguson). This 2016 book supplies the *concept* (budget, burn rate, the need for both a sensitivity and a precision window) but not the worked arithmetic table itself. This is exactly the kind of nuance the Continuous Delivery research file in this same directory already flagged from the other direction — it attributes canary's burn-rate gating to "later SRE-lineage thinking" without naming a source; this file is that source, and the honest answer is *two* Google texts, not one, with the Workbook doing the specific derivation.
  - **The derivation logic, reconstructed from the Workbook's method (since the skill uses it without ever showing where it comes from):** pick how much of the 28-day error budget a given burn pattern is allowed to consume before it must page, and over what long window that consumption is measured. For the fast-burn tier: "no more than 2% of the 28-day budget consumed within 1 hour" is the design goal. A burn rate is "the budget's target lifetime divided by *N*" — so consuming 2% of a 28-day budget inside a 1-hour window is the same rate as consuming 100% of it in `1h / 0.02 = 50h ≈ 2.08 days`. The multiplier `14.4` is nothing more than `28 days / 2.08 days ≈ 13.4×`... more precisely, the Workbook's own algebra is `burn rate threshold = (budget consumption fraction) × (SLO window / alerting window)`: for the fast-burn row, `0.02 × (28 days / 1 hour) = 0.02 × 672 = 13.44`, close to (and in the Workbook's published table, exactly) `14.4` once the window boundaries are set to the exact hour and the budget-consumption target is tuned to a round number (Google's own table rounds to 2%/1h precisely to land on 14.4). The **short window** (5m for fast burn, 30m for slow burn, 6h for the trickle) exists purely so the alert *stops firing quickly after the underlying problem is fixed* — a long window alone would keep paging for the length of the long window even after recovery, because the long window's average includes the now-resolved bad period. This confirms `alerting-rules-design`'s own framing ("long window: is this real? short window: is it still happening?") is exactly right; the skill's gap is only that it presents the 14.4/6/1 numbers as received wisdom with no derivation and no citation, which is a documentation-quality gap, not a correctness gap — the numbers are right, the "why 14.4 and not 15" question is currently unanswerable from the skill text alone.
- **The Four Golden Signals — latency, traffic, errors, saturation (Ch. 6, "Monitoring Distributed Systems," Rob Ewaschuk and Betsy Beyer).** If you can only measure four things about a user-facing system, measure these: **latency** (how long requests take — critically, *distinguishing successful-request latency from failed-request latency*, because a failed request can return very fast and silently drag a p99 down to a falsely comforting number); **traffic** (demand on the system, in whatever unit fits — requests/second, concurrent sessions, bytes ingested); **errors** (rate of requests that fail, explicitly or implicitly — a wrong-content 200 counts); and **saturation** (how "full" the service is — the resource closest to a limit, framed as a fraction of capacity, ideally with a forward-looking projection: "at this rate the disk fills in four hours," not just "disk is at 80% now"). The book's explicit advice for a team that has only a little monitoring maturity: start with these four and expand later, rather than building an ad hoc pile of dashboards with no organizing principle. This maps unevenly onto the repo: latency, traffic, and errors are all present and correctly symptom-based (SLO burn on availability and latency SLOs *is* traffic-normalized error and latency monitoring), but **saturation has no generic treatment** — see Applies To.
- **Toil — a formal, four-part-plus definition, not a vibe (Ch. 5, "Eliminating Toil," Vivek Rau).** Toil is operational work that is: **manual** (a human runs it by hand), **repetitive** (done the same way each time, not a novel investigation), **automatable** (a machine could do it — this distinguishes toil from genuinely irreducible engineering judgment), **tactical** (interrupt-driven, reactive to the moment, not part of a strategic plan), **devoid of enduring value** (the system is no better off after the task than before — a restart doesn't teach the system anything), and — the property that makes it dangerous rather than merely annoying — **scales linearly with service growth** (twice the traffic, twice the manual restarts), whereas the engineering work that reduces toil ideally scales sub-linearly. Google's target ceiling, stated as an organizational policy in the book: **SRE teams should spend no more than 50% of their time on toil**, with the rest protected for engineering work that reduces future toil — a formal, tracked, budgeted quantity, not an incidental complaint. The book's stronger position (beyond "write a good runbook") is: a manual procedure repeated three times is a candidate for automation, and the aspirational end-state for a well-run alert is not a well-drilled human response but *no human response required at all* — the system self-heals and the alert becomes unnecessary. `alerting-rules-design`'s "prune ruthlessly" monthly review discipline is aimed at the same target (a pager that is silent for weeks and then only ever right) but frames it entirely as *alert hygiene* (keep/tune/delete an alert), never as a *toil budget* (how much human manual effort did the surviving alerts actually cost, and is that shrinking or growing as the platform scales) — see Applies To for the precise gap.
- **Being On-Call — sustainable load, not heroics (Ch. 11, Andrew Stribblehill).** The book states concrete, if Google-scale, guidance: a healthy on-call rotation keeps the fraction of time spent in reactive/interrupt work (on-call plus its associated toil) below roughly 25–50% of an engineer's total time, and a single on-call shift should see events "sparingly" — the oft-cited rule of thumb is that a shift generating more than a couple of genuine incidents is itself a signal the system (or the alert design) is unhealthy, independent of whether each individual incident was handled well. Sustainable on-call, in the book's telling, is a *system property* the whole team is responsible for maintaining (via toil reduction, alert tuning, and automation), not a personal endurance trait of whoever happens to be holding the pager. The book's rotation mechanics — primary/secondary on-call, ideally spread across at least two, preferably more, geographically distributed engineers so no one shift exceeds a sustainable length, with clean handoffs and a documented escalation path to a secondary if the primary doesn't acknowledge in time — assume a *team*, and this is the sharpest team-vs-solo-operator tension in the whole book (see Caveats, following the precedent the DevOps Handbook research file in this same directory already set for its own "swarming" content).
- **Blameless Postmortems, Google's own chapter — distinct from, and more structurally detailed than, the DevOps Handbook's treatment (Ch. 15, "Postmortem Culture: Learning from Failure," John Lunney and Sue Lueder).** This repo's existing `research/devops-and-delivery/devops-handbook-kim-humble-debois-willis.md` already found a real gap in `alerting-rules-design` — its monthly review stops at "keep/tune/delete" and never asks whether a page that required genuine human action produced a structural corrective change — and credited that finding to the DevOps Handbook's Ch. 14, itself credited "substantially to Etsy's practice and John Allspaw's writing." This SRE book has its *own*, independently-run chapter on the identical practice, published the same year (March 2016, versus the DevOps Handbook's October 2016), grounded in Google's internal postmortem process rather than Etsy's. It is not a restatement — it adds three concrete mechanisms the DevOps Handbook finding does not mention at all:
  1. **Explicit, written trigger criteria** for when a postmortem is *mandatory*, not a judgment call made after the fact at a monthly review. Google's stated criteria (paraphrased from the book): user-visible downtime or degradation beyond an agreed severity threshold; any data loss; an incident that required a human to be paged and to actively intervene; a resolution time above a threshold; a monitoring or alerting failure (something broke and *nothing paged* — arguably the most important trigger of all, discussed below); and a repeat of a previously postmortem'd incident. This converts "should this have produced a structural fix?" from a retrospective monthly question into a same-day gate applied at incident-close time.
  2. **A specific document template**, more granular than the DevOps Handbook's "timeline, multiple contributing factors, corrective actions" summary: Summary, quantified customer-facing **Impact** (not "consumer lag exceeded threshold" but "X% of requests failed for Y minutes, affecting Z tenants"), **Root Cause(s)** stated in the plural, a **Detection** field (how was this actually noticed — monitoring, a customer report, a routine check?), a full **Timeline**, an **Action Items** table (task, owner, tracking-issue link, target date), and a closing **Lessons Learned** split into what went well / what went wrong / where the team got lucky (the last category is distinctively Google's own addition — it captures near-misses that worked out by chance, which a pure "what went wrong" framing would never surface).
  3. **Corrective-action tracking to actual completion**, not just to being written down. Google runs its postmortem action items as tracked issues with owners and due dates, and the book is explicit that a postmortem is not a completed process artifact the moment the document is written — it is complete when its high-priority action items close, and low completion rates are themselves treated as an organizational health signal worth escalating.
  The **Detection field** in particular closes the exact loop the DevOps Handbook finding names abstractly ("pages that fire and require real action should feed a postmortem whose corrective actions can include...alert changes") with a concrete mechanism: if a postmortem's Detection field reads "customer reported it, no alert fired," that sentence *is* the ticket to write a new symptom-based alert in `alerting-rules-design` — a structured trigger rather than a hoped-for connection.
- **Emergency response and effective troubleshooting as separable disciplines from monitoring (Ch. 12–14).** The book treats "how do you monitor" (Ch. 6), "how do you troubleshoot once paged" (Ch. 12), "how do you manage a live incident" (Ch. 13), and "how do you respond in a true emergency" (Ch. 14) as four distinct, sequential skills, each with its own chapter — troubleshooting is explicitly framed as a hypothesis-and-elimination discipline (not "try things until it works"), and incident management introduces the Incident Command System (borrowed, as the book says, from firefighting) with named roles (Incident Commander, Communications Lead, Operations Lead) for anything large enough to need coordinated response. This maps to `runbook-authoring`'s remediation decision trees reasonably well at the single-incident scale, but the Incident Command role structure is inherently a multi-person concept with no solo-operator analog beyond "the same person plays every role sequentially" — noted in Caveats rather than force-fit as a new skill.
- **Error budgets as the resolution to the Dev-vs-Ops incentive conflict (Ch. 3, "Embracing Risk," Marc Alvidrez).** The book's own framing, worth stating precisely because it is the philosophical core the burn-rate machinery serves: without an error budget, product development is incentivized to launch as fast as possible while SRE is incentivized to prevent outages, and the two goals are structurally opposed. An error budget converts the argument from a subjective negotiation ("is this safe enough to ship?") into an objective one ("is there budget left?") — when budget remains, launches proceed on the product team's schedule with no additional gate; when budget is exhausted, launches pause until the budget recovers or the underlying reliability problem is fixed, and this is agreed *in advance*, not litigated mid-incident. `slo-definition` already states this policy verbatim ("When the budget is exhausted, feature deploys pause and reliability work takes priority; that policy is written into the SLO document so the trade-off is decided in advance, not mid-incident") — a precise, correctly-scoped match to Ch. 3's argument, and correctly placed in `slo-definition` rather than `alerting-rules-design`, whose job is *detecting* burn, not *deciding policy* about it.

## Actionable Techniques

- Add a one-line derivation note (or a linked appendix) to `alerting-rules-design`'s burn-rate table showing the arithmetic — `budget-consumption-target × (SLO window / long window) = burn-rate multiplier` — so the 14.4/6/1 figures stop reading as received constants and become auditable if the SLO window or target ever changes (e.g., moving from a 28-day to a 30-day window shifts every multiplier and the skill currently gives no way to recompute them).
- Cite the SRE Workbook (2018) by name as the source of the specific multiwindow table, and this 2016 book as the source of the error-budget/burn-rate *concept* — the skill currently cites neither, and the distinction matters if a future rebuild ever needs to adjust the table's precision (the Workbook's chapter is the correct reference to consult, not this book).
- Reorganize `alerting-rules-design`'s existing symptom/cause table under the four golden signals as an explicit header dimension (Latency, Traffic, Errors already map to the existing SLO-burn alerts; Saturation currently has no generic row) — this doesn't change any existing content, it gives the existing content a named, citable organizing structure instead of an ad hoc "symptom vs. cause" framing invented for this skill alone.
- Add a generic, non-pipeline-specific **saturation** ticket alert pattern — `prometheus-metrics-design` already documents the exact PromQL (`db_pool_in_use / db_pool_max` approaching 1) as a dashboard-only pattern; wiring an equivalent trending alert (ticket severity, using `deriv()` the same way `PipelineConsumerLagGrowing` already does, to project time-to-exhaustion rather than just a static threshold) turns an existing query into the golden signal's fourth pillar instead of leaving saturation as dashboard-only for every non-pipeline resource.
- Split latency histograms (or at minimum the `histogram_quantile` recording rule) by success/failure outcome before computing p99 — the book's specific warning is that failed requests can return fast and pull a percentile down misleadingly; `prometheus-metrics-design`'s current `service:http_request_duration_seconds:p99_5m` recording rule sums across all requests regardless of `http_response_status_code`, and `slo-definition`'s latency SLI ("requests completing < 800ms / all requests") has the identical blind spot.
- Add a toil line to the monthly alert review in `alerting-rules-design` — not just "keep/tune/delete" per alert, but "did resolving this again require the same manual steps as last time, and if so, is this now a 3rd-plus repetition that should be scripted instead of paged" — the review already asks "did a human act?"; this adds "and should a human still have to?"
- Add explicit, written postmortem trigger criteria (user-visible downtime above a threshold, any data loss, a page that required active intervention, resolution time above a threshold, or — the most valuable one — an incident detected by something other than monitoring) as a gate applied at incident close, feeding a postmortem document distinct from `runbook-authoring`'s lightweight post-incident note on the runbook itself.
- Make "Detection" an explicit field on that postmortem document (how was this actually noticed) so "detected by customer report, not monitoring" becomes a structured, always-visible trigger for a new `alerting-rules-design` symptom alert, rather than something that has to be remembered and connected manually.
- Track corrective action items from postmortems as real, dated, owned tasks tracked to closure — not merely documented — mirroring the same discipline `runbook-authoring`'s freshness log already applies to runbooks themselves.
- Translate the book's sustainable-pager-load guidance into a solo-operator heuristic: if any given week produces more than a small, fixed number of real pages (not tunable noise), that volume is itself the trigger for an unscheduled alert-hygiene pass, rather than waiting for the monthly cadence — applying the book's "on-call load is a system health signal" idea without pretending a rotation exists to spread the load across.

## Candidate New Skills/Agents

One genuine candidate emerges, and it is not a new invention — it is the same recommendation the DevOps Handbook research file already made independently, now reinforced with enough distinct structural content (trigger criteria, a named template, corrective-action tracking, a Detection field) that it clears the bar for a standalone skill rather than a section added to an existing one:

- **`postmortem-authoring`** (deploy phase, owner `platform-engineer`): would own the postmortem *document* — trigger criteria (when one is mandatory), the template (Summary, Impact, Root Causes, Detection, Timeline, Action Items, Lessons Learned), blameless-framing style rules (contributing factors plural, no individual blame, same discipline `runbook-authoring` already applies to its own style rules), and a corrective-action tracker with closure discipline. It would explicitly **not** own: the runbook itself (`runbook-authoring` keeps the per-alert and per-procedure runbooks, and its lightweight "post-incident hook" note becomes a pointer *into* the postmortem rather than a duplicate of it); the alert definitions or the monthly alert-hygiene review (`alerting-rules-design` keeps "keep/tune/delete," and gains a new input — "does this alert's fired-and-actioned history have an open postmortem with unresolved action items?"); and the SLO/error-budget policy (`slo-definition` keeps the spend policy untouched). This mirrors the DevOps Handbook's own suggestion almost exactly, which is itself a reason to trust the call rather than a coincidence to explain away — two independent primary sources, on two different chapters/books, both point at the same missing artifact.

Everything else surfaced above is better handled as an addition to an existing skill, not a new one, and it is worth being explicit about why so this section doesn't pad itself with restatements of the Applies To table:

- **Saturation as a golden signal** and **success/failure-split latency percentiles** are additions to `alerting-rules-design` and `prometheus-metrics-design` respectively — they are new *rows in existing tables* (a new generic alert pattern, a filtered recording rule), not new domains of expertise requiring their own owner.
- **Toil tracking** is a new *column or check* in `alerting-rules-design`'s existing monthly review, not a new artifact type — there is nothing to author, only something additional to ask at a review that already exists.
- **On-call escalation and sustainable pager load** do not warrant a new skill for the same reason the DevOps Handbook file gave for not inventing team-coordination mechanics for "swarming": a solo operator has no rotation to formalize, so the honest output is a caveat and a small addition to `runbook-authoring`'s escalation table (an "unacknowledged page" leaf), not a fabricated on-call-scheduling skill serving a team that doesn't exist.

## Key Terminology

Checked against `skills/glossary-management/references/ubiquitous-language.md` (231 lines, ten sections) before proposing — its "Reliability and Operations" section already defines `Service Level Indicator`, `Service Level Objective`, `Service Level Agreement`, `Health Check`, `Observability`, `Monitoring`, `Metrics`, and several resilience terms, but has **no entry at all** for the error-budget/burn-rate vocabulary that `slo-definition` and `alerting-rules-design` both already use constantly, nor for anything from this book's toil, golden-signals, or postmortem chapters:

- **Error Budget** — `1 − SLO target`, the quantity of allowed unreliability, spent deliberately rather than merely tracked. Used throughout `slo-definition` and `alerting-rules-design` today with no canonical glossary definition backing it — arguably a higher-priority glossary gap than anything net-new this book introduces, since the term is already load-bearing in two existing skills.
- **Burn Rate** — the speed at which an error budget is being consumed, normalized as `observed error ratio / budget fraction`; a burn rate of 1 exhausts the budget exactly at the SLO window's end. Same situation as Error Budget: already in constant use, not yet defined.
- **The Four Golden Signals** — the organizing framework (Latency, Traffic, Errors, Saturation) for what to monitor first on any user-facing system. Distinct from, and narrower than, the already-glossaried `Metrics`/`Monitoring`/`Observability` entries — this is a specific selection framework, not a synonym for monitoring in general.
- **Saturation** — how "full" a service is relative to its most constrained resource, ideally expressed as a trend toward exhaustion rather than a static percentage. The fourth golden signal, and currently absent from the glossary as a term even though `prometheus-metrics-design`'s PromQL table already computes exactly this quantity for a DB pool without naming it.
- **Toil** — manual, repetitive, automatable, tactical, enduring-value-free operational work that scales linearly with service growth. A precise, load-bearing term the book insists on defining exactly (not "busywork" or "grunt work" loosely) because its automatability and its linear-scaling property are what make it a policy target (the 50% ceiling) rather than a complaint.
- **Postmortem Trigger Criteria** — the explicit, written thresholds (downtime duration, data loss, a page requiring intervention, resolution-time threshold, detection-by-non-monitoring-means) that make writing a postmortem mandatory rather than discretionary.
- **Corrective Action** (or **Action Item**) — a specific, owned, dated task produced by a postmortem, tracked to actual closure rather than merely recorded in the document.
- **On-Call** and **Escalation Policy** — surprisingly absent from the current glossary given how central both are to `runbook-authoring` and `alerting-rules-design`'s Alertmanager routing; worth defining even in this repo's honestly-scoped, solo-operator form (see Caveats) rather than only in a team-based sense the repo doesn't use.
- **Not proposed here — already flagged:** `Blameless Postmortem` itself is already listed as a candidate glossary term in `research/devops-and-delivery/devops-handbook-kim-humble-debois-willis.md`'s own Key Terminology section, credited there to that book's Ch. 14/Etsy lineage. This file does not re-propose it as new; it instead notes (in Core Concepts, above) that this book's Ch. 15 is an independent, same-year, Google-internal origin for the identical term and practice — worth citing alongside the DevOps Handbook's Etsy attribution if the term is ever added, not as a competing claim but as a second, equally valid lineage.

## Caveats

- **On-call rotation mechanics are the book's most team-shaped content, and the honest translation is scoping, not adaptation** — following the exact precedent `research/devops-and-delivery/devops-handbook-kim-humble-debois-willis.md` already set for its own "swarming" content. Primary/secondary on-call across multiple engineers and time zones, follow-the-sun coverage, and a rotation large enough (the book cites eight-plus engineers across multiple locations as a comfortable minimum) to keep any individual's on-call burden sustainable simply do not have a solo-operator analog — there is no secondary to escalate to, and no rotation to spread load across. Cite the book's *principle* (on-call load is a measurable system-health signal, not a personal endurance contest) rather than its mechanics (rotation scheduling, timezone handoffs), exactly as the DevOps Handbook file already recommended for team-coordination content generally.
- **The Incident Command System (Ch. 13–14) is inherently multi-role.** Incident Commander, Communications Lead, and Operations Lead as separate people coordinating a live incident has no meaningful solo-operator equivalent beyond "the same person performs each role in sequence, badly, under stress" — worth naming as an honest limitation in any future runbook-authoring rebuild rather than pretending role separation adds value at a headcount of one.
- **The precise multiwindow multi-burn-rate table is a 2018 SRE Workbook artifact, not this 2016 book's.** This book supplies the error-budget and burn-rate *concepts* (Ch. 3–4) that make the table possible, but the specific 14.4×/6×/1× multipliers and their paired windows were formalized in the sequel. Any future citation in `alerting-rules-design` or `slo-definition` should name both texts precisely rather than crediting this one book with content it only set up.
- **Chaos Engineering and continuous fault injection are conspicuously thin in this book** (a few paragraphs referencing Netflix, not a chapter) — the DevOps Handbook research file already covers this territory (crediting it, correctly, to that book's Ch. 14 Netflix case study) and this file adds nothing beyond confirming the same gap exists here; do not double-count it as a new finding.
- **The 50%-toil ceiling and the "SRE team" organizational model assume a distinct SRE function separate from product engineering** — Google's SRE org famously has the standing authority to hand a service back to its product-development team if reliability work exceeds the toil ceiling for too long. This repo has one `platform-engineer` agent that is simultaneously "SRE" and "the only ops function there is," with no product-dev team to hand a misbehaving service back to. The toil ceiling still works as a personal sustainability heuristic (a rising, unaddressed toil footprint is itself a signal worth surfacing to Shafi, since it is ultimately a headcount/tooling-investment decision only he can make), but the "give it back to Dev" escalation valve the book relies on does not exist here and should not be invented.
- **This book predates Kubernetes, containers-as-default, and OpenTelemetry (published 2016; Google's own internal tooling — Borgmon, its monitoring system — is described in the book and was never open-sourced).** The book's monitoring philosophy (the golden signals, symptom-over-cause alerting, burn-rate budgets) translates directly and is stack-agnostic; its literal tooling references do not. Worth noting as a point in this repo's favor rather than a gap: Prometheus (this repo's actual metrics backend, per `prometheus-metrics-design` and `CLAUDE.md`'s Tech Stack Defaults) was built by an ex-Google engineer explicitly inspired by Borgmon, so the repo's existing stack is already the open-source spiritual successor to the tooling this book describes — the philosophy transplant this file is doing was, in effect, already done once by Prometheus's own designers.
- **The postmortem chapter's "postmortem review committee" (a cross-team body that periodically checks postmortem quality and action-item completion rates across the whole engineering organization) has no solo-operator equivalent and should not be built as a formal recurring ceremony** — the useful residue is the discipline itself (mandatory triggers, a real template, tracked-to-closure action items), which one person can apply to their own past incidents without a committee to enforce it; a `postmortem-authoring` skill (see Candidate New Skills/Agents) should say so explicitly rather than gesturing at a review body that will never convene.
- **The book is now a decade old (2016) and Google has since published two follow-ups** (the *SRE Workbook*, 2018, cited above for the burn-rate table, and *Building Secure and Reliable Systems*, 2020) that refine several of this book's chapters with more field-tested specifics. Treat this book as the origin and the philosophical grounding — the "why" behind symptom-based alerting, error budgets, toil, and blameless postmortems — and treat the Workbook as the better source if a future research pass needs the more mechanical "how" (the burn-rate table's exact arithmetic is the clearest example already surfaced in this file).
