# Learning DevOps
**Author:** Mikael Krief
**Domain:** DevOps & Delivery — Tooling Practice

> **Confidence note:** This book (Packt; 1st ed. 2019, 2nd ed. 2022, "Learning DevOps: The Complete Guide to Accelerate Collaboration with Jenkins, Docker, Kubernetes, Terraform, and Azure DevOps") is a broad, tool-by-tool practical walkthrough rather than a principles text — its value is worked, runnable configuration, not novel theory. I have solid confidence in its toolchain scope and the practices below (they match well-established, widely-taught Terraform/Docker/Kubernetes/Ansible/CI-CD practice that this specific book is known for teaching hands-on); I do not cite chapter numbers, unlike the two sibling files, because I'm not confident enough in exact chapter boundaries across its two editions to assert them. Where a claim is closer to "standard practice this book teaches" than "a claim unique to this book," I've said so rather than overstating novelty.

## Core Concepts

- **The book teaches a toolchain, in pipeline order, not a philosophy.** Its structure mirrors an actual path from a developer's commit to a running, monitored system: source control (Git/GitFlow) → CI (Jenkins/Azure Pipelines/GitHub Actions) → configuration management (Ansible) → machine imaging (Packer) → Infrastructure as Code (Terraform) → containerization (Docker) → orchestration (Kubernetes/Helm) → security scanning woven throughout → monitoring (Prometheus/Grafana). Where *Continuous Delivery* argues for the pipeline as a concept and *The DevOps Handbook* argues for the culture around it, this book's contribution is the literal, tool-specific "how" at each stage — closer in altitude to this repo's own Skill files than either sibling.
- **Infrastructure as Code with Terraform: state as the map, not the territory.** The book teaches the practical mechanics this repo's `opentofu-module` already implements independently — a remote backend with locking so state isn't corrupted by concurrent applies, `plan` as a reviewable artifact before `apply`, and modules as the unit of reuse. Its treatment of **workspaces** is presented as the default multi-environment mechanism (one configuration, `terraform workspace select dev|staging|prod`), a lighter-weight but less explicit alternative to directory-per-environment.
- **Golden images via Packer + Ansible, composed with Terraform.** A recurring pattern in the book: Packer builds an immutable machine image (baking OS packages, agents, and configuration via an Ansible provisioner into the image at build time), and Terraform then provisions infrastructure *from* that pre-baked image rather than configuring a live instance after boot. This is the VM-era equivalent of "build once, promote everywhere" — the image is the artifact; instances are disposable copies of it, never hand-tuned after launch.
- **Ansible for configuration management: idempotent, agentless, declarative playbooks.** Playbooks describe desired host state (packages, files, services) and running them twice produces the same result as running them once — the same idempotency discipline this repo's ubiquitous language already uses in the event-driven sense, applied here to server configuration instead of message handling. Roles are the reuse unit (Ansible's rough equivalent of a Terraform module or a Helm chart).
- **Docker fundamentals as a craft, not just a `FROM` line.** Multi-stage builds to keep build tooling out of the final image, `.dockerignore` to keep the build context small, deliberate `COPY` ordering so dependency-install layers cache independently of source-code layers (changing application code shouldn't invalidate the expensive dependency-install layer), and `docker-compose` as the standard way to run a multi-container stack (app + database + broker) identically on every developer's laptop.
- **Kubernetes and Helm as the deployment target, taught object-by-object.** Pods, Deployments, Services, Ingress, ConfigMaps, Secrets, and RBAC (Roles/RoleBindings/ClusterRoles) are each introduced as a concrete YAML manifest before Helm is introduced as the templating and packaging layer over them — charts, values files per environment, and a chart repository (`helm repo index`/ChartMuseum-style hosting) as the distribution mechanism for teams that install charts rather than only GitOps-reconcile them.
- **DevSecOps: security tooling as pipeline stages, not a separate department.** Static analysis (SonarQube), dependency/secret scanning, and container image scanning (Trivy/Clair/Aqua) are taught as gates wired directly into the CI pipeline — conceptually the same "shift left" argument as *The DevOps Handbook*'s Ch. 20, but here realized as literal pipeline YAML rather than an organizational argument.
- **Monitoring closes the loop.** Prometheus for metrics collection and Grafana for dashboards are presented as the practical capstone — once something is deployed via the whole toolchain above, it must be observable, or none of the automation before it can be trusted to have worked.

## Actionable Techniques

- Order Dockerfile instructions so dependency-manifest copies (`go.mod`/`package.json`) and their install step precede the full source copy — an unchanged dependency set replays a cached layer instead of reinstalling on every build.
- Maintain a `.dockerignore` alongside every Dockerfile (`.git`, local env files, build output) so the build context stays small and secrets never enter it by accident.
- Provide a `docker-compose.yml` for local multi-container development (app + database + broker) so a new contributor's environment matches production topology without touching Kubernetes.
- Treat `terraform`/`tofu plan` output as a review artifact posted to the PR — the plan is what gets approved and applied, never a fresh re-plan at apply time.
- Pin provider and module versions explicitly; treat version bumps as reviewed PRs, never floating references.
- Where a golden-image workflow applies (VM-based compute, not pure containers), bake configuration into the image at build time via a provisioner rather than configuring a live instance after boot — the image is the artifact.
- Write configuration-management playbooks/roles to be idempotent — running twice must be a no-op the second time; test this explicitly, don't assume it.
- Author Kubernetes RBAC (Role/RoleBinding) per workload identity, scoped to the minimum verbs/resources that workload actually needs — least privilege applied to the API server, not just the network.
- Package and version Helm charts through a chart repository (or an OCI registry) when charts need to be installed outside a GitOps reconciler's direct Git reference — not only relevant to a Flux-only shop, but worth knowing as the more common Helm distribution path.
- Wire at least SAST, dependency scanning, and container image scanning into the CI pipeline as blocking stages, each producing a report artifact reviewable without re-running the scan.
- Stand up Prometheus scraping and a starter Grafana dashboard as part of "done" for a new service, not as a follow-up ticket.

## Key Terminology

Worth considering for `skills/glossary-management/`'s Ubiquitous Language — none of these currently appear in `CLAUDE.md`'s term lists:

- **Immutable Infrastructure** — the principle that a golden image or container image is never modified after being built, only replaced; this repo already practices it fully at the container layer (`dockerfile-patterns`) but has no glossary term naming the practice itself.
- **Golden Image** — a pre-baked, versioned machine or container image serving as the single build artifact instances are stamped from; relevant if this platform ever provisions VM-based node pools directly, and arguably a useful reframing of what a Dockerfile's final stage already produces.
- **DevSecOps** — security tooling embedded as pipeline stages rather than a separate audit; distinct from and narrower than "Zero Trust Architecture" (already glossaried), specifically naming the *pipeline-stage* mechanism rather than the runtime-trust model.
- **Chart Repository** — a versioned, queryable store of packaged Helm charts (via `helm repo index` or an OCI registry), distinct from this repo's current GitOps-direct-reference model; worth naming even if the deliberate choice remains not to use one.
- **Configuration Drift** — a nearly-glossary-ready term this repo already uses in substance (`opentofu-module`'s "drift is an incident," `environment-config`'s promotion-invariance gate) but never as a defined term — this book (like most IaC literature) treats it as foundational vocabulary.

## Caveats

- **This book is heavily Azure-centric.** Its CI/CD walkthroughs default to Azure DevOps Pipelines and Azure Key Vault, with Jenkins as the secondary tool and GitHub Actions receiving comparatively lighter treatment (more so in the 1st edition than the 2nd). This repo's GitHub Actions-only CI/CD (`ci-pipeline`, `cd-pipeline`) requires translating every pipeline-YAML example, not reading it directly — the underlying stage sequence (build → test → scan → publish → deploy) transfers cleanly; the exact YAML syntax does not.
- **Terraform, not OpenTofu.** Both editions predate OpenTofu's 2023 fork (a reaction to HashiCorp's license change); every example is written against Terraform directly. HCL syntax, `plan`/`apply` semantics, module and state mechanics are functionally identical between the two, so translation is close to a search-and-replace of the binary name — the one substantive difference worth checking against is provider registry resolution (OpenTofu can use its own registry or mirror the Terraform registry), which doesn't affect anything in `opentofu-module` as currently written.
- **Ansible is not in this repo's stack at all, and the translation is not trivial.** The book gives Ansible real weight as the configuration-management layer for VM-based compute. This platform is Kubernetes-native and container-immutable end to end (`dockerfile-patterns`, `kubernetes-manifest`) — there is no live host to converge configuration onto, so Ansible's core value proposition (idempotent convergence of *mutable* servers) doesn't map onto this stack's model at all. Don't adopt Ansible's playbook mechanics here; if a VM-level need ever arises (a self-managed node image, a bastion host), the closer conceptual fit is the book's Packer golden-image pattern, not Ansible's live-configuration pattern.
- **Jenkins-specific material (a substantial fraction of the book, especially the 1st edition) is not relevant** given this repo's committed GitHub Actions default — skip those chapters' mechanics entirely rather than translating them; they don't carry principles GitHub Actions lacks.
- **The book's security chapter names specific commercial/hosted tools (SonarQube Cloud, Azure Key Vault, Aqua) where this repo already has open-source-first equivalents in place or implied** (Trivy for scanning per `dockerfile-patterns`, Vault for secrets per `environment-config`'s config/secret split) — cite the book for the *practice* (scan in-pipeline, secrets never in Git) not for its specific tool endorsements, consistent with this repo's frugality constraint.
- **Kubernetes/Helm chapters use generic or cloud-managed clusters (AKS, Minikube) with no multi-tenancy concern.** This repo's physical per-tenant multi-tenancy model (`multi-tenancy-design`) and its consequences (directory-per-tenant OpenTofu, per-tenant Helm values, per-tenant NetworkPolicy) go well beyond anything in this book's Kubernetes coverage — treat the book as a source for base object mechanics (Ingress, RBAC) only, not for fleet-at-scale patterns already better covered by this repo's own skills.
