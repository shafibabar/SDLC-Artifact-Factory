# Continuous Delivery
**Authors:** Jez Humble and David Farley
**Domain:** DevOps & Delivery

## Core Concepts

- **The Deployment Pipeline Is a Single, Automated Path (Ch. 5).** Every change — code, configuration, or data migration — travels through one automated pipeline from commit to production; there is no second, manual, or "hotfix" path that bypasses stages. The pipeline's job is to make the release process itself boring, repeatable, and visible to the whole team, not just to engineers.
- **Staged Progression: Commit → Automated Acceptance → Manual/Exploratory → Release (Ch. 5, 7–10).** A build is a **release candidate** the moment it passes the fast **commit stage** (compile, unit tests, static analysis — minutes, not hours). It then runs through an **automated acceptance stage** (functional acceptance tests, and separately, nonfunctional tests — capacity, performance, security, per Ch. 9) against a deployed instance of the exact candidate. Only a candidate that clears both automated stages is eligible for **manual/exploratory testing** (UAT, showcases), and only then for **release**. A candidate that fails any stage stops there — it does not proceed on a schedule or a manager's say-so.
- **"If It Hurts, Do It More Often, and Bring the Pain Forward" (Ch. 1).** The book's central mantra. Anything painful — deployment, releasing, database migration, integration — should be done more frequently and earlier in the cycle, not deferred, because frequency is what forces automation and removes the pain; deferring it only lets the pain compound.
- **Build Once, Promote the Same Binary Everywhere (Ch. 2, 6).** A binary artifact is built exactly once by the commit stage and that same artifact — never rebuilt, never recompiled — is what every later stage tests and what production ultimately runs. Rebuilding "for staging" or "for prod" invalidates every test the earlier build passed; if it isn't the bytes you tested, you haven't tested what you're shipping.
- **Continuous Integration Means Trunk, Not Long-Lived Branches (Ch. 3).** Every developer commits to mainline at least once a day. The build (commit stage) runs on every commit, and if it breaks, fixing it is the team's first priority — not a queued ticket. Long-lived feature branches are explicitly discouraged because they defer integration pain (merge conflicts, divergent assumptions) to the worst possible moment; the book's preferred alternatives are **feature toggles** and **branch by abstraction**, not version-control branching.
- **Deploy and Release Are Different Activities (Ch. 10).** *Deploying* means putting a new version of software into an environment (it may be running, dormant, or toggled off). *Releasing* means making it available to (some or all) users. Conflating the two forces big-bang cutovers; separating them is what makes feature toggles, canary releasing, and blue-green deployment possible in the first place — you deploy continuously, and release on a separate, deliberate schedule.
- **Continuous Delivery vs. Continuous Deployment (Ch. 1, and throughout).** In **Continuous Delivery**, every change that passes the pipeline is *proven releasable* — a human still decides when to press the release button. In **Continuous Deployment**, that decision is also automated: every change that passes every stage ships to production with no human gate at all. The book's default recommendation is Continuous Delivery; Continuous Deployment is a further, optional step some teams take once they trust the pipeline completely.
- **Configuration Management Means Versioning Everything Needed to Recreate the System (Ch. 2).** Source, build scripts, environment/infrastructure definitions, dependency versions, and database migration scripts all belong in version control. Configuration that legitimately differs per environment (endpoints, credentials, feature states) is injected at deploy time and kept separate from the versioned binary — never baked in, never hand-edited on a live box.
- **Database Changes Ride Alongside Code, Never Ahead or Behind (Ch. 11).** Schema changes are automated, versioned migration scripts, applied through the same pipeline as code, and written so that the current and previous application version can both run against the migrated schema — the book's origin of the backward-compatible migration discipline this repo calls expand/contract.
- **Nonfunctional Requirements Are Pipeline Stages, Not End-of-Project Sign-Off (Ch. 9).** Capacity, performance, and security testing are automated gates inside the pipeline, run continuously against every candidate (or as often as feasible), not a manual audit bolted on right before a release date.

## Actionable Techniques

- Commit to mainline at least once a day; hide unfinished work behind a feature toggle instead of a long-lived branch.
- Keep the commit stage fast — the book's own rule of thumb is single-digit minutes — and push anything slower (full acceptance suites, load tests) into a later, asynchronous stage.
- Treat a broken commit-stage build as the team's top priority to fix — "stop the line" — before starting new work.
- Build the release candidate binary exactly once; every later stage and every environment consumes that identical artifact by reference, never by rebuild.
- Inject all environment-specific configuration at deploy time; never compile an environment assumption into the binary.
- Make the pipeline's state visible to the whole team — anyone should be able to see, at a glance, which stage the latest (or any) commit has reached.
- Gate progression past the commit stage on automated acceptance tests run against a deployed instance of the actual candidate — not a developer's local run.
- Automate at least one nonfunctional gate (capacity, performance, or security) as a pipeline stage rather than a manual pre-release checklist item.
- Separate "deployed" from "released": deploy continuously; decide exposure to users (all, some, none) as an independent, reversible action.
- Ship every schema change as a versioned migration script that keeps the current and immediately-prior application version compatible with the migrated schema.
- Rehearse rollback on a schedule — an untested rollback path is a belief, not a capability.
- Provision every non-production environment (test, staging) through the exact same automated process used for production, never a hand-built approximation of it.

## Key Terminology

Worth considering for `skills/glossary-management/`'s Ubiquitous Language — none of these currently appear in `CLAUDE.md`'s platform-terms list, which already has `GitOps`, `Blue-Green Deployment`, and `Canary Deployment` but nothing naming the pipeline-stage vocabulary those patterns operate inside of:

- **Deployment Pipeline** — the single, automated, end-to-end path from commit to production; this repo has the mechanics (split across `ci-pipeline` + `cd-pipeline`) but no glossary term naming the whole thing as one artifact.
- **Commit Stage** / **Automated Acceptance Stage** — precise names for what `ci-pipeline`'s PR gates and nightly suites already are, useful shorthand in review so "the fast gates" and "the slow gates" aren't reinvented as prose each time.
- **Continuous Delivery** vs. **Continuous Deployment** — distinct terms for "every build is provably releasable" vs. "every good build ships automatically." This repo implements the former; naming both prevents the two from being used interchangeably in future artifacts.
- **Trunk-Based Development** — the practice `ci-pipeline` and `feature-flag-design` both assume but never name; citable shorthand for "commit to main at least daily, no long-lived branches."
- **Deploy vs. Release** — the book's foundational distinction that `feature-flag-design`, `canary-deployment`, and `blue-green-deployment` all implicitly rely on (code can be present in production without being exposed to users) but no glossary entry currently separates the two verbs.

## Caveats

- **The book predates containers, Kubernetes, and GitOps entirely (2010).** Its "deployment pipeline" is normally realized as one CI/CD tool pushing binaries to servers via scripting (the authors co-created GoCD around this model). This repo's pull-based GitOps reconciliation (`cd-pipeline`) is a legitimate, arguably stronger, translation of the same underlying goals (versioned desired state, one path, provable promotion) — cite the book's *principles* (build once, promote the same artifact, make config external), not its push-based agent/script mechanics.
- **The book's single-tool pipeline visibility doesn't exist as one dashboard here.** Its emphasis on "everyone can see what stage every build is in" was originally a literal tool UI; in this repo that same value-stream visibility is achieved conceptually across GitHub Actions run status, Git history, and promotion PRs across three separate skills. That's a valid architectural translation, but it does mean no single artifact currently answers "where is this commit right now" — flagged above as `cd-pipeline`'s gap, not a caveat to dismiss.
- **Canary Deployment's SLO-burn-rate automation is barely in this book.** The 2010 edition predates widely-adopted SRE error-budget practice; its canary mention is a passing pattern reference, not a worked mechanism. Treat `canary-deployment`'s actual depth as informed primarily by later observability practice (Google SRE-lineage thinking), not this book — don't over-attribute its sophistication here.
- **Branching examples assume 2000s-era centralized-ish version control norms**, predating Git's dominance. Translate "commit to trunk daily" literally as `git commit && git push origin main` (or short-lived branch + same-day PR merge), not toward any specific branching model the book's era might have implied.
- **Chapter 11 (Managing Data)'s backward-compatible migration discipline is already fully implemented, under different naming, by the backend-engineer's `go-migration` expand/contract skill** (not one of the six read here) and consumed correctly by `blue-green-deployment` and `canary-deployment`. This document doesn't re-derive that content — it only notes the book as its origin.
