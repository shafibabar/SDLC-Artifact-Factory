# Mastering Context Engineering for AI Agents
**Authors:** Yuan Zhu
**Domain:** AI Agent Engineering / Context Window & Memory Management

**A note on scope, and how this file relates to the one before it:** `research/ai-agent-engineering/mastering-claude-code-agent-skills-gale.md` covers this repo's *skill architecture* — how one `SKILL.md` is shaped, split, versioned, and discovered. This book covers a different layer entirely: how an agent's **context window itself** is budgeted and how **state persists across sessions** that never share conversation history. Its subject in this repo is `sdlc-context.json` (896 lines, 94,912 bytes as read for this file) and `CLAUDE.md` (266 lines, 16,846 bytes) — the two files every session loads before doing anything else — plus, secondarily, the `Agent` tool's subagent-dispatch pattern used to spawn research tasks like this one. Every citation below points at what these two files actually contain today, not at a downstream product's artifacts.

## Core Concepts

- **A context window has three cost tiers, and this repo's own Session Startup rule collapses two of them into one always-resident load (Ch. 2–3).** Zhu's framing: content an agent needs falls into *always-resident* (loaded on every turn regardless of task), *session-resident* (loaded once per session, cheap relative to the task at hand), and *retrieved-on-demand* (fetched only when a specific need arises). `CLAUDE.md`'s "read on every session" instruction and `sdlc-context.json`'s "read this file at the start of every session — before reading anything else" instruction (its own `_meta.how_to_use`, step 1) both land in the *always-resident* tier — together **111,758 bytes / 1,162 lines** are read cold, in full, before a single line of actual work happens in any session, per CLAUDE.md's own "Session Startup" three-step rule. Compare this to `skills/*/SKILL.md`'s progressive disclosure (covered by the Gale file): skills defer their Tier 3 `references/` content until a trigger fires. `sdlc-context.json` has no equivalent tiering — its `build_checklist` array carries full deliverable lists for all 31 chunks, Chunk 1 through Chunk 31, every session, whether or not that session's work touches chunk 1's skeleton-folder history at all.
- **Structured memory across sessions is exactly what `sdlc-context.json` already is — the book's name for the pattern is "durable state object," and it specifies what such an object must guarantee (Ch. 4).** Zhu's requirement for a durable state object: it must be self-sufficient (restorable without the conversation that produced it), append-safe (new state doesn't corrupt old readers), and bounded (growth must have a stated ceiling or compaction plan). `sdlc-context.json`'s `_meta.purpose` states the self-sufficiency goal almost verbatim ("restores full project context... without re-deriving it from conversation history") and decision D008 records this as a deliberate architectural choice ("Context window compaction and new sessions lose conversation history. This file must be self-sufficient..."). The file satisfies self-sufficiency and append-safety completely. It satisfies **bounded growth not at all** — there is no stated ceiling anywhere in the file or in `CLAUDE.md`'s "Context File Maintenance" section, and the file has grown from a Chunk 1 skeleton to 896 lines across 31 chunks with no compaction event ever recorded.
- **Context rot is a *silent* failure mode — the book's warning is that a stale entry doesn't announce itself, it just keeps costing tokens while quietly becoming wrong (Ch. 6).** This is checkable, concretely, in this repo's own `decisions` array (27 entries, D001–D027). Several are narratively superseded by later ones but carry no machine-readable marker of that fact: D024 ("End the plan-start/plan-exit GitHub Project ceremony... for future individual skill refactors") is explicitly reversed *for one case* by D025 ("Re-introduced GitHub issue/sub-issue tracking... reversing D024's default, for this case"), which is itself corrected by D026 ("do not call `plan_status.py done`... only after Shafi has actually merged"). All three remain resident, undated as to which is "current default behavior," with the reader expected to reconstruct the chain from prose alone. There is no `status`, `superseded_by`, or `supersedes` field anywhere in the decision schema — `grep -in "supersed\|archive\|deprecat"` across the whole file returns only two incidental prose mentions of the word "superseded," both inside unrelated ADR/Principle discussion, never as a schema field on a decision itself.
- **The single clearest, most concrete instance of context rot in this repo is a rule the repo's own governing document states and its own state file already violates (Ch. 6, "documented-but-unenforced state hygiene").** `CLAUDE.md`'s "Context File Maintenance" section states plainly: *"The `open_questions` array is cleared when questions are resolved."* But `open_questions` currently holds **Q001**, whose `status` field literally reads `"resolved — see decision D016"` — resolved, per the file's own words, and still present, contradicting the stated maintenance rule in the very file that rule governs. (Q002 and Q003 are genuinely open, correctly retained.) This is not a hypothetical risk the book warns about in the abstract; it is a rule this repo wrote for itself and has already broken once, caught by direct inspection, not by inference.
- **Retrieval-vs-preload is a spectrum, and this repo currently sits at "preload everything, forever" for its two context files while its skills sit at "preload trigger surface, retrieve the rest" (Ch. 3, cross-referenced against the Gale file's progressive disclosure model).** The book frames this as a single continuum an architecture should deliberately place content along, not a binary. `CLAUDE.md` and `sdlc-context.json` are both placed at the preload-everything end by explicit design choice (D008's rationale is precisely that compaction/new-session loss makes preload necessary) — but the book's point is that preload-everything is a *reasonable choice for content that changes the shape of every decision* (methodology non-negotiables, tech stack defaults, the current build position) and a *poor one for content that is historical record* (chunk 1–20's already-shipped deliverable lists, decisions that have since been narratively superseded). This repo has not yet drawn that line inside its own preloaded files — everything in `build_checklist` and `decisions` gets the same preload treatment regardless of whether it's still decision-shaping or purely archival.
- **Multi-agent context handoff needs a declared contract, not just a shared filesystem convention — and `domain-modeler.md` already has the strongest example of one in this repo (Ch. 8).** Zhu's chapter on inter-agent handoff distinguishes *ambient* handoff (agent B just happens to read files agent A wrote, hoping the shape matches) from *contracted* handoff (agent A's output schema and agent B's input schema are the same declared shape, checked before B proceeds). `agents/domain-modeler.md`'s Inputs table (six named artifacts, each with a `Source` file-path pattern and a `Required?` column) plus its explicit halt rule ("If any required input is missing, the domain-modeler halts and reports the gap to Shafi before proceeding") is close to a contracted handoff — but the check is prose-enforced by the agent's own judgment at runtime, not validated against `schemas/sdlc-manifest.schema.json` or any other machine-checkable artifact-manifest entry before the agent starts. The three **Handoff Rules** sections at the bottom of the same file (to enterprise-architect, data-architect, test-strategist) are the *outbound* half of the same contract, equally prose-only.
- **Context isolation via subagent forking is a deliberate technique for keeping a long-running orchestrator's own window clean — and this exact pattern is what produced this file (Ch. 9).** The book's chapter on multi-agent context management distinguishes a *shared, ever-growing single context* (every subtask's full reasoning stays resident in the parent's window) from *forked, isolated contexts* (a parent dispatches a self-contained brief to a subagent, which returns only a distilled result — its intermediate reasoning, file reads, and tool calls never re-enter the parent's window). This conversation's own research-agent dispatches (this file, and the Gale file before it) are a real instance of Ch. 9's forking pattern: each research agent received one self-contained prompt naming exactly what to read and produce, ran its own full context window of file reads and reasoning in isolation, and returned only a short final report — none of the fork's intermediate reads (all 1,162 lines of `CLAUDE.md`+`sdlc-context.json`, both source books, `domain-modeler.md`) land in the parent conversation's window at all. This repo has no written convention naming this pattern or stating *when* to fork a subagent versus keep working in the parent's own context — it happens by construction whenever the `Agent` tool is invoked, but nothing documents the selection criteria (task is self-contained and its intermediate detail is disposable → fork; task needs to build incrementally on the parent's own accumulated context → don't).

## Actionable Techniques

- Add a `status` field (`active` | `superseded`) and, where applicable, a `superseded_by` field to every entry in `sdlc-context.json`'s `decisions` array going forward — retrofitting the D024→D025→D026 chain costs three one-line edits and immediately replaces "read three prose paragraphs to infer which is current" with a single queryable field.
- Enforce `CLAUDE.md`'s own stated rule literally: when an `open_questions` entry's `status` becomes `resolved`, remove it from the live array in the same edit that adds the resolving decision — Q001 should not still be present. If the intent going forward is "mark resolved and retain as history" rather than "clear," update the stated rule in `CLAUDE.md` to say that instead of leaving a written rule the data already contradicts.
- Adopt a **decision-archival threshold**: once the `decisions` array exceeds a stated count (this repo is already at 27 — a reasonable threshold to adopt now is 25, meaning archival is already due), move decisions with no unresolved forward reference (nothing later in the array cites them as still-governing) into a companion `decisions-archive.json` or a `DECISIONS-ARCHIVE.md`, leaving only a one-line pointer (`"D003 — superseded, see decisions-archive.json"`) in the live file. `CHANGELOG.md` already exists as a durable, dated record of every chunk — decisions tied 1:1 to a chunk already narrated there are a natural first archival candidate.
- Adopt an equivalent **compaction rule for `build_checklist`**: once a chunk's deliverables are fully narrated in `CHANGELOG.md` (which the build workflow already requires for every chunk) and no later chunk's `deliverables` text references it as a live dependency, collapse that chunk's entry in `sdlc-context.json` to `{chunk, title, status, date_completed}` only, dropping the full `deliverables` array — the detail is not lost, it already lives in `CHANGELOG.md`; it is only duplicated, at preload cost, in the state file.
- State an explicit **context-budget ceiling** for `CLAUDE.md` itself in `CLAUDE.md`'s own "Session Startup" or a new section — e.g., a soft line-count or byte target, checked manually at the end of any chunk that edits `CLAUDE.md`, with an explicit decision point ("this section is now big enough to move to a skill or a `references/`-style companion doc") rather than unconditional accretion. `CLAUDE.md` has already grown across at least Chunks 3, 4, 9 (implicitly, since methodology/glossary references were added), and 31 (Q003, D027) with no such check ever having been applied.
- Write down the **fork-vs-continue selection criteria** for subagent dispatch as a short rule of thumb (self-contained task, disposable intermediate detail, distillable to a short report → fork; task must build incrementally on context the parent already holds → continue in-place) so future sessions apply it deliberately rather than by whatever the `Agent` tool happens to make convenient in the moment.
- When `domain-modeler.md`'s (or any agent's) Inputs table lists a required artifact, treat "artifact exists at the stated path" as a mechanically-checkable pre-condition, not just a prose halt-rule the agent applies from memory — the manifest schema (`schemas/sdlc-manifest.schema.json`) already records enough to check this; nothing currently invokes it before an agent begins its Decision Process.

## Key Terminology

Worth considering for `skills/glossary-management/`'s Ubiquitous Language, or at minimum for the proposed `context-engineering-standards` skill's own vocabulary section:

- **Context Window Budget** — the total tier-weighted cost of everything loaded before a task begins; this repo's own always-resident budget is concretely 111,758 bytes / 1,162 lines (`CLAUDE.md` + `sdlc-context.json`) as of this writing, before any product-specific artifact is read.
- **Durable State Object** — the book's term for a file like `sdlc-context.json`: self-sufficient (restorable without conversation history — D008's own stated rationale), append-safe, and (the property this repo has not yet satisfied) bounded.
- **Context Rot** — state that remains resident and syntactically valid while becoming semantically stale or superseded, with no marker distinguishing it from still-current state. The D024→D025→D026 chain and the resolved-but-retained Q001 are both real, present-tense instances of this in `sdlc-context.json` today, not hypothetical risks.
- **Bounded Growth / Archival Threshold** — a stated ceiling (count, byte size, or line count) past which a durable state object must compact or archive rather than simply keep appending. Not currently stated anywhere for `decisions` (27 entries and counting), `build_checklist` (31 chunks and counting), or `CLAUDE.md` itself.
- **Context Contract** (contracted handoff, vs. **ambient handoff**) — a declared, machine-checkable input/output shape between two agents' work, as opposed to one agent's output happening to be readable by another's prose-stated expectations. `domain-modeler.md`'s Inputs table and Handoff Rules sections are close to a contract today but are checked only by the agent's own runtime judgment, not validated against `schemas/sdlc-manifest.schema.json`.
- **Context Isolation / Subagent Forking** — dispatching a self-contained brief to a subagent whose full intermediate context (file reads, reasoning, tool calls) never re-enters the parent's window, only its distilled final report does. This is exactly the pattern that produced this file and the Gale file before it, via the `Agent` tool — real, already in use, just never named or given selection criteria in this repo.
- **Preload vs. Retrieval-on-Demand** — the spectrum a piece of content sits on between "loaded for every session/task regardless of need" and "fetched only when a specific trigger requires it." `CLAUDE.md`/`sdlc-context.json` sit at the preload extreme by deliberate choice (D008); `skills/*/SKILL.md`'s `references/` tier (per the Gale file) sits at the retrieval extreme.

## Caveats

- **This candidate skill (`context-engineering-standards`) is deliberately kept separate from the Gale file's `skill-authoring-standards` candidate, but the two may eventually deserve merging.** Both would be cross-cutting, `factory-governance`-owned skills about how *this plugin's own components* should be shaped rather than about a downstream product's artifacts — `skill-authoring-standards` governs the shape and discovery of `SKILL.md` files; `context-engineering-standards` governs the shape and hygiene of `sdlc-context.json`/`CLAUDE.md`. They do not overlap in subject matter today (one is about prose-file architecture for skills, the other about state/memory persistence), but if this repo ever accumulates a third or fourth "how this plugin governs itself" skill, consolidating them under one `plugin-self-governance` umbrella (with these as its `references/` files) would be a natural future move — flagged here explicitly, per the instruction not to silently reproduce the same candidate under two names, rather than decided unilaterally now.
- **The book's chapters on distributed/sharded memory (multi-tenant agent fleets, vector-store-backed long-term memory, retrieval-augmented context assembly) assume a scale this repo does not have.** This is a single-operator, single-repository, single-Claude-session-at-a-time project — `sdlc-context.json` is read whole, by one agent, at the start of one session, not queried by a fleet of concurrent agents needing sharded access. Treat the book's memory-sharding and vector-retrieval chapters as inapplicable; the chapters on single-file durable state, context rot, and budget ceilings are the ones that actually apply here.
- **Do not read this book as license to add retrieval infrastructure (a vector database, an embedding-based search layer) to solve the growth problem.** The concrete finding above (27 decisions, 31 chunks, no archival) is real and worth fixing, but the fix that fits this repo's stated Budget and Frugality rule ("open-source over paid tooling... prefer simpler solutions") is a plain-JSON archival convention and a threshold check, not new infrastructure — the book itself distinguishes "you have an archival problem" from "you have a retrieval-at-scale problem," and this repo currently has only the former.
- **The book is agent-framework-agnostic and has never seen this repo's specific eight-phase structure, chunk-based build workflow, or `sdlc-context.json` schema.** Every mapping from "Ch. N's concept" to "this repo's file X, line Y" above is this document's own extrapolation onto real, already-existing content — including the specific byte/line counts, decision count, and the Q001 finding, all independently verified by reading the files rather than asserted from the book's own examples (which are necessarily generic).
- **This file and the Gale file were produced by exactly the technique Core Concept 6 describes** — each was a self-contained subagent dispatch whose full intermediate context (every file read while producing it) never entered the parent conversation's window, only this distilled report does. That is worth noting as a live demonstration of the pattern, not just an abstract description of it.
