# Mastering Claude Code Agent Skills: Building Modular, Production-Ready AI Systems with Skill-Based Architectures
**Authors:** Elbert Gale
**Domain:** AI Agent Engineering / Claude Code Skill Architecture

**A note on scope, before the notes themselves:** every other file in `research/` grounds a book against skills this plugin *teaches a downstream product to build* (Go handlers, DDD, testing, security). This book is different — its subject is the exact mechanism `CLAUDE.md` already codifies for *this repo's own* skills: flat `skills/<name>/SKILL.md` files, progressive disclosure via `references/`, frontmatter schemas, discovery, versioning, and testing. Every citation below points at this repo's own components, not at a downstream product's artifacts.

## Core Concepts

- **Progressive disclosure is a three-tier loading model, not just "split the file" (Ch. 3–4).** The book's central architectural claim is that a Skill has three distinct load tiers with different costs: (1) **metadata** — `name` + `description` — resident in context for every session regardless of whether the skill fires, so it must be cheap and dense; (2) the **SKILL.md body** — loaded only once the metadata triggers a match, so it can afford more detail but should still stay tight; (3) **`references/` files** — loaded only when the body explicitly points to them, so they can be as long and example-heavy as the task needs with zero cost to sessions that never need them. This repo already implements all three tiers correctly where it has been refactored: `skills/adr-authoring/SKILL.md` is a lean 147-line body that defers its full template and two worked examples to `references/output-format-template.md` and `references/worked-example.md`; `skills/access-control-model/SKILL.md` does the same, deferring Go constructor code and a trust-boundary checklist to `references/domain-primitives-and-enforcement.md`. `skills/event-storming-facilitation/SKILL.md`, read as the deliberate contrast case, has never been split — its worked example, full Output Format template, and card taxonomy all sit resident in a 273-line body every time the skill triggers, at Tier 2 cost for content that Tier 3 was invented for.
- **The `description` field is the discovery mechanism itself, not a summary written for humans (Ch. 5).** Gale's repeated warning is that teams write `description` as if it were documentation ("this skill covers ADRs") when its actual job is to be the trigger surface Claude's skill-matching logic scores against a live prompt — it needs the nouns, verbs, and named collaborators a real request would contain. This repo already does this well in practice without ever naming the principle: `skills/glossary-management/SKILL.md`'s description ends with the load-bearing trigger phrase "When any agent is unsure whether a term is canonical, they check this skill first" — a *usage condition*, not a topic summary. `skills/adr-authoring/SKILL.md`'s description names the owning agent ("Used by the enterprise-architect agent") and the phase, both of which sharpen matching. CLAUDE.md's Component Frontmatter table mandates the `description` field exists but never states *why* its wording matters or what makes one better than another — the discipline is present by convention, not by stated rule.
- **A Skill's single-responsibility boundary is a discovery problem, not just a tidiness one (Ch. 2).** The book frames "does this skill do one thing" as directly determining whether the matcher can confidently route to it — a skill that quietly covers two unrelated capabilities produces a description that is either too vague to match precisely or too long to stay cheap at Tier 1. This is the same boundary CLAUDE.md's Component Architecture table draws with "Skills = Expertise... No reasoning, no decisions" and the flat one-skill-per-directory rule (`skills/<name>/SKILL.md`) — but CLAUDE.md's stated reason is architectural purity (reasoning belongs in Agents), while the book adds a second, independent reason: single-purpose skills are also what keeps 141 concurrently-registered skills (this repo's actual count) individually matchable instead of degrading into an undifferentiated pile the matcher has to guess between.
- **Composability across skills needs an explicit dependency signal, not prose cross-references (Ch. 6).** Gale distinguishes *implicit* composability (Skill A's text happens to mention Skill B, hoping the reader — human or Claude — follows the pointer) from *declared* composability (a machine-checkable list of related/required skills a tool can validate). This repo currently only has the implicit form: `skills/access-control-model/SKILL.md` says "This is the same Domain Primitive concept `subdomain-distillation`'s `references/security-sensitive-subdomains.md` covers generally... this skill applies it specifically" — a prose pointer with no frontmatter field backing it, so nothing catches it if `subdomain-distillation` is ever renamed, split, or has that reference file moved. `agents/domain-modeler.md`'s `skills:` list is the closest existing structure to a declared-dependency mechanism in this repo, but it exists only on agents, never between two skills.
- **Skills are tested as behavior, not read for coherence (Ch. 9).** Because a Skill's content is prose consumed by a model rather than code executed by a runtime, the book insists the only trustworthy test is: give the loaded skill a real trigger prompt and assert on the resulting behavior/answer, exactly the way any other black-box system is tested. This is precisely what `tests/skills/adr-authoring.contract.sh` already does — it loads this repo live via `claude --plugin-dir`, asks the skill a narrow, near-verbatim-answerable question ("is a Principle superseded like an ADR, or revised in place?"), and asserts the substring `"revised in place"` appears in the response, per `tests/lib/harness.sh`'s own stated design rationale ("a narrow question with a quotable, near-verbatim expected substring... rather than an open-ended question"). CLAUDE.md's "Running Tests" section already documents this exact philosophy for skills, agents, commands, and hooks — the book gives it a named justification (prose cannot be statically verified; only invocation reveals whether it actually produced the intended behavior).
- **Versioning a Skill has two independently-breaking surfaces: the trigger surface and the content surface (Ch. 8).** The book's point is that a change to `description` can silently break discovery for prompts that used to match without a single word of the body changing — this is a different kind of "breaking change" than a body edit that changes what the skill teaches once triggered, and the two need different version-bump discipline. Every skill in this repo carries a `version` field (`skills/adr-authoring/SKILL.md` is at `2.0.0`, `skills/event-storming-facilitation/SKILL.md` at `1.1.0`, `skills/access-control-model/SKILL.md` at `2.0.0`) but CLAUDE.md never states a bump policy anywhere in its Component Frontmatter or Naming Conventions sections — there is no written rule for what makes a skill change MAJOR vs MINOR vs PATCH, and specifically no rule that a `description` edit is a different category of change than a body edit.
- **A `references/` file must be self-contained — loadable without the parent SKILL.md already in context (Ch. 4).** This is close to a direct quote of what this repo's own template already states: `skills/adr-authoring/references/output-format-template.md` opens with the line **"Self-contained — loadable without reading `SKILL.md` first."** The book frames this as a hard requirement of the progressive-disclosure model, not a nicety — if a Tier 3 file assumes Tier 2 context it can't rely on being present (Claude may re-load references/ independently on a later turn, or another skill may point at the same reference file), the whole tiering model breaks. This repo has already independently arrived at the rule; the book supplies the reason it must hold universally, not just for this one file.
- **There is no stated threshold for *when* a skill must split into `references/` — the book supplies one; this repo currently decides ad hoc (Ch. 4).** Gale's heuristic is a resident-content budget: once a SKILL.md body's Tier-2-resident content (prose the matcher-triggered load pulls in every time, regardless of whether the specific section is needed for this task) exceeds roughly what a reader would tolerate skimming in one sitting — his rule of thumb is content that is *reference material* (a full template, a worked example, an exhaustive checklist, generated code) rather than *decision-shaping guidance* (when to use X vs Y, what "good" looks like) — it should move to Tier 3. Checked against this repo: 19 of 141 skills currently have a `references/` directory (`find skills -maxdepth 2 -type d -name references` confirms the count); the other 122, including `event-storming-facilitation` at 273 lines with a full worked example and a full Output Format template both resident, have not been evaluated against any stated threshold — the split has so far happened only when a skill happened to come up for refactor, per the repo's own lightweight refactor process (read, gap-check vs. `research/`, split, rewrite, test, commit), not because a rule was checked and failed.
- **Production-readiness for a skill library is a function of scale, and 141 skills is already past the point where informal review scales (Ch. 10).** The book's "production concerns" chapter is explicitly about what changes once a skill library grows past a handful of entries: overlapping `description`s start competing for the same prompts (a **collision**), a renamed or removed skill leaves dangling prose pointers in every skill that mentioned it, and manual read-through review of "does this all still hang together" becomes infeasible. This repo is concretely at that scale (141 skills, 19 with `references/`, dozens of cross-skill prose mentions like the `access-control-model` → `subdomain-distillation` one above) but has no automated check for any of the three failure modes the book names — only per-skill behavioral contract tests (`tests/skills/*.contract.sh`), which by construction test one skill in isolation and cannot catch a collision between two skills or an orphaned cross-reference.

## Actionable Techniques

- Before writing or refactoring any `description`, draft it as if it were the *only* text available to decide whether this exact skill should fire for a given prompt — include the nouns a real request would use, the owning agent's name, and (if applicable) the command that invokes it (`skills/adr-authoring/SKILL.md`'s "Used by the enterprise-architect agent throughout the Design phase" is the pattern to generalize).
- Apply a resident-content test to every SKILL.md body before calling it done: for each section, ask "is this decision-shaping guidance (stays in the body) or reference material — a full template, a worked example, exhaustive generated code — that most invocations won't need in full (moves to `references/`)?" `event-storming-facilitation`'s full Output Format template and worked example are reference material sitting in the body today; they are the clearest concrete candidate for the next split.
- When a skill's prose says "see `other-skill`" or "per `other-skill`'s `references/x.md`" (as `access-control-model` does twice, for `subdomain-distillation`), treat that sentence as a dependency the refactor process should be able to check mechanically, not just read — at minimum, grep for the referenced skill/file's continued existence before closing out a refactor of either side.
- Adopt a two-part version-bump rule going forward for every skill: bump MAJOR when `description` changes in a way that could change what prompts trigger it, or when previously-taught guidance is reversed/removed; bump MINOR when body content is added (new section, new anti-pattern, new quality criterion) without changing the trigger surface; bump PATCH for wording-only fixes. Retrofit is optional; applying it forward is not.
- For every new `references/` file, open with the same self-containment sentence `adr-authoring`'s template already uses ("Self-contained — loadable without reading `SKILL.md` first") as a literal checklist item, not just a convention that happened to appear once.
- When a skill's `references/` directory grows past one file, keep the naming convention this repo has already converged on independently across two different skills — `output-format-template.md` for the copyable template, `worked-example.md` for the illustrative instance — rather than inventing a new name per skill.
- Periodically (not per-refactor) sample a handful of ambiguous prompts against the live plugin and confirm the skill that fires is the one intended, specifically for skills whose domains sit close together (e.g., `access-control-model` vs `zero-trust-design`, or the several DDD skills owned by `domain-modeler`) — a lightweight collision check, not a full regression suite, given the real-model-call cost of every `tests/skills/*.contract.sh` invocation.

## Candidate New Skills/Agents

Per Shafi's instruction to think expansively and not just patch existing content, these are meta-improvements to the plugin's *own* architecture — the same recursive move `adr-authoring` itself models (a skill about how to write one specific artifact type, cited by the agent that writes them). None of these touch a downstream product's artifacts.

1. **New skill: `skill-authoring-standards`** (proposed frontmatter: `phase: cross-cutting`, `owner: factory-governance` — the exact precedent already set by `skills/glossary-management/SKILL.md` and `skills/methodology-review/SKILL.md`, both cross-cutting governance skills owned by `factory-governance` rather than any phase-specific agent). This would be the canonical, citable distillation of this book's guidance for *this repo's own skill authoring*: the resident-content split threshold, the description-as-discovery-surface discipline, the two-surface versioning rule, the `references/` self-containment requirement, and the `output-format-template.md`/`worked-example.md` naming convention already in organic use. It is recursive by design — a skill about how to write skills, consulted by whoever runs the next skill-refactor session, exactly as Shafi's brief anticipated. It slots into the existing governance pair (`glossary-management`, `methodology-review`) as a third cross-cutting authority, this one about the plugin's own component shape rather than the products it builds.
2. **New `CLAUDE.md` section: "Skill Authoring & Progressive Disclosure."** Currently, the flat-file rule, the frontmatter schema, and the `references/` pattern are each documented in "Working in This Repository" and "Component Frontmatter," but the *decision rule* for splitting is nowhere — it lives only as tacit knowledge from the memory note "read, gap-check vs. research/, split, rewrite, test, commit." A short new section stating the resident-content threshold and the version-bump policy (both drafted above under Actionable Techniques) would make the rubric something a fresh session can check without having done a prior refactor — the same reason CLAUDE.md exists in the first place.
3. **New lightweight frontmatter convention: an optional `related:` field on Skill frontmatter.** Not a required field in the Component Frontmatter table (that table is explicitly closed — "no other shapes are permitted" — and changing it is a bigger decision than this file should force), but a documented, optional convention: a short list of skill names this skill's body references in prose (e.g., `access-control-model` would list `subdomain-distillation`). This is the minimum viable version of the book's "declared composability" — cheap enough to retrofit gradually during normal refactor passes, and it gives the validation script below (Candidate 4) something concrete to check.
4. **New script: `scripts/validate-skill-structure.sh`.** A deterministic, single-purpose script (fitting the existing Scripts = Actions rule exactly) that: (a) checks a `SKILL.md`'s frontmatter has the required fields in order per the Component Frontmatter table; (b) for every `references/<file>.md` the body text mentions, confirms the file exists on disk, and for every file that exists in `references/`, confirms the body actually points at it (catches orphans in both directions); (c) flags a body whose non-frontmatter line count exceeds a stated threshold (e.g., 200 lines) as a split candidate per Core Concept 9 above. This is exactly the kind of atomic, stateless check the existing `scripts/validate-artifact-structure.sh` already models for product artifacts, applied instead to the plugin's own skill files — and it is testable the same way other scripts are tested (`tests/scripts/`), with no live model call required.
5. **New governance hook entry (in `hooks/hooks.json`): a `PostToolUse` `command`-type handler matching `Write`/`Edit` on `skills/**/SKILL.md`, invoking `validate-skill-structure.sh`.** This closes the loop the book's "production-readiness" chapter argues for directly — catching an orphaned reference link or a malformed frontmatter field at the moment a skill file is edited, rather than at the next incidental refactor session that happens to touch that skill. Consistent with the existing Hooks rule ("Governance... Validate only. No business logic") and the existing `command`-handler pattern already used for other `scripts/*.sh`-backed hooks.
6. **Future consideration, not a proposal to build now: a periodic skill-collision sampling test.** A full regression suite testing all 141 skills' `description`s pairwise for ambiguity would be the most literal reading of the book's Ch. 10 production concern, but every invocation in `tests/skills/` is a real, billed model call via `claude --plugin-dir` — a full collision matrix is combinatorially expensive against this repo's own Budget and Frugality rule. Flagging this as a deferred idea (a small, targeted sample of known-adjacent skill pairs — e.g., `access-control-model` vs. `zero-trust-design` — run occasionally, not a standing CI gate) rather than a hard recommendation.

## Key Terminology

Worth considering for `skills/glossary-management/`'s Ubiquitous Language, or at minimum for the proposed `skill-authoring-standards` skill's own vocabulary section:

- **Progressive Disclosure** — already used informally in this repo's own layout note ("Chunk 19 onward") and in the skill-refactor campaign memory, but never formally defined as the three-tier (metadata / body / `references/`) model the book gives it. Naming the tiers explicitly (Tier 1/2/3, or metadata/body/reference) would let future skill-refactor sessions reason about *which tier* a piece of content belongs in, rather than just "should this move to references/ or not."
- **Trigger Surface** — the book's term for exactly `name` + `description`, the only content resident before a skill fires. Distinct from "the skill" as a whole; sharpens the distinction this repo's own strong `description` examples already embody without naming it.
- **Resident Content** — content loaded into context the moment a skill's trigger surface matches, regardless of whether the specific triggering task needs it. The book's split-threshold heuristic is entirely about minimizing this for content that is reference material rather than decision-shaping guidance.
- **Skill Collision** — two (or more) skills whose `description`s both plausibly match a single real prompt, so discovery becomes ambiguous or ties break on factors the author didn't control. Not currently named or tested for anywhere in this repo; becomes more likely, not less, as the skill count grows past 141.
- **Declared Composability** (vs. **Implicit Composability**) — the book's distinction between a machine-checkable cross-skill dependency (a field a script can validate) and a prose cross-reference (readable by a human or Claude, invisible to any test). This repo currently has only the implicit form between skills (the declared form exists, but only on agents, via the `skills:` frontmatter list).
- **Self-Contained Reference** — a `references/` file written so that loading it alone, with no assumption the parent `SKILL.md` body is also in context, is sufficient to use it. `skills/adr-authoring/references/output-format-template.md`'s opening line already states this property for itself; the book generalizes it as a hard requirement of the whole progressive-disclosure model, not a one-off note.

## Caveats

- **This book's assumed audience is building and shipping skill libraries at a scale and exposure this repo does not have.** Much of its "production-readiness" material (skill marketplaces, third-party distribution, consumers pinning specific skill versions, multi-tenant skill access control) assumes skills are a distributed artifact with external consumers. This repo's skills are consumed entirely within Shafi's own Claude Code sessions, never packaged or distributed to a third party — treat the distribution/marketplace chapters as inapplicable, and the versioning/testing/composability chapters (which apply regardless of distribution model) as the ones actually worth citing.
- **The book's collision-testing and load-testing guidance assumes cheap, frequent automated re-verification; this repo's own testing model is explicitly not cheap.** Every `tests/skills/*.contract.sh` invocation is a live, billed `claude -p` call per `CLAUDE.md`'s "Running Tests" section — the book's implicit assumption of frequent, broad automated re-testing across a skill library needs to be scaled down against this repo's own stated Budget and Frugality rule ("prefer simpler solutions... every added dependency must justify its presence"). This is why Candidate 6 above is flagged as a future consideration rather than a build-now recommendation.
- **Do not read the versioning chapter as implying this repo needs a package registry or dependency resolver.** The book's versioning guidance is written with an implicit assumption that *something* consumes a pinned skill version and needs a compatibility contract. This repo has no such consumer — skills are read live, in full, every session. The version field's actual value here is closer to a historical audit trail (paired with `CHANGELOG.md` and git history) than a compatibility guarantee; adopt the two-surface bump *rule* for its clarity, not because anything downstream depends on the number.
- **The book is written model-and-product-agnostic and does not know this repo's specific eight-phase structure, `factory-governance` ownership pattern, or the `adr-authoring`/`access-control-model` content itself.** Every mapping from "Ch. N's concept" to "this repo's file X" above is this document's own extrapolation onto real, already-existing repo content — not something the book states about this repo, which it has never seen. Treat the book as a source of vocabulary and structural discipline for *how a Skill should be shaped*, not as a source of opinions about DDD, ADRs, or ABAC, which is this repo's own domain content layered on top of the shape the book informs.
