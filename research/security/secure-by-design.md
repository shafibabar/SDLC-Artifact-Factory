# Secure by Design
**Authors:** Dan Bergh Johnsson, Daniel Deogun, Daniel Sawano
**Domain:** Security — Domain-Driven Security

## Core Concepts

- **Domain Primitives (Ch. 5–6).** The book's central technique: replace primitive-obsessed types (`string email`, `string tenantId`, `int amount`) with small, immutable, self-validating types that make illegal states unrepresentable — not just at the database boundary, but everywhere the value travels. A Domain Primitive is stricter than a Value Object: it must be (1) immutable, (2) unable to hold an invalid value once constructed (validity is a *type* guarantee, not a runtime check the caller must remember to run), and (3) free of business logic beyond its own invariants. `TenantId`, `Permission`, `Sensitivity`, `EmailAddress` are candidate Domain Primitives in this repo's domain — today they are plain `string`/`uuid.UUID` fields passed around, which is exactly the primitive obsession the book targets.
- **Assertions, not scattered validation (Ch. 4, 6).** Validity checks belong in the constructor/factory of the type, executed once, at the single place the type comes into existence — not re-checked ad hoc at every call site that happens to remember to check. The book calls this "totality": a constructor is either total (every input the type signature allows produces a valid instance) or it fails outright; there is no partially-valid instance in circulation. This directly targets a diffuse pattern where a "valid tenant ID" is only as trustworthy as the last function that remembered to validate it.
- **Pushing security into the Ubiquitous Language (Ch. 2–3).** The book's core DDD argument: access-control and trust concepts (e.g., "this operation requires Restricted-clearance", "this Aggregate crosses a trust boundary") should be named domain concepts discussed with domain experts — not unnamed infrastructure bolted on afterward. A security control that lives only in a middleware function with no domain-language name is invisible to the people who understand the actual risk. This repo's `access-control-model` skill already does real domain-language work here (`Subject`, `Resource`, `Action`, `Environment`, explicit `AccessPolicy` interface) — the book's contribution is framing this as a *language* discipline, not just a policy-table exercise: every permission (`data-assets:write`), every sensitivity tier (`Restricted`), and every trust-boundary crossing should be a term the business recognizes, reviewed the same way an Aggregate name is reviewed.
- **Aggregates as trust boundaries (Ch. 5, 9).** The book argues that deciding *which* Aggregate boundaries double as authorization boundaries is a first-class design decision, made explicitly and per-Aggregate — not an assumption that "the API layer handles it so the domain doesn't need to care." An Aggregate that can be reached from more than one context, or that represents a resource multiple tenants' data could flow through, is a candidate for carrying its own authorization check inside its invariants, not only outside them in a policy object.
- **Defense in depth is not an excuse for a shallow domain model (Ch. 1, 9).** The book's polemic point: firewalls, gateways, and middleware absolutely matter, but citing them ("the network already isolates this") to justify skipping a domain-level invariant is a documented anti-pattern the book names explicitly — every layer is expected to independently enforce its own boundary, not lean on the layer around it.
- **Security sensitivity is an axis orthogonal to Core/Supporting/Generic (Ch. 3).** A Subdomain's strategic classification (Core, Supporting, Generic) answers "how much competitive value does this create?" The book insists security-sensitivity is a separate question — a Generic Subdomain (e.g., authentication) can be maximally security-sensitive despite being strategically Generic, and a Core Subdomain (e.g., pricing) can be low security-sensitivity. Conflating the two axes leads teams to under-invest in security review of "boring" Generic/Supporting subdomains precisely because they're not where the product's competitive differentiation lives.

## Actionable Techniques

- **Domain Primitive worked example (Go idiom for the book's pattern):**

```go
// TenantID is a Domain Primitive: constructible only through NewTenantID,
// which is the single place validity is enforced. Once constructed, every
// caller can treat a TenantID as always-valid — no defensive re-checking.
type TenantID struct {
    value uuid.UUID
}

func NewTenantID(raw string) (TenantID, error) {
    id, err := uuid.Parse(raw)
    if err != nil {
        return TenantID{}, fmt.Errorf("invalid tenant id %q: %w", raw, err)
    }
    if id == uuid.Nil {
        return TenantID{}, fmt.Errorf("tenant id cannot be nil")
    }
    return TenantID{value: id}, nil
}

func (t TenantID) String() string { return t.value.String() }
func (t TenantID) Equal(other TenantID) bool { return t.value == other.value }
```

- **Assertion-style checklist for any new type carrying a security-relevant value:** (1) can this type ever hold a value the domain considers invalid or dangerous once constructed? If yes, it is not yet a Domain Primitive. (2) Is validity checked in more than one place? If yes, collapse to a single constructor. (3) Does the zero value (`TenantID{}` in Go) accidentally look valid? Guard against the zero-value trap explicitly (a `uuid.Nil` check, as above) since Go — unlike the book's Java examples — always has an accessible zero value.
- **Per-Aggregate trust-boundary decision checklist:** for every Aggregate, answer explicitly and record the answer: (a) can this Aggregate be reached by more than one tenant's request path? (b) does crossing into this Aggregate require an authorization decision beyond "the caller is authenticated"? (c) if yes to (b), is that check expressed as a domain concept (e.g., `Subject` + `AccessPolicy.Evaluate`) or left implicit in a middleware function with no name a domain expert would recognize?
- **Security-sensitivity tagging as a design artifact:** tag each Subdomain/Bounded Context with a security-sensitivity rating independent of its Core/Supporting/Generic classification, and review the two ratings side by side — a Generic Subdomain rated high-sensitivity should get the same design scrutiny as a Core Subdomain, not less.
- **Treat "the network/mesh already isolates this" as a prompt, not a conclusion:** whenever a design defers a check to an outer layer, require the same design to state what happens if that outer layer is misconfigured or bypassed — if the answer is "nothing, we're relying on it," that is the anti-pattern the book names.

## Key Terminology

- **Domain Primitive** — a Value-Object-like type, immutable and self-validating at construction, that makes an entire class of illegal state (including security-relevant illegal state) unrepresentable. Distinct from a generically-validated string or a bare Value Object that only wraps a value without a totality guarantee. Worth adding to `glossary-management` alongside the existing DDD terms (`Aggregate`, `Bounded Context`) since it is a precise, citable name this repo's Go code (Domain Primitive candidates: `TenantID`, `Permission`, `Sensitivity`) could reference directly in code review.
- **Assertion (this book's specific sense)** — a fail-fast, constructor-embedded validity check, as distinct from a generic "test assertion." Worth a glossary note distinguishing it from `Assertion` in the testing sense (`Test Fixture`, `Scenario` already exist in the glossary's testing cluster) to avoid the same word meaning two things across security and testing artifacts.
- **Totality** — the book's term for "every input the constructor's type signature accepts produces a valid instance, or construction fails outright." Candidate for the glossary if `access-control-model` adopts Domain Primitives, since "totality" gives reviewers a one-word test for whether a new type actually qualifies.

## Caveats

- **Confidence level, overall:** I am confident in the book's headline concepts (Domain Primitives, Assertions, pushing security into the Ubiquitous Language, Aggregates as trust boundaries, the anti-pattern of citing perimeter defenses to excuse a shallow domain model) — these are the book's well-known, widely cited contributions to domain-driven security and I have solid confidence in their substance. I am **less confident on exact chapter numbers and the precise order of topics** — the chapter references above are my best approximation of the book's structure, not verified against the physical text, and should be treated as approximate signposts rather than citations to check page numbers against.
- **The book's constructor pattern is written for languages with exceptions (primarily Java/Kotlin idiom in the book's examples).** "Throw on construction" — a constructor that throws rather than returns — has no direct equivalent in Go, which has no exceptions and no constructors that can fail without an explicit return path. The Go worked example above (`NewTenantID(raw string) (TenantID, error)`) is a genuine adaptation, not a literal port: Go's idiomatic "constructor-returns-error-or-value" pattern *is* the totality guarantee, just expressed as an explicit two-value return the caller must destructure rather than an exception the caller can ignore until it propagates. Where the book's examples show a constructor that simply throws and trusts the exception to halt bad data from propagating, the Go equivalent requires every call site to actually check the returned `error` — Go offers no automatic propagation, so a Domain Primitive constructor whose error return is discarded (`tid, _ := NewTenantID(raw)`) silently reintroduces exactly the illegal-state risk the pattern exists to prevent. This is a sharper requirement in Go than in the book's source language, not a weaker one.
- **This repo's physical multi-tenancy (separate namespace/deployment per tenant) already provides an isolation layer the book's examples don't assume.** The book's SaaS examples generally assume a single shared deployment where the *only* tenant isolation is application-layer (a `WHERE tenant_id = ?` clause or equivalent domain check) — so the book leans harder on Domain Primitives and Aggregate-boundary checks as the *sole* line of defense for tenant separation. This repo's `access-control-model` skill already correctly notes its own tenant-isolation check is "defence-in-depth... partially redundant" given physical isolation — the book's techniques here should be read as reinforcing a second independent layer, not as compensating for an isolation gap this repo doesn't have.
- **The book predates this repo's specific stack** (Go, chi, pgx, Linkerd, ABAC-over-JWT) — none of its code examples are Go, and its multi-tenant SaaS case studies are generic illustrations rather than references to this repo's actual architecture. Every Go-specific application above (the `TenantID`/`Secret`/`ExtractedEntitySummary` examples) is this document's extrapolation of the book's language-agnostic principles onto this repo's stack, not content literally present in the book.
