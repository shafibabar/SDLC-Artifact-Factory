# Unleash Your Creativity with AI
**Authors:** Nikita Bier and various contributors
**Domain:** Human-AI Collaboration / Creative Ideation Technique

**A note on scope, and how this file relates to the one before it:** `research/human-ai-collaboration/co-intelligence-mollick.md` covers the overall *working relationship* between Shafi and Claude Code — who decides what, how trust is built, centaur vs. cyborg division of labor. This book sits one level lower and narrower: it is a general-audience, practitioner-voice collection of concrete techniques for generating ideas *with* an AI in the room — divergent brainstorming, overcoming the blank page, prompting for range instead of the first plausible answer — aimed at solo creators (writers, marketers, indie builders), not at software delivery pipelines. It is not a technical reference and does not attempt to be one. The honest finding, stated up front: this repo's five Ideate/Strategy-phase structuring skills (`jtbd-analysis`, `impact-mapping`, `business-model-canvas`, `competitive-analysis`, `moscow-prioritization`) are, without exception, written as pure convergent/structuring techniques for ideas that already exist — they tell you how to organize, prioritize, and validate candidates, never how to generate a divergent spread of candidates in the first place, and none of them acknowledges that the entity running the technique is itself an AI rather than a human facilitator interviewing real people. That is a real, checkable gap. It is also a narrow one: this book's own content is general creative-writing/content-marketing advice, and most of it (image generation, social copy, video hooks, personal-brand content) has no plausible application to a Markdown-artifact SDLC factory at all.

## Core Concepts

- **Divergent generation and convergent structuring are different acts, and this book is entirely about the first while this repo's Ideate-phase skills are entirely about the second.** The book's operating premise (echoing decades of brainstorming literature it draws on, most visibly Osborn's original rules — defer judgment, aim for quantity, welcome wild ideas, combine and improve) is that idea *generation* and idea *evaluation* must be temporally separated: generate a wide, undisciplined spread first, then narrow. Every one of the five ideation skills read here does the opposite in emphasis — `moscow-prioritization` explicitly states its technique "only works when there is a defined scope limit" and operates on a backlog assumed to already exist; `competitive-analysis` analyzes a landscape that already exists; `business-model-canvas`'s block-by-block instructions ask you to *state* the value proposition, not to generate several candidate value propositions and choose among them. The book's core technique — separate generation from judgment — is not present anywhere in these five skills, because none of them are generation techniques at all; they are structuring and filtering techniques applied to inputs the skill assumes arrived from elsewhere (research, interviews, OKRs).
- **"Quantity before quality" as a deliberate prompting discipline, not a throwaway habit.** The book's most concrete, repeatable technique for working with a chat-based AI specifically is: ask for many candidates in one pass (the book's examples run from a dozen to a few dozen) before evaluating any of them, because the first 3–5 outputs from any generative model tend to be the most statistically average, expected completions, and genuinely novel candidates cluster later in the list. Applied to this repo: `impact-mapping`'s Level 4 (WHAT deliverables) comes closest to this instinct already — "List all candidates, then prioritise — not all must be built" — but gives no guidance on *how many* candidates constitute a real spread, or that the technique specifically depends on generating past the first few obvious ones. `business-model-canvas`'s Value Propositions step has no such instruction at all: "State it as: what the customer gains + what pain is relieved" reads as write-the-one-right-answer, not generate-several-and-select.
- **Constraint injection as a deliberate anti-anchoring technique.** A recurring move across the book's essays is to force range by imposing an artificial constraint on a subsequent generation pass after an unconstrained first pass — "now give me five that assume zero budget," "now five that assume the opposite of your first idea," "now five a skeptical customer would ridicule." The mechanism is explicit: a model's (or a person's) second attempt at the same prompt tends to anchor on its first attempt's framing unless a constraint deliberately breaks that anchor. This is the sharpest, most exportable idea in the book for this repo's specific failure mode — none of the five skills read here contain any instruction to re-run a generation step under a deliberately different constraint before settling on candidates.
- **Adversarial/persona reframing to surface non-obvious candidates.** A second recurring technique: explicitly prompt the AI to answer *as* a specific skeptical or adversarial persona ("what would a customer who thinks this is overpriced say the value prop should be instead?") rather than only as a neutral generator, on the reasoning that a neutral prompt tends to produce safe, generic completions while a persona prompt forces specificity. This is distinct from this repo's existing persona work — `jtbd-analysis` and `user-persona` (not re-read here, referenced by `jtbd-analysis`) use personas to describe *who the product serves*, never as a prompting device to generate more divergent *candidate* ideas from the AI itself.
- **Iterative refinement loops treat the first output as a draft, never a deliverable.** The book's implicit workflow for any creative task is: generate → react critically → regenerate with the critique folded in → repeat until diminishing returns, typically 2–4 cycles. This is compatible with, but not identical to, this repo's approval-gate model (`CLAUDE.md`'s "if Shafi has not said 'go'... do not build") — the book's loop is a *pre-gate* private iteration the AI (or the person prompting it) runs alone before presenting anything to another person, whereas this repo's gates are the point at which Shafi reviews Claude Code's output. The two are complementary, not competing: nothing here argues Shafi's review gates should change, only that what arrives *at* a gate could itself be the product of an internal divergent-then-convergent pass rather than a single first-draft pass.
- **The book has almost nothing to say about structuring, prioritizing, validating, or governing ideas once generated** — its entire compass is the generation half of the problem. This is precisely the inverse of what `impact-mapping`, `moscow-prioritization`, `business-model-canvas`, and `competitive-analysis` already do extremely well (measurable goals, coherence checks, the 60% Must rule, defensibility criteria). The two are genuinely complementary bodies of technique with almost no overlap — which is exactly why the honest yield here is narrow: this repo already has strong convergent technique and needs, at most, a small divergent-technique addition bolted onto the front of a couple of existing steps, not a parallel skill covering ground already covered.
- **None of the five skills acknowledge that an AI agent, not a human facilitator, executes the technique.** This is the specific thing the assignment asked to check for, and the answer is unambiguous: every one of the five skills is phrased as instructions to a human running a workshop or interview ("ask," "for each primary persona," "list every distinct customer segment," "aim for 3–6 direct competitors, 3–4 indirect"). None flags that when `requirements-analyst` or `product-strategist` executes these steps, the entity doing the asking-of-itself is a language model, which is exactly the situation this book's anti-anchoring and quantity-first techniques were built to correct for. This absence is not a defect in the skills as structuring techniques — they do that job well — but it is a genuine blind spot for the specific step where an agent must originate candidates (deliverables in `impact-mapping`, value propositions in `business-model-canvas`) rather than analyze or prioritize inputs that already exist.

## Actionable Techniques

- Before finalizing WHAT deliverables in `impact-mapping`'s Level 4, generate at least 2–3x the number of candidates that will ultimately be selected, and explicitly generate a second pass under an inverted or budget-zero constraint before narrowing — do not accept the first plausible deliverable per impact as the only candidate considered.
- Before finalizing a Value Proposition in `business-model-canvas`'s Step 2, generate 3–4 candidate phrasings (not one) and select among them using the block's existing pain-relieved/gain-created test, rather than writing a single value proposition directly.
- When an agent (not a human) is the one applying `jtbd-analysis`'s Step 1 ("what progress is this person trying to make?"), explicitly run the question twice — once straightforwardly, once from an adversarial "what would a skeptical user say this job is really about?" framing — before committing to the core job statement, since there is no real interview subject to anchor the answer against ground truth.
- Do not import the book's iterative-refinement loop as a new formal gate — treat it as something that can happen silently before a skill's existing Quality Criteria table is checked, not as a new checkpoint requiring its own approval step. Adding ceremony here would violate CLAUDE.md's "prefer simpler solutions" frugality rule for a technique whose entire value is lightweight, private iteration.

## Candidate New Skills/Agents

**No new skill or agent is warranted.** The honest read: this book's substantive, exportable content is three techniques (quantity-first generation, constraint injection against anchoring, persona/adversarial reframing) that apply to exactly one narrow moment across all five skills examined — the point where an idea-generation skill asks the executing agent to originate a candidate (a deliverable, a value proposition, a core job statement) rather than structure or prioritize inputs that already exist. That moment is small enough, and concentrated enough in two files, that it does not justify a new skill domain:

1. **Deferred, narrow candidate: a short "Generate Before You Narrow" note added to `impact-mapping`'s Step 4 and `business-model-canvas`'s Step 2, not a new file.** A 3–5 sentence addition per skill instructing the executing agent to produce multiple candidates (ideally under at least one deliberately varied constraint) before applying the skill's existing selection criteria. This is the only concrete, worth-doing change this book earns — and it is a wording addition to two existing steps, not a new component, matching the discipline applied in the three prior thin-yield files in this batch.
2. **Not recommended: a standalone `ai-assisted-ideation` or `divergent-brainstorming` skill.** The three techniques above do not add up to enough independent, reusable procedure to justify a directory of their own — `impact-mapping` and `business-model-canvas` already own the convergent half of the WHY/WHO/HOW/WHAT and nine-block structures respectively; grafting a generic brainstorming skill alongside them would either duplicate their existing structure or float unanchored to any consumer, the same trap the Mollick file's deferred `human-ai-collaboration-patterns` candidate correctly avoided by requiring a real future consumer before building.

No new agent is warranted at all. Divergent generation is something an existing agent (`requirements-analyst`, `product-strategist`) does *within* a step of a skill it already owns — it is not a distinct reasoning role.

## Key Terminology

These are creative-technique terms, not domain/DDD terms, and do not belong in `skills/glossary-management/`'s canonical Ubiquitous Language (which governs terms used *inside* artifacts the plugin produces for a product). If the narrow addition above is ever made, these are the precise terms worth using:

- **Divergent Generation** — producing a wide, unfiltered spread of candidate ideas before any evaluation is applied. Absent from all five skills read here, every one of which begins from evaluation/structuring criteria.
- **Convergent Structuring** — narrowing, prioritizing, or validating a set of candidates against explicit criteria. What all five Ideate/Strategy skills read here already do, and do well.
- **Quantity-First Generation** — the specific discipline of deliberately generating more candidates than needed (the book's examples run to a dozen or more) before narrowing, on the premise that the earliest outputs of any generation pass tend to be the least novel.
- **Constraint Injection** — deliberately imposing an artificial limitation ("assume zero budget," "assume the opposite framing") on a second generation pass specifically to break anchoring on the first pass's framing.
- **Anchoring (in generation, not evaluation)** — a narrower usage than the anchoring-bias concept implicit elsewhere in this repo's research: here specifically the tendency for a second attempt at the same generative prompt to resemble the first attempt unless a deliberate constraint interrupts it.

## Caveats

- **This is the thinnest-yield book in this cluster, and that is the honest, correct call, not an undersell.** The book is a general-audience creativity guide covering image generation, short-form video hooks, personal-brand content, and social copywriting — the overwhelming majority of its material (arguably 80%+) has no plausible mapping onto a Markdown-artifact SDLC pipeline building an enterprise data-compliance platform, and this file does not attempt to force one. The one genuinely exportable thread — divergent generation technique for the narrow moment where a skill must originate candidates rather than structure existing ones — is real but small, matching the calibration set by the three preceding thin-yield files in this batch (Raschka, Russell & Norvig, Mollick).
- **The book's techniques are written for a human prompting a chat interface interactively, turn by turn — not for an agent executing a Markdown skill file end-to-end in one pass.** Applying "ask for a dozen, then narrow" inside a skill step is this document's own extrapolation of the book's spirit, not a literal instruction the book gives for this context. Nothing in the book anticipates a Claude Code agent reading its own skill file and generating candidates for itself with no human in the interactive loop at all.
- **Do not conflate the anchoring risk named here with Mollick's "jagged frontier" concept from the prior file in this cluster.** Mollick's frontier is about unpredictable variation in AI *competence* across tasks; this book's anchoring risk is about a generation pass narrowing prematurely regardless of competence — a shallow-spread problem, not a wrong-answer problem. They are both real but distinct risks and should not be merged into one finding.
- **This repo's structuring skills were never broken by this gap** — `impact-mapping`, `business-model-canvas`, `jtbd-analysis`, `competitive-analysis`, and `moscow-prioritization` all remain sound, well-built convergent techniques exactly as read. The finding here is additive and optional (a candidate generation habit worth adopting at one or two specific steps), not corrective of anything currently wrong.
