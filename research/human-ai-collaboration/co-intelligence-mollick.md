# Co-Intelligence: Living and Working with AI
**Authors:** Ethan Mollick
**Domain:** Human-AI Collaboration / Working Practice

**A note on scope, and how this file relates to the ones before it:** every other file in `research/` grounds a *technical* book against a skill, an agent's domain knowledge, or the plugin's own mechanics (`research/ai-agent-engineering/context-engineering-for-ai-agents-zhu.md` covers how `CLAUDE.md` and `sdlc-context.json` budget the context window; `research/software-architecture/fundamentals-of-software-architecture-richards-ford.md` covers how services and ADRs get designed). This book is not technical. It is about how a human and an AI should actually divide labor, build trust, and stay honest about the AI's limits when working together over time — and its subject here is not a skill or a diagram convention but the lived collaboration pattern this repo's own `CLAUDE.md` already encodes between Shafi (PM, not a programmer) and Claude Code (does all engineering). The honest finding, stated up front so it isn't buried: this repo independently arrived at good practice on most of Mollick's four rules before ever reading the book — what the book adds is vocabulary, a name for one genuine, checkable gap (capability-reliability boundaries versus decision-ownership boundaries), and a concrete account of *why* one existing but unnamed practice (D024's loosening of process ceremony) is not drift but a textbook instance of the book's fourth rule.

## Core Concepts

- **Rule 2, "be the human in the loop," is not aspirational here — it is a literal, load-bearing instruction Claude Code reads on every single session.** Mollick's second rule says the human must remain the point of final judgment and consent, not a rubber stamp consulted after the fact. `CLAUDE.md`'s "Session Startup" section states this as the third of three non-negotiable steps: *"If Shafi has not said 'go' on the next chunk — do not build. Discuss the plan, get approval, then build."* This is a stronger, more literal version of the book's rule than most of the book's own case studies describe (Mollick mostly discusses knowledge workers choosing when to check AI output; this repo makes the check a precondition of *any* work happening at all). Nothing needs to be added here — the book validates existing practice and gives it a name.
- **The gate is not one blanket rule but a designed sequence of checkpoints, already granular per artifact type — `agents/domain-modeler.md` is the clearest evidence.** The book's rule 2 is usually illustrated as a single "review the output before you trust it" habit. This repo does something more disciplined: `domain-modeler.md`'s Execution Sequence places two named approval gates mid-workflow (*"PRESENT TO SHAFI: Event Storming findings, BC candidates, language candidates ... Wait for approval before continuing"* after step 5, and again after the Context Map at step 7), while steps 8–11 (Aggregate Design through Read Model Design) explicitly "may be produced in sequence without individual approval gates, with a final review at the end." The same file's Inputs table ends with *"If any required input is missing, the domain-modeler halts and reports the gap to Shafi before proceeding"* — a halt-on-missing-input rule that is itself a human-in-the-loop checkpoint, distinct from the plan-approval gate. This repo already varies gate density by how consequential and hard-to-reverse a decision is (domain boundaries and the Context Map get gates; per-aggregate detail work does not) — exactly the discrimination the book asks for, rather than gating everything uniformly.
- **Rule 1, "always invite AI to the table," is already maximal here in a way the book doesn't quite anticipate.** Mollick's first rule is aimed at a reader who under-uses AI out of habit or distrust, and who should experimentally try it on more tasks. This repo has no such reader to persuade: "Who Shafi Is" states plainly that *"Claude Code does all engineering"* and Shafi *"provides the problem statement, reviews artifacts, and approves each chunk of work."* This is a 100%-invited, pre-committed division of labor, not an exploratory one. The book's rule mostly confirms the direction already taken; it adds nothing actionable because there's no residual under-use to correct.
- **The "jagged frontier" is real in this repo's own domain, but the repo's existing boundary-setting mechanism (Escalation Rules) answers a different question than the book's.** Mollick's jagged frontier describes unpredictable, task-by-task variation in AI competence that doesn't track human intuitions about difficulty — you must map it empirically rather than assume uniform skill. All 13 agent files in `agents/` carry a named `## Escalation Rules` section (verified by grep across the directory) listing specific conditions under which that agent stops and asks Shafi rather than deciding — e.g. `domain-modeler.md`'s *"An NFR constraint... forces a BC boundary that makes the domain model awkward — this is an architectural trade-off that requires a product decision"* or `enterprise-architect.md`'s *"A technology default... cannot be applied to a specific Bounded Context and a non-standard choice is required — write an ADR and present for approval."* These are genuinely fine-grained, domain-specific frontier boundaries — but every single one is framed as a **decision-ownership** question (who should own this trade-off, given it's a product/business judgment call), never a **capability-reliability** question (is Claude Code actually likely to get this specific kind of call right, or is this a place where confident-sounding output has historically been wrong?). The book's frontier concept is about the latter. This repo has excellent coverage of the former and none, currently, of the latter, named as such.
- **The clearest place this capability-reliability gap actually bites is the split between mechanically-verified components and prose-only ones — and CLAUDE.md already draws this line without naming why it matters.** "Working in This Repository" states that commands, hooks, and scripts "are tested by actually invoking them, not just read for coherence," while "skills and agents have no build step or test suite (they are prose consumed by Claude)." The rationale given for the mechanical side is explicit and concrete: *"Every hook and every backing script in `scripts/` was individually triggered with a real (or synthetic) input and its outcome verified — not just read for coherence; this caught 3 real bugs... that a read-only review missed."* (Decision D015 names the bugs: a naming-convention regex that silently let invalid names through, a hook resolving paths from the wrong directory, unprotected JSON parsing — all found only by running the thing, not by reading it and judging it coherent.) That same discipline has no equivalent for skill or agent *content* — a skill's DDD guidance or an agent's escalation logic can be just as confidently wrong as that regex was, but the only check available is a human reading prose and judging it plausible, which is precisely the failure mode D015 demonstrates doesn't work. Mollick's jagged frontier is the reason this matters: fluent, confident prose is not evidence of correctness, and the places least amenable to a runnable test (domain judgment, methodology guidance) are exactly where that gap is most dangerous, not least.
- **That gap is only partially open — the skill-refactor process already does an informal version of the check the book asks for.** Per this session's own memory of the refactor process (read, gap-check against `research/`, split, rewrite, test, commit), every skill rewrite is checked against a primary source in `research/` before being trusted, which is a real (if manual, book-unnamed) mitigation for exactly the prose-correctness risk described above. This is a genuine partial answer already in place — it just isn't described anywhere as "how we handle the jagged frontier for skill content," so it isn't consistently required, dated, or checked for coverage the way `tests/run-smoke-tests.sh` mechanically is for scripts.
- **Rule 3, "treat AI like a person, while knowing it isn't," maps cleanly onto this repo's agent design and its counterweight.** Mollick's rule 3 has two halves: give the AI a role/persona and interact with it as a team member (because this measurably improves output), while never forgetting it has no persistent memory, no real understanding, and no consistent self across sessions. The "half" of giving AI a role is fully present: every agent file has a named role (`domain-modeler`, `enterprise-architect`), a first/third-person `## Purpose` section describing what it "owns" and "does not own," and boundaries against overlapping another agent's domain (CLAUDE.md's "Agent Behaviour Rules": *"Never overlap with another agent's domain"*). The "knowing it isn't" half is present too, just in a different section: the D015-driven live-verification discipline above is precisely a refusal to take an agent's or a script's fluent output at face value — the opposite of naive anthropomorphism. The two halves of rule 3 are both independently present in this repo; they just live in different sections (`Component Architecture`/agent files vs. `Working in This Repository`) and are never connected to each other as two sides of one stated principle.
- **Rule 4, "assume this is the worst AI you'll ever use," is not stated anywhere in CLAUDE.md as a principle — but decision D024 is a real, dated instance of Shafi applying it in practice.** Mollick's fourth rule: capability keeps improving, so don't calibrate your working relationship permanently around today's model's limits — expect to loosen constraints as trust and demonstrated competence accumulate. `sdlc-context.json`'s D024 records exactly this: Shafi ended the plan-start/plan-exit GitHub Project ceremony for skill refactors, with the rationale *"given the accumulated research and demonstrated judgment across DDD and this chunk."* That is a track-record-based trust recalibration, done once, informally, as a single decision-log entry — not a named, recurring practice. (The `research/ai-agent-engineering/context-engineering-for-ai-agents-zhu.md` file already flags, from a different angle, that D024 was itself partially reversed by D025 and then corrected by D026 with no machine-readable marker of which is current — the same underlying event this book would call "a trust recalibration cycle" is visible in that other file only as an unresolved-looking chain of decisions, because nothing frames it as a deliberate, expected, periodic act.)
- **Centaur vs. cyborg: this repo is a clean, deliberate centaur, and that's the right call for this specific human, not a gap to close.** Mollick distinguishes centaur collaboration (clear task division — human and AI each own separate, well-bounded parts of the work, like a chess centaur where the human sets strategy and the AI calculates) from cyborg collaboration (tight, moment-to-moment interleaving inside the same task, e.g. co-writing sentence by sentence). This repo is unambiguously centaur: Shafi owns problem statements, review, and approval; Claude Code owns 100% of the engineering; the two never interleave inside a single artifact's construction (there is no described mode where Shafi edits code mid-generation or Claude drafts half a sentence of a PM decision). This is the correct choice given "Who Shafi Is" — cyborg collaboration presupposes the human can operate directly inside the AI's output medium (code), which a self-described non-programmer structurally cannot. The book would validate this as the right pattern for this participant mix, not flag it as underdeveloped.

## Actionable Techniques

- When adding a new agent's `## Escalation Rules` section going forward, add a second, explicitly distinct list alongside it — not merged into it — naming *capability-reliability* concerns for that agent's domain (where has this kind of judgment call gone wrong before, or where is there no way to verify it beyond reading it) as distinct from *decision-ownership* concerns (whose call is this, given business stakes). The two lists answer different questions from Mollick's book and conflating them hides the reliability gap inside what currently reads as a complete list.
- Treat the skill-refactor "gap-check against `research/`" step as the explicit, load-bearing mitigation for jagged-frontier risk in prose-only components, not an incidental process detail — name it that way the next time the refactor process itself gets documented, so its purpose (catching confidently-wrong domain guidance, the D015 failure mode applied to prose instead of scripts) survives independently of any one session's memory.
- The next time a chunk loosens a gate, a ceremony, or an approval-density rule (as D024 did), record in the same decision entry what specific track record justified it — turning what is currently an ad hoc, retroactively-inferable pattern (visible only by reading D024/D025/D026 together) into a named, intentional act each time it happens.
- When mapping a new agent's domain (the next time a chunk adds one), spend five minutes explicitly asking "where in this domain would Claude Code's output look equally confident whether right or wrong?" before finalizing the Escalation Rules section — this is the book's jagged-frontier-mapping exercise, done once per agent rather than left implicit.
- Do not add a formal, permanent "trust budget" mechanism prematurely — the book's rule 4 argues for periodic recalibration as capability and track record grow, not for a one-time loosening followed by never revisiting gate density again. If gate density is ever revisited, it should be revisited again later, not treated as solved by D024.

## Candidate New Skills/Agents

The honest read, per the assignment's own instruction: **this book does not earn a new skill or agent today.** Its yield is two specific, small `CLAUDE.md` wording additions and one explicitly-deferred, conditional candidate:

1. **A short new subsection distinguishing decision-ownership escalation from capability-reliability escalation.** Add a few sentences near "Agent Behaviour Rules" or as a new subsection clarifying that an agent's `## Escalation Rules` section answers "whose call is this" and inviting (not yet mandating, to avoid retrofitting 13 files in one pass) a second, separate note per agent on where that agent's domain judgment is hardest to verify by any means other than reading it. This is a wording/organizational change to an existing section, not a new component.
2. **A one-line acknowledgment, near "Running Tests" or "Session Startup," that skill/agent prose correctness is checked only by human read-through and the `research/`-gap-check convention, unlike commands/hooks/scripts.** This makes explicit what D015's rationale already argues implicitly for mechanical components, extended honestly to the one place it doesn't currently reach. Again a wording addition, not new prose or a new component — the mitigation (gap-checking against `research/`) already exists in practice; it just isn't named as serving this purpose.
3. **Deferred: a lightweight `human-ai-collaboration-patterns` skill teaching the centaur model, the jagged-frontier-mapping exercise, and the escalation-rules convention to a downstream product team — only once decision D023's confirmed multi-product/multi-team future actually materializes as a second team onboarding to this plugin.** Building it now would be inventing a downstream audience that doesn't yet exist, which the assignment explicitly warns against. If and when a second team is real, this skill would have a genuine home and a genuine reader; today it would sit unread, competing with the domain skills that do have an active consumer (Claude Code itself, this session).

No new agent is warranted at all — none of this book's concepts require reasoning about a product domain; they are about how the existing collaboration is conducted, which is CLAUDE.md's job, not an agent's.

## Key Terminology

These are collaboration-process terms, not domain/DDD terms — they do **not** belong in `skills/glossary-management/`'s canonical Ubiquitous Language, which governs terms used *inside* artifacts the plugin produces for a product (Bounded Context, Domain Event, etc.), not terms describing how Shafi and Claude Code work together. If any of the `CLAUDE.md` wording changes above are adopted, these are the precise terms to use consistently:

- **Jagged Frontier** — the book's term for AI competence varying unpredictably across tasks that look similarly hard to a human, such that "harder for a person" does not predict "harder for the AI." Distinct from this repo's existing "escalate when it's a product trade-off" logic, which is about decision ownership, not competence.
- **Centaur** — a collaboration pattern with clear, swim-laned division of labor between human and AI, each owning separate parts of the work. This repo's entire Shafi/Claude Code split is a centaur pattern by this definition.
- **Cyborg** — a collaboration pattern with tight, moment-to-moment interleaving of human and AI work inside a single task (e.g., co-writing sentence by sentence). Not present anywhere in this repo's described workflow, and not a good fit given Shafi's stated non-programmer role.
- **Human in the Loop** — keeping a human as the point of final judgment/consent before an AI-produced action takes effect. `CLAUDE.md`'s Session Startup gate and `domain-modeler.md`'s "halts and reports the gap" rule are both concrete instances of this already.
- **Capability-Reliability Escalation** (not the book's own term, but the gap named above needs one) — a proposed pairing term for the missing half of this repo's existing `## Escalation Rules` sections: escalation triggered by where an agent's output is empirically or structurally hard to trust, as distinct from where it's triggered by whose call a decision is.

## Caveats

- **This is a light-touch, honest-yield book application, not a discovery of buried problems.** The strongest finding here is confirmation: `CLAUDE.md`'s existing approval-gate structure, agent-persona design, and mechanical-verification discipline already implement three of the book's four rules well, independently of ever having read it. Resist any temptation to inflate the one real gap (capability-reliability vs. decision-ownership escalation) into more than it is — it is a naming and organization gap inside an already-good structure, not a structural failure.
- **The skill/expertise-atrophy risk Mollick raises for over-delegating judgment does not apply on the engineering side here, because there was never programmer expertise to atrophy.** Shafi is explicitly a PM, not a programmer, per "Who Shafi Is" — the book's warning is aimed at people delegating away skills they already had. The more honest version of this risk for this specific pairing is whether Shafi's *review* judgment (his side of the approval gates) could become pro-forma over time as trust accumulates — but nothing read in this repo shows evidence of that; D023's own decision trail (*"Claude flagged this tension explicitly before proceeding rather than silently building content contradicting the repo's own prior research"*) shows deep, substantive engagement at a recent gate, not rubber-stamping. Flag this as something to keep an eye on as the decision log grows, not as a finding.
- **The "worst AI you'll ever use" rule is genuinely hard to operationalize as a standing CLAUDE.md rule without inviting exactly the kind of unbounded, ungoverned gate-loosening the rest of CLAUDE.md is designed to prevent.** Do not read the D024 discussion above as license to relax the Session Startup gate itself — that gate exists for reasons (single-operator review capacity, PM-not-programmer review depth) that don't change with model capability. The rule 4 lesson applies to secondary process ceremony (as D024 already correctly demonstrates), not to the core "get approval before building" gate.
- **This book predates and is written independently of Claude Code, subagents, or any specific tool this repo uses** — its case studies are general knowledge-work examples (consulting, writing, coding at a high level), not Claude Code-specific. Every mapping above from the book's general vocabulary onto this repo's specific mechanisms (Escalation Rules sections, D015's live-verification rationale, D024's ceremony change) is this document's own extrapolation, not literal book content.
- **Do not conflate this cluster with `research/ai-agent-engineering/`.** That cluster's `context-engineering-for-ai-agents-zhu.md` file already covers *mechanics* adjacent to this one (subagent forking/context isolation, in its Ch. 9 discussion) — this file deliberately does not re-cover that ground; it stays at the level of the working *relationship* (who decides what, how trust is built and re-earned), not the token-budget or context-architecture mechanics of how any single agent invocation is constructed.
