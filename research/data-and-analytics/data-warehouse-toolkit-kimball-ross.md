# The Data Warehouse Toolkit
**Authors:** Ralph Kimball and Margy Ross
**Domain:** Data & Analytics — Dimensional Modeling

## Core Concepts

- **Dimensional modeling / star schema.** The book's central technique: a presentation-layer model built from wide, denormalized **dimension tables** (the descriptive "who/what/when/where/why/how" context of a business event) joined to narrow **fact tables** (the numeric measurements of that event), each fact row carrying foreign keys to its dimensions plus one or more measures. Chosen over normalized (3NF) modeling for the presentation layer specifically because it optimizes for business-user comprehensibility and query performance, at the cost of controlled redundancy.

- **The Four-Step Dimensional Design Process.** Every star schema is designed in a fixed order: (1) select the business process being measured, (2) declare the grain, (3) identify the dimensions, (4) identify the facts. Skipping or reordering these steps is the book's most-repeated failure mode — dimensions and facts chosen before the grain is nailed down produce an ungovernable schema.

- **Grain declaration as the first modeling step.** The grain is a single, precise sentence stating what one fact-table row represents ("one row per data asset per sensitivity scan," not "asset sensitivity data"). Every dimension and fact must be consistent with that one sentence. This is the same discipline `dashboard-specification` already applies to *metric* precision, but the book applies it one level earlier — to the row itself, before any metric is computed from it.

- **Fact table types.** Three canonical shapes, each answering a different question: **transaction fact tables** (one row per discrete event, e.g. one row per compliance gap opened), **periodic snapshot fact tables** (one row per entity per fixed time interval, e.g. sensitivity distribution captured once a day), and **accumulating snapshot fact tables** (one row per process instance, with columns updated in place as the process passes milestones, e.g. one row per remediation workflow with a column per pipeline stage's completion timestamp).

- **Additive, semi-additive, and non-additive facts.** A fact's aggregation behavior must be classified explicitly: **additive** facts sum correctly across every dimension (a count of gaps); **semi-additive** facts sum across some dimensions but not others — classically, a balance or a point-in-time count sums across entities but not across time periods (summing today's and yesterday's asset counts double-counts); **non-additive** facts (ratios, percentages) can never be summed and must be recomputed from underlying additive components.

- **Slowly Changing Dimensions (SCD).** The book's technique for handling dimension attributes that change over time: **Type 0** (attribute never changes, or changes are ignored), **Type 1** (overwrite in place — no history retained), **Type 2** (insert a new dimension row with a new surrogate key and effective/expiration dates or a current-row flag — full history preserved, and facts join to the dimension row that was true when the fact occurred), **Type 3** (add a column holding the previous value — limited, single-step history). Later editions cover hybrid Types 4–7 combining these.

- **Surrogate keys.** Dimension tables use a meaningless, sequentially-assigned key (not the source system's natural/business key) as their primary key. This is what makes SCD Type 2 possible (the same real-world entity can have multiple dimension rows, one per historical version) and insulates the star schema from source-system key reuse or change.

- **Conformed dimensions and the enterprise data warehouse bus matrix.** A conformed dimension is built once with a consistent structure, meaning, and set of values, then shared across every fact table/data mart that needs it — this is what lets two different reports be compared and combined ("drilled across") without disagreeing about what "Tenant" or "Sensitivity Level" means. The **bus matrix** is the master planning artifact: business processes as rows, conformed dimensions as columns, marked cells showing which process uses which dimension. It lets a warehouse be built incrementally, one business process/mart at a time, without creating incompatible stovepipes — the book's answer to building a coherent whole without a big-bang enterprise schema.

- **Degenerate dimensions, junk dimensions, role-playing dimensions.** A **degenerate dimension** is an operational identifier (e.g. an order number) kept as a bare column on the fact table with no corresponding dimension table. A **junk dimension** bundles several low-cardinality flags/indicators into one small dimension table to keep the fact table narrow. A **role-playing dimension** is one physical dimension table (classically `Date`) used multiple times in a single fact table under different aliases/views (order date, ship date, delivery date).

- **Snowflaking is generally discouraged.** The book advises keeping dimension tables flat and denormalized (a "star," not a "snowflake") even though this creates redundancy — the redundancy is a deliberate trade for simpler joins and better query performance for business users, not an oversight.

- **Factless fact tables.** Some business processes are worth recording even with no numeric measure — e.g. "this control was tested and passed" as an event with no associated number. The fact table exists purely to record that a combination of dimension values co-occurred.

## Actionable Techniques

- Before designing any analytical table, write the grain as one literal sentence and reject any dimension or fact that doesn't fit it.
- Run the four-step process explicitly and in order for every new dashboard widget or report section that aggregates data: business process → grain → dimensions → facts.
- Classify every fact column as additive, semi-additive, or non-additive, and state the rule for how it may (or may not) be summed across each dimension, especially time.
- Name the fact table type (transaction / periodic snapshot / accumulating snapshot) for every Read Model that backs a dashboard or report — this determines whether "today's value" and "last month's value" can be safely compared or must be looked up as distinct snapshot rows.
- For every dimension attribute that can change over its entity's lifetime, decide and document its SCD type — don't let "overwrite" be the silent default where a historical "as of" answer is actually required.
- Use surrogate keys plus effective/expiration timestamps for any dimension carrying SCD Type 2 history, so a fact can join to "the dimension row true at the time the fact occurred."
- Build (or at least sketch) a bus matrix once more than one dashboard/report exists, so shared dimensions (tenant, person, data source, sensitivity level, compliance framework) are verified structurally consistent, not just verified by convention.
- Prefer denormalized, flat dimension tables over normalized/snowflaked ones in the presentation layer, even at the cost of redundancy.
- Use a role-playing pattern (one dimension, multiple aliased views) instead of duplicating a dimension table (e.g. `Date`) for every date role a fact needs.
- Reach for a factless fact table when the requirement is "record that this happened," not "measure how much."

## Caveats

- This book's dimensional modeling is designed for the OLAP/BI **presentation layer** — a star schema meant to be queried by business users or BI tools, often physically separate from operational systems. This repo's operational domain model is explicitly DDD Aggregate-shaped (`data-model-design`'s Aggregate-to-Schema Rule: one root per Aggregate, FKs never crossing Aggregate boundaries) and must stay that way — dimensional modeling must never be applied to the operational schema.
- Where it *does* apply is narrower and later in this repo's pipeline than in the book: the Read Model / dashboard / reporting layer already owned by `dashboard-specification` and `reporting-spec`. Adopting the book's techniques here means applying grain/fact-type/SCD discipline to existing PostgreSQL projection tables — not standing up a separate physical data warehouse, a columnar store, or an ETL "back room," none of which this repo's stack (PostgreSQL+pgx, Apache AGE, Redpanda) currently includes or needs.
- Surrogate keys and full SCD Type 2 versioning add real complexity (versioned rows, effective-dated joins) that should only be introduced where a genuine "as of a past date" question is a real requirement (e.g. compliance evidence needing historical sensitivity state) — not applied reflexively to every dimension-like table.
- The bus matrix is most useful once this product has more than one or two dashboards/reports sharing dimensions; with only `dashboard-specification` and `reporting-spec` currently populated, it's worth keeping in mind rather than forcing into either skill immediately.
- Apache AGE's graph model (vertices/edges for the estate relationship graph) is a different tool for a different job — graph traversal, not aggregation — and this book has nothing to say about it; no dimensional concepts should be forced onto the graph model.
