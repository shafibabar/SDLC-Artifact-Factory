## Additional Key Terminology

None of these appear in either original file's Key Terminology list; cross-checked against both before adding, per instructions. Note: `Change Data Capture` is already canonical Ubiquitous Language per `CLAUDE.md`'s event-driven terms list — it is not a new glossary candidate, only newly connected here to a concrete design gap (reprocessing/replay).

- **Derived Data** — Kleppmann's Part III framing term: any dataset that is some function of a source dataset (a cache, index, materialized view, or the estate graph itself). Naming it explicitly would sharpen `data-lineage-design`'s and `data-classification`'s implicit use of the concept.
- **Materialized View (Incrementally Maintained)** — the general name for what `data-classification`'s `effective_level` computation and `data-pipeline-design`'s Read Models both are: a derived view kept up to date by applying deltas, not recomputed from scratch on every change.
- **Reprocessing / Replay** — resetting a consumer group's offset to the start of a log topic to rebuild or backfill derived state; the missing vocabulary for `data-pipeline-implementation`'s backfill gap above.
- **Log Compaction** — a Redpanda/Kafka-family retention policy that keeps only the latest record per key, as distinct from time-based retention that keeps a full history window; the two are a real per-topic choice this repo's pipeline contracts don't currently name.
- **Unbundling (of Databases)** — Kleppmann's Ch. 12 framing for composing specialized stores (search index, graph, cache) that are each kept in sync as derived data from one source of truth, rather than one monolithic database doing everything — a good one-line justification for why this repo's polyglot store choices (AGE, Elasticsearch) are each treated as derived/rebuildable rather than as an additional system of record.

## Caveats

- **Batch processing (Ch. 10) remains out of scope, confirmed again on this pass.** The original Part 1 file already flagged this; nothing in `data-pipeline-implementation` or the other four new skills introduces an offline/batch analytics need — all five are stream/event-choreography or design-time artifacts. No new gap to report here beyond the original caveat.
- **The reprocessing/backfill gap is a real but currently latent risk, not an active defect.** No pipeline stage has yet needed a logic change requiring historical replay; the gap matters the first time one does, and is cheap to close now (a topic-retention line in each stage's contract) versus expensive to discover later (a stage silently log-compacted, with no history left to replay against).
- **This cross-check found less genuinely new content than the original two-file pass, and that itself is informative.** Two of the five new skills (`data-classification`, `analytics-requirements`) have little to no direct overlap with this book's subject matter — they belong to privacy/compliance and product-analytics disciplines Kleppmann does not write about. This is not a failure to find gaps; it is the honest result of checking a systems-engineering book against skills outside its domain.
- **The ML-model-erasure caveat for `data-retention-policy` is speculative and forward-looking.** This repo's current extraction pipeline does not train models on tenant data (rule-based and library-based extraction only, per `data-pipeline-implementation`'s Python-exception framing) — the caveat is worth recording now so it is not missed later, not because it applies to anything built today.
