# Building Micro-Frontends: Distributed Systems for the Frontend
**Author:** Luca Mezzalira
**Domain:** Micro-Frontends — Architecture

## Applicability Note

This repo's frontend default (`CLAUDE.md`, Tech Stack Defaults) is a single React + TypeScript application — one Vite build, one `estate-ui/` project, feature-based folders under one deployable unit (`skills/react-project-structure/SKILL.md`). There is no micro-frontend decomposition anywhere in this repo's current scope, and no multi-team frontend organisation that would motivate one. This file distills the book as forward-looking reference material — the kind of thing worth having on file if a future product genuinely outgrows a single-app frontend — not as a proposal to change this repo's architecture now. See Caveats for why the book's own decision criteria argue against adopting this today.

## Core Concepts

- **Vertical split is the book's actual recommendation, not horizontal split.** Mezzalira distinguishes decomposing a frontend *horizontally* (one team owns the header, another the footer, another a shared cart widget — components sliced across every page) from decomposing it *vertically* (one team owns everything for a business subdomain, from a route down through its own components, state, and even its own build/deploy pipeline). The book treats horizontal split as usually the wrong shape — it multiplies cross-team coordination on every page — and vertical split, aligned to a business subdomain end-to-end, as the pattern that actually delivers team independence.
- **Boundaries come from Domain-Driven Design's Subdomains, not from arbitrary page splits.** A recurring thread in the book is that micro-frontend boundaries should mirror the same Subdomain/Bounded Context boundaries a DDD-informed backend already has — a frontend vertical slice and a backend Bounded Context should line up, so one cross-functional team can own a business capability top to bottom without touching another team's code.
- **The decision to adopt micro-frontends is organizational, not technical.** Mezzalira frames the central question as: does the organisation have a genuine team-scaling problem — multiple teams that need to ship a large frontend independently, on their own release cadence, without blocking on each other's PRs and deploys? If the answer is a single team (or no scaling pain at all), the book's own position is that the added operational complexity (multiple builds, a composition layer, cross-app contracts, more moving parts to deploy and observe) is not worth paying for. This is explicitly a Conway's Law argument: the frontend's shape should mirror the team shape, and per Team Topologies-style thinking, you build micro-frontends to match already-independent, stream-aligned teams — not to manufacture team independence that doesn't otherwise exist.
- **Composition pattern is a separate decision from decomposition strategy, and each has different trade-offs:**
  - **Client-side composition** — an app shell in the browser loads independently-built remote applications at runtime (Webpack Module Federation, single-spa-style registration). Most flexible, most in the "current SPA mental model," but the composition and routing logic lives in the client, so a broken remote can be a partial or full runtime failure, and the shell has to coordinate shared dependencies (React itself, a design system) to avoid shipping duplicate copies.
  - **Edge-side composition** — a CDN/edge layer (traditionally Edge Side Includes, or a modern equivalent) assembles the page from fragments before it reaches the browser. Pushes composition off the client entirely, which helps first-paint performance, but the edge layer becomes a new piece of infrastructure the team must own, and it's a poorer fit for pages that need heavy client-side interactivity after composition.
  - **Server-side composition** — a server (or a dedicated composition layer/gateway) fetches and stitches fragments together as part of generating the HTML response (frameworks like Podium/Tailor exemplify this in the book's era). Best first-paint and SEO characteristics, closest to classic SSR, but requires every micro-frontend to expose a server-renderable fragment, which raises the bar for what each team's stack must support.
  - **Build-time composition** — assembling micro-frontends as npm packages compiled together into one artifact at build time. The book calls this out as usually defeating the purpose: it reintroduces a shared build and a shared release train across teams, which is exactly the coupling micro-frontends exist to remove. It's presented as a trade-off of last resort, not a first-class pattern.
- **Independent deployability is the pillar that validates the whole exercise.** A micro-frontend that cannot be built, tested, and deployed on its own — without coordinating a release with any other team — hasn't actually achieved the goal, no matter which composition pattern is chosen. The book returns to this as the litmus test repeatedly.
- **Cross-application communication should stay shallow.** Mezzalira favors custom browser events / a lightweight pub-sub layer for the rare cases micro-frontends must talk to each other, and explicitly warns against a shared client-side state store spanning micro-frontend boundaries — that re-creates the tight coupling (a shared mutable dependency every team must coordinate around) that vertical decomposition was meant to eliminate.
- **The strangler fig pattern is the book's migration path off an existing monolithic frontend** — carve out one vertical slice at a time behind the existing shell, proving the composition pattern on a low-risk slice before committing the whole frontend to the architecture.

## Actionable Techniques

(Only relevant if this repo ever adopts micro-frontends for a specific future product — see Caveats for why that bar isn't currently met.)

- Decompose along Subdomain/Bounded Context boundaries first; only consider a horizontal split for truly shared, rarely-changing chrome (e.g. a global nav shell), and keep even that as thin as possible.
- Pick one composition pattern deliberately (client-side, edge-side, or server-side) based on the product's actual performance and SEO requirements — don't default to client-side composition just because it matches the current SPA mental model.
- Treat "can this micro-frontend deploy on its own, right now, without a coordinated release" as the acceptance test for every boundary drawn — if the answer is no, the boundary is wrong or the shared dependency needs to move.
- Give every micro-frontend team ownership of its own pipeline, its own versioning, and its own runtime contract (props/events) with the shell — document that contract like an API.
- Ban a shared client-side state store across micro-frontend boundaries; use narrow custom events or a documented pub-sub channel for the few cases of real cross-slice communication.
- Extract a shared design system (tokens, primitives) as its own independently versioned package consumed by every micro-frontend — this is how visual consistency survives independent deployability, and it directly extends what a single design system would already be doing in a monolithic app.
- If migrating an existing monolithic frontend, use the strangler fig approach: pick the lowest-risk vertical slice, prove the composition pattern end-to-end on it, then expand slice by slice — never a big-bang rewrite.
- Build a contract-testing layer between the shell and each remote (interface/props contract, not full e2e) so an independent deploy of one micro-frontend can't silently break another team's slice.

## Key Terminology

Not proposed for `glossary-management` at this time — none of these terms appear anywhere in this repo's actual artifacts, and adding them without a real micro-frontend architecture to anchor them would violate the glossary's own principle of using only terms the domain model actually needs. Recorded here only for future reference, if a product genuinely adopts this pattern:

- **Vertical Split** / **Horizontal Split** — the two decomposition axes; the book's stated preference is vertical.
- **App Shell** — the thin composition/orchestration layer client-side composition depends on.
- **Micro-Frontend** — a vertically-sliced, independently deployable frontend unit owned end-to-end by one team.
- **Composition Layer** — whichever of client-side, edge-side, or server-side mechanism assembles fragments into one page.

## Caveats

- **This repo's actual current context is a solo operator (Shafi) plus Claude Code, generating one frontend per product — there is no multi-team frontend organisation for micro-frontends to solve.** Mezzalira's own decision framework is explicit that the primary justification for micro-frontends is organizational: multiple teams needing to ship a shared frontend independently, without blocking on each other. That condition does not exist here now, and won't exist simply because a product's frontend grows larger — team count, not codebase size, is the book's actual trigger.
- **Applying vertical-split thinking as a single-app *internal* discipline is already happening, and is sufficient.** `react-project-structure`'s feature-based folders with enforced module boundaries get most of the practical benefit (isolation, clear ownership lines, no cross-feature coupling) that vertical-split micro-frontends provide across teams — without paying the operational cost (multiple builds, a composition layer, cross-app contract testing, more moving parts to deploy and observe) that only pays off once more than one team needs to ship independently.
- **Do not force this file's applicability.** If a future product genuinely reaches the point of multiple frontend teams shipping independently, this file is the place to start the actual design conversation — but that is a real architectural decision requiring Shafi's explicit approval (a new `sdlc-config.json` tech-stack decision, per this repo's Budget and Frugality rule: added complexity must justify its presence), not something to infer from a book being researched.
- **The book is not primarily a performance-optimization or component-design reference** — its actual subject is deployment/team topology, and most of its concrete guidance (composition mechanics, contract boundaries, migration strategy) only becomes relevant once the organizational precondition is met. Treating it as a source of frontend *implementation* patterns for the current single-app frontend would be a misread of what the book is actually for.
