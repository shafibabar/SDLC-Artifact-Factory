# The Art of Micro Frontends
**Author:** Florian Rappl
**Domain:** Micro-Frontends — Implementation Patterns

## Applicability Note

Same framing as the sibling file (`building-micro-frontends-mezzalira.md`): this repo's frontend default (`CLAUDE.md`, Tech Stack Defaults) is a single React + TypeScript application — one Vite build, one `estate-ui/` project (`skills/react-project-structure/SKILL.md`). There is no micro-frontend decomposition in this repo's current scope, and (per the sibling file's Caveats) no multi-team frontend organisation that would justify one. Where the Mezzalira book argues the *organizational* case for whether to decompose at all, Rappl's book is the *mechanical* reference for how to actually build the integration once that case is made — runtime composition techniques, styling isolation, cross-application communication, and dependency/versioning management. This file is forward-looking reference material only, not a proposal to change this repo's architecture now.

## Core Concepts

- **The integration-technique spectrum, and their real trade-offs.** Rappl's book is organized around comparing the concrete mechanisms available to compose independently-built frontend fragments into one page at runtime — this is the book's actual center of gravity, distinct from Mezzalira's team-topology framing:
  - **iframes** — the oldest and most isolated technique: a truly separate browsing context (separate DOM, separate JS realm, separate CSS cascade) embedded via `<iframe>`. Strongest isolation guarantee available in a browser (no style bleed, no global-scope collision, a crash in one frame can't take down the shell), but at a real cost: no shared routing/history without extra plumbing, awkward responsive sizing, degraded accessibility (focus management, screen-reader traversal across frame boundaries), and communication reduced to `postMessage` — no direct DOM or JS-object access across the boundary. The book treats iframes as the right tool when isolation is the priority above all else (e.g. embedding untrusted or fully independent third-party content), not the default choice for a cooperative multi-team product.
  - **Web Components / Custom Elements** — register a framework's output as a native custom element (`customElements.define`) so any host page can drop in `<my-widget>` regardless of what framework built it inside. This is the book's preferred "same-document" integration mechanism: it runs in the same JS realm as the shell (unlike an iframe), so DOM-level interop is direct, while Shadow DOM (see below) still gives style and markup encapsulation. The friction point the book is candid about: React does not map cleanly onto the Custom Elements lifecycle (props vs. attributes, event dispatching, and re-render timing all need a wrapper layer), so a genuinely framework-agnostic host that mixes React/Vue/Angular fragments pays a real integration tax per framework.
  - **JavaScript-bundling / runtime composition (Module Federation and peers)** — the shell and each micro-frontend are separately built and deployed bundles that resolve each other's exposed modules *at runtime* rather than at a shared build step. Webpack 5's Module Federation (and later esbuild/Vite equivalents, and frameworks the book surveys such as single-spa, qiankun, and Rappl's own Piral) is presented as the mechanism that most closely preserves "normal" JS-application ergonomics — no iframe boundary, no custom-element wrapper tax — while still shipping fragments independently. Its central operational hazard is **shared singleton dependencies**: if the shell and every remote don't agree to treat `react`/`react-dom` (and any shared design-system package) as a runtime singleton, the page silently loads multiple copies of React, breaking hooks and duplicating bundle weight. The book returns to singleton configuration as the single most common Module Federation misstep.
  - **Server-side / edge-side composition** — covered as a complementary axis, not a competing one: the *runtime* techniques above answer "how do fragments combine in the browser," while server/edge composition answers "who assembles the response and when" (this overlaps with, but is more implementation-detailed than, the sibling file's edge/server composition summary).
- **Styling isolation is a first-class problem, with a small set of real solutions.** Because independently-built fragments share one page (except behind an iframe), uncontrolled CSS collides — the book treats this as inevitable without a deliberate strategy, and lays out the actual options:
  - **Shadow DOM** — the browser-native encapsulation boundary (paired naturally with Web Components): styles inside a shadow root neither leak out nor get leaked into. Strongest guarantee short of an iframe, but complicates theming (design-token/CSS-custom-property propagation across a shadow boundary needs deliberate piercing, e.g. `::part()` or CSS custom properties, which do cross shadow boundaries by design).
  - **CSS Modules / scoped class names** — build-time class-name hashing (`.button_a3f2b1`) so each fragment's styles are namespaced without a runtime boundary. Works with any bundler, zero runtime cost, but only as strong as the build pipeline enforcing it — a fragment that ships un-scoped global CSS (a raw `@import`, a third-party stylesheet) still leaks.
  - **Naming-convention discipline (BEM and similar)** — a low-tech, no-tooling fallback: prefix every class with the owning micro-frontend/team name so collisions are avoided by human convention rather than machine enforcement. The book is explicit this is the weakest guarantee of the three — it depends on every team actually following it — but is sometimes the only option when the shell can't mandate a build tool across teams.
  - **CSS-in-JS** with per-component scoping (styled-components/Emotion-style generated class names) is covered as a variant of the CSS Modules approach — runtime- or build-time-generated unique selectors, same collision-avoidance goal, different mechanism.
- **Cross-fragment communication and shared state.** Once fragments are independently built, they still sometimes need to talk — the book's treatment (aligned with, and more mechanically detailed than, the sibling file's "keep it shallow" principle):
  - **Custom DOM events** (`CustomEvent` + `dispatchEvent`/`addEventListener`) as the lowest-coupling channel — a fragment announces something happened without knowing who's listening.
  - **A shared event bus / pub-sub layer** provided by the shell — one step more structured than raw DOM events, still without a shared mutable store.
  - **Props/attributes at mount time** — for one-directional shell → fragment configuration (the fragment's initial context: tenant id, locale, feature flags), the simplest and most common channel.
  - **A deliberately shared, minimal state slice** (e.g. current user, current tenant, auth token) exposed by the shell through a well-defined API — the book allows this narrow case (unlike Mezzalira's blanket ban on any shared store) but is emphatic it must be a small, versioned, read-mostly surface — not a general-purpose cross-fragment store that re-creates monolith-style coupling.
- **Versioning and independent deployability as an ongoing operational concern, not a one-time decision.** Because each micro-frontend ships on its own cadence, the book spends real attention on: semantic-versioning the shell↔fragment contract (props/events/exposed-module shape) so a fragment can deploy without breaking the shell or siblings; dependency-version negotiation for shared singletons (which version of React "wins" when fragments were built against different minors); and feature-flagging / canary rollout of a single fragment's new version behind the same shell, so an independent deploy can be rolled back without touching any other fragment.
- **Testing across the fragment boundary.** The book treats contract/integration testing between shell and fragments (not full e2e as the only safety net) as necessary precisely because unit tests inside one fragment's repo can't catch a shell-contract break introduced by another team's independent deploy.

## Actionable Techniques

(Only relevant if this repo ever adopts micro-frontends for a specific future product — see Applicability Note and Caveats.)

- If runtime composition is chosen over iframe isolation, default to Module Federation (or its current-generation equivalent) over Web Components for a same-framework (all-React) product — it avoids the Custom Elements/React lifecycle friction the book documents, at the cost of being less framework-agnostic.
- Reserve iframes for genuinely untrusted or fully independent third-party content, not for cooperating internal teams' fragments — the isolation is real but the routing/a11y/communication costs are steep.
- Mark shared runtime dependencies (`react`, `react-dom`, the shared design-system package) as **singletons** explicitly in the module-federation config; treat a duplicate-React-copy warning as a build-breaking error, not a lint warning.
- Choose one styling-isolation mechanism deliberately and enforce it in CI, don't leave it to convention: CSS Modules (or equivalent build-time scoping) as the default for a same-framework Module Federation setup; Shadow DOM only if using Web Components; naming-convention (BEM) only as a last resort when no shared build tooling can be mandated.
- Version the shell↔fragment contract (exposed props, events, and any exposed module signatures) with semantic versioning, and treat a breaking contract change the same as a breaking API change — coordinated, documented, never silent.
- Expose at most a small, explicit, versioned "shell context" (current user, tenant, feature flags) via a documented API — never a general shared store fragments can read/write arbitrarily.
- Build shell↔fragment contract tests (mount the fragment against the shell's actual exposed context/props shape) as a required CI gate for each fragment's independent deploy pipeline — this is the mechanical counterpart to the sibling file's "contract-testing layer" recommendation.
- Canary/feature-flag a fragment's new version behind the existing shell before full rollout, so an independent deploy of one fragment has a rollback path that doesn't touch any other fragment or the shell.

## Key Terminology

Not proposed for `glossary-management` at this time, for the same reason the sibling file gives — none of these terms appear in this repo's actual artifacts, and the glossary's own principle is to carry only terms the current domain model needs. Recorded here only for future reference:

- **Module Federation** — the webpack 5 (and equivalent) mechanism for runtime-resolving separately-built JS modules across a shell/remote boundary.
- **Singleton dependency** — a shared runtime dependency (e.g. `react`) explicitly deduplicated across shell and remotes so only one copy loads.
- **Shadow DOM** — the native browser DOM/CSS encapsulation boundary used by Web Components for style isolation.
- **Custom Element** — a browser-native component registered via `customElements.define`, the Web Components integration unit.
- **Shell↔Fragment Contract** — the versioned props/events/exposed-module surface a host and an independently-deployed fragment agree to.

## Caveats

- **This file is more confident about durable, well-established micro-frontend implementation mechanics (iframes, Web Components/Shadow DOM, Module Federation's singleton-sharing problem, CSS-scoping strategies) than about the book's exact chapter structure or specific code listings.** Rappl (a Piral maintainer) is a credible primary source for these techniques and the book is understood to cover Piral and comparable frameworks (single-spa, qiankun) as worked examples, but this file leans on well-established, cross-source-corroborated industry practice for the specifics above rather than asserting precise chapter-by-chapter content — flagged here rather than fabricated with false precision.
- **This book is the mechanical complement to the sibling Mezzalira file, not a replacement for its organizational reasoning.** Knowing *how* to wire up Module Federation singletons or scope CSS across fragments does not answer *whether* this repo should ever have more than one deployable frontend for a product — that question is still Mezzalira's team-topology argument (and per the sibling file's Caveats, this repo's current single-operator context does not meet that bar).
- **Do not let this file's styling-isolation content bleed into today's single-bundle styling decisions.** This repo's current frontend has one CSS cascade by construction (one Vite build); CSS Modules/Shadow DOM/BEM as *collision-avoidance-across-independently-built-fragments* techniques are a different problem from ordinary component-scoped styling in a single app. If a plain (non-micro-frontend) styling skill is ever written for `react-component-design` or a new `react-styling` skill, it should be sourced from ordinary single-app CSS/styling practice, not from this file.
- **As with the sibling file, adopting any of this is a real architectural decision requiring Shafi's explicit approval** (a new `sdlc-config.json` tech-stack decision, per the Budget and Frugality rule) — this file exists to have the mechanics on record if that day comes, not to propose it now.
