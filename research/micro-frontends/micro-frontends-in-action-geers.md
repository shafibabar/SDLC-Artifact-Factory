# Micro Frontends in Action
**Author:** Michael Geers
**Domain:** Micro-Frontends — Team Ownership & Production Practice

## Core Concepts

- **The book teaches through real production case studies, not abstract patterns first.** This is the book's actual organizing device and its main point of difference from the two sibling books: rather than deriving decomposition/composition rules in the abstract and illustrating them afterward, Geers walks through how specific companies actually shipped micro frontends in production — the specific integration technique each org picked, why, and what broke. This is the "strong practical case-study angle" the prompt flags, and it's the book's real value: it substitutes "here is what actually shipped and what it cost" for "here is the theoretically correct pattern."
- **Self-Contained Systems (SCS) as the more radical sibling concept.** Geers gives real weight to SCS — an architecture (with roots in the German consulting community, INNOQ) that goes further than "micro frontend" as commonly used elsewhere in this research: an SCS owns its own UI *and* its own backend logic *and* its own datastore, end to end, as one autonomous unit. Cross-system UI composition is deliberately minimized — an SCS mostly links to other SCSs via ordinary hyperlinks rather than embedding another system's fragment inline. The book treats SCS as the purer, less-coupled expression of the same team-autonomy goal micro frontends chase at the frontend layer only.
- **Team-to-fragment ownership, grounded in real organizations rather than asserted as theory.** Like Mezzalira, Geers argues each team should own a vertical slice — a page or a cluster of pages — end-to-end, including its own deployment pipeline, so releasing doesn't require cross-team coordination. What Geers adds is naming which real companies actually organize this way and what technology each one picked to make it work (e.g., Zalando's early work with a server-side composition tool, IKEA's use of native web components to let independently-built fragments interoperate, DAZN's adoption of a client-side single-spa-style shell to run multiple frameworks' fragments side by side). The book's case-study chapters are explicit that the *organizational* commitment (one team, one fragment, one pipeline) came before the technical integration choice, not after.
- **Routing strategies span a wider spectrum than "one client-side router," including the simplest possible option.** The book is candid that the lowest-tech, lowest-risk way to let independently-owned fragments coexist is plain **hyperlinks between separately deployed pages** — a genuine multi-page-app style handoff with no shared client-side router, no shared shell state, and no JS-level composition at all. A user clicking from one team's page to another's is a full page navigation. Geers presents this not as a compromise but as a legitimate first rung: it costs nothing to build, has zero cross-fragment coupling, and is what a surprising amount of real production micro-frontend traffic actually uses for the boundary between, say, a marketing site and an authenticated app. From there the book covers progressively more integrated routing: a shared app-shell that owns the URL-to-fragment mapping and lazy-loads each fragment's bundle client-side (closer to Rappl's client-side composition), and server/edge-side routing where a reverse proxy or edge layer decides which team's backend serves a given path before any HTML is composed — page-level ownership enforced at the infrastructure layer rather than in a JS router.
- **Production trade-offs are treated as real costs paid, not hypothetical risk.** The book does not hedge on this: independently-built fragments frequently mean the same framework runtime (React, or whatever each team chose) ships multiple times across a single user session as they navigate between fragments, inflating page weight and hurting repeat-load performance compared to one shared bundle. Design/branding drift across independently-shipped fragments is called out as a recurring real-world failure mode, not a theoretical one — teams' fragments visibly diverge over time without active enforcement. The book's honest position is that these costs are the price of the team-autonomy and independent-deploy gains, and a team adopting micro frontends should go in expecting to actually pay them, not assume tooling will make the trade-off free.
- **A shared pattern/component library as the practical answer to visual drift** — published and versioned independently of any single fragment's release cadence, consumed by every team's fragment, so visual consistency survives independent deployability without requiring a shared deploy. This runs parallel to Mezzalira's and Rappl's design-system points but the case-study framing here makes it a genuinely observed practice at the companies profiled, not a first-principles recommendation.
- **Full-stack, cross-functional teams as the organizational precondition** — echoing the "Inverse Conway Maneuver" idea (deliberately shaping team boundaries to get the system architecture you actually want, rather than discovering after the fact that your architecture mirrors an accidental team structure). This is the same Conway's-Law argument the Mezzalira file already covers; Geers's distinguishing move is anchoring it in the specific team structures of the companies profiled rather than as an abstract organizational-design principle.

## Actionable Techniques

(Only relevant if this repo ever adopts micro-frontends for a specific future product — see Caveats for why that bar isn't currently met.)

- Before reaching for any JS-runtime composition (Module Federation, web components, single-spa), evaluate whether plain hyperlinks between independently deployed pages actually satisfy the requirement — it is the cheapest, lowest-coupling option and this book treats it as a legitimate first choice, not a fallback.
- If a future product's decomposition goes deep enough that a "fragment" would also need its own backend and datastore (not just its own frontend bundle), evaluate the Self-Contained Systems framing explicitly rather than defaulting to a frontend-only micro-frontend split — it's a different (and more autonomous) boundary shape with different consequences for data ownership.
- Before picking a composition technique, look for a comparable real-world case study with a similar team-count and traffic profile (the book's own method) rather than choosing a pattern in the abstract — match the technique to precedent at a similar scale.
- Budget for framework-duplication page weight explicitly if fragments are independently built with independent bundlers — measure it, don't assume a shared "smart" build step will dedupe it away by default.
- Stand up a shared, independently-versioned pattern/component library from day one of any multi-fragment effort — treat visual drift as a certainty to actively counter, not an edge case.
- Confirm the Inverse-Conway precondition explicitly before adopting: does the team structure already look like (or is it about to be deliberately reshaped into) the fragment boundaries being proposed? If not, the technical work will not fix the coordination problem.
- Choose the routing layer (hyperlink handoff / client-side shell / server-side edge routing) based on how tightly the fragments' navigation needs to feel unified to the end user — don't default to the most technically sophisticated option (a shared client-side shell) when a simpler hyperlink boundary meets the actual UX bar.

## Key Terminology

Not proposed for `glossary-management` at this time — none of these terms appear anywhere in this repo's actual artifacts, and adding them without a real micro-frontend or Self-Contained Systems architecture to anchor them would violate the glossary's own principle of using only terms the domain model actually needs. Recorded here only for future reference, if a product genuinely adopts this pattern:

- **Self-Contained System (SCS)** — an autonomous unit owning its own UI, backend logic, and datastore end-to-end, communicating with sibling systems mostly via hyperlinks and asynchronous messaging rather than inline UI composition. The most distinctive term this book contributes beyond what either sibling file's terminology already covers.
- **Fragment** — an independently built and deployed piece of a composed page, owned by one team (used across all three books but central to how Geers frames team ownership).
- **Inverse Conway Maneuver** — deliberately shaping team boundaries to produce a desired system architecture, rather than discovering the architecture already mirrors an accidental team structure.
- **Page-based composition** — assembling a full page from a single owning team/system rather than compositing fragments from multiple teams on one page; the book's term for the vertical-split idea applied at the page-ownership level.

## Caveats

- **This book's team-ownership model assumes multiple independent teams shipping independently — a precondition this repo does not meet at any level.** This repo has one `frontend-engineer` agent producing one frontend for one product; there is no second team, no second pipeline, and no organizational scaling problem for the book's case studies to solve. The book's own case-study framing makes this more, not less, obvious than the sibling files do: every named example (Zalando, IKEA, Spotify, DAZN, OpenTable, HelloFresh) is a company with multiple product teams that needed to stop blocking on each other — a situation with no current analogue in this repo's single-operator, single-agent structure.
- **This file is less certain about exact chapter organization, page numbers, or verbatim case-study details than about the book's general shape and recurring themes** (case-study-driven pedagogy, SCS as a named alternative/extension, the hyperlink-as-legitimate-routing-option stance, and candor about production performance costs). These are treated here as well-corroborated characterizations of the book's actual content and stance, not verbatim quotations — flagged rather than asserted with false precision, consistent with how the Rappl sibling file handles the same uncertainty.
- **Self-Contained Systems is a materially different — and larger — boundary than anything else in this research set.** Both sibling files (and most of this book's own micro-frontend chapters) treat the boundary as frontend-only: one team owns a UI slice, the backend contract is still shared/negotiated separately (per this repo's `enterprise-architect`-owned `openapi.yaml`). SCS instead bundles frontend, backend, and datastore into one autonomous unit per team — a genuinely different (and more invasive) architectural commitment than a frontend-only micro-frontend split. If this ever became relevant, it would be a whole-system decomposition decision, not a frontend-implementation one, and would need to be evaluated against this repo's backend architecture (Go services, PostgreSQL, `enterprise-architect`'s contract ownership) as much as against the frontend.
