This file should therefore be read as a technical supplement to Rappl's Module Federation paragraph, not a restatement of it.

**Distinctly worth separating from the full-micro-frontend question:** Module Federation is also directly usable **without adopting micro-frontend architecture at all** — the narrower case is sharing a component library, a design system, or a single feature module across otherwise-independent applications (e.g. two unrelated internal tools that both want the same `Button`/`DataTable` set, or a "properties panel" plugin shared between two products) without those applications becoming a decomposed single product's fragments. This narrower use case does not require adopting vertical-slice team ownership, an app shell, or any of the organizational precondition Mezzalira's book gates full micro-frontend adoption on — it is closer to "a smarter npm package" that resolves at runtime instead of install time. The book is explicit that this is a legitimate, standalone use of the technology, and it is the more plausible near-term relevance for this repo (a shared `shared/ui/` design-system package consumed by a second, currently-hypothetical product) than a full vertical-slice decomposition of one product's frontend.

## Core Concepts

- **`exposes` / `remotes` / `shared` are the three configuration primitives, and their precise semantics matter more than the sibling files' survey treatment suggests.** In a Webpack 5 `ModuleFederationPlugin` config:
  - `exposes` maps a public name (e.g. `./Button`) to a local module path in the app acting as a **remote** (the app being consumed by others). Each exposed entry becomes its own async chunk, loaded on demand — not bundled into the remote's own initial load unless something in the remote's own app also imports it.
  - `remotes` maps a name to a **container location** (a URL to the remote's `remoteEntry.js`, the small federation runtime manifest each remote emits) in the app acting as a **host** (the app consuming others). The host's bundler does not need the remote's source at build time — resolution happens in the browser at runtime.
  - `shared` is the dependency-negotiation layer: for each package listed (`react`, `react-dom`, a design-system package), Module Federation decides at runtime whether the host's copy or a remote's copy satisfies everyone's semver range, and only loads the copies that don't. This is the single most consequential config surface in the book and the one place a shallow reading (as in Rappl's survey) undersells the actual mechanics.
- **Runtime federation vs. build-time integration — the book's central distinction.** Unlike a monorepo or an npm-published component library (resolved by the bundler at **build time**, baked into one artifact), Module Federation resolves which code loads at **runtime**, in the browser, after both the host and the remote are already independently deployed. This is what makes independent deployability real: a remote can ship a new version and every host picks it up on next page load without a host rebuild — but it also means integration bugs that a build-time system would catch at compile time (a removed export, an incompatible prop shape) surface at runtime instead, in production, for a real user. The book treats this trade — deploy-time independence purchased with weaker static guarantees — as the central cost/benefit ledger of the whole technology, more explicitly than any sibling file frames it.
- **Dynamic remote loading (not just static `remotes` config) is a first-class technique the book covers in depth.** Beyond declaring a fixed `remotes` map at build time (`remoteName: 'remoteName@https://host/remoteEntry.js'`), Module Federation supports resolving a remote's URL **at runtime** — e.g. fetching a manifest that lists which remote URL to use per environment/tenant, or letting a host discover and mount a remote it didn't know about at its own build time (`import(/* webpackIgnore: true */ ...)` patterns, or the more ergonomic dynamic-remote APIs the book walks through). This is materially more advanced than Rappl's survey-level "shell and remotes resolve each other at runtime" framing — it is the mechanism that would let, for example, one shell serve different per-tenant remote bundles from the same deployed shell image, which is directly relevant to this repo's physical multi-tenancy default if this pattern were ever adopted for a UI-plugin use case.
- **Version negotiation and singleton conflict resolution — the book's deepest and most load-bearing content, going well past the sibling files' single-paragraph "avoid duplicate React" warning.** The mechanics, as the book lays them out:
  - `shared: { react: { singleton: true, requiredVersion: '^18.0.0', eager: false } }` — `singleton: true` tells Module Federation there must be exactly **one** instance of this module shared across host and every remote at runtime, no matter how many separately-built bundles request it (this is required for React specifically, because multiple React copies break the Hooks dispatcher — a component from one copy calling into another copy's runtime is undefined behavior, not just "wasted bytes," which is a stronger claim than the sibling files' "duplicate bundle weight" framing).
  - **Version negotiation without `singleton`**: when a shared package is *not* marked singleton, Module Federation allows multiple versions to coexist, each remote getting the version it was built against — this is a deliberate escape hatch for packages where the sibling files' "ban duplicates" instinct doesn't actually apply (a chart library two remotes both use independently, not as a shared runtime object).
  - **The "first loaded wins" resolution hazard**: without careful `requiredVersion` ranges, Module Federation's negotiation can non-deterministically select whichever copy of a singleton dependency happens to be requested first at runtime (a function of load order, which can depend on network timing) — the book is explicit that this is a real production failure mode, not a theoretical one, and treats a strict `requiredVersion` semver range plus a `strictVersion: true` (fail loudly rather than silently pick an incompatible version) as the actual fix, not just "mark it singleton and hope."
  - **`eager: true` vs. lazy shared modules**: eager sharing bundles the dependency into the initial chunk (needed for the host's own first render before any remote has loaded, since the negotiation itself needs *something* loaded to compare candidate versions against); lazy (the default) defers loading the shared module until first actually needed, which is right for most remotes but wrong for a host that must render before any remote federation call completes.
- **Bidirectional / omnidirectional federation — a host can simultaneously be a remote, and the book treats this as normal, not exotic.** Nothing about Module Federation restricts an app to being purely a "shell" or purely a "remote" — an app can declare both a `remotes` map (consuming others) and an `exposes` map (being consumed by others) in the same `ModuleFederationPlugin` config. The book's term for this fully-meshed topology (every app both a host and a remote to every other) is **omnidirectional federation**, contrasted with the simpler **unidirectional** shell→remotes topology the sibling files implicitly assume when they describe "the shell" as a distinct role. This matters practically: it means Module Federation is equally suited to a hub-and-spoke topology (one shell, many remotes, as the sibling files describe) and to a peer-to-peer mesh of apps that each expose some things and consume others — the narrower "share a component library between two otherwise-independent apps" use case in the Applicability Note above is exactly this bidirectional/mesh shape, not the hub-and-spoke shape the sibling files' full-micro-frontend framing defaults to.
- **The federation runtime itself (`remoteEntry.js`) is a small, separately-cacheable manifest, not the remote's full bundle.** Each federated app emits a container entry that exposes an async `get`/`init` API the host's federation runtime calls to negotiate shared scope and then fetch the actually-requested exposed module's chunk. Understanding this two-step (negotiate shared scope, then fetch the specific chunk) is what makes the version-negotiation mechanics above legible — negotiation happens before any of the remote's actual feature code downloads.
- **Type safety across the federation boundary is a real, separately-solved problem the book addresses directly** (largely absent from the sibling files, which are JavaScript/architecture-focused rather than TypeScript-specific): since a host's bundler never sees a remote's source at build time, TypeScript has no way to type-check a dynamically federated import by default. The book covers the practical workaround — generating and publishing a remote's `.d.ts` types as a side artifact (a separate small package, or a federation-aware plugin that fetches types at dev time) so the host gets real autocomplete/type-checking against a remote it doesn't build. This is directly relevant to any TypeScript-strict repo (this repo's `typescript-types` skill bans `any` outright) that might ever consume a federated remote.

## Actionable Techniques

(Only relevant if this repo ever adopts Module Federation — for either the narrower shared-library use case or full micro-frontend decomposition; see Applicability Note and Caveats.)

- For the narrower use case (sharing a component library across independent apps, not full micro-frontend decomposition): expose only the specific components/hooks meant to be public (`exposes: { './Button': './src/shared/ui/Button' }`), never the whole `shared/ui/` barrel — mirrors this repo's own `react-project-structure` rule against eager barrel exports, for the same tree-shaking reason.
- Mark every shared UI-runtime dependency (`react`, `react-dom`, and any shared design-system package) `singleton: true` with an explicit `requiredVersion` semver range and `strictVersion: true` — never rely on the default negotiation to "figure it out," per the book's documented first-loaded-wins hazard.
- Use `eager: true` only for the specific shared dependency the host needs before any remote's federation call resolves (typically the shared UI runtime itself); leave everything else lazy to preserve the code-splitting benefit.
- If per-tenant remote resolution is ever needed (this repo's physical multi-tenancy default), use dynamic remote loading (resolve the remote URL from a runtime manifest/config, not a build-time `remotes` map) rather than baking tenant URLs into the host's build — this is the same "build once, inject config at runtime" principle `react-dockerfile` already applies to `window.__APP_CONFIG__`, extended to remote resolution.
- Generate and publish a remote's TypeScript declarations as a build artifact consumed by the host's typecheck step — do not let a federated import silently degrade to `any`, which `typescript-types`' lint ban would otherwise be defeated by.
- Decide unidirectional (hub-and-spoke shell/remotes) vs. bidirectional/omnidirectional topology deliberately, based on whether apps genuinely need to expose to each other symmetrically — default to the simpler unidirectional shape unless a real mesh requirement exists.
- Version the `exposes` surface with the same discipline the sibling files recommend for the shell↔fragment contract generally (semver, documented, breaking changes coordinated) — Module Federation's runtime resolution does not remove the need for a contract, it just moves where the contract is checked (runtime, not compile time).

## Key Terminology

Not proposed for `glossary-management` at this time — none of these terms appear anywhere in this repo's actual artifacts, and the glossary's own principle is to carry only terms the current domain model needs. Recorded here only for future reference, if the narrower shared-library use case or full micro-frontend decomposition is genuinely adopted:

- **Container / Remote Entry** — the small runtime manifest (`remoteEntry.js`) a federated app exposes, distinct from its full bundle; the two-step negotiate-then-fetch mechanism depends on this being loaded first and separately.
- **Singleton (shared dependency)** — a shared runtime dependency constrained to exactly one loaded instance across host and all remotes (mandatory for React/React DOM); distinct from ordinary version-permissive sharing.
- **Version Negotiation** — the runtime process by which Module Federation selects which loaded copy of a shared dependency satisfies every consumer's semver range.
- **Eager Sharing** — bundling a shared dependency into the initial chunk (rather than lazily loading it) so it's available before any remote's federation call resolves.
- **Bidirectional / Omnidirectional Federation** — a topology where an app is simultaneously a host (consuming remotes) and a remote (exposing modules to others), as opposed to a strict shell/remotes hierarchy. The most distinctive term this book contributes beyond what any sibling file's terminology covers — none of the three siblings name this topology explicitly.
- **Dynamic Remote** — a remote whose location is resolved at runtime (from a manifest/config) rather than fixed in the host's build-time `remotes` config.

## Caveats

- **This book is Webpack-5-specific; this repo's actual build tool is Vite, not Webpack** (confirmed directly: `skills/react-project-structure/SKILL.md`'s Vite Build section specifies `vite.config.ts` / `defineConfig` / `rollupOptions` — there is no Webpack configuration anywhere in this repo). Module Federation as this book describes it is a `webpack.ModuleFederationPlugin` mechanism; a Vite-based repo would reach for `@originjs/vite-plugin-federation` or the newer, more actively maintained **Module Federation 2.0 / `@module-federation/vite`** tooling (part of the broader Module Federation project's move to be bundler-agnostic, spanning Webpack, Rspack, and Vite) rather than adopting Webpack purely to get Module Federation. The core *concepts* this file distills (`exposes`/`remotes`/`shared`, singleton negotiation, dynamic remotes, bidirectional federation) transfer across these implementations since they're the same underlying protocol/runtime concept — but exact config syntax and available options (e.g. Vite plugin maturity/edge cases) differ from this book's Webpack-specific code and would need to be verified against the Vite-side tooling's own docs before implementation, not assumed identical.
- **This file is confident about the durable, well-corroborated mechanics** (the `exposes`/`remotes`/`shared` config model, the runtime-vs-build-time distinction, singleton negotiation and its first-loaded-wins failure mode, dynamic remote loading, bidirectional/omnidirectional federation as a named topology, and the cross-boundary TypeScript problem) as accurately representing what a book by Module Federation's own creator would cover in depth — these are the mechanism's actual, well-documented characteristics, not filler. It is less certain about this specific book's exact chapter structure, code-listing verbatim syntax, or page-level organization, which is flagged here rather than asserted with false precision, consistent with how the Rappl and Geers sibling files handle the same kind of uncertainty.
- **Do not let this file's depth read as an implicit recommendation to adopt Module Federation.** Going deeper on one mechanism's mechanics is not the same claim as Mezzalira's organizational precondition being met (it still isn't — see the sibling files) — this file exists so the *narrower* shared-library use case (which does not require the organizational precondition) has a technically accurate reference on hand, and so the full-micro-frontend mechanics are on record if that day ever comes.
- **As with all three sibling files, adopting any of this — even the narrower shared-library use case — is a real architectural decision requiring Shafi's explicit approval** (a new `sdlc-config.json` tech-stack decision, per the Budget and Frugality rule: added complexity, including a new build plugin and a new class of runtime failure mode, must justify its presence) — this file exists to have the mechanics on record if that day comes, not to propose it now.
