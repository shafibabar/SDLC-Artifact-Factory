# Learning Domain-Driven Design
**Authors:** Vlad Khononov
**Domain:** Domain-Driven Design — Tactical Pattern Selection, Event Sourcing, CQRS, and the Boundary Hierarchy (the modern, post-microservices synthesis)

## Core Concepts

- **An Aggregate is a consistency boundary shaped by two independent forces that must both be checked before the pattern is even reached for — a true multi-object invariant (Evans/Vernon's own criterion, already covered in `research/domain-driven-design/implementing-ddd-vernon.md`) *and* enough business-logic complexity in the containing subdomain to justify the Domain Model pattern at all.** This is Khononov's sharpest reframing of Aggregate design: where Evans and Vernon both write as if "does a true invariant exist here" is the only question, Khononov inserts a prior, cheaper gate — is this subdomain even one where the Aggregate/Domain Model pattern is the right *implementation pattern* in the first place, independent of whether a candidate invariant can be found? A Generic or simple Supporting subdomain can have something that superficially looks like a cross-object invariant and still not warrant an Aggregate, because the correct fix might be to simplify the business rule (or accept a validation-not-enforcement stance) rather than build tactical DDD machinery around it. `skills/aggregate-design/SKILL.md` currently has only the first gate (Rule 1's invariant test) and no version of the second — it assumes every Bounded Context reaching this skill already needs Aggregates, an assumption this book directly challenges (see next concept).
- **Khononov's Business Logic Design Pattern catalog names four patterns, not one, and Aggregate-based Domain Model is only the most expensive of them — Transaction Script, Active Record, Domain Model, and Event-Sourced Domain Model, each matched to a different point on the complexity/investment spectrum already established by `skills/subdomain-distillation/SKILL.md`'s Core/Supporting/Generic classification.**
  - **Transaction Script** — each use case is a single procedure that operates directly against persistence (a script, not an object model); no Entities, no Value Objects, no invariant-enforcing methods. The business logic *is* the procedure. Appropriate for Generic subdomains (a bought/adopted capability with a thin custom wrapper) and the simplest Supporting subdomains where a use case is a handful of sequential steps with no rule that spans multiple records atomically.
  - **Active Record** — an object per persisted row (or per table), carrying both data and simple CRUD-shaped behaviour, with the object structure mirroring the storage structure directly (no separation between domain shape and persistence shape). More structure than Transaction Script, but still no enforced cross-object invariant and no isolation from the persistence schema. Appropriate for Supporting subdomains with moderate, mostly single-record logic — meaningfully more logic than pure CRUD, not yet complex enough to need a domain model isolated from storage.
  - **Domain Model** — the full DDD tactical pattern set this repo's `skills/aggregate-design/SKILL.md` and `skills/go-domain-model/SKILL.md` already implement: Aggregates, Entities, Value Objects, invariants enforced by the root, isolation from persistence. Reserved for Core Domain and complex Supporting subdomains where the business rules are genuinely intricate enough that the modeling investment pays for itself.
  - **Event-Sourced Domain Model** — the Domain Model pattern with one additional decision layered on top: the Aggregate's state is derived by replaying its own event history rather than read from a current-state row (see below). A further, optional escalation reserved for Core Domain subdomains with a specific additional need (temporal/audit query, retroactive read-model construction), not a default even within the Core Domain.
  This four-pattern catalog is genuinely new content: `skills/subdomain-distillation/references/classification-techniques.md`'s Khononov section already cites "Core Domain subdomains often warrant full DDD tactical patterns... Generic subdomains warrant the simplest viable approach — even a plain CRUD/Transaction Script" but names only the two extremes and never Active Record as the deliberate middle rung, nor Event-Sourced Domain Model as a fourth, separate rung above Domain Model. Neither `aggregate-design` nor any Go implementation skill read in this session names Transaction Script or Active Record as legitimate, deliberately-chosen alternatives at all — as currently written, this repo's skill set reads as if the Domain Model/Aggregate pattern is simply *the* way to implement business logic, with no cheaper alternative ever discussed for the Supporting/Generic subdomains this repo's own `subdomain-distillation` worked example identifies (Storage Connectors' non-security-sensitive slice, Authentication/Authorization's build-vs-buy slice).
- **Pattern selection happens per Aggregate or per use case, not once per Bounded Context — a single Bounded Context routinely mixes patterns.** Khononov is explicit that a Bounded Context is not required to implement every one of its Aggregates with the same pattern: a Classification Bounded Context's core classification-decision logic might warrant full Domain Model treatment while a same-context "connector configuration" use case is simple enough for Active Record, without either choice implying anything about the other. This directly matters for `CLAUDE.md`'s Non-Negotiable Methodology table, which states DDD applies to "every service, every bounded context" with no per-Aggregate granularity — Khononov's guidance would treat that as over-application by default for any Bounded Context containing a mix of Core and Supporting/Generic-shaped use cases, not a correction to make to `CLAUDE.md` (out of this research file's scope to alter) but a real tension worth surfacing for whoever eventually revisits that rule, in the same spirit `subdomain-distillation`'s own Khononov section already flags a milder version of this tension around investment level rather than pattern *choice* itself.
- **Aggregate boundaries and Bounded Context boundaries are not the same kind of boundary drawn at different granularity — they are shaped by fundamentally different forces, and Khononov is precise about naming what those forces are.** A Bounded Context boundary is a *linguistic and model-consistency* boundary: it exists because a Ubiquitous Language term can mean something different on either side, and the boundary is where translation (an Anti-Corruption Layer, a Context Map relationship) becomes necessary. An Aggregate boundary is a *transactional-consistency* boundary: it exists because a specific invariant must be enforced atomically, full stop — it has nothing to do with language or team ownership per se. `skills/bounded-context-mapping/SKILL.md`'s "Context per Aggregate" anti-pattern (already confirmed correct and unmodified by `research/domain-driven-design/domain-driven-design-evans.md`'s review) states the *consequence* of this distinction well, but this book supplies the *reason* the consequence holds: conflating the two boundary types isn't merely "too fine-grained," it's a category error — asking a transactional-consistency mechanism to also do the job of a language/model-consistency mechanism, two jobs that don't correlate 1:1 even when they're usually nested correctly (many Aggregates inside one Bounded Context).
- **Khononov extends this into an explicit hierarchy of boundary types, each driven by a distinct force, that a system accumulates as it's decomposed: Subdomain (business-strategy boundary — is this differentiating?), Bounded Context (linguistic/model boundary — where does the Ubiquitous Language change meaning?), Aggregate (transactional-consistency boundary — what must be atomic?), and Module/microservice (physical/deployment boundary — what is independently deployable, independently scalable, independently owned by a team?).** These commonly nest cleanly (one microservice implements one Bounded Context containing several Aggregates spanning one or more Subdomains) but Khononov's point is that nothing *forces* clean alignment, and misalignment at any layer is a specific, nameable failure mode: a microservice boundary drawn without regard to the Bounded Context boundary underneath it reproduces a distributed monolith (chatty cross-service calls to complete what's really one model's work); a Bounded Context spanning multiple genuinely different Subdomains blurs a Core Domain's Ubiquitous Language with a Generic one's. This repo's `skills/container-diagram/SKILL.md` (not re-read in this session but referenced in `research/software-architecture/fundamentals-of-software-architecture-richards-ford.md`'s own Applies To table) is the artifact where the microservice/module layer of this hierarchy gets decided — worth checking, in a future research pass, whether it explicitly checks its service-boundary decisions against the Bounded Context boundary from `bounded-context-mapping` rather than deriving boundaries independently.
- **Event Sourcing is a persistence strategy for an Aggregate, not a different modeling pattern — the Aggregate's public behaviour (its command methods, its invariant enforcement) is unchanged; what changes is that its state is derived by replaying its own history of events rather than read from a current-state row.** Concretely: an event-sourced Aggregate holds no directly-settable fields corresponding to its current attributes; instead, it exposes an `Apply` (or `When`) method per event type it can experience, and its current state is whatever results from folding every event in its stream, in order, through those `Apply` methods — used identically whether the Aggregate is being rebuilt from storage (replay the whole stream) or being updated in memory after a new command succeeds (apply the one new event immediately, so the in-memory instance stays consistent without a second read). `skills/go-domain-model/SKILL.md`'s `NewDataAsset`/`Reconstitute` split and its current-state field layout (`sensitivity SensitivityLevel`, `version int64` as a row-level optimistic-concurrency counter) is a faithful, well-executed *current-state* Aggregate — the default this repo's `skills/cqrs-pattern/SKILL.md` already explicitly commits to ("Aggregates store state in a standard PostgreSQL table... Event Sourcing is an option... not the default") — but no skill in this repo shows what the *alternative* shape actually looks like in Go, so a modeler who reaches for Event Sourcing for a specific Bounded Context (as `cqrs-pattern` already permits) has no worked pattern to follow.
- **Khononov's criteria for choosing Event Sourcing over current-state persistence, stated as a genuine cost/benefit decision rather than a Core-Domain default:** adopt it when (a) the business genuinely needs to reconstruct historical state at an arbitrary past point in time (not merely "we log changes" — literally rehydrating what the Aggregate looked like at time T), (b) new Read Models/projections need to be constructed *retroactively* from data that already existed before that projection was designed (impossible with current-state persistence, since the intermediate states are already gone), or (c) the audit trail requirement is strict enough that the event stream itself, not a separate audit log bolted on afterward, should be the authoritative record. Its costs, named explicitly rather than left implicit: replay performance at load time (mitigated by snapshotting, see below), schema evolution of *events themselves* being harder than schema evolution of a current-state table (see the upcasting concept below), and materially higher implementation and mental-model complexity that is not worth paying for subdomains that don't need any of the three benefits. This repo's own product context — a compliance/audit-heavy first product where `Audit Trail`, `Auditability`, and `Non-Repudiation` are all canonical glossary terms — makes classification history (`DataAssetClassified` events) a plausible genuine candidate for this trade-off, but nothing currently in `aggregate-design`, `go-domain-model`, or `domain-event-catalog` states this decision criterion in a form a modeler could actually apply to decide.
- **Snapshotting is Event Sourcing's standard mitigation for unbounded replay cost — periodically persist the Aggregate's derived current state alongside (not instead of) the event stream, so a rebuild only needs to replay events *since* the last snapshot, not the entire history.** A snapshot is never authoritative on its own — the event stream remains the single source of truth, and the snapshot is purely a load-time performance optimisation that must be safely discardable and regeneratable from the stream at any time. This is a direct structural parallel to `skills/read-model-design/SKILL.md`'s already-correct principle that a Read Model is never authoritative and can always be rebuilt from events — the same "this is a derived, disposable acceleration structure, not a second source of truth" discipline applied one layer earlier, to the write side's own state reconstruction rather than the read side's query optimisation.
- **Internal (persistence) events used to reconstruct an event-sourced Aggregate's state are not automatically the same thing as the external Domain Events published for cross-context integration, and conflating the two is a specific, named coupling hazard.** An Aggregate's internal event stream may need to be more granular than what's business-meaningful externally — e.g., a `DataAssetClassificationAttempted` / `DataAssetClassificationSucceeded` pair used purely to reconstruct in-progress state correctly, where only `DataAssetClassified` (the coarser, business-meaningful fact) is worth publishing to Compliance Intelligence and Graph consumers. Publishing every internal state-transition event as if it were an external contract couples other Bounded Contexts to this Aggregate's storage implementation detail (that it happens to use event sourcing, and exactly how it happens to model its own internal state machine) — precisely the kind of hidden coupling `research/domain-driven-design/domain-driven-design-evans.md`'s "Model-Driven Design" concept and this repo's own `Anti-Corruption Layer` glossary term exist to prevent, just occurring at the write side rather than at a context integration seam. Khononov's resolution: either maintain a deliberate translation step (an internal event or a dedicated small "event enricher" step that derives the coarser public Domain Event from one or more internal events before it reaches the outbox) or, where the internal and external shapes genuinely coincide, treat that coincidence as a design choice to be revisited if the two ever need to diverge — not an assumption to bake in permanently. Neither `skills/domain-event-catalog/SKILL.md` nor `skills/aggregate-design/SKILL.md` currently distinguishes "events this Aggregate uses to persist/reconstruct itself" from "events this Bounded Context publishes for external consumption" at all; both bodies of guidance are currently written as if there is only one category of event, which is a safe assumption only as long as no Aggregate in this repo adopts Event Sourcing — the moment one does, the distinction becomes load-bearing.
- **Event Sourcing's schema-evolution discipline is strictly harder than, and different from, `domain-event-catalog`'s existing additive/breaking versioning strategy for integration events — because a stored, already-persisted internal event can never be deleted, rewritten, or "run in parallel with v2" the way an in-flight integration event can.** `skills/domain-event-catalog/SKILL.md`'s versioning table (additive → minor bump, tolerant readers; breaking → major bump, run both versions in parallel until consumers migrate) works because integration events are transient messages on a topic with a retention window — old and new versions can coexist on the wire during a migration, and once every consumer has migrated, the old version can genuinely stop being emitted. An event-sourced Aggregate's own persisted history has no such luxury: every event ever appended to that Aggregate's stream must remain loadable and interpretable *forever*, because it is not a transient integration message, it is the permanent system of record for that Aggregate's state. Khononov's named technique for this is **event upcasting**: a versioned transformation function that converts an old-shape event, read from storage, into the shape the current code expects, applied transparently at load time before the Aggregate's `Apply` methods ever see it — the stored bytes are never rewritten; only the in-memory representation during replay is upgraded. This is a genuinely distinct concern from anything currently in `domain-event-catalog`, and worth flagging clearly as a *different* versioning discipline that only applies to the internal/event-sourced case, not a correction to the existing integration-event versioning guidance (which remains correct for its own scope).
- **An event-sourced Aggregate's optimistic-concurrency check is the same underlying mechanism as `skills/go-repository-pattern/SKILL.md`'s compare-and-swap `UPDATE ... WHERE version = $N`, just expressed against an append-only store instead of a mutable row: the append of new events is conditioned on the stream still being at the expected version/sequence number at append time, and a mismatch means a concurrent writer already appended — the same "0 rows affected ⇒ concurrent modification" signal, restated as "append rejected ⇒ concurrent modification."** This is worth stating explicitly precisely because it's easy to assume Event Sourcing requires an unfamiliar concurrency model — Khononov's point (and this repo's existing CAS pattern already gets this right for the current-state case) is that it's the identical Vernon-sourced default (optimistic concurrency, version-checked write), applied to a different storage shape, not a different concurrency philosophy.
- **CQRS exists on a spectrum, and this repo's existing skills already commit to the *heaviest* point on that spectrum without naming the lighter options as legitimate, deliberate starting points.** Khononov distinguishes at least two meaningfully different levels commonly both called "CQRS": a lighter form where the write model and read model are simply separate *types/DTOs* with separate responsibilities, but may still be built from the same underlying tables (even the same database) with no event-driven projection machinery at all — versus the heavier form `skills/cqrs-pattern/SKILL.md` and `skills/read-model-design/SKILL.md` already implement, where Read Models are asynchronously projected from Domain Events into wholly separate storage. Both are legitimate CQRS; the lighter form is often the right starting point for a Bounded Context whose read/write shapes have diverged enough to earn separate types but whose query load doesn't yet justify a projection pipeline. `skills/read-model-design/SKILL.md`'s anti-pattern table entry "Querying the Write Model for reads... Reintroduces the coupling CQRS exists to remove" states this as an unconditional failure — Khononov's own framing would treat a query-time join against Aggregate tables, done through a dedicated read-side type rather than exposing the Aggregate itself, as the legitimate lightweight CQRS starting point, escalating to full event-driven projection only once query load, multiple-consumer needs, or an audit-trail requirement (this repo's own "When to Apply CQRS" table) actually arrive. This is a genuine, actionable nuance the current anti-pattern entry is currently too absolute to permit.
- **Read Model design should be driven entirely by the querying client's needs and should have no structural dependency on Aggregate boundaries whatsoever — the two are decoupled by design, not merely by different storage.** Because a Read Model (in the heavier, event-projected form this repo already defaults to) is built by folding Domain Events into whatever shape best serves a query, it never needs to respect the Aggregate boundary the events originated from: one Read Model can freely combine fields sourced from several different Aggregates' events (a `DataAssetListItem` denormalising in a `storageSourceName` string, as `skills/read-model-design/SKILL.md`'s own worked example already does), and one Aggregate's events can feed several structurally unrelated Read Models. This is already a strong point of alignment — `read-model-design`'s existing "Read Model / API Contract Connection" section ("shape the Read Model at projection time... if the API response requires significant transformation, the Read Model is wrongly shaped") is exactly Khononov's decoupling principle in practice — the gap is only that neither skill states the underlying reason (Read Models answer to query needs, full stop; Aggregate boundaries answer to transactional-consistency needs; there is no reason these two decompositions should ever need to match, and expecting them to is itself a modeling mistake worth naming).
- **Aggregate sizing is a coupling/cohesion trade-off, not a one-directional "always smaller" instinct — Khononov frames this using the same coupling vocabulary the book applies to Bounded Context and module design generally, not a bespoke Aggregate-only heuristic.** A larger Aggregate increases *cohesion* (more of the rules that must hold together are enforced by one mechanism, one lock, one transaction) at the direct cost of *coupling* between every operation that touches any part of that Aggregate (they all now contend for the same lock/version, exactly as `skills/aggregate-design/SKILL.md`'s existing "Sizing Under Contention" section already covers from the performance angle, and `research/domain-driven-design/implementing-ddd-vernon.md`'s Rule 2 covers from the performance/scalability/collaboration angles). Khononov's contribution is naming this explicitly as the same coupling-cohesion trade-off that governs every other boundary decision in the book (Bounded Context size, module size, service size) rather than a bespoke Aggregate-specific rule of thumb — useful because it lets a modeler reuse one mental model ("what does drawing the boundary here cost in coupling, and what does it buy in cohesion?") across Aggregate, Bounded Context, and service-boundary decisions alike, instead of learning three unrelated-feeling heuristics.
- **Khononov names premature or unjustified Event Sourcing adoption as a specific, common failure mode distinct from the God Aggregate or Anemic Aggregate anti-patterns this repo's `skills/aggregate-design/SKILL.md` already documents — teams reach for Event Sourcing because a subdomain is Core (and therefore "deserves the best patterns"), not because any of the three genuine justifying needs (temporal query, retroactive projection, audit-as-source-of-record) actually exist for that specific Aggregate.** This is the Event Sourcing-specific instance of the same over-application error `skills/subdomain-distillation/references/classification-techniques.md`'s Khononov section already names for tactical patterns generally ("Applying rich tactical patterns uniformly regardless of classification is itself the named mistake") — worth stating as its own anti-pattern in a future `aggregate-design` rebuild specifically, since Event Sourcing carries costs (upcasting discipline, snapshotting infrastructure, replay-based testing) that a Core-Domain-but-otherwise-ordinary Aggregate (say, `DataAsset` itself, absent a specific temporal-query requirement) would pay for no corresponding benefit.
- **Data ownership — the rule that only the Aggregate (or the service/Bounded Context) that owns a piece of data may write to its authoritative copy, and everyone else either references it by ID or consumes a Read Model / integration event derived from it — is the same discipline expressed at the Aggregate level (Rule 2, reference by ID) and at the Bounded Context level (the existing glossary's `Data Ownership` term, "Only the owning context may write to that entity's master record").** Khononov treats this as one principle applied recursively at every boundary layer in the hierarchy above, not two coincidentally similar rules — worth noting as confirmation that `skills/glossary-management/references/ubiquitous-language.md`'s existing `Data Ownership` definition already generalises correctly across both scales, even though no skill currently states the recursive pattern explicitly.

## Actionable Techniques

- Before applying the Domain Model/Aggregate pattern to any subdomain, run Khononov's prior gate explicitly: given this subdomain's classification from `subdomain-distillation` (Core/Supporting/Generic) and its actual logic complexity, would Transaction Script or Active Record serve the use case adequately? Document the choice per Aggregate/use case, not once for the whole Bounded Context — a Core-classified context can still contain individual use cases that don't warrant full Aggregate treatment.
- For any Bounded Context under design, explicitly enumerate which of the four Business Logic Design Patterns (Transaction Script, Active Record, Domain Model, Event-Sourced Domain Model) each of its Aggregates/use cases uses, and require a one-line justification per choice — treat a Bounded Context where every single use case defaults to full Domain Model without exception as a signal to re-check the classification, not evidence the context is unusually complex.
- Before drawing an Aggregate boundary, separately name which force is actually driving it: is this a transactional-consistency decision (an Aggregate boundary question) or a linguistic/model-consistency decision (a Bounded Context boundary question)? Never let a single diagram or a single worksheet answer both questions with the same boundary line without checking they actually coincide for the right reasons.
- Before adopting Event Sourcing for any specific Aggregate, require an explicit answer to Khononov's three-question justification test in writing: do we need to reconstruct state at an arbitrary past point in time; do we need to build new Read Models retroactively from history that already happened; or must the event stream itself (not a bolted-on audit log) be the authoritative compliance record? If none apply, use current-state persistence (this plugin's existing default) regardless of the subdomain's Core-Domain status.
- When an Aggregate does adopt Event Sourcing, design its internal/persistence events and its externally-published Domain Events as two explicitly distinct lists from the start, even if some internal events pass through unchanged — write down which internal events are published as-is, which are translated/coarsened before publication, and which never leave the Aggregate at all.
- For any event-sourced Aggregate, define its snapshotting cadence and rebuild procedure explicitly (how many events before a snapshot is taken; how a snapshot is invalidated/regenerated) as part of the Aggregate's design artifact, the same way `read-model-design` already requires a documented rebuild procedure for every Read Model.
- When a schema change is needed for an event-sourced Aggregate's own internal events, use an upcaster (a versioned load-time transformation) rather than `domain-event-catalog`'s additive/breaking-version-with-parallel-emission strategy — that strategy is correct for transient integration events and actively wrong for permanently-stored internal events, which can never be deleted or reissued under a new version.
- Before treating "Read Model built by projecting events" as the only acceptable CQRS shape, check the actual read/write divergence and query load against `cqrs-pattern`'s own "When to Apply CQRS" table — where the divergence is real but the load is modest, a lighter CQRS (separate read-side types, still sourced by a query-time join against write tables through a dedicated read-side type, not the Aggregate itself) is a legitimate, cheaper starting point, not an anti-pattern to avoid on principle.
- When sizing any Aggregate, state the trade-off in coupling/cohesion terms explicitly — name what enforcing this boundary here buys in cohesion (which rules become atomically guaranteed) and what it costs in coupling (which otherwise-independent operations now contend for the same lock/version) — reusing the same trade-off vocabulary for Bounded Context and service-boundary sizing decisions elsewhere in the same design.

## Key Terminology

Missing from `skills/glossary-management/references/ubiquitous-language.md` entirely — recommended additions to the "Domain and DDD" table, sourced from this book:

- **Business Logic Design Pattern** — Khononov's umbrella term for the four implementation strategies available for a subdomain's business logic (Transaction Script, Active Record, Domain Model, Event-Sourced Domain Model), selected per Aggregate or use case based on the subdomain's classification and actual logic complexity, not applied uniformly across a whole Bounded Context.
- **Transaction Script** — a business logic pattern where each use case is implemented as a single procedure operating directly against persistence, with no domain object model and no cross-object invariant enforcement. Appropriate for Generic subdomains and the simplest Supporting subdomains.
- **Active Record** — a business logic pattern where an object per persisted row/table carries both data and simple CRUD-shaped behaviour, with no separation between the object's shape and the persistence schema, and no enforced invariant spanning multiple objects. A middle rung between Transaction Script and full Domain Model.
- **Domain Model** *(business-logic-pattern sense, distinct from the general DDD "domain model" usage already implicit throughout this repo)* — the full DDD tactical pattern set: Aggregates, Entities, Value Objects, invariants enforced by the Aggregate Root, isolated from the persistence schema. Reserved for Core Domain and complex Supporting subdomains.
- **Event-Sourced Domain Model** — the Domain Model pattern with the Aggregate's state derived by replaying its own persisted event history rather than read from a current-state row. An additional, optional escalation above Domain Model, justified only by a specific temporal-query, retroactive-projection, or audit-as-record need.
- **Event Sourcing** — the persistence strategy underlying the Event-Sourced Domain Model pattern: an Aggregate's authoritative state is its append-only stream of past events, not a mutable current-state row; current state is always a derived, rebuildable projection of that stream.
- **Event Store** — the append-only storage mechanism for an event-sourced Aggregate's event stream, supporting append-if-expected-version (the event-sourced equivalent of optimistic-concurrency compare-and-swap) and full-history read for replay.
- **Snapshot** *(event-sourcing sense, distinct from any infrastructure/backup sense)* — a periodically-persisted copy of an event-sourced Aggregate's derived current state, used purely to bound replay cost at load time; never authoritative on its own and always regeneratable from the event stream.
- **Event Upcasting** — a versioned, load-time transformation applied to an event read from an event store, converting an old-shape stored event into the shape current code expects, without ever rewriting the stored event itself. The event-sourcing-specific schema-evolution discipline, distinct from `domain-event-catalog`'s additive/breaking-version-with-parallel-emission strategy for transient integration events.

## Caveats

- **Confidence level, overall:** I am confident in this book's headline structural contributions — the four-pattern Business Logic Design Pattern catalog, the explicit Aggregate-boundary-vs-Bounded-Context-boundary force distinction, the boundary hierarchy (Subdomain/Bounded Context/Aggregate/Module), and the CQRS-as-a-spectrum framing — these are among the most-cited, distinctive contributions of this specific book relative to Evans and Vernon. I am **less confident on exact chapter numbers and precise terminology the book itself uses for some sub-concepts** (e.g., whether "event enricher" or a materially different term is the book's own phrase for the internal-vs-integration-event translation step) — this document was written from working knowledge of the book's actual arguments and structure rather than a fresh line-by-line re-read, and specific wording should be treated as substantively faithful but not verified-by-quotation, consistent with the confidence caveats already present in both prior books' research files.
- **This document deliberately does not re-derive the Core/Supporting/Generic classification, the complexity-vs-strategic-importance axis, or the "match pattern richness to classification" principle at the subdomain-investment level** — `skills/subdomain-distillation/SKILL.md` and its `references/classification-techniques.md` already cite this book correctly and in reasonable depth for that content, per this research assignment's explicit instruction to avoid duplicating what's already absorbed there. This document's scope is the Aggregate-specific tactical content (the four-pattern catalog's *implementation* detail, Event Sourcing, the boundary-hierarchy precision, and CQRS/Read Model depth) that `subdomain-distillation` does not cover.
- **Event Sourcing content here is not a recommendation to change this repo's default architecture.** `skills/cqrs-pattern/SKILL.md` already correctly states current-state persistence with CQRS is this plugin's default, and Event Sourcing is an available, deliberate, per-Bounded-Context option — this document's content is meant to make that already-permitted option actually implementable (a worked Go pattern currently doesn't exist anywhere in this repo) and to supply the justification criteria for choosing it, not to argue the default should change.
- **Khononov's own book uses C#/.NET-flavoured code examples throughout** (consistent with his professional background), not Go — every Go-specific detail in this document (the `Apply` method shape, the conditional-append repository pattern, the interplay with `pgx`) is this document's own extrapolation of the book's language-agnostic principles onto this repo's existing Go conventions, in the same spirit as the prior two books' research files' own Go extrapolations, not a literal port of the book's sample code.
