# Domain-Driven Design: Tackling Complexity in the Heart of Software
**Authors:** Eric Evans
**Domain:** Domain-Driven Design — Tactical and Strategic Design (the foundational text)

## Core Concepts

- **The Aggregate exists to solve a specific problem: invariant enforcement across object clusters without distributed locking (Part IV, Ch. 6).** Evans introduces the Aggregate not as a data-modeling convenience but as the answer to a hard question: when a business rule spans several objects (e.g., "the sum of line items must equal the order total"), what mechanism guarantees that rule holds after every transaction? His answer: draw a boundary around exactly the objects that participate in the rule, appoint one of them the root, and forbid any transaction from touching more than one such cluster at a time. `skills/aggregate-design/SKILL.md`'s Rule 1 ("Enforce Invariants Within the Aggregate Boundary") and Rule 3 ("One Aggregate per Transaction") restate this almost verbatim but never state the *reason* the two rules are actually one rule — a transaction boundary is only correct because it was drawn to contain an invariant in the first place. The skill's worked example ("A DataAsset may only be marked Restricted if its storage source is confirmed active") is exactly the kind of cross-object rule Evans uses to motivate the whole pattern, but the skill presents the rule as a design heuristic rather than crediting it as the pattern's entire raison d'être.
- **The Aggregate Root is the only member of the cluster held by reference from outside (Ch. 6, "Aggregates").** Evans is precise about what this buys: because external references (and hence external mutation paths) can only reach the root, the root can serve as a genuine enforcement checkpoint. If any internal Entity were externally referenceable, the root's invariant checks could be bypassed by mutating that Entity directly through the other reference — the whole pattern would be theater. `skills/aggregate-design/SKILL.md` Rule 4 ("The Root Controls All Modifications") and its Anti-Patterns table entry "Leaking internal collections" capture the *consequence* of this rule well, but the skill never states the underlying reason in Evans' terms: a reference is a capability, and the boundary is only as strong as the set of objects that hand out references to their internals. This is worth stating explicitly because it clarifies *why* even a read-only getter that returns a live internal slice (rather than a copy) is a boundary violation — it hands out a capability, not just data.
- **Global identifiers only for Aggregate Roots; everything inside is locally identified relative to the root (Ch. 6).** Evans distinguishes two kinds of identity: Entities that need to be found and referenced from anywhere in the system (roots) get a global, unique identifier; Entities that only make sense as part of their containing Aggregate (e.g., an order's line items) need only be unique *within* that Aggregate — they are found by traversing the root, never by an independent global lookup. `skills/aggregate-design/SKILL.md`'s worksheet ("Entities within the Aggregate — why it needs identity") gestures at this but doesn't name the local-vs-global identity distinction, which is the actual test for "does this collaborator need its own Entity identity, or can it be demoted to a Value Object or a locally-scoped child Entity." `skills/go-domain-model/SKILL.md`'s example never shows a child Entity (only Value Objects and the root) — a genuine gap, since Evans' own canonical example (Order → OrderLineItem) is precisely a root with locally-identified child Entities, a shape this repo's worked examples don't currently demonstrate anywhere.
- **Entity vs. Value Object is decided by whether continuity of identity matters to the application, not by any property of the data itself (Ch. 5, "A Primer on the Design of Object Models").** Evans' test is behavioral: ask whether two objects with identical attributes should be treated as interchangeable (Value Object) or as distinct individuals that must be tracked through change over time (Entity) — the same real-world "thing" (e.g., a person) can be modeled as a Value Object in one Bounded Context (an "Address" attached to a shipment, where only the current attribute values matter) and an Entity in another (a "Customer" whose relationship history must be tracked). `skills/aggregate-design/SKILL.md`'s Entity/Value Object table (Identity / Mutability / Equality / Examples) correctly states the mechanics but frames the distinction as an intrinsic property of a concept ("`DataAsset`, `StorageSource`, `User`" are listed as Entities, full stop) rather than as a per-Bounded-Context modeling decision Evans insists it always is. This matters concretely for this repo: `User` inside an Identity & Access context is unambiguously an Entity, but a `User` reference embedded in an audit record (only the name and role *at the time* matter, never "this same user changing over time") is a legitimate Value Object snapshot — the skill's table as written would mislead a modeler into treating "User" as always-Entity everywhere.
- **Evans states a strong, explicit preference for Value Objects and gives the reason: Value Objects are simpler to create, test, understand, and share, precisely because they have no lifecycle to manage (Ch. 5).** "Prefer Value Object to Entity when possible" is one of the book's clearest actionable defaults — not because Value Objects are trendy, but because every Entity carries the ongoing cost of identity management (equality by ID, careful handling of concurrent modification, lifecycle tracking) that a Value Object never incurs. `skills/aggregate-design/SKILL.md`'s line "Prefer Value Objects. A concept that can be modelled as a Value Object should be" already states this conclusion correctly and is the single strongest point of alignment between the existing skill and this book — the gap is only that the skill doesn't explain the *cost asymmetry* (identity-management overhead) that makes the preference correct, so a modeler has a rule to follow but not the reasoning to apply it to a novel, ambiguous case.
- **Value Objects should be immutable, and this immutability is what makes sharing them across Aggregates safe (Ch. 5).** Evans notes a specific benefit the aggregate-design skill's table lists as a bare fact ("Immutable — replacement, not mutation") without connecting it to *why* it matters: because a Value Object never changes after construction, multiple Aggregates (or multiple threads) can safely hold copies of "the same" Value Object without any risk of one holder's mutation corrupting another's view — this is precisely why a `SensitivityLevel` or a `DateRange` can be freely passed around and even shared by reference in a garbage-collected runtime without defensive copying. `skills/go-domain-model/SKILL.md`'s "Value vs Pointer" table correctly derives the Go-specific consequence (never pointer-to-Value-Object) but frames it as an idiom rather than tracing it back to this book's immutability-enables-sharing argument.
- **A Value Object may reference other Value Objects, and may even reference Entities, but Evans warns this must not be a route to bypass Aggregate boundaries (Ch. 5–6).** A Value Object that holds a reference to an Entity in another Aggregate reintroduces exactly the coupling the Aggregate boundary exists to prevent — Evans' guidance is that Value Objects composed of other Value Objects are safe and common (a `DateRange` made of two `Date` Value Objects), but a Value Object that embeds a live Entity reference is really a disguised Aggregate-boundary violation wearing a Value Object's syntax. `skills/aggregate-design/SKILL.md`'s Rule 2 ("Reference Other Aggregates by ID Only") is stated only for the Aggregate Root's own fields — it does not extend the rule to Value Objects nested inside the Aggregate, which is a real gap: a `StorageSourceSummary` Value Object embedded in `DataAsset` that happened to hold a `*StorageSource` pointer would violate the same principle Rule 2 protects, just one level deeper in the object graph, and the current skill's Go example wouldn't catch it because it only checks the root's own struct fields.
- **The Repository pattern exists to give the illusion of an in-memory collection of Aggregates, and Evans is explicit that there is exactly one Repository per Aggregate type — never one per Entity (Ch. 6, "Repositories" and Ch. 12 in some editions).** The reason is structural, not stylistic: since only the Aggregate Root is globally referenceable, only the root has an identity a Repository could meaningfully query by; a "Repository" for an internal Entity would have to expose a way to fetch that Entity independent of its root, which directly punches a hole through the encapsulation the Aggregate pattern exists to provide. `skills/aggregate-design/SKILL.md`'s Anti-Patterns table already names "Repository per Entity" as a violation ("Callers load and mutate internals without the root, bypassing invariants") — this is a strong, correct restatement of Evans' rule; the gap is that it doesn't explain the *reason* (repositories query by identity, and only roots have queryable identity), which would help a modeler recognize the anti-pattern in less obvious forms, e.g., a `FindLineItemByID(id)` query method added "for convenience" to an `OrderRepository`.
- **A Repository's contract belongs to the domain layer even though its implementation belongs to infrastructure — Evans calls this "the illusion of an in-memory collection," and the domain must never see the storage mechanism (Ch. 6).** This is precisely the Ports-and-Adapters shape `skills/go-repository-pattern/SKILL.md` already implements ("The interface lives with the consumer (`internal/domain/ports.go`); the implementation lives in `internal/infrastructure/postgres/`") — a genuinely strong, faithful implementation of this exact Evans concept, executed in idiomatic Go with pgx, that the existing skill never credits to the source pattern. The skill's rule "No business logic. A repository loads, saves, and translates" is Evans' own scoping of the pattern's responsibility, stated correctly.
- **The Factory pattern is for construction complex enough to itself become a design responsibility — Evans' criterion is whether the *rules and invariants of construction* are substantial enough to deserve encapsulation separately from ongoing lifecycle operations, not merely "the constructor has more than N parameters" (Ch. 6, "Factories").** Evans' worked distinction: a simple Aggregate whose constructor validates a couple of required fields needs no Factory — a constructor or "creation method" on the object itself suffices. A Factory becomes warranted when (a) construction must enforce invariants that span multiple objects being created together, (b) the constructing code needs to choose between several concrete implementations of an abstraction, or (c) reconstitution from storage requires bypassing constructor-time invariant checks that only apply to genuinely new instances. `skills/aggregate-design/SKILL.md` never mentions Factories at all — it goes straight from "invariants" to Go struct sketches, silently assuming a plain constructor always suffices. `skills/go-domain-model/SKILL.md`'s `NewDataAsset` (construction, emits `DataAssetRegistered`) vs. `Reconstitute` (no events, no re-validation) is, in Evans' own vocabulary, a **factory method pair** — precisely the construction/reconstitution split his Factory pattern chapter recommends for exactly the reason the skill states ("conflating it with creation would double-fire events on every load") — but the skill never names this as a Factory responsibility, so nothing in this repo currently gives a modeler a rule for *when* a single-Aggregate case needs to graduate from "a constructor" to "a dedicated Factory type," e.g., when creating a `DataAsset` requires simultaneously creating and validating a companion `AuditRecord` as one atomic operation.
- **Factories should be as simple as possible and belong either on the object being created (a static factory method) or on a dedicated Factory object only when construction logic doesn't naturally belong to any single created object (Ch. 6).** Evans warns against over-engineering: most factory needs are met by a factory method on the Aggregate Root's own type (`NewOrder(...)`) or on the Repository (`repository.NextIdentity()`-style methods that also help with identity assignment). A dedicated, separate Factory *class* is reserved for cases where the constructed objects don't have an obvious single home for the logic (e.g., a Factory that must choose which of several concrete Aggregate types to build based on runtime configuration). This repo's `skills/go-domain-model/SKILL.md` already uses the lightweight form (factory *methods* on the domain package, not separate Factory types) — which is the correct default per Evans' own guidance — but because the skill doesn't cite the Factory pattern by name, there's no documented escalation path for the rarer case where a dedicated Factory type actually becomes warranted (e.g., a `DataAssetFactory` that must decide, based on file type and source, which of several concrete Aggregate Root shapes to construct).
- **The Ubiquitous Language must be enforced inside the Aggregate's method names and type names specifically, not only at the Bounded Context level — Evans calls this "Intention-Revealing Interfaces" (Ch. 10, "Supple Design").** A method or class name should describe its effect and purpose in the domain expert's own words, without forcing the reader to look at the implementation to understand what it does — `Classify(level, classifiedBy)` (from this repo's own `skills/aggregate-design/SKILL.md` example) is a strong instance of this discipline already in practice; contrast it with a hypothetical `SetSensitivityAndAudit(level, id)` which would leak an implementation detail (that classification also touches an audit trail) into the interface name. Neither `skills/aggregate-design/SKILL.md` nor `skills/go-domain-model/SKILL.md` names "Intention-Revealing Interfaces" as a discipline, even though both already practice it faithfully in their worked examples — this is a citation gap, not a substance gap, and naming the discipline would let future skill authors check new method names against a named standard rather than "it felt right."
- **Side-Effect-Free Functions and the separation of Commands from Queries — Evans' domain-specific framing of what later became CQS (Command-Query Separation) (Ch. 10).** Evans argues that functions returning a result should not change observable state (a query is safe to call any number of times with no consequence), while methods that change state ("Commands" in his terminology, predating and distinct from the later CQRS "Command" concept but closely related) should be clearly named as such and, ideally, not return a domain result beyond success/failure. This repo's `skills/go-domain-model/SKILL.md` Aggregate accessor methods (`ID()`, `TenantID()`, `Sensitivity()`, `Version()`) are pure, side-effect-free queries, and its mutating methods (`Classify`) return only `error` — a textbook-correct instance of this discipline already in the code — but the pattern is unnamed in the skill, so there's no standard to hold a *future* method to if someone writes a mutator that also happens to return a computed value (a smell Evans specifically flags).
- **Assertions make the pre- and post-conditions of an operation explicit, ideally in the Ubiquitous Language, as a design discipline distinct from ad hoc runtime validation (Ch. 10).** Evans frames an Assertion as a documented, ideally-testable claim about what must be true before and after an operation runs — this predates and is a close ancestor of *Secure by Design*'s "totality" argument (already captured in `research/security/secure-by-design.md`'s "Assertions, not scattered validation" concept) but Evans' version is broader: it's a general supple-design discipline for *all* invariants, not specifically a security hardening technique. `skills/aggregate-design/SKILL.md`'s Quality Criteria table ("Invariants documented — Every Aggregate has at least one named invariant") is functionally an assertion-discipline requirement already, but frames invariants only as a static list to document rather than as pre/post-conditions attached to specific *operations* (Evans' sharper unit — "before `Classify()` runs, X must hold; after it returns without error, Y must hold").
- **Closure of Operations: an operation on a type that returns the same type is easier to understand, test, and compose, because the caller never has to reason about a different, unfamiliar result type (Ch. 10).** Evans' example is arithmetic-like operators on Value Objects — e.g., a `DateRange` operation that returns another `DateRange` rather than a bare pair of dates. This repo doesn't currently have a strong worked example of this in the aggregate-design or go-domain-model skills (their Value Object examples, like `SensitivityLevel`, are simple enums with comparison methods, not composable operations) — a genuine candidate gap for a future Value-Object-focused reference: a `DateRange.Overlap(other DateRange) (DateRange, bool)` style method would be the idiomatic Go instance of Closure of Operations, and none of the read skills currently model this shape.
- **Specification pattern: an explicit object that encapsulates a business rule as a predicate over a candidate object, usable to (a) test whether a candidate satisfies the rule, (b) select a subset from a collection matching the rule, and (c) express a rule as a first-class, named, composable, and testable object rather than scattered `if` statements (Ch. 9, "Making Implicit Concepts Explicit").** Evans introduces the Specification pattern specifically to solve the problem of a business rule that recurs in several places (a query filter, a validation check, and a UI-facing eligibility check) drifting out of sync because each occurrence is implemented separately. None of the skills read here — `skills/aggregate-design/SKILL.md`, `skills/go-domain-model/SKILL.md`, `skills/go-repository-pattern/SKILL.md` — name or use the Specification pattern; this repo's closest analog is `skills/data-quality-rules/SKILL.md`-style rule definitions (not read in this research session) or the Domain Event `Invariants:` sections in `skills/domain-event-catalog/SKILL.md`, which document rules as prose rather than as reusable, named, testable objects. This is the single clearest candidate for genuinely new content this repo does not have anywhere yet — see Candidate New Skills/Agents below.
- **Making Implicit Concepts Explicit is Evans' broader diagnostic habit that Specification is one instance of (Ch. 9).** The general technique: when a rule, a process, or a role keeps surfacing as scattered conditional logic or a recurring but unnamed parameter combination, that's a signal an unnamed domain concept is trying to emerge — pull it out and give it a name from the Ubiquitous Language, turning implicit knowledge into an explicit, reviewable model element. This is the generative principle behind Evans' entire pattern catalog (Specification, Policy objects, closure of operations) and is itself worth stating as a standalone modeling habit distinct from any single pattern — it's the reasoning move a domain-modeler agent should be running continuously, not just the specific patterns it produces.
- **Model-Driven Design: the code *is* the model, not a translation of a separately-maintained model (Ch. 3–4).** Evans' foundational argument, predating and motivating everything downstream (Aggregates, Ubiquitous Language, Bounded Contexts): if the implementation and the conceptual model drift apart, the model becomes documentation nobody trusts, and the code becomes an unprincipled pile of logic nobody can reason about in domain terms. `CLAUDE.md`'s "Component Architecture" (Skills = Expertise, Agents = Reasoning) and the whole plugin's insistence that "every artifact must be reviewable by a PM without IDE tooling" is a structurally different but philosophically compatible descendant of this argument — Shafi reviews Markdown artifacts as the model, and `go-domain-model`/`aggregate-design` are supposed to keep the Go code a faithful, undistorted rendering of that same model. No skill currently states Model-Driven Design as a named principle the domain-modeler agent is accountable for maintaining across the design→implement handoff.
- **Continuous Integration of the model (a distinct, older sense of the term than the CI/CD sense) — Evans' insistence that the team refactor and merge conceptual model changes constantly so the model stays a single, coherent whole rather than silently forking into incompatible per-developer mental models (Ch. 4, "Isolating the Domain," and Ch. 14, "Maintaining Model Integrity").** This is a genuine terminology collision risk for this repo, since `Continuous Integration (CI)` is already a canonical, differently-defined term in `skills/glossary-management/references/ubiquitous-language.md`'s "Delivery and Platform" section ("developers integrate code changes into a shared branch frequently..."). Any future skill drawing on this Evans concept must name it distinctly (e.g., "Model Integration" or explicitly qualify it as "Evans' modeling-integrity sense") to avoid silently overloading the existing glossary term — this is exactly the kind of drift the glossary's own "Never substitute synonyms" rule exists to prevent, and the collision is currently latent (unnoticed) rather than resolved.
- **Refactoring Toward Deeper Insight / the "Breakthrough" — Evans' account of how a model evolves: long periods of incremental refinement punctuated by a sudden reconceptualization that dramatically simplifies the model once a previously implicit concept is finally named and extracted (Ch. 8, "Breakthrough").** This is a qualitative, non-mechanical process Evans is candid cannot be scheduled or forced — but he does give a precondition: it only happens to a team that is already practicing Model-Driven Design and Continuous (model) Integration closely enough to *notice* when the current model is straining. No skill in this repo currently addresses this — it is squarely a process/mindset concept rather than a checklist, and the closest existing artifact (`skills/subdomain-distillation/SKILL.md`'s note that "classification is a hypothesis, not a one-time verdict") is compatible with but doesn't state this specific idea.
- **Large-Scale Structure: when a system is too large for a single Bounded Context's model or a single Context Map to give teams a shared sense of the system's overall shape, Evans proposes lightweight structural conventions layered on top of the Context Map (Ch. 16).** The two techniques most relevant to Aggregate-boundary-drawing at scale: **Responsibility Layers** (grouping the domain into broad strata — e.g., Decision-Making Layer, Policy Layer, Operations Layer — so an Aggregate's boundary decision can be sanity-checked against "which layer does this concept's rate of change and reason for existing actually belong to") and the **Knowledge Level** (a distinct, higher-level set of objects that describe or configure the behavior of a base-level set of objects — e.g., a `ClassificationRule` Aggregate that configures how `DataAsset` Aggregates get classified, rather than encoding all classification logic directly into `DataAsset` itself). Neither concept appears in any read skill. This is genuinely relevant to Aggregate-boundary-drawing precisely because a large-scale structure question — "is this rule-configuration concept its own Aggregate (Knowledge Level) or does it belong folded into the Aggregate it governs?" — is a boundary-drawing decision `skills/aggregate-design/SKILL.md`'s heuristics don't currently address; the compliance-rule-engine subdomain this repo's own product context depends on (`skills/subdomain-distillation/SKILL.md`'s "Compliance Rule Engine — Core") is close to a textbook Knowledge Level candidate that nothing in the current skill set names as such.
- **The four-layer architecture (User Interface, Application, Domain, Infrastructure) and the rule that the Domain layer must be isolated and ignorant of the other three (Ch. 4, "Isolating the Domain").** Evans' Application layer is thin — it coordinates, it does not contain business rules (they belong in the Domain layer) — and is a distinct, named layer from both UI and Domain, responsible for transaction demarcation, security checks that aren't domain rules, and orchestrating calls to the domain and infrastructure. `skills/go-domain-model/SKILL.md`'s "Purity Rule" ("The domain package imports only stdlib, uuid, and time... no framework, no I/O") is a strict, faithful, and correctly-scoped implementation of Evans' Domain-layer isolation — again a case where the practice is already excellent but the pattern is never named or attributed. There is no equivalent named "Application Layer" skill read in this session (`skills/go-service-layer/SKILL.md` was not read but is the likely home for it) — worth verifying that skill states the same thin-orchestration-only boundary Evans specifies, since "business logic leaking into the application/service layer" is exactly the Anemic Aggregate anti-pattern already named in `skills/aggregate-design/SKILL.md`'s Anti-Patterns table, but from the opposite direction (rules present in the wrong layer, not absent from the right one).
- **Aggregates and Bounded Contexts operate at different granularities and solve different problems — an Aggregate is a transactional/consistency concept inside a single model; a Bounded Context is a linguistic/team/deployment concept that may contain many Aggregates (Ch. 6 vs. Ch. 14–15).** Evans is explicit that these must not be conflated: a Bounded Context is never sized to a single Aggregate, and an Aggregate never spans multiple Bounded Contexts. `skills/bounded-context-mapping/SKILL.md`'s Anti-Patterns table already states this correctly and explicitly ("Context per Aggregate — every Aggregate is declared its own 'context'... A Bounded Context holds a whole consistent model — usually several Aggregates sharing one Ubiquitous Language") — this is a strong, precise restatement of Evans' own distinction and needs no correction; it is cited here only to confirm the two granularities are already kept properly separate across `aggregate-design` and `bounded-context-mapping` and this book's content should reinforce, not blur, that separation when the aggregate-design rebuild proceeds.
- **Domain Events were not in Evans' original 2003 text; they are Evans' own later addition (a widely-cited 2013 blog post and talk, "What I Learned About DDD Since the Book," often folded into modern editions' discussion and DDD curricula alongside the original tactical patterns) — Evans himself has stated Domain Events would be a first-class tactical pattern (alongside Entity, Value Object, Aggregate, Repository, Factory, Service) if he were to rewrite the book today.** The core addition: an event is the record of something that happened, is immutable, and — crucially for Aggregate design — gives Aggregates a way to communicate consequences to the rest of the system *without* the requesting Aggregate needing a direct reference to, or synchronous call into, any other Aggregate, which is precisely the escape hatch Evans' original Rule 3 (one Aggregate per transaction) requires for any multi-Aggregate consequence. `skills/aggregate-design/SKILL.md`'s Rule 3 already names this exact resolution ("Emit an event from the first Aggregate; the second updates asynchronously") and `skills/domain-event-catalog/SKILL.md`'s entire skill is a mature, thorough treatment of the pattern — the gap is purely one of attribution/provenance: nothing in either skill notes that Domain Events are Evans' own later refinement to the original book's pattern set, which matters for a research file whose job is precisely to trace which concepts trace back to which part of the DDD literature.
- **Aggregates should be sized considering not just invariants but the true transactional needs of use cases — Evans explicitly cautions against both over-large Aggregates (unnecessary locking, poor performance) and over-small Aggregates (invariants that can't actually be enforced, because the objects that must agree aren't in the same transactional boundary) as symmetric failure modes, not just the "keep them small" advice in isolation (Ch. 6).** `skills/aggregate-design/SKILL.md`'s "Rule 5: Keep Aggregates Small" and its later "Sizing Under Contention" section (added, per the skill's `version: 1.1.0`, evidently after the original v1.0 skill) already treat this symmetrically and in real depth — including concurrency-specific guidance (optimistic-concurrency contention, unbounded-collection splitting) that goes well beyond what Evans' original text covers, since Evans wrote before optimistic-concurrency-via-version-field was the default persistence idiom it is today. This is a case where the existing skill's *practical* sizing guidance already exceeds the book's own depth — the book supplies the underlying *why* (invariant-scope-defines-boundary), the skill correctly supplies the *how* for a modern Go/pgx stack.

## Actionable Techniques

- Before finalizing any Aggregate boundary, state the *specific invariant* that justifies drawing the boundary where it is — not just "these objects seem related" — and check whether that invariant could be relaxed to eventual consistency instead of forcing a wider transactional boundary (Evans' own escape hatch, formalized in this repo's Rule 3 already but worth requiring as an explicit written justification per Aggregate, not just an implicit design instinct).
- For every collaborator inside a candidate Aggregate, apply the local-vs-global identity test explicitly: "would any code outside this Aggregate ever need to look this object up directly, by its own ID, independent of the root?" If no, it's either a Value Object or a locally-identified child Entity reachable only by traversing the root — never a second Repository-eligible root hiding inside the cluster.
- When a Value Object embeds a reference to another domain object, check whether that referenced object is itself an Aggregate Root in a different Aggregate. If so, the Value Object must hold that reference by ID only (extending Rule 2 one level deeper than the root's own fields) — the same discipline the Aggregate Root itself is already held to.
- Apply the Factory-vs-constructor decision test explicitly for every Aggregate: does construction (a) enforce invariants spanning multiple objects created together, (b) need to select among several concrete implementations, or (c) need a materially different code path for reconstitution-from-storage versus genuine creation? If any answer is yes, document the Factory responsibility (even if implemented as a plain constructor function in Go, as `NewDataAsset`/`Reconstitute` already are) rather than leaving it implicit.
- When a business rule recurs in more than one place (a query filter, a validation guard, and a UI-facing eligibility check, for instance), extract it into a named Specification object/function rather than reimplementing the predicate three times — and write one test against the Specification itself, reused by all three call sites.
- Run an Intention-Revealing-Interface check on every new Aggregate method name before merging: can a domain expert (not an engineer) predict what the method does and what it changes, from the name and signature alone, with zero knowledge of the implementation? If not, rename before implementing, not after.
- Apply Command-Query Separation explicitly to every Aggregate method: a method that queries state returns a value and changes nothing; a method that changes state returns only success/failure (an `error` in Go) and no other domain value. Flag any method that does both as a design smell requiring a split.
- For any new Value Object with more than one operation, check for Closure of Operations: do the type's own operations return the same type wherever mathematically/semantically sensible (e.g., `DateRange.Intersect(DateRange) (DateRange, bool)` rather than returning two bare `time.Time`s), rather than always degrading to primitive return types.
- When a domain concept is described using words like "rule," "policy," or "configuration" that govern *another* Aggregate's behavior, run the Knowledge Level test: should this be modeled as its own configuring Aggregate (a Knowledge Level object, e.g. `ClassificationRule`) referenced by ID from the Aggregate it governs, rather than folding the rule directly into the governed Aggregate's own invariants?
- Before naming anything "Continuous Integration" in a future skill or artifact that draws on this book, disambiguate explicitly from the existing CI/CD glossary term — use "Model Integration" or an explicitly qualified phrase, to avoid a silent term collision with `skills/glossary-management/references/ubiquitous-language.md`'s existing `Continuous Integration (CI)` entry.
- When citing this book's tactical patterns in a rewritten skill, attribute Domain Events explicitly as Evans' later addition (not part of the original 1-9-chapter tactical pattern set) — this preserves an accurate provenance trail distinguishing "the original book" from "what Evans and the community added afterward," which matters for a research file whose purpose is precise sourcing.

## Key Terminology

Missing from `skills/glossary-management/references/ubiquitous-language.md` entirely — recommended additions to the "Domain and DDD" table, sourced from this book:

- **Entity** — A domain object defined by a thread of continuity and identity, tracked through change over time, rather than by its attribute values. Two Entities are equal only if their identities match, regardless of attribute differences. Whether a given concept is modeled as an Entity or a Value Object is a per-Bounded-Context decision, not an intrinsic property of the concept. *(Note: this repo's prior research flagged a DDD-vs-Clean-Architecture collision risk for this term — verified during this research session that the term is currently simply absent from the glossary rather than defined-and-colliding; the collision risk is real but latent, since Clean Architecture's "Entities" layer (enterprise-wide business rules) is a different scope than DDD's per-Aggregate Entity, and a future glossary entry should disambiguate the two senses explicitly rather than pick one silently.)*
- **Value Object** — A domain object defined entirely by its attribute values, with no identity of its own. Two Value Objects with identical attributes are interchangeable and considered equal. Value Objects are immutable — a change is modeled as replacement, not mutation. Preferred over Entity wherever an object's identity does not need to be tracked, due to the lower lifecycle-management overhead.
- **Repository** — An object that gives client code the illusion of an in-memory collection of Aggregates, hiding the underlying persistence mechanism. Exactly one Repository exists per Aggregate type, keyed by the Aggregate Root's identity; there is never a Repository for an internal Entity or Value Object.
- **Factory** — An object or method responsible for encapsulating the logic of creating complex Aggregates or Value Objects, especially when construction must enforce invariants across multiple objects being created together, or must choose among several concrete implementations. Distinguished from reconstitution (rebuilding an existing instance from storage), which uses a separate code path that does not re-run creation-time invariant checks or emit creation events.
- **Specification** — An explicit, named, testable predicate object encapsulating a business rule, usable to test a candidate object against the rule, to select matching objects from a collection, and to express the rule as a first-class, reusable model element rather than scattered conditional logic.
- **Intention-Revealing Interface** — A method or class name and signature that communicates its purpose and effect in the Ubiquitous Language, without requiring the reader to inspect the implementation to understand what it does.
- **Side-Effect-Free Function** — A function or method that returns a result without changing any observable state; safe to call any number of times. Contrasted with a Command, which changes state and, ideally, returns only success/failure.
- **Closure of Operations** — An operation defined on a type that returns a value of the same type, making the operation easier to understand, test, and compose without introducing an unfamiliar result type.
- **Assertion** *(Evans' general design sense, distinct from the testing-assertion sense already in the glossary's Testing cluster and distinct from Secure by Design's security-hardening sense)* — an explicit, ideally-documented pre- or post-condition attached to a specific operation, stating what must be true before it runs and what it guarantees is true after it completes without error.
- **Model-Driven Design** — the practice of keeping the implementation a direct, undistorted expression of the conceptual domain model, rather than treating the model as separate documentation that the code merely approximates.
- **Continuous Integration (model sense)** *(Evans' pre-CI/CD usage — requires explicit disambiguation from the glossary's existing DevOps-sense `Continuous Integration (CI)` entry if ever added)* — the practice of merging and reconciling a team's evolving mental models of the domain frequently, so the model stays one coherent whole rather than silently forking.
- **Large-Scale Structure** — a lightweight organizing convention (e.g., Responsibility Layers, Knowledge Level) applied across an entire system, above the level of a single Bounded Context's Context Map, to give a large team a shared sense of the system's overall shape.
- **Knowledge Level** — a distinct set of objects that describe, configure, or govern the behavior of a separate "base level" set of objects, used when a rule or policy is itself complex and variable enough to deserve its own explicit model rather than being folded into the object it governs.
- **Responsibility Layer** — a broad stratification of a domain (e.g., Decision-Making, Policy, Operations) used at Large-Scale-Structure granularity to check whether a concept's boundary placement matches its actual role and rate of change.

## Caveats

- **Confidence level, overall:** I am highly confident in this book's headline tactical patterns (Aggregate, Entity, Value Object, Repository, Factory, the five-rules framing of Aggregate design, Ubiquitous Language, Bounded Context) — these are the most widely cited, foundational content in the entire DDD literature and this repo's existing skills already demonstrate close, careful familiarity with them. I am **somewhat less confident on precise chapter numbers** (editions and reprintings of this book vary slightly in structure) and on the exact wording Evans uses for some of the "Supple Design" chapter's less commonly cited sub-patterns (Intention-Revealing Interfaces, Side-Effect-Free Functions, Closure of Operations, Assertions) — the substance of these concepts is well-established and correct, but chapter/section numbers given above are best-approximation signposts, not verified page citations.
- **This book predates optimistic-concurrency-via-version-field, event-driven microservices, and message brokers as default persistence/integration idioms.** Evans wrote primarily against a backdrop of session-based ORMs (Hibernate-era Java) and largely synchronous, single-database systems; `skills/aggregate-design/SKILL.md`'s "Sizing Under Contention" section (optimistic concurrency, unbounded-collection splitting, hotspot analysis) is this repo's own necessary and well-executed extrapolation beyond what the book itself covers, not a gap the book fills — treat that section as already ahead of the book's own depth, not behind it.
- **Domain Events, as covered here, are substantially reconstructed from Evans' widely known post-2003 supplementary material (talks, the "What I Learned About DDD" writing) rather than the original book's nine core patterns.** Where this book's original text is silent or thin (it discusses "Domain Events" only in passing, if at all, depending on edition), this repo's own `skills/domain-event-catalog/SKILL.md` is already a mature, independently-developed treatment that stands on its own merits — this document's contribution there is provenance and connective tissue to the Aggregate-boundary rules, not new substantive content.
- **The Repository pattern as Evans describes it assumes an ORM-and-session-based persistence idiom (`repository.add(entity)`, unit-of-work, lazy loading) that doesn't map cleanly onto this repo's raw-SQL-via-pgx approach.** `skills/go-repository-pattern/SKILL.md`'s explicit `FindByID`/`Save` methods, manual row-scanning, and explicit transaction/outbox handling are a deliberate, idiomatic Go adaptation of the *pattern's contract* (illusion of a collection, one-per-Aggregate, domain-owns-the-port) without adopting the book's implied ORM machinery — this is the correct adaptation, not a deviation to flag as wrong, but any future skill content citing this book's Repository chapter directly should be careful not to imply this repo should adopt session/unit-of-work machinery it deliberately doesn't have.
- **The book is written for a single, cohesive team modeling one system together over a long period ("Hands-On Modelers," Ch. 2) — its social/process chapters (getting developers and domain experts to model together, organizational patterns for keeping a team's model coherent) don't translate directly to this repo's solo-operator, single-agent-driven structure.** Content on team dynamics and the "Hands-On Modelers" practice should be read for its underlying principle (whoever writes the code must be close to the model, not handed a model to blindly implement) rather than adapted literally — this repo's Command → Agent → Skill hierarchy is already a different but compatible mechanism for keeping "the modeler" and "the implementer" from drifting apart, since in practice they're the same agent reading the same skill.
- **This document deliberately does not re-derive Bounded Context / Context Map depth already owned by `skills/bounded-context-mapping/SKILL.md` and `skills/context-map-patterns/SKILL.md`**, per this research assignment's explicit scope instruction — strategic-design content here is limited to what's specifically relevant to Aggregate-boundary-drawing (Large-Scale Structure, Knowledge Level) rather than a full re-coverage of the nine Context Map relationship patterns, which both of those skills already treat thoroughly and accurately.
