Formalize `sdlc-config-management` and `artifact-manifest`'s already-documented Output Format shapes exactly, rather than inventing a new one. `sdlc-manifest.schema.json` validates only the per-product manifest instance — the master artifact-type catalog stays a prose table (extended over time, so `type` is a free-form pattern, not a closed enum). Both schemas' own worked examples validated cleanly against the real `jsonschema` library, and 15 deliberately invalid instances across both were correctly rejected (missing required fields, bad enum values, malformed dates/semver/slugs, unknown fields, empty traceability refs).

Replaces an originally proposed 14-role "DDD orchestrator" with its own duplicated reference tree and embedded test suite — see `sdlc-context.json` decision D020 and GitHub issue #48 for the full analysis.

- 2026-07-22 — `skills/miro-board-notation/SKILL.md` (new, MVP scope, Chunk 26, PR #76) — cross-cutting, owner `factory-governance`: the 5 real Miro element types (sticky note, shape, frame, connector, text) and a standard two-table Board Spec Format any skill can adopt. Adopted as a deliberate decision anticipating multiple future products and teams (the organizational precondition every source book in `research/micro-frontends/` gates adoption on) — see decision D023 for the full reasoning, including a tension Claude flagged and Shafi resolved before proceeding. CSS Modules chosen as the isolation mechanism over Shadow DOM, since fragments are same-framework React via Module Federation, not Web Components.
