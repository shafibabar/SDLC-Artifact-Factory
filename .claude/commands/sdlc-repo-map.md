# Command: /sdlc-repo-map

## Purpose
Generate the multi-repo map from the bounded context map. Creates the repository inventory, naming conventions, and a bootstrap script that initialises all repositories in GitHub. This is the concrete output of the DDD decomposition into deployable units.

Usage:
```
/sdlc-repo-map [--create-repos]
```

Options:
- `--create-repos`: Actually create the GitHub repositories (requires GitHub CLI `gh` to be authenticated)

Without `--create-repos`: displays the repo map and bootstrap commands without executing them.

---

## Pre-Flight Checks

### 1. Required artifact
- `artifacts/design/bounded-contexts.md` must exist
- If not: BLOCK with message to run `/sdlc-event-storm` or `/sdlc-artifact design/bounded-context-map` first

### 2. Config check
- `sdlc-config.json` must have `github_org` populated
- If not: ask user to provide it now; update sdlc-config.json

---

## Process

### Step 1: Read bounded context map
Parse `artifacts/design/bounded-contexts.md` — extract:
- All bounded context names
- Their classifications (Core / Supporting / Generic)
- Their primary aggregates and language

### Step 2: Generate repository inventory

Apply naming convention: `{github_org}/{product-codename}-{bc-name-kebab}`

Product codename: derived from `product_name` in sdlc-config.json (slugified, max 20 chars).

Special repositories always included:
- `{github_org}/{product-codename}-platform` — Infrastructure (OpenTofu + Helm)
- `{github_org}/{product-codename}-contracts` — Shared event schemas and API specs (read-only shared artefacts)
- `{github_org}/{product-codename}-docs` — Architecture decision records, documentation

### Step 3: Write repo map artifact

Output to `artifacts/design/multi-repo-map.md`:

```markdown
# Multi-Repo Map

**Product:** {product_name}
**Organisation:** {github_org}
**Phase:** Design
**Artifact:** Multi-Repo Map
**Version:** 1.0
**Date:** {date}
**Status:** Draft

## Repository Bootstrap Script

```bash
#!/bin/bash
# Generated by /sdlc-repo-map
# Run: bash artifacts/design/repo-bootstrap.sh

set -euo pipefail
ORG="{github_org}"
PRODUCT="{product-codename}"

repos=(
  "${PRODUCT}-file-domain:Go service — File Domain"
  "${PRODUCT}-entity-domain:Go service — Entity Domain"
  "${PRODUCT}-compliance-domain:Go service — Compliance Domain"
  "${PRODUCT}-graph-domain:Go service — Graph Domain"
  "${PRODUCT}-identity-domain:Go service — Identity Domain"
  "${PRODUCT}-alert-domain:Go service — Alert Domain"
  "${PRODUCT}-audit-domain:Go service — Audit Domain"
  "${PRODUCT}-platform:Infrastructure — OpenTofu + Helm"
  "${PRODUCT}-contracts:Shared contracts — event schemas and API specs"
  "${PRODUCT}-docs:Architecture docs and ADRs"
)

for entry in "${repos[@]}"; do
  name="${entry%%:*}"
  desc="${entry##*:}"
  echo "Creating ${ORG}/${name}..."
  gh repo create "${ORG}/${name}" \
    --private \
    --description "${desc}" \
    --add-readme
  echo "  ✓ ${ORG}/${name}"
done

echo ""
echo "All repositories created. Next steps:"
echo "  1. Clone each repo and run the service scaffold for Go services"
echo "  2. Configure branch protection rules on main branch"
echo "  3. Set up CI/CD workflows"
```

## Branch Strategy

All repositories follow this branch model:

| Branch | Purpose | Protection |
|--------|---------|-----------|
| `main` | Production-ready code | Require PR; require CI; require review |
| `develop` | Integration branch | Require CI pass |
| `feature/{ticket-id}-{description}` | Feature work | None |
| `fix/{ticket-id}-{description}` | Bug fixes | None |
| `release/{version}` | Release candidates | Require CI; tag on merge |

## Cross-Repo Contract Management

Cross-context event schemas live in `{org}/{product-codename}-contracts`:

```
contracts/
├── events/
│   ├── FileProcessed.v1.schema.json
│   ├── EntitiesExtracted.v1.schema.json
│   └── ...
├── api/
│   ├── file-domain-api.yaml (OpenAPI 3.1)
│   └── ...
└── CHANGELOG.md
```

Service repositories consume contracts as a git submodule or by pinning a contracts release tag in CI.

Schema changes to `contracts` require:
1. PR with changelog entry
2. Review from at least one maintainer of each consuming service
3. Version bump if breaking change
```
