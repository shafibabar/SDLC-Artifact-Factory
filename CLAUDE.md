# SDLC Artifact Factory — Plugin Standards

## What this plugin is

The SDLC Artifact Factory is a Claude Code plugin that takes a problem statement and drives a product through its full software development lifecycle — Strategy → Ideate → Design → Implement → Data → Quality → Deploy → Customer Validation — emitting real, reviewable artifacts at every step.

The plugin is domain-agnostic. It produces artifacts for any software product. The methodology it enforces and the ubiquitous language it uses are consistent across every domain and every product run.

The plugin itself is built using the same methodology it enforces. Every artifact in this repository was produced by following these standards.

---

## How to use this plugin

1. Run `/sdlc-start` to begin a new product run. Answer the interactive questionnaire. A `sdlc-config.json` and `sdlc-manifest.json` are created.
2. The current phase's skills generate artifacts. Use `/sdlc-artifact <skill-name>` to invoke a specific skill.
3. Use `/sdlc-status` to see current phase, artifacts produced, and DoD gaps.
4. Use `/sdlc-review` to trigger a methodology compliance check on current phase artifacts.
5. When the phase DoD is satisfied, run `/sdlc-next` to advance. A DoD check runs; any gaps are surfaced as advisory warnings requiring acknowledgment.
6. Continue through all eight phases.

Skills, hooks, and agents are defined in `.claude/skills/`, `.claude/settings.json`, and the agent roster in `.claude-plugin/plugin.json`.

## Non-Negotiable Methodology Rules

Every artifact generated by this plugin MUST conform to these rules. Violations are defects.

### Domain-Driven Design (DDD)
- Every product begins with a bounded context mapping exercise driven by Event Storming.
- Every artifact uses the ubiquitous language of its bounded context. Terms from another bounded context's model are never used directly — an Anti-Corruption Layer (ACL) or translation is always explicit.
- Aggregates enforce their own invariants. No service reaches into another aggregate's data store.
- Domain Events are named in past tense, are immutable once written, and carry all data needed by consumers.
- Read Models are separate from Write Models (CQRS). A query never goes through a command handler.
- Data Ownership is declared per bounded context. Cross-context access is via published API or event — never by sharing a database table.

### Test-Driven Development (TDD)
- Tests are written before implementation code. No implementation artifact is generated without a corresponding TDD spec.
- Red → Green → Refactor. A failing test is required before a passing implementation.
- The `pre-implement` hook enforces this: if no TDD spec exists for a feature, the implement skill warns before generating code.
- Integration tests use real infrastructure (real database, real queue). Mocks are not used at integration test boundaries.

### Behaviour-Driven Development (BDD)
- Every user story has a corresponding BDD feature file in Gherkin (Given/When/Then) before implementation begins.
- Acceptance criteria in user stories are written as verifiable examples, not prose assertions.
- Example Mapping is the technique used to derive BDD scenarios from stories before writing feature files.

### SOLID Principles
- **Single Responsibility**: Each class, module, and service has one reason to change.
- **Open/Closed**: Behaviour is extended through new code, not by modifying existing code.
- **Liskov Substitution**: Subtypes are substitutable for their base types without altering correctness.
- **Interface Segregation**: Clients depend only on the interfaces they use. Large interfaces are split.
- **Dependency Inversion**: High-level modules depend on abstractions, not concretions.

### Event-Driven Architecture
- Services communicate through events on the message broker (Redpanda). No direct synchronous calls between services for domain operations.
- The Transactional Outbox pattern is used for all event publishing that must be atomic with a database write.
- All event handlers are idempotent. Processing the same event twice produces the same result as processing it once.
- Dead Letter Queues are mandatory for all consumer groups. No event is silently discarded.
- Circuit Breakers are applied to all calls to external systems or unreliable dependencies.

## Canonical Ubiquitous Language

All agents, skills, commands, and artifacts MUST use these terms consistently and precisely. Using a synonym, an undefined term, or a term from a different bounded context without explicit translation is a terminology defect, flagged by the `post-artifact-created` hook.

### Domain & DDD

**Ubiquitous Language** — A shared vocabulary between domain experts and engineers used consistently in code, documents, and conversation within a Bounded Context. There is no global ubiquitous language — it is always scoped to a context.

**Domain** — The sphere of knowledge and activity around which the business operates and the software is built. The problem space the organisation owns.

**Subdomain** — A coherent part of the Domain. Classified as: Core (primary competitive advantage — build it); Supporting (necessary but not differentiating — build simply); Generic (commodity — buy or use open source).

**Bounded Context** — An explicit boundary within which a domain model applies consistently. Inside a Bounded Context, every term has one precise meaning. The same word in a different Bounded Context may have a different meaning.

**Domain Events** — Facts that occurred in the domain, named in past tense (e.g. `FileProcessed`, `EntityExtracted`). Immutable. Carry all data consumers need. Represent things the business cares about, not technical implementation steps.

**Read Model** — A projection of domain state optimised for querying. Part of CQRS. Separate from the Write Model. Can be eventually consistent with the Write Model.

**Write Model** — The authoritative source for domain state changes. Commands are handled here. Domain Events are emitted here. Consistency is enforced here.

**Canonical Data Model (CDM)** — A shared data model for entities that cross Bounded Context boundaries. Lives in the Shared Kernel. Prevents translation errors at integration points.

**Data Ownership** — Each piece of data has exactly one owning Bounded Context. Other contexts consume it via published contract (API or event). Cross-context direct database access is prohibited.

### Design Principles

**Separation of Concerns** — Each module, service, or layer is responsible for exactly one concern. Cross-cutting concerns (logging, auth, tracing) are handled explicitly via middleware or shared libraries, not scattered through domain code.

**Single Responsibility Principle** — A class or module has one and only one reason to change. If two different actors could cause a class to change, it has two responsibilities and should be split.

**Don't Repeat Yourself (DRY)** — Every piece of knowledge has a single, authoritative representation. Duplication of logic, not just code, is the defect.

**Model-View-Controller (MVC)** — Separates Model (data and domain logic), View (presentation), and Controller (input handling and routing). The Model is the authority; the View and Controller are thin.

**CQRS (Command Query Responsibility Segregation)** — Commands (writes that change state) and Queries (reads that return state) are handled by separate models. A method either changes state or returns state — never both.

**Chain of Responsibility** — A request passes through a chain of handlers. Each handler decides whether to process the request or pass it to the next handler. Used for middleware, hooks, and validation pipelines.

**API-First Design** — APIs are designed and documented (OpenAPI spec) before any implementation begins. The spec is the contract. Implementation must conform to the spec, not the reverse.

**Contract-First Design** — Interfaces, schemas, and event payloads are defined before any implementation. The contract drives both producer and consumer implementations independently.

**Consumer-Driven Contracts** — API and event contracts are defined by consumer requirements, not producer convenience. Verified by automated contract tests that run in the consumer's CI pipeline and break if the producer violates them.

**Backward Compatibility** — Changes to a published API or event schema do not break existing consumers. New fields are additive and optional. Breaking changes require a new API version or a new event type.

**API Gateway** — A single entry point that handles routing, authentication, rate limiting, and protocol translation for all external requests to internal services.

### Event-Driven & Distributed

**Event-Oriented Architecture** — System components communicate through events rather than direct synchronous calls. Producers and consumers are decoupled in time and space.

**Event Choreography** — Services react to events independently without a central orchestrator. Each service subscribes to the events it needs and emits events when its work is done.

**Event Storming** — A collaborative workshop technique for rapidly exploring complex domains. Domain Events are placed on a timeline; Commands, Aggregates, Policies, and Read Models are identified through facilitated discussion.

**Event Modeling** — A technique for designing complete system behaviour by describing the full sequence of events, commands, read models, and user interactions over time. Complements Event Storming with implementation detail.

**Eventual Consistency** — A distributed system guarantee that all replicas will converge to the same value in the absence of new updates — but not necessarily immediately. Acceptable in the read layer; not acceptable for compliance-critical writes.

**Transactional Outbox** — A pattern ensuring that a database write and a message publish are atomic. The event is written to an outbox table in the same ACID transaction as the domain write, then relayed to the message broker by a separate process.

**Change Data Capture (CDC)** — Detecting and reacting to database changes by reading the transaction log. An alternative to the Transactional Outbox for propagating changes to downstream systems.

**Idempotency** — Processing the same message or command multiple times produces the same result as processing it once. Enforced via idempotency keys stored at each processing stage.

**Atomic Transaction** — A set of operations that either all succeed or all fail as a unit. No partial state is observable or durable.

**Immutable Events** — Domain Events cannot be modified after they are written. Corrections are new events that supersede prior ones. The original event remains in the log.

**Dead Letter Queue (DLQ)** — A holding queue for messages that repeatedly fail processing after the configured retry limit. Every failed event is routed here — none are silently discarded. DLQ contents are monitored and must be resolved.

**Retry and Backoff Strategies** — Failed operations are retried with increasing delays (exponential backoff with jitter) to prevent thundering herd against recovering dependencies.

**Circuit Breaker** — A pattern that stops calls to a failing dependency after a failure threshold is reached (Open state), then probes with a single request after a backoff (Half-Open), and resumes normal operation on success (Closed).

**Rate Limiting** — Constraining the number of requests a client can make in a time window. Applied at the API Gateway and service level.

**Throttling** — Reducing the processing rate of a consumer to match downstream capacity. Applied at the consumer side of the message broker.

### Data Lifecycle

**Data Lineage** — The complete history of a data item: where it came from, every transformation applied, and every system it passed through. Traceable from source to current state.

**Data Provenance** — The origin and custody chain of a piece of data. Who created it, from what source, at what time, under what authority.

**Audit Trail** — An immutable, chronological record of all actions taken on data or within a system. Append-only. Tamper-evident. Required for compliance and forensic investigation.

**Soft Delete** — Marking a record as deleted without physically removing it. Preserves lineage and enables recovery. A `deleted_at` timestamp is set; the record is excluded from normal queries but retained in the audit log.

**Data Retention Policies** — Rules governing how long data is stored before archival or deletion. Derived from compliance requirements (GDPR, HIPAA) and business need. Enforced by automated lifecycle management jobs.

**Data Lifecycle Management** — The complete governance of data from creation through active use, archival, and deletion. Includes classification, retention enforcement, and disposal records.

**Data Classification** — Categorising data by sensitivity level (e.g. Public, Internal, Confidential, Restricted, Regulated). Controls for storage, access, transit encryption, and retention are derived from the classification tier.

**Data Sovereignty** — Data is subject to the laws of the jurisdiction where it is stored and processed. Customer data must remain in customer-controlled or jurisdictionally appropriate infrastructure. Never moves to a shared infrastructure without explicit contractual consent.

### Reliability & Operations

**Observability** — The ability to understand the internal state of a system from its external outputs (logs, metrics, distributed traces). A property designed into the system from the start, not added by tooling.

**Monitoring** — Watching system metrics against thresholds and alerting when thresholds are breached. Reactive. Complements observability.

**Logging** — Structured, timestamped records of discrete events. Always machine-parseable (JSON). Always correlated via trace IDs. Log level (DEBUG, INFO, WARN, ERROR) is applied consistently.

**Metrics** — Quantitative measurements of system behaviour over time. RED framework: Rate, Errors, Duration. USE framework: Utilisation, Saturation, Errors. All services emit Prometheus-compatible metrics via OpenTelemetry.

**Distributed Tracing** — Tracking a request across multiple services by propagating a trace context (trace ID + span ID). Collected via OpenTelemetry, stored in Tempo, visualised in Grafana.

**Health Checks** — Liveness (is the process running?) and Readiness (is the service ready to handle traffic?) probes. Required on all services. Used by Kubernetes and Linkerd.

**SLIs (Service Level Indicators)** — Quantitative measures of service behaviour. Examples: request success rate, p99 latency, data freshness. The raw measurements.

**SLOs (Service Level Objectives)** — Targets for SLIs. The internal commitments. More aggressive than SLAs to allow error budget headroom.

**SLAs (Service Level Agreements)** — External contractual commitments to customers. Derived from SLOs minus the error budget buffer.

**Fault Tolerance** — The ability to continue operating correctly after the failure of one or more components. Achieved through redundancy, circuit breakers, graceful degradation, and retry logic.

**Resilience Engineering** — Proactively designing systems to anticipate, absorb, and recover from failures. Includes chaos testing, failure mode analysis, and runbook-driven incident response.

**Graceful Degradation** — When a non-critical component fails, the system continues to function with reduced capability rather than complete failure. Core user journeys remain available.

**High Availability** — A system design that minimises downtime through redundancy, automatic failover, and zero-downtime deployment. Defined by the availability SLO (e.g. 99.9%).

**Vertical Scalability** — Increasing capacity by adding resources to a single node (more CPU, RAM). Bounded by hardware limits. Complements, does not replace, horizontal scaling.

**Load Balancing** — Distributing traffic across multiple service instances to prevent any single instance from becoming a bottleneck. Handled by Kubernetes Services and Linkerd.

**Capacity Planning** — Forecasting resource requirements based on growth projections and observed usage patterns. Documented in the capacity plan artifact and reviewed quarterly.

**Disaster Recovery** — The process of restoring a system after a catastrophic failure. Defined by RTO (Recovery Time Objective — how long recovery takes) and RPO (Recovery Point Objective — how much data loss is acceptable).

**Business Continuity** — Ensuring critical business functions continue during and after a disaster. Broader than Disaster Recovery — includes people, process, and communication plans.

### Security & Compliance

**Security by Design** — Security is built into the system from inception. Every design decision explicitly considers security implications. Security is never an afterthought or a separate layer.

**Privacy by Design** — Privacy protections are architectural commitments, not compliance checkboxes. Data minimisation, purpose limitation, and user rights are designed in from the start.

**Principle of Least Privilege** — Every component, service, and user has only the minimum access required to perform its function. Access is explicit and time-bounded where possible.

**Zero Trust Architecture** — No implicit trust based on network location or service identity alone. Every request is authenticated and authorised regardless of origin (internal or external). mTLS is enforced between all services via Linkerd.

**Authentication** — Proving identity. Who are you? Implemented via OAuth 2.0 + OIDC for human users, and mTLS for service-to-service communication.

**Authorization** — Proving permission. What are you allowed to do? Implemented via Attribute-Based Access Control (ABAC) enforced at the API layer, not the UI layer.

**Attribute-Based Access Control (ABAC)** — Access decisions based on attributes of the subject (user), resource (data), action, and environment. More expressive than role-based control. Claims in the JWT drive authorization decisions.

**Encryption at Rest** — Data is encrypted when stored on disk. Protects against physical media theft and unauthorised storage access. Applied to all databases and object storage.

**Encryption in Transit** — All data in transit is encrypted via TLS (external) or mTLS (internal via Linkerd). No plaintext communication between services.

**Key Management** — The secure generation, storage, rotation, and destruction of cryptographic keys. Managed by a dedicated secrets management system (not hardcoded, not in environment variables).

**Secrets Management** — Secure storage and access control for credentials, API keys, certificates, and connection strings. Never in source code, never in plain-text environment variables.

**Non-Repudiation** — The guarantee that a completed action cannot be denied after the fact. Achieved through cryptographic signatures on events and an immutable audit trail.

**Auditability** — The ability to reconstruct what happened, when, by whom, and to what data. A first-class architectural requirement. The Audit Trail is a tamper-evident, append-only record.

**Compliance as Code** — Compliance controls and policies are implemented and verified programmatically. Compliance test suites run in CI. Compliance posture is measurable, not manual.

### Testing & Quality

**TDD (Test-Driven Development)** — Tests are written before implementation code. The cycle: write a failing test (Red), write the minimum code to pass it (Green), then refactor (Refactor).

**BDD (Behaviour-Driven Development)** — Tests describe system behaviour in business language (Given/When/Then). Gherkin feature files are the shared specification between product and engineering.

**Specification by Example** — Requirements are expressed as concrete, testable examples of system behaviour. Examples become automated tests. Ambiguity is resolved at the example level, not through prose.

**Shift Left Testing** — Testing is moved earlier in the development cycle. Defects are cheaper to fix when found at the unit test stage than at production. TDD is the primary shift-left mechanism.

**Unit Test** — Tests a single unit of behaviour in complete isolation. No real database, no real network, no real filesystem. Deterministic. Milliseconds to run.

**Integration Test** — Tests the interaction between two or more components using real infrastructure (real database, real message broker). Verifies that components work correctly together. No mocks at the integration boundary.

**Contract Test** — Verifies that a producer's API or event schema satisfies consumer expectations. A consumer defines the contract; the producer runs tests against it. Prevents silent breaking changes in the event-driven pipeline.

**Component Test** — Tests a service in isolation from other services but with its real internal dependencies (real database, real cache). Faster than E2E, more realistic than unit.

**End-to-End Test** — Tests the complete system flow from user action to final state, across all services and real infrastructure. Validates that the system works as a whole.

**Acceptance Test** — Verifies that the system meets business requirements. Maps to BDD scenarios. Written from the user's perspective.

**Performance Test** — Measures system behaviour under expected load. Validates SLOs. Run against a production-like environment.

**Load Test** — Measures behaviour at peak and above-peak expected load. Identifies the point at which performance degrades below SLO thresholds.

**Stress Test** — Pushes the system beyond designed capacity to identify failure modes and recovery behaviour. Answers: how does the system fail, and does it recover?

**Chaos Test** — Deliberately injects failures (network partition, pod kill, disk full, latency injection) to verify resilience engineering and runbook effectiveness.

**Mutation Testing** — Verifies test suite quality by introducing small code mutations (flipping a boolean, changing an operator) and checking that tests detect them. A mutation that survives indicates a gap in test coverage.

**Test Pyramid** — Unit tests are the base (many, fast, cheap). Integration tests in the middle (fewer, slower). E2E tests at the top (fewest, slowest, most expensive). Invert the pyramid and the test suite becomes slow and brittle.

**Test Automation** — All tests run without manual intervention, triggered by CI on every commit. No manual test steps in the standard delivery pipeline.

**Continuous Testing** — Tests run at every stage of development and delivery, not just pre-release. Coverage is measured continuously. Regressions are caught within minutes of introduction.

### Delivery & Platform

**Infrastructure as Code (IaC)** — All infrastructure is defined in version-controlled code (OpenTofu). No manual cloud console configuration. Infrastructure changes go through the same PR review as code.

**GitOps** — The git repository is the single source of truth for both application and infrastructure desired state. A reconciliation process (Argo CD, Flux, or equivalent) applies the declared state to the actual environment continuously.

**Continuous Integration (CI)** — Code is integrated into the shared branch frequently (at least daily). Every commit triggers an automated build and full test suite via GitHub Actions. A failing build blocks the merge.

**Continuous Deployment (CD)** — Every commit that passes all CI checks is automatically deployed to production without manual intervention.

**Continuous Delivery** — Every commit that passes CI is deployable to production, but the deployment decision is made by a human. Production deployments require explicit approval.

**Feature Flags** — Runtime toggles that enable or disable features without a deployment. Enable safe rollouts, A/B testing, and instant rollback of a feature without reverting code.

**Progressive Delivery** — Features are released to a subset of users first (canary, percentage rollout, cohort), then expanded as confidence grows. Reduces the blast radius of defects.

**Blue-Green Deployment** — Two identical production environments. The old version (blue) continues to serve traffic while the new version (green) is deployed and validated. Traffic switches atomically.

**Canary Deployment** — The new version receives a small percentage of live traffic. Metrics are monitored. Traffic shifts gradually to the new version as quality is confirmed. Rollback is instant if metrics degrade.

**Immutable Infrastructure** — Servers and containers are never modified after deployment. Updates replace instances rather than patching them. Prevents configuration drift.

**Platform Engineering** — Building internal platforms and tools that increase developer productivity, enforce standards, and abstract infrastructure complexity. The SDLC Artifact Factory is a platform engineering product.

**Internal Developer Platform (IDP)** — A self-service layer that abstracts infrastructure, deployment, and observability complexity from application developers. Reduces cognitive load and improves developer experience.

**Developer Experience (DX)** — The quality and efficiency of the engineering workflow: how fast and how easily a developer can build, test, deploy, and debug. A first-class product concern for the platform team.

**Environment Parity** — Development, staging, and production environments are as similar as possible in configuration, infrastructure, and data shape. Prevents defects that only manifest in production.

**Configuration as Code** — All environment configuration is version-controlled code. No configuration that affects behaviour is set manually or stored outside the repository.

### Product & Discovery

**Impact Mapping** — A visualisation technique linking a business goal to the actors who can affect it, the impacts those actors could have, and the deliverables (features) that would produce those impacts. Prevents building features disconnected from outcomes.

**User Story Mapping** — Organising user stories along the horizontal axis of user activities (the narrative flow) and the vertical axis of story slices (from essential to enhancement). Reveals scope and guides release planning.

**Example Mapping** — A structured conversation technique using coloured index cards to clarify story scope: Rules (yellow), Examples (green), Questions (red), Stories (blue). Produces BDD scenarios before development begins.

**Domain Storytelling** — A collaborative modelling technique where domain experts narrate workflows as stories. A facilitator captures the narrative using a pictographic language of actors, work objects, and activities.

**Outcome-Driven Development** — Development is guided by desired outcomes (measurable changes in user behaviour or business metric), not feature lists. Features are hypotheses; outcomes are the validation criteria.

**Jobs To Be Done (JTBD)** — Users hire products to do a job. Understanding the functional job (what they do), social job (how they want to be perceived), and emotional job (how they want to feel) drives better product decisions than feature analysis.

**Product Discovery** — The process of identifying what to build before committing to building it. Involves user research, prototyping, and assumption testing. Discovery reduces the risk of building the wrong thing.

**Product Validation** — Testing product assumptions with real users during or before development. Includes usability testing, concept testing, and beta programs.

**Product Metrics** — Quantitative measures of product value delivery and health. Tied to OKRs and derived from the North Star Metric. Instrumented in code from day one.

**North Star Metric** — The single metric that best captures the core value the product delivers to customers. All feature decisions are evaluated against their impact on this metric.

**OKRs (Objectives and Key Results)** — A goal-setting framework. Objectives are qualitative, inspiring directions. Key Results are quantitative, measurable indicators of progress toward the objective.

**Value Stream Mapping** — Visualising all steps required to deliver value to the customer — from idea to production. Identifies waste, delays, handoffs, and bottlenecks.

### Artifact Naming Rules
- All filenames: lowercase, hyphen-separated. No spaces, no underscores.
- ADRs: zero-padded three-digit prefix (`001`, `002`), followed by a short slug (`001-use-postgresql-for-acid-store.md`).
- Reports: include a timestamp suffix in ISO format (`test-run-2026-06-20T14-30.md`).
- Bounded context names in directory paths: lowercase, hyphen-separated, matching the name in `bounded-contexts.md`.

## Phase Definitions of Done

A phase is not complete until all items in its DoD are satisfied. The `pre-phase-advance` hook checks these before allowing `/sdlc-next` to proceed. Gaps are surfaced as advisory warnings. Unresolved gaps require explicit acknowledgment before advancing.

### Strategy DoD
- [ ] Vision statement produced
- [ ] North Star Metric defined with measurement method
- [ ] OKRs defined for at least one planning horizon
- [ ] Strategic roadmap covering at least two delivery phases
- [ ] GTM strategy covering: target segment, channel, positioning, pricing model
- [ ] Stakeholder map identifying all key decision-makers and their concerns
- [ ] All artifacts pass terminology scan (no undefined or misused ubiquitous language terms)

### Ideate DoD
- [ ] Functional requirements document complete
- [ ] NFRs defined with measurable targets per quality attribute
- [ ] At least one user persona produced
- [ ] User Story Map produced covering all major user activities
- [ ] Epics defined and linked to OKRs
- [ ] At least three user stories per epic, each with acceptance criteria
- [ ] Example mapping completed for at least two stories per epic
- [ ] MoSCoW prioritisation applied to all epics
- [ ] All artifacts pass terminology scan

### Design DoD
- [ ] Domain context documented: domain, subdomains, core/supporting/generic classification
- [ ] Event Storming completed: domain events, commands, aggregates, policies, read models catalogued
- [ ] Bounded Context map produced with relationship types (ACL, OHS, PL, CF, SK)
- [ ] Ubiquitous Language defined per Bounded Context (extends canonical glossary)
- [ ] C4 diagrams: system context, containers, and at least one component diagram
- [ ] At least one ADR per major architectural decision
- [ ] OpenAPI contracts for all public-facing and inter-service APIs
- [ ] Event schemas defined for all domain events (payload, versioning strategy)
- [ ] Data architecture documented: stores, ownership per BC, lineage strategy
- [ ] Threat model (STRIDE) completed for all service boundaries
- [ ] Compliance design produced for each framework in scope
- [ ] Deployment architecture documented for all three targets (cloud, hosted, local)
- [ ] Multi-tenancy design documented (if applicable)
- [ ] Observability design documented (logging, metrics, tracing, alerting strategy)
- [ ] All artifacts pass terminology scan

### Implement DoD
- [ ] Coding standards documented and finalised
- [ ] Multi-repo structure documented with BC-to-repo mapping
- [ ] TDD specs written for all user stories before code generation
- [ ] BDD feature files written for all acceptance criteria
- [ ] Service skeletons generated for all bounded context repos

## Multi-Repo Conventions

Bounded Contexts drive repository boundaries. The `design/bounded-context-map` artifact and the `/sdlc-repo-map` command produce the authoritative BC-to-repo mapping.

### Repository Naming
```
{org}/{product-codename}-{bounded-context-name}    # One per bounded context
{org}/{product-codename}-shared-kernel             # Shared event schemas, proto definitions, common types
{org}/{product-codename}-platform                  # OpenTofu IaC, Helm charts, CI/CD templates, runbooks
```

### Per-Repo Standards
Each bounded context repo contains:
- `CLAUDE.md` — extends the factory CLAUDE.md with BC-specific standards and ubiquitous language
- `.github/workflows/ci.yml` — CI pipeline (generated by `deploy/ci-pipeline` skill)
- `helm/` — Helm chart for this service
- `iac/` — OpenTofu module for this service's infrastructure
- `docs/` — BC-specific ADRs and design notes

### Cross-Repo Contract Management
- Event schemas live in `{product}-shared-kernel/schemas/`
- Each consumer repo runs contract tests against the shared kernel schemas in its CI pipeline
- A schema change in the shared kernel that breaks a consumer's contract test blocks the merge
- API contracts (OpenAPI specs) live in the producer's repo under `api/`; consumers reference them via contract test tooling

### Branch Strategy
- `main` — protected; no direct pushes
- All changes via pull request; CI must pass before merge
- Feature branches: `feature/{story-id}-{short-description}`
- Release branches: `release/{version}`
